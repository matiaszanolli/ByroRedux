# ReSTIR DI — The Foundational Paper

**Paper**: "Spatiotemporal reservoir resampling for real-time ray tracing
with dynamic direct lighting"
**Authors**: Benedikt Bitterli (Dartmouth), Chris Wyman, Matt Pharr, Peter
Shirley, Aaron Lefohn (NVIDIA), Wojciech Jarosz (Dartmouth)
**Venue**: ACM Transactions on Graphics / SIGGRAPH 2020
**PDF**: `restir_di.pdf` (gitignored)

This is **the original ReSTIR paper**. Everything in the ReSTIR family (GI,
PT, Volumetric, Area, etc.) builds on this. Critical for understanding:
- The reservoir data structure
- The streaming RIS algorithm
- The MIS-based bias correction (the math we were missing)
- The biased vs. unbiased trade-off

## Results at a Glance

- **6×–60× faster** than Moreau et al. 2019 (state-of-the-art light BVH)
- **35×–65×** with the biased variant
- Scenes with **thousands to millions** of dynamic emissive triangles
- **Under 50 ms per frame** at 1080p (on 2019 Titan RTX / RTX 2080 Ti)
- Up to **8 rays per pixel** max
- **No scene data structures** — only image-sized buffers

## Problem Being Solved

Direct lighting with many lights is expensive. At N lights, each pixel
needs to know which light contributes most. Traditional approaches:
- BVH over lights (Moreau 2019) — needs maintenance for dynamic lights
- Uniform/importance sample all lights — scales poorly
- VPLs — need data structures

ReSTIR sidesteps this entirely. **No light data structures**. Just reservoirs
per pixel that collect + combine samples across space and time.

## Section 2: Mathematical Preliminaries

### The Direct Lighting Integral (Equation 2)

```
L(y) = ∫_A f(x) dx    where f(x) = ρ(x) · Le(x) · G(x) · V(x)
```

Where:
- ρ = BSDF at shading point y
- Le = emitted radiance from light point x
- G = geometry term (inverse square distance + cosines)
- V = visibility (0 or 1)
- A = surface of all emitters

### Importance Sampling (IS)

```
⟨L⟩_is = (1/N) · Σ f(x_i) / p(x_i)
```

Works if p(x) > 0 wherever f(x) > 0. Better when p correlates with f.
**Problem**: You can't practically sample `p(x) ∝ f(x)` because visibility
V is unknown without tracing a ray.

### Multiple Importance Sampling (MIS)

Combine M different sampling strategies p_s (e.g., BSDF vs light sampling):
```
⟨L⟩_mis = Σ_s (1/N_s) · Σ_i w_s(x_i) · f(x_i) / p_s(x_i)
```

As long as Σ_s w_s(x) = 1 (partition of unity), unbiased.
The **balance heuristic** `w_s(x) = N_s·p_s(x) / Σ_j N_j·p_j(x)` is the
standard choice — provably good.

### Resampled Importance Sampling (RIS) — THE KEY IDEA

Instead of sampling from a single PDF, generate **M candidates** from a
cheap source PDF p(x), then select one proportionally to a target PDF p̂(x):

**Step 1**: Generate candidates x₁, ..., x_M ~ p(x) (cheap)
**Step 2**: Select index z with probability:
```
P(z | x) = w(x_z) / Σ_i w(x_i)    where    w(x) = p̂(x) / p(x)
```
**Step 3**: Use 1-sample RIS estimator (Equation 6):
```
⟨L⟩_ris = f(y) / p̂(y) · (1/M · Σ w(x_j))
```

**Intuition**: The chosen sample y has approximate distribution p̂ (better
PDF than p), but we don't know p̂ exactly. The parenthesized term corrects
for the approximation error.

**Why this matters**: We can use p̂ = unshadowed path contribution
(`ρ · Le · G`) which is MUCH better than just sampling lights by power.
Generating from p (light area sampling) is cheap; evaluating p̂ is also
cheap (just math, no rays). We get good samples without tracing M shadow rays.

### Weighted Reservoir Sampling (WRS) — THE STREAMING TRICK

RIS naively stores all M candidates. WRS achieves the same result with
just 1 sample in memory:

```
Reservoir:
    y       // currently selected sample
    wsum    // running sum of weights
    M       // number of candidates seen
    W       // final RIS weight after processing

update(sample x_new, weight w_new):
    wsum += w_new
    M += 1
    if random() < w_new / wsum:
        y = x_new   // stochastically replace
```

Each new candidate replaces the current sample with probability
`w_new / Σw` (Equation 9). The invariant: at any time, sample i occurs
in the reservoir with probability `w_i / Σw`.

**Memory**: O(1) per reservoir, regardless of M. **This is what makes
ReSTIR GPU-feasible.**

## Section 3: Streaming RIS with Spatiotemporal Reuse

### 3.1 Streaming RIS (Algorithm 3)

Wrap RIS in reservoir sampling. Per pixel:
```
Reservoir r
for i in 1..M:
    x_i ~ p                           // generate candidate
    r.update(x_i, p̂_q(x_i) / p(x_i))  // RIS weight

r.W = r.wsum / (r.M · p̂_q(r.y))       // Equation 6
final_color = f_q(r.y) · r.W          // shade
```

**Default M**: 32 candidates per pixel. Generated by importance sampling
emissive triangles by power.

**Visibility reuse** (§ after Alg 5): After initial RIS, **test visibility
of the selected y** by tracing a shadow ray. If occluded, set W=0. This
propagates only visible samples to neighbors — dramatic quality improvement
for the cost of 1 extra ray per pixel.

### 3.2 Spatial Reuse (Algorithm 4)

**Core insight**: Reservoirs can be *merged* in O(1) time — no need to
store candidate streams!

```
combineReservoirs(q, r1, r2, ..., rk):
    s = new Reservoir
    for each r in {r1..rk}:
        s.update(r.y, p̂_q(r.y) · r.W · r.M)
    s.M = r1.M + r2.M + ... + rk.M
    s.W = s.wsum / (s.M · p̂_q(s.y))
    return s
```

**What's happening**: Each neighboring reservoir's sample is re-weighted
for pixel q's target PDF, then treated as a new candidate with weight
`p̂_q(y) · W · M`. The `M` factor accounts for the number of candidates
the neighbor already accumulated.

**Reweighting explanation** (§4.1): The weight `p̂_q(y) · W · M` is
equivalent to the standard RIS weight `p̂_q(y) / p(y)` when we don't know
the exact PDF `p(y)` — we use `W` (Equation 12) as an estimate of `1/p(y)`.

**Iteration**: n rounds of spatial reuse, each gathering from k neighbors.
After n iterations, each pixel effectively sees `k^n · M` candidates.

**Typical values**: k=5 neighbors, n=2 iterations, M=32 initial candidates.
→ 25 · 32 = 800 effective candidates per pixel.

**Neighbor selection**:
- NOT a box pattern (causes artifacts)
- **k = 5 random points in a 30-pixel radius** (biased version)
- **k = 3** for unbiased version (more expensive per-sample)
- Low-discrepancy sequence for the random points

**Geometric similarity test** (biased variant only):
- Reject neighbor if depth difference > 10% of current depth
- Reject neighbor if normal angle > 25°

### 3.3 Temporal Reuse (within Algorithm 5)

After rendering a frame, **store each pixel's final reservoir** for reuse
next frame.

```
// At start of next frame:
foreach pixel q:
    q_prev = backprojection(q)  // motion vector
    reservoirs[q] = combineReservoirs(q, reservoirs[q], prevFrameReservoirs[q_prev])
```

**Critical: clamp previous frame's M to 20× current frame's M.**
Otherwise temporal M grows unbounded and stale samples dominate.
This limits the influence of any single sample to ~20 frames of history.

### The Full Algorithm (Algorithm 5)

```
function reservoirReuse(prevFrameReservoirs):
    // 1. Generate initial candidates (RIS with M=32)
    foreach pixel q:
        reservoirs[q] = RIS(q)

    // 2. Visibility reuse
    foreach pixel q:
        if shadowed(reservoirs[q].y):
            reservoirs[q].W = 0

    // 3. Temporal reuse
    foreach pixel q:
        q_prev = backprojection(q)
        reservoirs[q] = combineReservoirs(q, reservoirs[q], prevFrameReservoirs[q_prev])

    // 4. Spatial reuse (n=2 iterations)
    for i in 1..n:
        foreach pixel q:
            Q = pickSpatialNeighbors(q)  // k=5 random in 30px radius
            R = {reservoirs[q'] for q' in Q}
            reservoirs[q] = combineReservoirs(q, reservoirs[q], R)

    // 5. Shade
    foreach pixel q:
        Image[q] = f_q(reservoirs[q].y) · reservoirs[q].W
```

## Section 4: Eliminating Bias (The Math We Were Missing)

### The Bias Problem (§4.1)

Naive spatial reuse is **biased**. Why?

When you combine reservoirs from different pixels, each pixel q_i has
its own source PDF p_i(x) (which depends on visible surface normal, BSDF,
etc.). A sample from pixel A's reservoir may have zero probability at
pixel B (e.g., the sample is below B's hemisphere). The standard RIS
weight `1/M` doesn't account for this — it assumes uniform weighting
across candidates, but some candidates can't produce certain samples.

**Mathematical explanation** (Equation 19):
```
E[W(x,z) | x_z = y] = (1/p(y)) · (|Z(y)| / M)
```

Where `Z(y)` is the set of candidate PDFs that are non-zero at y.
If all PDFs are non-zero at y, then |Z(y)| = M and the estimator is unbiased.
**But if some are zero, |Z(y)|/M < 1 → the inverse PDF is underestimated
→ the result is darker than it should be.**

This is exactly the "darkening at geometric discontinuities" artifact.

### Unbiased RIS with MIS (§4.3)

**The fix**: Replace `1/M` with an MIS weight `m(x_z)`:
```
W(x, z) = (1/p̂(x_z)) · m(x_z) · Σ w_i(x_i)
```

As long as `Σ_{i∈Z(y)} m(x_i) = 1`, the estimator is unbiased.

**Naive approach**: `m(x_z) = 1 / |Z(x_z)|` — divide by number of
non-zero candidates. Unbiased but **extremely noisy** when a candidate
PDF is near-zero but not exactly zero.

**MIS-based approach (Equation 22)** — the paper's contribution:
```
m(x_z) = p_z(x_z) / Σ_i p_i(x_z)
```

This is the **balance heuristic** over the candidate PDFs. Unbiased AND
low-variance. Cost is O(M) instead of Talbot et al. 2005's O(M²).

### Practical Unbiased Algorithm (Algorithm 6)

```
function combineReservoirsUnbiased(q, r1..rk, q1..qk):
    s = new Reservoir
    for each r in {r1..rk}:
        s.update(r.y, p̂_q(r.y) · r.W · r.M)
    s.M = r1.M + r2.M + ... + rk.M

    // The bias correction:
    Z = 0
    for each q_i in {q1..qk}:
        if p̂_{q_i}(s.y) > 0:     // Evaluate neighbor's target PDF
            Z += r_i.M            // Include its candidate count
    m = 1 / Z                     // Or use MIS weights

    s.W = s.wsum · m / p̂_q(s.y)
    return s
```

**The key difference from biased**: The `Z` term counts only neighbors
where the chosen sample `s.y` has non-zero probability (is above hemisphere,
not occluded, etc.). This corrects the darkening bias.

**Cost**: For unbiased visibility reuse, evaluating `p̂_{q_i}(s.y) > 0`
requires **tracing an additional shadow ray per neighbor** to check
occlusion. That's k extra rays per spatial reuse step.

## Section 5: Implementation Details (The Numbers We Need)

### Candidate Generation
- **M = 32** initial candidates per pixel
- Importance sample **emissive triangles by power**
- **25% environment map samples** if env map present
- Use alias tables for fast weighted sampling (Walker 1974)

### Target PDF
- `p̂(x) = ρ(x) · Le(x) · G(x)` — unshadowed path contribution
- BSDF ρ = dielectric GGX microfacet + Lambertian diffuse

### Neighbor Selection
- **Spatial**: k=5 random points in 30-pixel radius (biased)
               k=3 for unbiased (more expensive)
- **Temporal**: motion-vector backprojection to prev frame

### Geometric similarity thresholds (biased only)
- Depth difference < 10% of current pixel's depth
- Normal angle < 25°

### Reuse counts
- **Spatial iterations**: n=2 biased, n=1 unbiased
- **Evaluated samples**: N=1 for interactive, N=4 for biased (4 reservoirs/pixel)

### Temporal M clamping
- **20× current frame's M** (so temporal M ≤ 20 · 32 = 640)

### Sample data per pixel (biased, N=4)
- 4 reservoirs × (sample_id + W + M + some metadata)
- Typical: ~32-48 bytes per reservoir × 4 = 128-192 bytes per pixel
- At 1080p: ~250-400 MB for reservoir storage

## Section 6: Results Summary

### Equal-time comparisons (vs Talbot 2005 streaming RIS)

| Scene | Talbot RMAE | Unbiased ReSTIR | Biased ReSTIR |
|-------|------------|-----------------|---------------|
| Bistro day | 0.70 | 0.42 | **0.26** |
| Burger Restaurant | 1.11 | 0.54 | **0.34** |
| Subway | 1.16 | 0.45 | **0.25** |
| Zero Day | 0.75 | 0.56 | **0.33** |

**Biased is ~2× better than unbiased at equal time**. At longer render
times (1s+), bias accumulates and unbiased eventually wins, but for
real-time (15-50ms) **biased is the clear choice**.

### Long-term convergence

Biased ReSTIR converges fast but plateaus at a bias floor.
Unbiased ReSTIR converges slower but reaches 0 error.
**Crossover point: ~1 second of compute**.

## Key Takeaways for ByroRedux

### What We Now Understand

1. **The math behind reservoir merging**: `p̂_q(r.y) · r.W · r.M` is the
   correct way to treat a neighbor's reservoir as a new candidate. Our
   previous mental model (simple EMA blend) was wrong.

2. **The M clamping trick**: Without clamping temporal M at 20×, history
   accumulates unboundedly and prevents new samples from ever being accepted.
   This is why our temporal experiments felt "stuck".

3. **Biased ≫ Unbiased for real-time**: Skipping visibility checks during
   spatial reuse causes slight darkening at geometric discontinuities
   but is 2-3× faster. **Ship biased.**

4. **Visibility reuse is cheap and essential**: Trace 1 shadow ray for
   the selected sample after initial RIS. This alone eliminates most of
   the "chose a sample that was occluded" noise.

5. **Spatial reuse neighbors are RANDOM**, not a fixed pattern. k=5 random
   points in 30px radius, low-discrepancy sequence.

6. **Geometric similarity rejects bad neighbors**: normal within 25°,
   depth within 10%. This is what keeps spatial reuse artifact-free.

7. **Initial M=32 is the baseline**, not 4 or 8. Don't skimp on initial
   candidates — they're cheap (no rays) and quality matters.

8. **The target PDF p̂ is the unshadowed path contribution** —
   `ρ · Le · G`, NOT the luminance we were using in our GI experiments.
   The BSDF factor matters.

### What This Paper Does NOT Cover (Important)

1. **Indirect illumination** — this is ReSTIR DI only. For GI, read
   ReSTIR GI (Ouyang 2021, which we already have notes on).

2. **Denoising the output** — ReSTIR output is still noisy. It pairs
   with SVGF or similar. We have notes on SVGF.

3. **Motion vectors** — assumed to exist. We don't have them yet.
   **This is still our biggest blocker**.

4. **G-buffer** — assumed to exist with depth, normal, material.
   **Also still missing in our forward-only pipeline**.

### Connection to ReSTIR GI

The ReSTIR GI paper (Ouyang 2021) applies the same reservoir mechanics
to **indirect lighting**. Key differences:
- Samples are **hit positions on surfaces**, not light samples
- Target PDF is the reflected radiance, not unshadowed path contribution
- The same M clamping, geometric similarity, spatial/temporal reuse apply
- Adds a **Jacobian determinant** because visible point geometry varies
  between pixels (not relevant in DI since samples are lights, not surfaces)

Understanding ReSTIR DI first makes ReSTIR GI's Jacobian trick make sense.

## Critical Files If We Implement This

Blockers we need first:
1. **Motion vectors** — R16G16_SFLOAT render target, requires prev viewProj
2. **G-buffer**: depth (have), normal (missing separate RT), material/BSDF data
3. **Visibility query infrastructure** — we have TLAS, need to trace from
   shading point to chosen light sample during visibility reuse

New data structures:
1. **Reservoir buffer** — double-buffered storage image (ping-pong for temporal)
   - Per pixel: selected light sample (index + position/normal), W, M, wsum
   - At 1080p × 4 reservoirs × 48 bytes = 400 MB (would need to compact)
2. **Previous frame reservoir** — for temporal reuse

New passes:
1. **Initial RIS**: 32-candidate sampling + visibility check (1 shadow ray)
2. **Temporal merge**: backproject + combine with prev reservoir
3. **Spatial merge** × 2: 5 neighbors random in 30px radius + similarity test
4. **Final shade**: use selected sample

**Estimate**: ~20-30 hours for a basic biased implementation, assuming
motion vectors + G-buffer infrastructure already exists (which it doesn't).

### Why ReSTIR DI Matters Less Than SVGF for Our Current Problem

Our immediate problem is **indirect lighting noise**. ReSTIR DI solves
**direct lighting efficiency with many lights** — a different problem.

For us, the priority order based on current needs:
1. **Motion vectors + G-buffer** — unblocks everything
2. **SVGF denoising** — fixes our current noisy GI
3. **ReSTIR GI** — replaces our ad-hoc GI with a principled approach
4. **ReSTIR DI** — only useful if we get to scenes with many dynamic lights

That said, reading this paper clarifies the ReSTIR framework and gives us
the math foundations for when we do implement ReSTIR GI.
