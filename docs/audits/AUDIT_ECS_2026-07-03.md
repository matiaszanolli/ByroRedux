### Fresh verification of the two new `SkinSlotPool` methods (dim 7/8)

- **Rollback correctness**: `try_mark_pose_dirty` records the pre-image only when
  `dirty` is true, via `or_insert(old)` — so a re-mark within the same frame
  preserves the *frame-start* baseline, which is exactly what rollback must
  restore. `rollback_pending_pose_commits` restores `Some(old)` → `insert`,
  `None` → `remove` (first-sight). The per-frame ordering
  (`clear_pose_dirty` → `try_mark_pose_dirty` → `draw_frame` → conditional
  rollback) is honored in the App render loop and rollback only ever undoes the
  *current* frame's commits (map is cleared at the start of the next frame).
  Pinned by three new tests (`rollback_restores_prior_hash_so_next_frame_stays_dirty`,
  `rollback_of_first_sight_restores_always_dirty_invariant`,
  `rollback_is_a_no_op_when_nothing_pending`).
- **Requeue correctness**: the caller (`main.rs:1768-1794`) keeps
  `pending_for_requeue` as the *filtered* subset actually about to be uploaded
  (entries whose `SkinnedMesh` is already gone are correctly excluded), and
  requeues via `std::mem::take` only when `!ctx.skin_dispatch_ran`. Prepend
  ordering makes the overdue entries the next drain's first candidates. Pinned by
  `requeue_pending_restores_entries_for_the_next_drain` +
  `requeue_pending_is_a_no_op_for_an_empty_list`.
- **Caller wiring**: `draw_frame`'s two early-return guards (empty framebuffers,
  `ERROR_OUT_OF_DATE_KHR`) return through the `Ok` arm; `skin_dispatch_ran` is
  reset `false` before both guards and set `true` only in
  `record_skinned_blas_refit` (`draw.rs:1462, 2042`) — so the flag is a correct
  signal for both fixes. The `Err(e)` arm calls `event_loop.exit()`, so skipping
  rollback/requeue there is harmless (process terminating, no next frame to
  corrupt).
- **No hot-path regression (dim 8)**: `rollback_pose_hash` is `clear()`ed each
  frame (capacity retained) and inserted-into only per already-dirty entity —
  one extra `HashMap` op on entities already being inserted into `pose_dirty`/
  `last_pose_hash`. The caller's `pending_for_requeue` is
  `Vec::with_capacity(pending.len())`, and on steady-state frames
  `drain_pending` returns empty (`pending.len() == 0` → zero-capacity Vec, no
  heap allocation); it only allocates during cell-load first-sight bursts,
  bounded by `MAX_PENDING_BIND_INVERSE_UPLOADS_PER_FRAME`. Not a per-frame
  steady-state regression.
- **No lock surface (dim 1)**: `SkinSlotPool` is a direct field on `App`
  (`skin_slot_pool: SkinSlotPool`, `main.rs:324`), never `insert_resource`d into
  the `World`, and accessed single-threaded in the render loop — no `RwLock`,
  no lock-ordering or ABBA concern despite living in the `ecs::resources`
  module.
