## Executive Summary

The NIF parser is in **strong shape** at HEAD. Across 5 audit dimensions and ~291 dispatch arms × 7 game variants, **zero CRITICAL and zero HIGH findings** were produced. Three previously-tracked findings landed during the 04-26 → 04-28 window (#714 NiTransformData Order, #715 NiSourceTexture UseInternal, #717 four zero-field BSShaderProperty subtypes, #718 walk-light/particle NiSwitchNode descent, #719 BSEffectShaderProperty env_map forwarding). **Headline count: 0 CRITICAL · 0 HIGH · 3 MEDIUM · 11 LOW · 4 carryovers verified clean.**

The remaining surface area splits into three buckets:

1. **Allocation-safety hardening** (1 MEDIUM + 3 LOW) — file-driven `u32` length fields that bypass `allocate_vec` / `check_alloc`. Today's blast radius is bounded by inner-read bounds checks but the pre-allocation itself can OOM-panic. Surfaces under untrusted-NIF parsing (mod loaders, network DLC, asset hot-reload).
2. **Coverage long-tail** (4 LOW + existing #710 / #720 / #728) — ~3,600 NiUnknown blocks across FO4 + FO76 + Skyrim SE Meshes1 closable with ~120 LOC of stub parsers.
3. **Pre-Bethesda compat / dead-code** (8 LOW) — NetImmerse 4.x / Morrowind-era version gates and unused helpers. None of the 7 supported games hit any of these paths.

The two open MEDIUM findings outside allocation safety are:
- **NIF-D2-03**: `BSShaderPPLightingProperty.Emissive Color` not parsed for BSVER > 34 — mostly cosmetic on FO3/FNV (those are BSVER ≤ 34); affects rare cross-version Skyrim PPLighting fallback only.
- **NIF-D4-05**: `BSEffectShaderProperty` additive-blend (ONE/ONE) for `Own_Emit` flagged surfaces never engages — fire/glow VFX on Skyrim+ over-attenuate. Source comment defers to follow-up.

## Findings — by severity

### MEDIUM (3)

#### NIF-D2-03: `BSShaderPPLightingProperty.Emissive Color` field never parsed for BSVER > 34
- **Dimension**: Block Parsing × Version Handling
- **Game Affected**: Skyrim LE/SE only via cross-version PPLighting fallback (rare). FO3/FNV (BSVER ≤ 34) are unaffected.
- **Location**: [crates/nif/src/blocks/shader.rs:55-87](crates/nif/src/blocks/shader.rs#L55-L87)
- **Status**: NEW (carryover from 2026-04-26 dim_2 draft; not in issues)
- **Description**: `BSShaderPPLightingProperty::parse` reads `texture_set_ref`, `refraction_*`, `parallax_*`, but never reads the trailing `Emissive Color: Color4` (16 bytes) gated by `vercond="#BS_GT_FO3#"` per nif.xml line 6250. Cross-version mods that ship a Skyrim-era PPLighting block on disk would have this field present; parser leaves 16 bytes unread, masked by `block_sizes` recovery.
- **Impact**: 16-byte under-read on any BSShaderPPLightingProperty with bsver > 34. Drops the emissive channel from the rare Skyrim-era PPLighting fallback. BSLightingShaderProperty (typical Skyrim shader) has its own correct emissive_color path at [shader.rs:480](crates/nif/src/blocks/shader.rs#L480) — unaffected.
- **Suggested Fix**: After `parallax_scale`, add a gated `if stream.bsver() > 34 { read 4×f32 emissive_color }` block. Pin via Skyrim PPLighting fixture in `shader_tests.rs`.

#### NIF-D3-09: `read_block_ref_list` bypasses `allocate_vec` budget guard — DoS via corrupt count
- **Dimension**: Stream Position × Allocation Safety
- **Game Affected**: All games (worst case Skyrim+ where lists are hot — `NiNode.children/effects`, `BsTreeNode.bones_1/bones_2`, `NiSkinInstance.bones`)
- **Location**: [crates/nif/src/stream.rs:439-446](crates/nif/src/stream.rs#L439-L446)
- **Status**: NEW
- **Description**: `read_block_ref_list` does raw `Vec::with_capacity(count as usize)` without going through `allocate_vec` / `check_alloc`. Every other size-prefixed reader in the file routes through the budget guard (cap = 256 MB single allocation, count ≤ remaining stream bytes). 8 call sites depend on this helper (base.rs:44/101, node.rs:102/109/313/314, particle.rs:906/910); `NiNode::parse` fires it twice per node — a 1000-node skeleton calls it 2,000 times.
- **Evidence**:
  ```
  $ grep -nE "with_capacity|reserve" crates/nif/src/stream.rs
  179:    /// Used in place of the raw `Vec::with_capacity(count as usize)`
  203:        Ok(Vec::with_capacity(count as usize))     # allocate_vec — guarded
  441:        let mut refs = Vec::with_capacity(count);  # read_block_ref_list — UNGUARDED
  ```
- **Impact**: Untrusted-NIF parsing (mod loader, asset hot-reload, network DLC) can OOM-panic the process from any block carrying a corrupt block-ref-list count. The OOM is in `Vec::with_capacity` *before* any read happens, so block-size recovery can't help.
- **Suggested Fix**: Route the count through `allocate_vec` like every other bulk reader:
  ```rust
  let count = self.read_u32_le()?;
  let mut refs = self.allocate_vec(count)?;
  for _ in 0..count { refs.push(self.read_block_ref()?); }
  ```
  No call-site changes; signature unchanged.

#### NIF-D4-05: BSEffectShaderProperty src/dst blend factors hardcoded — additive flags ignored
- **Dimension**: Import Pipeline
- **Game Affected**: Skyrim, FO4, FO76, Starfield
- **Location**: [crates/nif/src/import/material/walker.rs:303-312](crates/nif/src/import/material/walker.rs#L303-L312)
- **Status**: Existing follow-up to #354 (called out in source comment at the site)
- **Description**: When BSEffectShaderProperty triggers the implicit alpha-blend override the importer leaves `src_blend_mode / dst_blend_mode` at their MaterialInfo defaults (SRC_ALPHA / INV_SRC_ALPHA) — but BSEffectShader content commonly authors additive blending (ONE / ONE) for fire / glow / emissive surfaces via `Own_Emit`. Source comment explicitly defers the additive branch.
- **Impact**: Skyrim+ fire/magic glow surfaces blend over their background instead of brightening it. Cumulative VFX (shield stacking, fire layering) over-attenuate to grey.
- **Suggested Fix**: After the alpha-blend override, test `shader.shader_flags_1 & OWN_EMIT` (Skyrim/FO4) with a CRC32 OWN_EMIT fallback for FO76/Starfield (mirror `is_two_sided_from_modern_shader_flags`). When set, `src_blend_mode = 0 (ONE), dst_blend_mode = 0 (ONE)`. Symmetric `EnvMap_Light_Fade` covered by same plumbing.

#### NIF-D3-10: Six `.reserve()` / `with_capacity()` calls on file-driven u32 counts bypass the alloc cap
- **Dimension**: Stream Position × Allocation Safety
- **Game Affected**: Skyrim+ (BSVER ≥ 132 path), pre-Skyrim NiTransformData
- **Location**: [shader.rs:443/447/902/906/1502/1506](crates/nif/src/blocks/shader.rs), [interpolator.rs:304](crates/nif/src/blocks/interpolator.rs#L304), [bs_geometry.rs:465](crates/nif/src/blocks/bs_geometry.rs#L465)
- **Status**: NEW
- **Description**: Six pre-allocations read `u32` length straight from stream and call `Vec::reserve` / `Vec::with_capacity` without `allocate_vec`. Plain `reserve` aborts on capacity overflow (no `try_reserve` here). Practical attack surface: deliberately oversized `Num SF1`/`Num SF2` in modder-shipped NIFs.
- **Suggested Fix**: Replace each with `stream.allocate_vec(count)?`. Mechanical cleanup.

#### NIF-D3-11: `until="N.M.O.P"` boundary uses `<=` not `<` — off-by-one on exact-version match
- **Dimension**: Stream Position × Version Gate
- **Game Affected**: None of the supported games hit either boundary in vanilla content (Bethesda jumps over both 10.1.0.0 and 20.1.0.1).
- **Location**: [interpolator.rs:294](crates/nif/src/blocks/interpolator.rs#L294), [properties.rs:233](crates/nif/src/blocks/properties.rs#L233)
- **Status**: NEW
- **Description**: niftools' `until=` semantics is exclusive (`version < until_version`). Current Rust gates use `<=`, so at exactly `version == until_version` the field is read though nif.xml says it's absent. Two known instances: `NiTransformData.Order` (`until="10.1.0.0"`) and `NiTexturingProperty.Apply Mode` (`until="20.1.0.1"`).
- **Suggested Fix**: Flip both `<=` to `<`. Add boundary-exact regression test.

## Methodology and reproduction

This iteration carried forward the 2026-04-26 corpus histograms rather than re-running the 9-archive sweep (~10 min wall-clock; would not change conclusions). The dispatch-table-vs-XML diff is the load-bearing step and was re-run from scratch at HEAD `c1a7f55`.

Reproduce dispatch-table inventory:

```bash
python3 -c '
import re
text = open("crates/nif/src/blocks/mod.rs").read()
m = re.search(r"match type_name \{(.*?)^    \}", text, re.DOTALL | re.MULTILINE)
body = m.group(1)
cleaned = []
in_block_comment = False
for line in body.splitlines():
    s = line.strip()
    if in_block_comment:
        if "*/" in s: in_block_comment = False
        continue
    if s.startswith("/*"):
        if "*/" not in s: in_block_comment = True
        continue
    if s.startswith("//"): continue
    idx = line.find("//")
    if idx >= 0: line = line[:idx]
    cleaned.append(line)
arms = set(re.findall(r"\"([A-Za-z_][A-Za-z0-9_:]*)\"", "\n".join(cleaned)))
arms -= {"NiUnknown"}
print(len(arms))
'
```

## Caveats

The NiUnknown bucket conflates two recovery paths:

- **(A)** `parse_block` falling through to `_ =>` because the type name has no arm (true coverage gap).
- **(B)** `parse_block` returning `Err(_)` because a parser bombed mid-block, triggering `lib.rs:356-388` to insert NiUnknown via `block_size`-driven recovery.

Both produce a `NiUnknown` whose `type_name` field is the original wire type. A row showing a name that IS in the dispatch table indicates path (B) — the gap is in the parser, not the dispatch. **#601 (FO4-DIM5-01)** tracks bundling improvement and would auto-classify the residual FO4 MeshesExtra counts (`BSMeshLODTriShape`, `BSTriShape`) into "parser-error" vs "missing dispatch".
