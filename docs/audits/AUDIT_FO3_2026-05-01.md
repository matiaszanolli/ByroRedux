## Dimension 1 — NIF v20.2.0.7 Parser Correctness

### FO3-1-01: Parallax Max Passes / Scale gate off-by-one against nif.xml
- **Severity**: MEDIUM
- **Location**: `crates/nif/src/blocks/shader.rs:96`
- **Status**: NEW
- **Description**: `BSShaderPPLightingProperty::parse` reads Parallax
  Max Passes + Parallax Scale when `bsver >= 24`. nif.xml line
  6247-6248 specifies `vercond="#BSVER# #GT# 24"` (strictly greater
  than 24). The correct gate is `bsver >= 25` (== `bsver > 24`). FO3
  ships content at BSVER=24 (`V20_2_0_7__11_4`, nif.xml:203) — those
  meshes consume 8 phantom bytes of parallax scalars they don't
  carry on the wire.
- **Impact**: On FO3 BSVER=24 PPLighting blocks the parser advances
  8 bytes past block end. FO3's `block_sizes` table re-aligns the
  outer dispatch loop, hiding the defect at the recoverable-rate
  metric. `parallax_max_passes` / `parallax_scale` end up holding
  two unrelated f32 words, then flow through `import/material/walker.rs`
  into the parallax fragment branch — cosmetic-only wrong
  displacement on those meshes.
- **Fix sketch**: Change `bsver >= 24` to `bsver >= 25` and update
  the inline comment to match the `#GT# 24` source phrasing. Add a
  synthetic dispatch_test driving a BSVER=24 PPLighting block to
  lock the regression.

### FO3-1-02: Refraction gate comment uses inverted phrasing
- **Severity**: LOW
- **Location**: `crates/nif/src/blocks/shader.rs:87,89`
- **Status**: NEW (cosmetic; arithmetic is correct)
- **Description**: `bsver >= 15` is mathematically equivalent to
  nif.xml's `BSVER #GT# 14` (line 6245-6246). The comment says "for
  bsver >= 15", which doesn't mirror the source-of-truth `> 14`
  phrasing. The pattern of inverting the inequality is what drove
  the off-by-one in the parallax gate (FO3-1-01).
- **Impact**: None at runtime. Risk: next reader extends pattern,
  gets next gate wrong by analogy.
- **Fix sketch**: Either flip both gates to the `bsver > N` form
  (matching nif.xml verbatim) or rewrite comments as `bsver > 14` /
  `bsver > 24` so the spec phrasing sits next to the code.

## Dimension 4 — Rendering Path for FO3 Shaders

### FO3-4-01: FO3 PPLighting `texture_clamp_mode` not propagated to `MaterialInfo`
- **Severity**: HIGH
- **Location**: `crates/nif/src/import/material/walker.rs:471-538`
- **Status**: NEW
- **Description**: `BSShaderPPLightingProperty.texture_clamp_mode: u32`
  is parsed at `crates/nif/src/blocks/shader.rs:48` (read line 84
  via `BSShaderPropertyData::parse_fo3`). The walker FO3 branch
  reads texture set, parallax, and decal flags but never assigns
  `info.texture_clamp_mode = shader.texture_clamp_mode`. Compare
  BSEffectShader at line 267 (`info.texture_clamp_mode =
  effect.texture_clamp_mode;` — #610) and NiTexturingProperty at
  lines 455–458 (which only mirrors per-slot
  `base_texture.flags & 0xF`).
- **Impact**: FO3 / FNV meshes that bind a `BSShaderPPLightingProperty`
  with CLAMP / MIRROR `clamp_mode` (terrain LOD swatches, billboarded
  posters, scope reticles, glow planes, decals where
  NiTexturingProperty is absent and the PP block is the sole
  authority) silently fall back to default WRAP. Edge bleeding /
  wrap-tiling artefacts.
- **Fix sketch**: Inside the PPLighting branch, after line 514, add
  `info.texture_clamp_mode = shader.texture_clamp_mode as u8;`. Cast
  is safe (nif.xml enum range is 0..=3).

### FO3-4-02: FO3 PPLighting `env_map_scale` (BSShader base) not propagated
- **Severity**: HIGH
- **Location**: `crates/nif/src/import/material/walker.rs:471-538`
- **Status**: NEW
- **Description**: `BSShaderPropertyData.env_map_scale: f32`
  (`crates/nif/src/blocks/base.rs:180`, read at line 207 inside
  `parse_base`) is the legacy env-map intensity scalar for FO3 / FNV
  PPLighting / NoLighting / TileShader. The walker PPLighting branch
  routes `texture_set[4]` (env cube) and `[5]` (env mask) to
  `info.env_map` / `info.env_mask` (#452, lines 503–513), but the
  scalar that modulates them is dropped. The Skyrim+ ParallaxOcc /
  EnvironmentMap path captures it via `apply_shader_type_data`
  (`material/shader_data.rs:65`); the BSEffectShader path captures
  it at `walker.rs:214` — only the FO3 PPLighting path is missing.
- **Impact**: FO3 / FNV glass bottles, polished metal, T-45/T-51
  power armor, brass / copper plumbing — anything using PPLighting
  env-cube reflections — render with `env_map_scale = 0.0`
  (`MaterialInfo::default()`) and lose all reflection intensity even
  with a valid env cube bound. The texture path arrives but the
  multiplier is zero.
- **Fix sketch**: Same branch, add
  `info.env_map_scale = shader.shader.env_map_scale;`. Field path:
  `BSShaderPPLightingProperty.shader: BSShaderPropertyData → .env_map_scale`.

  `embed_name_toggle` (#616) which is a no-op on uniform vanilla
  policy.

  `TileShaderProperty`, `BSShaderTextureSet`,

  controller variants, Havok constraint stubs.

  same MaterialInfo walker — both FO3-4-01 (texture_clamp_mode) and
  FO3-4-02 (env_map_scale) are FNV bugs of identical shape (the
  walker branch fires for both FO3 and FNV PPLighting).
- **CLI**: no `Fallout3.esm` / `FalloutNV.esm` -specific code paths
  in `main.rs` / `scene.rs` / `cell_loader.rs`.

**FO3-only gaps in shared paths**: none in the codebase. The two NEW
HIGH items (FO3-4-01, FO3-4-02) affect FO3 + FNV identically; FO3
just happens to ship more PPLighting content.
