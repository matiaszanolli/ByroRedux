# Fallout 3 Compatibility Audit — 2026-07-16

**Engine**: ByroRedux (Rust + Vulkan Gamebryo/Creation-engine rebuild)
**Scope**: Readiness of the shared FNV code path for **Fallout 3** content, hunting
the FO3-vs-FNV **divergences** (not a re-run of `/audit-fnv`). 7 dimensions,
orchestrated as Task agents (max 3 concurrent). Real FO3 GOTY data validated where
possible: `/mnt/data/SteamLibrary/steamapps/common/Fallout 3 goty/Data/`.

---

## Executive Summary

**Compatibility level: HIGH — parity with FNV, one real visual-completeness gap
found, no new correctness defects.** FO3 rides the FNV path end-to-end (both share
the single `GameKind::Fallout3NV` enum variant, and both ship NIF v20.2.0.7 at
bsver 34), so nearly every code path audited is *structurally* the same code FNV
already exercises — divergence is only possible where an explicit game-kind branch
exists, and every such branch was traced. This is the fifth consecutive FO3 audit
cycle (following 2026-06-23, 06-28, 07-02, 07-03, 07-05) to come back CLEAN or
near-CLEAN, re-verified against live code rather than carried forward on trust.

**Delta vs FNV**: The two functional FO3-distinctive facts are (1) the **Obscript
scripting runtime gap** — 1,257 SCPT records parse, no runtime executes them yet
(M47.0/M47.1 landed, M47.2 in progress, owned by `/audit-scripting`) — a known,
tracked blocker, not a bug; and (2) a **newly-identified distant-object LOD gap**
(FO3-D4-01, MEDIUM) — the Oblivion-style per-cell placement-LOD scheme that closed
issue #1726 wired for `GameKind::Oblivion | GameKind::Fallout3NV` turns out to never
fire on real FO3 *or* FNV data, because neither game ships a `distantlod\` folder in
any vanilla archive (base or DLC). This is a genuine scope gap in a closed issue's
premise, not a regression.

**Findings: 3 total — 0 CRITICAL, 0 HIGH, 1 MEDIUM, 2 LOW.** No behavioral change
is required for FO3 interior/exterior content to render at FNV parity; the one
MEDIUM finding is a visual-completeness gap (missing distant-object silhouettes),
not a crash/corruption/parse-failure risk.

## Dimension Findings

### Dimension 1 — FO3 Rendering Path (Inline Shaders) · CLEAN (0 new findings)

All 7 checklist items verified correct against current code:
1. Legacy single-u32 flag layout confirmed (`ShaderFlags::classify` selects
   `LegacyFo3Fnv`; decal via `fo3nv_f1` bits 26/27 + `fo3nv_f2` bit 21, two-sided via
   `NiStencilProperty`, alpha-test via `NiAlphaProperty`) — no FO4 u32-pair leakage,
   cross-era bit collisions firewalled by compile-time asserts.
2. Normal maps resolve from the dedicated `BSShaderTextureSet` slot 1; the Oblivion
   bump-slot fallback only fires when the dedicated slot is absent — no regression.
3. `BSShaderNoLightingProperty` routes through `MATERIAL_KIND_NO_LIGHTING = 102`,
   with an explicit shader branch (`triangle.frag:708`) that short-circuits scene
   lighting/GI/distance falloff — self-illumination does not dim with distance.
4. NIFAL single boundary confirmed: `translate_material` is the sole
   `ImportedMesh → Material` site; `Material.metalness`/`roughness` are plain `f32`,
   filled at import time by `classify_pbr_keyword`, no second per-game resolution site.
5. `EmissiveSource::Material` confirmed for FO3 (from sibling `NiMaterialProperty`);
   the Skyrim+ `Lighting` / FO4+ `Effect` variants are gated inside a BSVER≥83 field
   FO3 NiTriShapes never populate — no scale-factor bleed.
6. `MAT_FLAG_PBR_BSDF` (Disney-BSDF gate) is **structurally unreachable** for FO3:
   it is only set via `merge_bgsm_into_mesh`, which never fires without a resolved
   `.bgsm`/`.bgem` path, and FO3 authors none. FO3 fragments always take the plain
   Lambert path.
7. `WaterShaderProperty` produces distinct `GpuMaterial` entries, not collapsed with
   glass (water texture paths don't match the glass keyword list).

One pre-existing OPEN issue (#1856, `MaterialInfo.water_shader_flags` dead-ending)
touches this dimension's water path but was confirmed FO3-inert — the non-BS
`WaterShaderProperty` struct carries no flags field to drop in the first place; not
re-reported.

### Dimension 3 — ESM Record Coverage (`Fallout3.esm`) · 1 LOW

#### FO3-D3-003: XATO/XTNM/XTXR REFR overlay path has zero FO3/FNV test coverage
- **Severity**: LOW
- **Location**: `byroredux/src/cell_loader/refr_texture_overlay_tests.rs` (0 FO3/FNV mentions in 469 lines); parse arm at `crates/plugin/src/esm/cell/walkers.rs:938-983`
- **Status**: Existing: #1887 (closed) — unresolved test-completeness checklist item from that issue, not a new code bug
- **Description**: Closed issue #1887 established that FO3/FNV REFRs never carry a resolvable XATO texture-set FormID (it's an FNV-only Activation-Prompt subrecord being misread, functionally inert), and the shipped fix documented the provenance caveat in code comments. The issue's own completeness checklist called for an FNV/FO3 test fixture (positive or negative) pinning the arm's game-scope; that checkbox was never actually completed — `refr_texture_overlay_tests.rs` remains 100% FO4-synthetic.
- **Impact**: No functional impact today (misparse is inert on both FO3 and FNV). Pure regression-guard gap: a future change that widens the XATO lookup could start resolving spurious overlays on FNV/FO3 content without any test catching it.
- **Related**: #1887, #584, #1654.
- **Suggested Fix**: Add one negative-path test asserting a `GameKind::Fallout3NV` REFR with a plausible activation-prompt-shaped XATO payload does not produce a resolvable overlay.

**Everything else verified OK, including a HIGH-value check**: the SCPT SCHR
`flags` field (issue #1654) reads via `u16_or_default()` landing exactly at cursor
offset 18-20, not the buggy u32 read that would fail on every real FO3 script — a
dedicated regression test is specifically shaped to catch a u16/u32 confusion. Live
record count reconciled at 44,666 (+9 benign drift over the 44,657 baseline, already
documented). CLAS/ACBS FO3-vs-FNV subform width handling, DIAL/INFO fixed-offset
layout, `sub_reader.rs` soft-truncation semantics, CELL XCLL 36-vs-40-byte
FO3-vs-FNV dispatch, WATR/WTHR/CLMT shared-parser byte layouts, and
`GameKind::from_header`'s HEDR-band classification of FO3 GOTY (0.94) were all
independently confirmed correct with no `unreachable_patterns` warnings in
`walkers.rs`. Two prior FO3-D3 findings from the 2026-07-05 audit (#1887, #1888)
were reconfirmed CLOSED with fixes present.

### Dimension 4 — FO3 Cell Loading End-to-End (interior + exterior) · 1 MEDIUM

#### FO3-D4-01: `PlacementLodProvider` distant-object LOD never fires on FO3 or FNV — vanilla archives ship zero `distantlod\*.lod` files
- **Severity**: MEDIUM
- **Location**: `byroredux/src/cell_loader/placement_lod.rs:283-353` (`stream_placement_lod_blocks`, gated `GameKind::Oblivion | GameKind::Fallout3NV`); `docs/engine/exal.md:240,576-613`
- **Status**: NEW (scope-mismatch in closed #1726's premise — not a regression of the Oblivion-side code #1726 landed, which is correct)
- **Description**: `placement_lod.rs` implements the `DistantLOD\<World>_<x>_<y>.lod → _far.nif` per-cell placement scheme and gates it to run for both `GameKind::Oblivion` and `GameKind::Fallout3NV`, per `exal.md`'s claim that the distant-object source is "same" across Oblivion/FO3/FNV. The format itself was reverse-engineered and validated **only against Oblivion** (9,889 real `.lod` files). Direct probes of every real FO3 and FNV archive (base game + all DLC) show **zero `distantlod\` matches in any of them**. `Fallout - Meshes.bsa` has only 2 `_far.nif` entries total (`washmonumentlod_far.nif`, `ravecity_far.nif` — one-off landmark assets, not a systematic scheme). `placement_lod_archive_path` therefore always builds a path that never resolves, and `spawn_placement_lod_cell` returns `None` on every call — no crash, the streaming ring just silently inserts empty sentinels forever.
- **Evidence**: `bsa_grep` for `"distantlod"` against `Fallout - Meshes.bsa` and all 13 FNV BSAs (base + 12 DLC) returns 0 matches everywhere. FO3's Meshes.bsa instead carries a large `meshes\landscape\lod\<worldspace>\...` terrain-LOD block tree (including named landmark sub-folders like `washmontop`, `dcworld03/08/09`), suggesting Bethesda folded landmark-object LOD into the terrain-LOD block system for Fallout titles rather than shipping Oblivion's separate per-cell object-placement format.
- **Impact**: Distant static-object LOD (buildings, rocks, landmark silhouettes beyond the loaded-cell radius) is completely absent on FO3 *and* FNV exteriors — the same class of gap `object_lod.rs` already documents for the Skyrim+/FO4 `.bto` path, just via the sibling provider. Visual-completeness only (terrain horizon still renders via heightmap LOD); no parse failure, no crash, no REFR mis-placement. Notably, FNV — this audit suite's own reference/baseline title — has *never* had this path runtime-validated either, only Oblivion has.
- **Related**: #1726 (closed; premise correct for Oblivion, incomplete for FO3/FNV), #1731 (VWD flag, same subsystem)
- **Suggested Fix**: Either (a) confirm via a wider GECK/xEdit cross-check that FO3/FNV genuinely ship no per-object placement LOD and narrow the `exal.md` §5 table + the `GameKind::Fallout3NV` gate to "Oblivion only," folding the Fallout landmark case into the existing terrain-LOD block path instead; or (b) if FO3/FNV do encode object LOD via the `meshes\landscape\lod\<landmark-subworld>\` convention, extend `terrain_lod.rs`'s landmark-tile handling to spawn renderable object geometry from it. Either way, update closed #1726 to note the FO3/FNV gap explicitly.

**Everything else verified OK**: Megaton interior REFR baseline (929, confirmed via
`parse_real_fo3_megaton_cell_baseline`); exterior worldspace/origin math is fully
data-driven with no FNV-hardcoded string or coordinate leaking into FO3's path;
`"wasteland"` in the preferred-default worldspace list is FO3's own Capital
Wasteland EDID, not an FNV artifact; no FNV-only branch exists anywhere in the
shared cell loader (FO3 and FNV share the single `GameKind::Fallout3NV` variant
everywhere in this path, making divergence structurally impossible without an
explicit split, and none exists); WTHR→CLMT→WTHR sun-position curve resolution is
fully climate-record-driven (the pre-#1012 hardcoded arc bug that specifically hit
FO3 Capital Wasteland is fixed and stayed fixed); `NifImportRegistry` is a
process-lifetime, path-keyed cache with correct LRU eviction and no per-cell-cycle
leak; the Skyrim+/FO4 `.bto` object-LOD path correctly excludes FO3/FNV, deferring
to the (now-flagged-gapped) placement-LOD sibling.

## FNV-Shared Surface

The following record types, block types, and shader paths are fully shared between
FO3 and FNV with no FO3-specific gap found this cycle:

- **ESM parser**: entire `crates/plugin/src/esm/` tree — SCPT (SCHR u16 flags),
  CLAS/ACBS subform widths, DIAL/INFO fixed-offset layouts, CELL XCLL (36-vs-40-byte
  dispatch), WATR/WTHR/CLMT, NAVM, `GameKind::from_header` HEDR-band classification.
- **NIF parser**: `BSShaderPPLightingProperty`/`BSShaderNoLightingProperty` field
  decode, `BSSegmentedTriShape`, typed particle emitters (`NiPSysEmitter` family),
  all gated on per-file `bsver`/`version`, not game-variant predicates — byte-
  identical wire format for FO3 and FNV.
- **Material translation**: single NIFAL `translate_material` boundary, Disney-BSDF
  gate, `EmissiveSource` discriminator, `WaterShaderProperty` handling.
- **Havok collision**: `CollisionAuthoring::Classic` classification, shape
  resolution (`BhkMultiSphereShape`, `BhkConvexListShape`, etc.), `havok_motion_type`
  canonical enum mapping, `havok_scale = 7.0`.
- **Cell loading**: interior/exterior CLI, worldspace resolution, weather/climate
  resolution, `NifImportRegistry` caching — all keyed on the single
  `GameKind::Fallout3NV` variant, structurally identical for both games.
- **Animation/NPC spawn**: b-spline fallback, clip-registry dedup, hand-mesh
  loading, skeleton resolution.

FO3-only gaps found *inside* this shared surface this cycle: none rise above LOW
(the XATO test-coverage gap, FO3-D3-003, and the NifVariant sample-data note,
FO3-D6-01, are both documentation/test-completeness items, not functional bugs).

## FO3-Distinctive Gaps

1. **Inline-shader-only material universe** — confirmed structurally correct and
   complete (Dimension 1); Disney-BSDF/BGSM paths are provably unreachable, not
   merely untested.
2. **Capital Wasteland worldspace** — confirmed fully data-driven, no FNV-hardcoded
   assumption found (Dimension 4).
3. **Distant-object LOD gap (NEW this cycle, FO3-D4-01, MEDIUM)** — the
   Oblivion-style placement-LOD scheme never fires on FO3 *or* FNV vanilla data;
   both Fallout titles are missing distant static-object silhouettes, a gap that
   predates this audit but was not previously identified because it had never been
   runtime-probed against real archives for either title.
4. **1,257-SCPT scripting runtime gap** — the largest FO3-specific functional gap,
   by design and already tracked (M47.0/M47.1 landed, M47.2 in progress). Quest and
   world interactivity driven by Obscript is not yet executable. Not a bug; owned by
   `/audit-scripting`.
