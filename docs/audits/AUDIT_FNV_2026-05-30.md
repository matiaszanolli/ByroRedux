### Dimension 2 — ESM Record Parser: 1 MEDIUM, 1 LOW

#### D2-01: FO3/FNV WEAP DNAM mislabels Min Spread / Spread as ap_cost / min_spread
- **Severity**: MEDIUM
- **Location**: `crates/plugin/src/esm/records/items.rs:209-215`
- **Status**: NEW
- **Description**: The DNAM decode stores the u32 at offset 16 as `ap_cost` and the value at
  offset 20 as `min_spread`, but the FO3/FNV WEAP DNAM layout has **Min Spread (f32)** at offset
  16 and **Spread (f32)** at offset 20. Proven by dumping the Varmint Rifle's raw DNAM: the field
  read as `ap_cost` is a float ~0.024 reinterpreted as a garbage u32.
- **Impact**: Latent — these fields have no runtime consumer yet and the integration test asserts
  nothing about them — but the stored values are wrong and would surface the moment AP-cost /
  spread is wired to gameplay.
- **Suggested Fix**: Re-map per the FNV DNAM layout (Min Spread f32 @16, Spread f32 @20, locate the
  real Action Points cost field) and add a Varmint-Rifle assertion to the parse test.

#### D2-02: read_sub_records does not handle the XXXX extended-size override
- **Severity**: LOW
- **Location**: `crates/plugin/src/esm/reader.rs:500-521`
- **Status**: NEW
- **Description**: Sub-records larger than 0xFFFF use an `XXXX` sub-record carrying the true u32
  size of the following field. The reader doesn't special-case it. FalloutNV.esm has exactly one
  (a 177 KB WRLD `OFST`); benign today because no parser reads OFST and outer-stream alignment is
  force-restored, so no count/data regression. Real format gap that would bite a typed parser
  needing a sub-record positioned after an XXXX-prefixed oversized one (higher exposure on FO4/Skyrim).
- **Suggested Fix**: Handle `XXXX` in `read_sub_records` — when seen, read the u32 override and use
  it as the size of the next sub-record instead of its own 16-bit length field.

**Verified clean:** all 107 dispatch arms present (incl. every #808/#809/#810 long-tail arm); the
`unknown_records` catch-all is fully removed; `cell/walkers.rs` recompiles with zero
`unreachable_patterns`; FO4 SCOL/MOVS/PKIN/MSWP are gated behind `is_fo4_plus` and never steal FNV
dispatch; CELL XCLL fog fields are per-field gated with no over-read.

### Dimension 3 — Cell Loading End-to-End: 4 HIGH, 1 MEDIUM (all NEW)

> The cell-load *machinery* is structurally sound (process-lifetime NIF Arc cache, deduped clip
> registry, `[first..last)` ID-span unload key, correct M38 water-plane spawn + `submersion_system`).
> The bugs are in the runtime unload / transition lifecycle, which the startup-load baseline numbers

#### D3-01: Water-plane normal-map texture refcount leaked on every cell unload
- **Severity**: HIGH
- **Location**: `byroredux/src/cell_loader/water.rs:187-201` (acquire) vs `byroredux/src/cell_loader/unload.rs:82-118` (walk that misses it)
- **Status**: NEW
- **Description**: `spawn_water_plane` resolves the WATR TNAM normal map via `resolve_texture`
  (bumps the registry refcount) but stores the index only in `WaterMaterial.normal_map_index`. The
  water entity carries no `TextureHandle`/`NormalMapHandle` component, so the unload victim walk
  can't reach it and never calls `drop_texture`.
- **Impact**: One never-freed GPU texture + bindless slot per water-bearing cell unloaded, for the
  process lifetime. Self-caps at the count of distinct WATR textures but is a genuine leak in the
  M40 streaming / door-walk path.
- **Suggested Fix**: Attach `NormalMapHandle(resolved_normal_idx)` (when `!= 0`) on the water entity;
  the existing unload walk then drops it automatically — zero new unload code.

#### D3-03: Worldspace transition re-acquires sky cloud/sun textures without releasing the prior set
- **Severity**: HIGH
- **Location**: `byroredux/src/scene/world_setup.rs:252-311` (re-acquire); `byroredux/src/main.rs:1180-1223` (transition handler); `byroredux/src/cell_loader/unload.rs:125-137` (the #1199 deferral comment)
- **Status**: NEW (realizes the leak #1199 left as a known follow-up; #1199 is CLOSED with the deferral baked in)
- **Description**: `apply_worldspace_weather` resolves 4 cloud layers + 1 CLMT sun sprite (refcount
  bumps), intentionally surviving per-cell unload (#1199). On a door transition to an Exterior
  destination, the handler tears down the interior and drains streaming — neither releases the old
  worldspace's sky textures — then calls `apply_worldspace_weather` again, re-acquiring a fresh set
  while the old set stays pinned. The `SkyParamsRes::texture_indices()` accessor built for this
  release is dead on the release side.
- **Impact**: Up to 5 GPU textures (bindless slots + VkImages) leaked per worldspace crossing.
  Self-caps per unique worldspace but accumulates for a player door-walking between Mojave/DLC regions.
- **Suggested Fix**: In the Exterior transition handler, before the new `apply_worldspace_weather`,
  read `SkyParamsRes::texture_indices()` and `drop_texture` each non-zero/non-fallback handle — the
  boundary release #1199 deferred.

#### D3-04: Door-walk + debug-console interior load discards CellLoadResult.lighting → stale exterior lighting
- **Severity**: HIGH
- **Location**: `byroredux/src/cell_loader/transition.rs:222-231` and `byroredux/src/debug_load.rs:207-215` (both discard); contrast `byroredux/src/scene.rs:178-211` (startup path applies it)
- **Status**: NEW
- **Description**: The startup interior loader reads `result.lighting` and installs
  `CellLightingRes::from_cell_lighting(..., is_interior=true)`. The door-walk transition path and
  the `cell.load` debug command both call the same loader but discard the `CellLoadResult` (the
  lighting is parsed, then thrown away). So a runtime-loaded interior renders with the **previous**
  cell's `CellLightingRes`.
- **Impact**: Any interior reached at runtime (the M40 gameplay path) gets wrong ambient/fog,
  exterior clear color, and the directional sun leaking into a sealed interior — the exact
  sun-shaft failure #1282 closed for the gate-on-`is_interior` case. Hits FNV's reference interiors
  (saloon, vault, Doc Mitchell's) once door-walking is exercised.
- **Suggested Fix**: Move the `result.lighting → CellLightingRes` block into a shared helper called
  by the startup, transition, AND debug-console paths (or return `CellLoadResult` for each caller to
  install).

#### D3-05: GreyscaleLutHandle texture refcount leaked on cell unload
- **Severity**: HIGH
- **Location**: `byroredux/src/cell_loader/spawn.rs:907-911` (acquire) vs `byroredux/src/cell_loader/unload.rs:76-118` (victim walk omits it)
- **Status**: NEW
- **Description**: `spawn_placed_instances` resolves a BSEffectShaderProperty greyscale LUT
  (refcount bump) and attaches `GreyscaleLutHandle`, but the unload victim walk queries every other
  handle component except `GreyscaleLutHandle`, so the LUT texture is never dropped.
- **Impact**: One texture refcount leaked per distinct greyscale-LUT texture per unloaded cell, for
  the process lifetime. The component was simply missed when added under #890.
- **Suggested Fix**: Add `GreyscaleLutHandle` to the unload walk (one-line-per-site, mirroring the
  existing `NormalMapHandle` handling).

#### D3-02: Terrain splat-layer texture refcounts leaked on mesh-upload / no-allocator early returns
- **Severity**: MEDIUM
- **Location**: `byroredux/src/cell_loader/terrain.rs:273` (acquire) vs `terrain.rs:350` + `:369` (early returns before the tile-slot binds them)
- **Status**: NEW
- **Description**: `build_cell_splat_layers` resolves up to 8 splat textures (refcount bumps)
  *before* the two early-return guards (`allocator` absent; mesh-upload `Err`). On either early
  return, no `TerrainTileSlot` is allocated, so the refcounts have no owner the unload walk can reach.
- **Impact**: Up to 8 pinned texture refcounts per **failed** terrain-cell load. Trigger paths are
  degraded states (headless/test, OOM/mesh-budget cap), so not steady-state — but a real leak when
  it fires.
- **Suggested Fix**: Move `build_cell_splat_layers` below the mesh-upload + allocator guards
  (no dependency), matching the water-plane ordering convention; or `drop_texture` on each early return.

### Dimension 6 — Animation & Skinning + M41 NPC Spawn: 1 MEDIUM (all guards hold)

#### D6-01: BSPSysSimpleColorModifier authored particle colors dropped (FO3/FNV-era stacks)
- **Severity**: MEDIUM
- **Location**: `crates/nif/src/blocks/particle.rs:492-499` (parser); `crates/nif/src/import/walk/mod.rs:617-637` (`extract_first_color_curve`)
- **Status**: NEW (adjacent to #707, which only wired the legacy `NiPSysColorModifier` → `NiColorData` path)
- **Description**: `BSPSysSimpleColorModifier` is the dominant FO3/FNV-era particle-color modifier
  and carries its 3 colors INLINE (start/mid/end Color4). The parser is byte-correct but discards
  every value, and `extract_first_color_curve` only matches the legacy `NiPSysColorModifier` chain.
  So an FNV NIF whose color is authored solely via the simple-color modifier falls back to the
  name-heuristic preset color.
- **Impact**: FNV/FO3 particle effects (geysers, steam, many weapon/spell FX) render with generic
  preset colors instead of the authored ramp. Visible-but-not-fatal; kinematics/size/rate unaffected.
- **Suggested Fix**: Capture `Colors[0]`/`Colors[2]` in a typed struct and extend
  `extract_first_color_curve` to fall back to the simple-color modifier, emitting the same
  `ParticleColorCurve` so both spawn sites' existing override picks it up unchanged.

### Dimension 7 — NIFAL Canonical Translation: 1 MEDIUM, 1 LOW (all 6 invariants hold)

#### D7-01: PBR keyword classifier runs at IMPORT, not at the NIFAL translate() boundary — doc framing claims otherwise
- **Severity**: MEDIUM
- **Location**: `crates/nif/src/import/mesh/ni_tri_shape.rs:184,246-247` (physical classify site); doc claims at `byroredux/src/material_translate.rs:58-61` and `crates/core/src/ecs/components/material.rs:568-590`
- **Status**: NEW
- **Description**: For FNV legacy inline-shader content the keyword classifier
  (`classify_pbr_keyword`) physically runs at NIF import time and writes
  `metalness_override: Some(...)`/`roughness_override: Some(...)` onto the `ImportedMesh`. At the
  boundary, `resolve_pbr` sees a non-NaN value, **skips its classifier**, and only clamps. So the
  structure is "classify-at-import + clamp-at-translate", not "classify-at-translate" as the docs
  imply. No behavioral bug — single-source-of-truth is preserved by delegation to the shared free
  fn — but the docs blur the *construction* site (single, at the boundary) with the *classification*
  site (upstream, at import), which can mislead a future change into a real bug.
- **Suggested Fix**: Doc-only. Amend `resolve_pbr` + `material_translate.rs` docs to state the
  classifier already ran at import and `resolve_pbr` is a clamp + sentinel-backstop. Optionally
  rename `resolve_pbr` → `clamp_and_backstop_pbr`.

#### D7-02: Deleted Material::classify_pbr cited as a live "per-frame draw build" call — missed site of CLOSED #1321
- **Severity**: LOW
- **Location**: `crates/nif/src/import/material/mod.rs:988-989`
- **Status**: Existing: #1321 (missed site — fix landed in core `material.rs`, skipped this nif-crate site)

## Draw-distance / exterior render-budget (raised during this audit)

A live exterior screenshot during this run surfaced a draw-distance concern. This is **not a
regression** (it is inherent to the current fixed `--radius` cell ring + no distant-LOD path), and
is being investigated separately from the regression findings above. It is connected to D5-02: the
exterior already spawns 9577 entities / ~4500 draws at radius 3, and every static mesh carries an RT
BLAS, so naively extending the cell ring multiplies both the raster draw count and the TLAS/BLAS
budget. A proper fix is a distance-LOD / impostor tier (the engine has no LOD record consumer yet —
ROADMAP M35 "terrain LOD" is the closest planned work) rather than simply widening the radius. See
the follow-up discussion below this report.
