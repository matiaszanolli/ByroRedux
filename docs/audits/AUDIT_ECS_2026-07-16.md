### 7. Component Lifecycles — 1 LOW finding

**New this session**: `WanderBehavior`/`WanderState`/`WanderPhase`,
`TravelBehavior`/`TravelState`/`Traveled`, `FollowBehavior`/`FollowState`,
`EscortBehavior`/`EscortState`/`Escorted`, `GuardBehavior`/`GuardState`,
`PatrolBehavior`/`PatrolState` — all six new `SparseSetStorage` component
pairs, all six new opt-in exclusive systems (`byroredux/src/systems/{wander,
travel,follow,escort,guard,patrol}.rs`), plus the shared `step_toward`
primitive (`systems/locomotion.rs`, pure function, no ECS access).

**Verified correct**:
- **Storage**: all twelve new components are `SparseSetStorage` (confirmed by
  reading every `impl Component for … { type Storage = SparseSetStorage<Self>; }`
  line) — a growing actor population cannot force any onto `PackedStorage`.
- **Mutual exclusivity by construction**: `npc_spawn.rs`'s spawn-tail
  (lines ~1628–1711) runs seven independent `if runs_X { world.insert(…) }`
  blocks, each gated by `active_package_is_X`, all of which call the *same*
  `active_package()` (single `.find()` over the priority-ordered package list)
  and then test the *one* resolved `PackRecord`'s `procedure_type` against a
  distinct constant (`PROCEDURE_SANDBOX=12`, `_WANDER=5`, `_TRAVEL=6`,
  `_FOLLOW=1`, `_ESCORT=2`, `_GUARD=14`, `_PATROL=13` —
  `crates/plugin/src/esm/records/misc/ai.rs:137–191`, all distinct). Pinned
  end-to-end by
  `active_package_is_sandbox_wander_travel_follow_escort_guard_and_patrol_are_mutually_exclusive`.
  No actor can receive two of the seven Behavior components.
- **One-shot terminal markers**: `Traveled` (`travel_system`) and `Escorted`
  (`escort_system`) are inserted exactly once on arrival and checked via a
  `.contains(entity)` skip-guard on every subsequent tick (`travel.rs:131`,
  `escort.rs:136`) — pinned by
  `travel_system_tags_traveled_on_arrival_and_then_stops_moving` and
  `escort_system_tags_escorted_on_arrival_and_then_stops`, both of which run
  the system twice and assert position is unchanged on the second call.
- **Indefinite, non-terminal state**: `WanderState`/`PatrolState` genuinely
  mutate every tick (`target`/`phase`/`pick_count` all change on arrival —
  `wander.rs::step_oscillating_wander`); `GuardState` is read every tick for
  the leash check but its `anchor` field is resolved once and frozen (see
  finding ECS-2026-07-16-01 below for a doc-accuracy issue on this point —
  the *behavior* is correct, only the doc comment overstates it).
- **Live vs. frozen resolution**: `FollowState`/`EscortState` (mid-collect)
  re-read `GlobalTransform` fresh every tick via `world.get::<GlobalTransform>`
  inside the Pass-1 loop (`follow.rs:119`, `escort.rs:170`) — confirmed by
  `follow_system_tracks_a_moving_target_live`, which moves the target between
  two `follow_system` calls and asserts the actor turns to the *new* position.
  `TravelState::destination`, `EscortState::destination` (once leading), and
  `GuardState::anchor` are all resolved once and never re-read from a live
  entity afterward — confirmed by code inspection (`state: None` on all three
  "already resolved" branches means Pass 2 never rewrites them).
- **Shared logic, separate storage**: `patrol_system` calls
  `wander_system::step_oscillating_wander` directly (`patrol.rs:90-103`) and
  `PatrolState` reuses `WanderPhase` verbatim (`components/patrol.rs:22,55`) —
  no divergent copy of the phase-transition state machine exists.
  `guard_system`/`escort_system` both call
  `travel_system::resolve_destination`/`resolve_travel_destination`
  (`pub(crate)`, generic over primitive fields, per M42.7's extraction) rather
  than reimplementing the resolve-or-pick-once logic — confirmed Guard's
  fallback is deliberately its own position (not Travel's random pick), per
  the documented revert of that alternative.
- **Opt-in gating, all seven independent**: `boot.rs:721-805` gates each of
  the seven systems behind its own env var
  (`BYRO_SANDBOX_SIT`/`_WANDER`/`_TRAVEL`/`_FOLLOW`/`_ESCORT`/`_GUARD`/
  `_PATROL`), all registered `add_exclusive(Stage::PostUpdate, …)` — none
  unconditionally registered, none in the parallel batch (so no
  `undeclared_parallel_count` regression per Dimension 5b).
- **Lock discipline**: every one of the six new systems follows the
  established two-pass (all-reads, then all-writes) structure — each
  `world.query::<T>()` / `world.query_mut::<T>()` call acquires and releases
  its own single-type lock; no `query_2_mut`/`resource_2_mut` paired
  acquisition is used anywhere in this code, so the TypeId-ordering rule
  doesn't apply and there is no ABBA surface. The outer `behavior_q` read
  guard (e.g. `WanderBehavior`, `GuardBehavior`, …) is held for the whole
  function including Pass 2, which is harmless (a read lock on a type Pass 2
  never touches) and matches the pre-existing `sandbox_seat_system` pattern —
  not a new risk.

#### ECS-2026-07-16-01: `GuardState` doc comment overstates its write frequency

- **Severity**: LOW
- **Dimension**: 7 (Component Lifecycles)
- **Location**: `crates/core/src/ecs/components/guard.rs:55-62`
- **Status**: NEW
- **Description**: The doc comment on `GuardState` says: *"Unlike Travel,
  there is no terminal marker: guarding continues indefinitely, so this state
  is read **and written** every tick, the same shape
  [`super::wander::WanderState`] has."* This is inaccurate. `guard_system`
  (`byroredux/src/systems/guard.rs:99-167`) only ever sets
  `GuardDecision::state` to `Some(GuardState { anchor })` on the branch where
  `existing_state` is `None` (first sight); once resolved, every later tick
  takes the `Some(s) => (s.anchor, None)` branch, and Pass 2's write loop
  (`if let Some(state) = d.state { sq.insert(…) }`) is a no-op for that
  entity from then on. `WanderState`, by contrast, genuinely has `target`,
  `phase`, and `pick_count` change on nearly every tick via
  `step_oscillating_wander`. The same file's own preceding sentence — *"anchor
  is resolved or picked exactly once … and then frozen — mirrors
  [`TravelState::destination`]"* — and the parallel module doc in
  `byroredux/src/systems/guard.rs:47-50` (*"Anchor is frozen once
  resolved/picked, exactly like `TravelState::destination`"*) both correctly
  describe the frozen-once discipline, directly contradicting the later
  "written every tick" sentence in the same doc block.
- **Evidence**:
  ```rust
  // crates/core/src/ecs/components/guard.rs:55-62
  /// ... `anchor` is resolved or
  /// picked exactly once (lazily, the first tick `guard_system` sees the
  /// actor) and then frozen — mirrors
  /// [`super::travel::TravelState::destination`]. Unlike Travel, there is no
  /// terminal marker: guarding continues indefinitely, so this state is read
  /// *and* written every tick, the same shape
  /// [`super::wander::WanderState`] has.
  ```
  ```rust
  // byroredux/src/systems/guard.rs:120-126 — write only on first sight
  let (anchor, state) = match existing_state {
      Some(s) => (s.anchor, None),
      None => {
          let anchor = resolve_anchor(world, behavior, current);
          (anchor, Some(GuardState { anchor }))
      }
  };
  ```
- **Impact**: Documentation-only; `guard_system` is the sole consumer of
  `GuardState` (confirmed via repo-wide grep — no other file reads or writes
  it) and its tests (`guard_system_holds_position_once_within_leash_and_returns_when_displaced`)
  correctly exercise and assert the frozen-anchor behavior, so runtime
  behavior is unaffected. Risk is future maintainers reading only the
  component doc and assuming `GuardState` needs per-tick write handling
  (e.g. when adding a save/load snapshot path or a future displacement
  system), which could lead to unnecessary work or a wrong assumption about
  change-detection granularity if `TRACK_CHANGES` is ever enabled on it.
- **Related**: None (new code, no prior issue).
- **Suggested Fix**: Reword the `GuardState` doc's last sentence to match its
  actual discipline, e.g.: *"Unlike Travel, there is no terminal marker —
  `guard_system` reads this state every tick for the leash check, but
  `anchor` itself is written only once, the same frozen-after-resolution
  discipline `TravelState::destination` has, not `WanderState`'s continuous
  mutation."*
