Full 7-dimension sweep (run as part of the `comprehensive` audit-suite). HEAD `435e265d`,
clean `main`. Real FO3 GOTY data present (`Fallout3.esm`, `Fallout - Meshes.bsa`,
`Fallout - Textures.bsa`, all 5 DLC archives). Static + offline-data validation; no

for FO3 reachability.

#### FO3-1-01: `intern_texture_path` does not guard whitespace-only paths
- **Severity**: LOW
- **Dimension**: Inline Shaders / Material
- **Location**: `crates/nif/src/import/material/mod.rs:33-39`
- **Status**: NEW
- **Description**: `intern_texture_path` collapses a texture-slot string to `None` only when
  `path.is_empty()`. A non-empty all-whitespace string (`" "`, `"\t"`) passes the guard and is
  interned to `Some(sym)`.
- **Evidence**: `if path.is_empty() { None } else { Some(pool.intern(path)) }` — no `.trim()`,
  no whitespace check. The **normal** slot is still safe (spawn-side `h != fallback()` guard at
  `byroredux/src/cell_loader/spawn.rs:883` → index 0), but the **diffuse** slot binds
  unconditionally (`spawn.rs:721`/`:846`, no `!= fallback()` filter), so a whitespace-only
  `BSShaderNoLightingProperty.file_name` binds the magenta checker placeholder instead of the
  neutral fallback (the `None`→`neutral_fallback()` early-out at `asset_provider.rs:580-582` is
  bypassed because the value is `Some(" ")`, not `None`).
- **Impact**: Near-zero vanilla incidence (FO3 slot strings are either populated or zero-length);
  defensive/robustness gap on malformed (hand-edited / broken-exporter) content. Cross-game
  (single chokepoint).
- **Related**: asset_provider F2 placeholder policy (`asset_provider.rs:569-579`). No existing issue.
- **Suggested Fix**: `if path.trim().is_empty()` at the single chokepoint; add a unit test
  asserting `intern_texture_path(pool, "  ") == None`.

#### FO3-2-02: Modern `NiParticleSystem` local TRS discarded at parse
- **Severity**: LOW
- **Dimension**: NIF parser / particle import
- **Location**: `crates/nif/src/blocks/particle.rs` (`parse_particle_system` binds `_av` then
  drops it); `ImportedParticleEmitter` (`crates/nif/src/import/walk/mod.rs:516-522`) has no
  transform field
- **Status**: Existing: #1333 (confirmed FO3-reachable — 422 particle systems). Cross-ref FNV D6-01.
- **Description**: `parse_particle_system` reads `NiAVObjectData` (incl. the block's local TRS)
  purely for byte-correct stream advancement, then discards it. Emitters authored with a
  non-identity offset on the particle block itself spawn at the host node origin.
- **Impact**: Block-local-offset emitters mis-place (flame above a torch cup, offset steam vent).

  is the minority that offset on the particle block. No crash, no parse loss.
- **Suggested Fix**: Capture `_av.transform` onto `NiParticleSystem`, surface it on
  `ImportedParticleEmitter`, compose onto host `GlobalTransform` at spawn.

#### FO3-3-01: FO3 SCOL records (54) silently dropped by the `is_fo4_plus` gate → 389 unresolvable REFRs
- **Severity**: HIGH
- **Dimension**: ESM Parser (import-pipeline)
- **Location**: `crates/plugin/src/esm/records/mod.rs:175-242` (gate + `b"SCOL" if is_fo4_plus`
  arm + skip arm); `parse_scol_group` at `crates/plugin/src/esm/cell/support.rs:351-400`; count
  at `crates/plugin/src/esm/index.rs` (`total()`/`categories()`)
- **Status**: NEW (FO3 instance). Same root cause as **FNV D4-01** (`AUDIT_FNV_2026-06-14.md`).
  Effectively a regression of pre-#1277-task3 behaviour (gate landed `dbabd880`, 2026-05-28,
  AFTER the 2026-05-26 baseline).
- **Description**: The `is_fo4_plus` gate (`mod.rs:185-188`) is `Fallout4 | Fallout76 | Starfield`.
  The `b"SCOL"` arm parses only when `is_fo4_plus`; for FO3 (`GameKind::Fallout3NV`) the fallback
  arm (`mod.rs:229-242`) logs "SCOL is FO4+ only — skipping" and calls `reader.skip_group`. The
  gate comment (`mod.rs:175-178`) claims SCOL "were all introduced in Fallout 4 and don't exist in
  vanilla Oblivion / FO3 / FNV / Skyrim masters." **This is false for SCOL** — SCOL (Static
  Collection) is a Gamebryo-Fallout record present in vanilla FO3. (PKIN/MOVS/MSWP genuinely are
  FO4+ and remain correctly gated — byte-scan confirms FO3 ships none of those three GRUPs.)
- **Evidence** (independent re-confirmation against real data, not inherited):
  - Raw byte-scan of `Fallout3.esm`: one top-level `b"SCOL"` GRUP with exactly **54 SCOL records**.
  - REFR `NAME` base-resolution scan (zlib-decompressing compressed records): **389 REFRs** reference
    one of the 54 SCOL bases → no `StaticObject` to resolve at cell-load → missing world geometry.
  - The drop arithmetic closes exactly: `parse_scol_group` inserts each SCOL into BOTH `statics`
    and `scols`; `total()` counts both → 54 × 2 = **108** = the observed 44,657 → 44,549 delta.
  - CI is blind: `FO3_TOTAL_FLOOR = 30,000` (`tests/parse_real_esm.rs:53`) sits ~14.5K below the
    drop, so `parse_rate_fo3_esm` stays green at 44,549.
- **Impact**: 54 SCOL base records dropped (combined-mesh static collections — rubble piles, rock
  clusters, modular building assemblies); 389 REFR placements across FO3 cells/worldspaces render
  as nothing. Predominantly visible world geometry. Not a parse crash.
- **Related**: FNV D4-01 (98 FNV SCOL / 1084 orphaned REFRs), #1277 Task 3 (`dbabd880`). No GitHub issue.
- **Suggested Fix**: SCOL is Gamebryo-Fallout, not FO4+. Widen the SCOL arm to
  `is_fo4_plus || matches!(game, GameKind::Fallout3NV)` (or an `is_fallout_or_later` predicate for
  the SCOL arm only), keeping PKIN/MOVS/MSWP FO4+-gated. Correct the `mod.rs:175-177` /
  `support.rs` doc comments. Re-cut the FO3 baseline to 44,657 and add a tight floor
  (`index.cells.scols.len() >= 50`) so a future re-gate can't pass CI.

#### FO3-4-01: Ragdoll multibodies leak on cell unload — `remove_ragdoll` has zero production callers
- **Severity**: HIGH
- **Dimension**: Cell Loading (import-pipeline)
- **Location**: leak site `byroredux/src/cell_loader/unload.rs:365-388`
  (`release_victim_rapier_bodies` walks only `RapierHandles`); unreached teardown
  `crates/physics/src/ragdoll.rs:319` (`remove_ragdoll`); component
  `crates/physics/src/components.rs:156-161`; attach `byroredux/src/ragdoll.rs:222-236`
- **Status**: NEW (introduced by PHYSAL `0a0bc3ce`, 2026-06-14). Related: FNV D1-01 / D7-01. NOT
  covered by CLOSED #1520.
- **Description**: `unload_cell` releases Rapier bodies only via `release_victim_rapier_bodies`,
  which walks the `RapierHandles` component (one `{body,collider}` per static-trimesh ghost, the
  #1520 path). An activated ragdoll stores its N bodies + multibody joints in a *separate*
  `Ragdoll` component, created via direct `pw.bodies.insert(...)` with no `RapierHandles` row. The
  unload victim walk never queries `Ragdoll`, so the actor entity is despawned (ECS row freed) while
  its ~10-20 rigid bodies, colliders, and multibody joints stay resident in the Rapier sets and
  broad-phase forever.
- **Evidence**: `grep -rn "remove_ragdoll"` → exactly two hits (the `fn` definition + a doc-comment
  cross-reference at `components.rs:154`) — **zero production call sites**. The teardown doc even
  says "Mirrors the #1520 no-leak discipline so a cell unload mid-ragdoll doesn't strand bodies" —
  written, never wired. The actor IS an unload victim (spawned in the `[first..last)` span, stamped
  with `CellRoot`).
- **FO3 reachability**: The ragdoll path is purely NIF-driven and not game-gated. `RagdollTemplate`
  attaches whenever `imported.ragdoll.is_some()` (`nif_loader.rs:1044`) — true for FO3 NPC/creature
  skeletons carrying `bhkRagdollConstraint` (PHYSAL "Oblivion/FO3/FNV/Skyrim converged"). The
  `ragdoll <id>` console command operates on any FO3 actor with a template, no `GameKind` check.
- **Impact**: Unbounded physics-set leak; compounds per cell-cycle (the exterior `PhysicsWorld` is
  never reset under radius streaming). Today's trigger is the console command (manual); becomes
  severe once AI death-ragdolls auto-fire (every Capital Wasteland corpse). Applies to both interior
  and exterior unloads (both route through `unload_cell`).
- **Suggested Fix**: In `unload_cell`, add a `release_victim_ragdolls` pass mirroring
  `release_victim_rapier_bodies` that walks victims for `Ragdoll` and calls `pw.remove_ragdoll`
  before despawn. Add a `body_count()`-baseline regression test alongside `rapier_release_tests.rs`.

#### FO3-4-02: Terrain LOD ring leaks entirely on worldspace-state drain
- **Severity**: HIGH
- **Dimension**: Cell Loading
- **Location**: leak site `byroredux/src/streaming_helpers.rs:24-40` (`drain_streaming_state` drains
  only `state.loaded`); resident state `byroredux/src/streaming.rs:177` (`lod_blocks`); reclaim
  `byroredux/src/cell_loader/terrain_lod.rs:280-283` (`unload_lod_block`)
- **Status**: NEW (gap left by CLOSED #1373, which fixed only the re-entry/teleport leak, never the
  drain path). Related: FNV D1-02.
- **Description**: `drain_streaming_state` (invoked on a worldspace transition — exterior→interior
  door-walk mid-session) drains only `state.loaded` (per-cell roots, via `unload_cell`). It never
  iterates `state.lod_blocks`. LOD blocks deliberately carry no `CellRoot` (`terrain_lod.rs:278-279`),
  so `unload_cell`'s `CellRootIndex` victim walk can't reach them either. A worldspace-state drain
  therefore frees zero LOD blocks — the entire resident distant-terrain ring leaks.
- **Evidence**: `drain_streaming_state` body = `state.loaded.drain()` + `unload_cell` loop +
  `shutdown`; no `lod_blocks` iteration. `unload_lod_block` callers are exclusively
  `terrain_lod.rs:209/226/237` (inside `stream_lod_blocks`). Drain callers are real transitions
  (`main.rs:1289,1339`, `debug_load.rs:220,304`).
- **FO3 reachability**: Terrain LOD is not game-gated; `stream_lod_blocks` runs for every exterior
  worldspace (FO3 Capital Wasteland streams the ring identically to FNV). Every FO3
  exterior→interior transition with a resident ring leaks it. (The `object_lod_blocks` half is
  Skyrim/FO4-gated and does NOT affect FO3 — only the terrain half does, which is enough for HIGH.)
- **Impact**: Up to ~625 blocks (`LOD_RADIUS_BLOCKS=12` Chebyshev square) leaked per transition,
  each leaking a global-geometry SSBO range (`MeshHandle`, never `drop_mesh`'d), a ground
  `TextureHandle` refcount (FO3-4-03), plus its ECS row. Unbounded across a session — the common
  "enter a building" case.
- **Suggested Fix**: In `drain_streaming_state`, before `shutdown`, iterate `state.lod_blocks.drain()`
  → `terrain_lod::unload_lod_block` (and `state.object_lod_blocks.drain()` → the object-LOD reclaim,
  a harmless no-op for FO3).

#### FO3-4-03: `unload_lod_block` never releases the base ground texture refcount
- **Severity**: MEDIUM
- **Dimension**: Cell Loading
- **Location**: reclaim `byroredux/src/cell_loader/terrain_lod.rs:280-283` (`drop_mesh` + `despawn`
  only); struct `byroredux/src/streaming.rs:64-69` (`LodBlock { entity, mesh_handle, hole_mask }` —
  no texture field); acquire `terrain_lod.rs:429-437`/`:461-463`
- **Status**: NEW (same bug class as the #1338/#1341/#1343 handle-leak fixes, missed in the LOD
  path). Related: FNV D1-03.
- **Description**: `spawn_lod_block` acquires the base ground texture via `resolve_texture`
  (refcount bump) and attaches it as a `TextureHandle`. `unload_lod_block` calls `drop_mesh` +
  `despawn`, but `LodBlock` stores no texture handle and `World::despawn` is pure-ECS, so the
  refcount is never dropped.
- **FO3 reachability**: Fires during NORMAL FO3 play (not just transitions) — every boundary-block
  retain/regen as the player walks the wasteland (`terrain_lod.rs:209/226/237`). Not game-gated.
- **Impact**: One ground-texture refcount leaked per boundary-block regen; the few distinct
  Capital-Wasteland landscape textures never reach refcount 0 → VkImages + bindless slots pinned for
  the session. Self-caps at the count of distinct landscape textures (hence MEDIUM).
- **Suggested Fix**: Add a `texture_handle: u32` field to `LodBlock`, populate at spawn, and have
  `unload_lod_block` call `ctx.texture_registry.drop_texture` (skip `0`/fallback).

#### FO3-5-01: Non-Ragdoll/non-LimitedHinge constraints silently dropped from the FO3 ragdoll chain
- **Severity**: MEDIUM (escalates toward HIGH if a vanilla FO3 creature authors one of these joints
  as the sole link between load-bearing bones — unverifiable here without on-disk skeleton analysis)
- **Dimension**: Collision / PHYSAL ragdoll
- **Location**: `crates/nif/src/import/collision.rs:319` (the `BhkConstraintData::Other => continue`
  arm in `extract_ragdoll`); decode at `crates/nif/src/blocks/collision/constraints.rs:379-386`
- **Status**: NEW. Related: FNV D7-02 (shared `collision.rs` code, confirmed FO3-reachable). No GitHub issue.
- **Description**: `extract_ragdoll` surfaces only constraints decoded as `Ragdoll` or
  `LimitedHinge`. `bhkHingeConstraint`, `bhkPrismaticConstraint`, `bhkStiffSpringConstraint`,
  `bhkBallAndSocketConstraint` all decode to `BhkConstraintData::Other` → hit
  `BhkConstraintData::Other => continue` and are dropped with **no log line** — unlike every other
  block-drop in this file, which `log::debug!`/`log::warn!`s.
- **FO3 reachability (real data)**: FO3 ships `bhkHingeConstraint` ×6, `bhkPrismaticConstraint`
  ×2+6(Anchorage)+1(PointLookout), `bhkStiffSpringConstraint` ×1(PointLookout) — all silently drop.
  The dominant FO3 humanoid joints (397 Ragdoll + 401 LimitedHinge + 60 Malleable) DO survive; the
  gap is the long-tail hinge/prismatic/stiff-spring joints (machinery/creature rigs).
- **Impact**: A dropped sole-link edge makes `orient_tree` (`crates/physics/src/ragdoll.rs:176`)
  produce a disconnected forest; `build_ragdoll` then builds each component as an independent
  free-floating multibody → detached part free-falls/spins with **no diagnostic**. The silent drop
  is the dangerous part.
- **Suggested Fix**: At minimum `log::warn!` when a constraint linking two ragdoll bodies is dropped
  as `Other` (name the type + both bone names). Better: post-`orient_tree`, warn when the result is a
  forest. Long-term: decode plain hinge → `LimitedHinge { min: -PI, max: PI }` and ball-and-socket →
  the canonical cone joint.

#### FO3-5-02: Dynamic trimesh ragdoll body has degenerate inertia (advisory)
- **Severity**: LOW
- **Dimension**: Collision / PHYSAL ragdoll
- **Location**: `crates/physics/src/convert.rs:156-176` + `crates/physics/src/ragdoll.rs:110-124`
- **Status**: NEW (note only). Related: FNV D7-03 (shared code).
- **Description**: If a ragdoll body's shape resolves to `CollisionShape::TriMesh`, `build_ragdoll`
  attaches a trimesh collider to a dynamic body; Rapier's auto inertia for a possibly-open trimesh
  is unreliable. The `.mass(part_mass)` override sets total mass but not a sane inertia tensor.
- **FO3 reachability**: FO3 has abundant `bhkPackedNiTriStripsShape` (5634) but overwhelmingly as
  static architecture, not ragdoll bones; vanilla FO3 ragdoll bones author capsules/boxes/convex
  hulls. Likely doesn't fire on stock content — advisory.
- **Suggested Fix**: For ragdoll bodies, substitute a convex-hull/bounding-capsule when the resolved
  shape is `TriMesh`, or set a principal-inertia tensor from the collider AABB.
