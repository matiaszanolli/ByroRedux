**Current state: NIF + BA2 at 100% (Recoverable) / 96.46% clean. ESM architecture records,
BGSM parser, SCOL/PKIN expansion, and PreCombined-Mesh Stage A spawn are all shipped and
correct.**

The FO4 content stack is substantially landed. All regression guards from prior audits hold.

current functionality. The main engineering gap is the **CSG companion reader**: vanilla FO4
`_oc.nif` precombines carry no vertex data inline (it lives in `Fallout4 - Geometry.csg`),

cleanly to per-REFR rendering via the honest `pc_spawned == 0` gate.

### Dimension 3 — FO4 Shader Flags + BGSM + Disney BSDF: **1 MEDIUM, 1 LOW**

#### D3-04-MEDIUM: SkinTint (type 5) / HairTint (type 6) missing explicit slot-map arms
- **Severity**: MEDIUM
- **Location**: `crates/nif/src/import/material/walker.rs:233` (default `_` arm of `match shader.shader_type`)
- **Status**: NEW
- **Description**: Types 5 (SkinTint) and 6 (HairTint) fall into the `_` default arm which routes slot 4 → `env_map`. Per nif.xml, SkinTint/HairTint declare no TS texture slots 4 or 5. Vanilla content leaves those slots empty (`.filter(|s| !s.is_empty())` guard saves it), but a modded or future NIF with a non-empty slot 4 on a SkinTint mesh would erroneously import as an env map.
- **Impact**: Vanilla FO4 is unaffected (empty slot guard). Modded content or a poorly-exported NIF could get an env map when it shouldn't.
- **Suggested Fix**: Add explicit match arms for types 5 and 6 that skip slots 4/5.

#### D3-05-LOW: `pack_bgsm_material_flags` doc comment stale — flags ARE consumed by shader
- **Severity**: LOW
- **Location**: `byroredux/src/cell_loader.rs:171-173`
- **Status**: NEW
- **Description**: Comment says "The bits are populated today but **not yet read by any shader**." Five flags are actively consumed by `triangle.frag`: `MAT_FLAG_PBR_BSDF` (lines 2610/2861), `MAT_FLAG_TRANSLUCENCY` (2889), `MAT_FLAG_MODEL_SPACE_NORMALS` (1218), `THICK_OBJECT` (2909), `MIX_ALBEDO` (2922).
- **Suggested Fix**: Update doc comment to reflect Phase 2b landed and all five flags are consumed.

All other D3 guards hold (u32 pair at BSVER 130, CRC arrays at 132, stopcond only for FO76+, material_path propagation, roughness-once, FO4_ENV_SCALE fully renamed, BSShaderPropertyBaseOnly wired).

**BGSM PBR note (informational, not a bug):** Vanilla FO4 BGSMs essentially never set `is_pbr=true` (0 of 793 sampled Materials BA2), so `MAT_FLAG_PBR_BSDF` is correctly plumbed but won't fire for vanilla. The spec-glossiness F0 derivation path handles vanilla FO4 correctly.

#### D7-03-MEDIUM: Vanilla FO4 BGSM content does not route through Disney diffuse lobe
- **Severity**: MEDIUM
- **Location**: `byroredux/src/asset_provider.rs:1183-1186`; `crates/renderer/shaders/triangle.frag:2610`
- **Status**: NEW
- **Description**: `MAT_FLAG_PBR_BSDF` is only set when `bgsm.pbr == true`. Vanilla FO4 BGSMs essentially never set `pbr=true` (0 of 793 sampled). So FO4 BGSM-authored content gets correct spec-glossiness → metallic-roughness conversion and GGX specular, but takes the Lambert diffuse path instead of the Disney path. Bethesda's FO4 content was authored for a modified Blinn-Phong pipeline; the Disney diffuse lobe would be a quality upgrade, but the existing Lambert path is not wrong — it's what the content was authored for.
- **Suggested Fix** (future): Set `mesh.is_pbr = true` when `mesh.from_bgsm == true` after RenderDoc visual validation comparing the two paths on FO4 content.

#### D8-07-MEDIUM: `BgsmFile.greyscale_texture` (palette LUT) silently dropped
- **Severity**: MEDIUM
- **Location**: `crates/bgsm/src/bgsm.rs` (parse) → `byroredux/src/asset_provider.rs::merge_bgsm_into_mesh` (no destination field)
- **Status**: NEW
- **Description**: `greyscale_texture` (palette-LUT slot 4, v≤2) and `BgemFile.grayscale_texture` are parsed but not forwarded — `ImportedMesh` has no destination field for them. FO4 lit-material palette tinting (NPC/creature color variants authored via this slot) will be missing. `grayscale_to_palette_scale` is also orphaned.
- **Impact**: Creature and NPC color-variant materials show the base texture rather than the palette-tinted variant. Similar to how the FNV greyscale-LUT handle was missing from the unload walk (#1341), but here the data is dropped at import time.
- **Suggested Fix**: Add a `greyscale_palette_texture: Option<String>` slot to `ImportedMesh`, forward from `merge_bgsm_into_mesh`, propagate through `translate_material` → draw pipeline (alongside the existing `GreyscaleLutHandle` pattern from #890/#1341).
