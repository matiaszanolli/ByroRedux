## Dimension Findings (by severity)

### MEDIUM

#### FO4-D5-01 — FO4 shader-flag-only alpha test is inert (threshold defaults to 0.0) · NEW
- **Location**: `crates/nif/src/import/material/walker.rs:341` (sets `info.alpha_test = true`); consumed by `crates/renderer/shaders/triangle.frag:217`
- **Status**: NEW — incomplete fix of the **closed** #1592 (not a dedup of any open issue)
- **Mechanism**: #1592 wired FO4 `F4SF2` bit 25 (`Alpha_Test`) to OR `info.alpha_test = true` for inline FO4 NIFs that signal a cutout via the shader flag alone (no `NiAlphaProperty`). But that branch sets only the boolean — it leaves `info.alpha_threshold` at the `MaterialInfo::default()` value `0.0` (`import/material/mod.rs:952`). The fragment shader gates the entire alpha-test discard on `if (aThresh > 0.0)` (`triangle.frag:217`), so a threshold of 0.0 disables the discard: `alpha >= 0.0` is always true. `apply_alpha_flags` (the only writer of a usable threshold) runs *only* when a `NiAlphaProperty` is present — precisely the case this flag branch exists to cover the absence of. The walker comment ("the GREATEREQUAL/default threshold stands in") is factually wrong.
- **Impact**: An FO4 inline NIF requesting a cutout purely via `F4SF2` bit 25 (foliage/fence/grate/chain-link with no sibling `NiAlphaProperty`) renders as a solid opaque quad — silhouette lost. Blast radius is **limited**: vanilla FO4 cutout meshes almost universally ship a `NiAlphaProperty` (which sets the threshold correctly), so this bites inline/modded content only. Not an active regression vs pre-#1592 (that state also produced no cutout) — an incomplete fix: the flag is consumed but has no effect. The test `fo4_alpha_test_flag_sets_field` asserts only the boolean, never that a usable threshold reaches the shader, so the gap is untested.
- **Suggested Fix**: When setting `info.alpha_test = true` from the FO4 flag with no `NiAlphaProperty`, seed a usable default threshold (`info.alpha_threshold = 128.0/255.0`, Bethesda's conventional cutout) and keep `alpha_test_func = 6`. Add a threshold assertion to `fo4_alpha_test_flag_sets_field`.

### LOW

#### FO4-D1-01 — CSG middle-chunk length not pinned to 65,536; a short interior chunk silently mis-addresses PSG space · NEW
- **Location**: `crates/bsa/src/csg.rs:221-250` (`chunk_bytes`), consumed by `read_psg`
- **Mechanism**: The PSG addressing model assumes chunk `i` occupies uncompressed bytes `[i*65536, (i+1)*65536)` — `read_psg` derives `idx = pos / CHUNK_SIZE`, `local = pos % CHUNK_SIZE` from that invariant. `chunk_bytes` rejects only *over-size* inflated chunks (`raw.len() > CSG_CHUNK_SIZE`); it never asserts a **non-last** chunk equals exactly `CSG_CHUNK_SIZE`. A corrupt/truncated CSG whose interior chunk inflates short would make every subsequent PSG offset point into the wrong chunk-local position.
- **Impact**: Malformed-input robustness only. On all validated real data every non-final chunk is exactly 65,536 (zlib 64 KiB blocks), so this never fires on vanilla/DLC content. When it would fire, mis-addressed bytes decode as garbage `u16` indices and are **rejected downstream** by the #1533 decode-time `index >= num_verts` guard (object skipped → REFR fallback) — fails closed, not corrupt render. Hence LOW.
- **Suggested Fix**: In `chunk_bytes`, when `idx < self.chunks.len() - 1` assert `raw.len() == CSG_CHUNK_SIZE`, else return `InvalidData` — turns a silent wrong-data path into an explicit reject, matching the existing over-size guard.

#### FO4-D2-01 — #1476 metalness-from-saturation formula is inlined and un-unit-tested · NEW
- **Location**: `byroredux/src/asset_provider/material.rs:728-748` (inside `merge_bgsm_into_mesh`)
- **Mechanism**: The non-pbr saturation metalness derivation (`metalness = (max-min)/max` of `specular_color`) and the pbr-branch luminance path are computed inline, not extracted into a pure helper. The two *adjacent* helpers this dimension protects — `conductor_diffuse_tint` (#1591) and `bgsm_blend_to_gamebryo` (#1823) — are extracted with dedicated regression tests. The metalness formula — **the exact code that regressed once** (luminance→chrome concrete, fixed in `08ed03be`) — has no equivalent guard. A future edit reintroducing `spec_lum` into the `else` (non-pbr) branch would compile, pass the whole suite, and silently re-chrome vanilla concrete.
- **Impact**: Defense-in-depth gap only; current code is correct (`grep 0.2126` = one hit, correctly inside `if leaf.pbr`). The regression this pin guards against is invisible to `cargo test`.
- **Suggested Fix**: Extract `pub(crate) fn bgsm_metalness(spec:[f32;3], mult:f32, pbr:bool) -> f32` and assert `spec=[1,1,1] pbr=false → 0.0` (concrete dielectric) and `spec=[1,0.85,0.70] pbr=false → > 0.1` (tinted conductor). Mirrors the existing `conductor_diffuse_tint` / `bgsm_blend_to_gamebryo` pattern.

#### FO4-D3-01 — Stale internal line-number references in `ba2.rs` comments (doc rot) · NEW
- **Location**: `crates/bsa/src/ba2.rs:541`, `:642`, `:643`
- **Mechanism**: Three comments cite sibling code by hard-coded line number; all three drifted after the #1825 `chunk_hdr_len != 24` warn block was inserted (line 541 says "~line 496" → actually 604; line 642 says "512-519" → 591-598; line 643 says "490-495" → 546-551). Cross-file refs (`archive/open.rs:40-48`) are correct — only same-file numeric refs drift.
- **Impact**: None at runtime; a maintainer following the comment lands on unrelated code.
- **Suggested Fix**: Replace hard line numbers with symbolic references ("the `num_mips == 0` warn below", etc.) that don't rot. Sibling of the recurring TD7-* stale-path class.
