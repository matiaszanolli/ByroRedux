#### NIF-D2-01: #1277 Task 5 silently reverted the #982 fix — variant-helper gates reintroduced at both `NiTriShape` sites, contradicting comments left in place
- **Severity**: MEDIUM
- **Dimension**: Version Gating
- **Game Affected**: no retail title (all seven shipping games classify onto the aligned side of the gate); divergence hits unclassified/hybrid `(user_version, bsver)` tuples only — e.g. `(uv=11, bsver 35..=82)` → routed `Unknown` → helper returns false while nif.xml's `#BS_GT_FO3#` (`bsver > 34`) authors the fields (8-byte under-read of shader/alpha refs, 4-byte under-read of material CRC)
- **Location**: `crates/nif/src/blocks/tri_shape/ni_tri_shape.rs:135` (`has_shader_alpha_refs()`), `crates/nif/src/blocks/tri_shape/ni_tri_shape.rs:341` (`has_material_crc()`)
- **Status**: **Regression of #982** (filed as NIF-D2-NEW-07 in the 2026-05-12 audit)
- **Description**: Commit `01957517` (Fix #982) deliberately replaced `variant().has_shader_alpha_refs()` / `variant().has_material_crc()` at these two sites with raw `stream.bsver() > 34`, adding comments explaining the BSVER 35..=82 Unknown-corner false-negative that the variant helper carries. Sixteen days later, commit `2bd447d5` (a refactor landed under #1277 Task 5, described as "six existing helpers now called from production") migrated both sites **back** to the variant helpers — while leaving the raw-bsver-mandating comments from the #982 fix completely unmodified, so the code now visibly contradicts its own adjacent comment.
- **Evidence**:
  ```rust
  // ni_tri_shape.rs:128-135 (comment unchanged since #982, code flipped by #1277 Task 5)
  // Query the file's actual bsver rather than the routed game variant —
  // `variant().has_shader_alpha_refs()` returns false for BSVER in 35..=82
  // … See NIF-D2-NEW-07
  if stream.variant().has_shader_alpha_refs() { ... }
  ```
  `git show 01957517` introduces `if stream.bsver() > 34` + the explanatory comments at both sites. `git show 2bd447d5` flips both back to the helper call, without touching the comments. nif.xml gates both fields purely on BSVER (`#BS_GT_FO3#`), never on `user_version` — the raw-bsver read is the spec-correct one.
- **Impact**: Functional divergence is confined to non-retail hybrid headers (the original #982 rated this "forward-compat hygiene", not a shipping-game bug). Rated MEDIUM rather than LOW because (a) it silently undoes a shipped, deliberate audit fix rather than being a fresh gap, (b) the surviving contradictory comments actively mislead the next maintainer into thinking the code still matches the fix, and (c) the 2026-06-23 audit certified this exact dimension clean over precisely this reverted state, i.e. the regression was invisible to the last full pass.
- **Related**: #982 / NIF-D2-NEW-07 (original fix), #1277 Task 5 / `2bd447d5` (the reverting commit), #160 / #1331 (the raw-bsver doctrine used at the sibling `base.rs`/`node.rs` sites, which is what #982 aligned these two sites to)
- **Suggested Fix**: Restore `stream.bsver() > crate::version::bsver::FO3_FNV` at both sites (matching the still-present comments). Add a regression test constructing a synthetic `(user_version=11, bsver=50)` header so a future "migrate raw bsver to helpers" sweep trips a red test instead of silently reverting again.

#### NIF-D5-02: `BSEffectShaderProperty` has no Starfield tail-capture path — 32-byte under-read on every full-body Starfield instance
- **Severity**: MEDIUM (the outer `block_sizes` reconciliation in the dispatch loop covers the drift — Starfield always ships a `block_sizes` table, so this is silent data loss, not a truncation/crash risk)
- **Dimension**: Collision/Shader Parsing (stream position)
- **Game Affected**: Starfield (`bsver >= STARFIELD = 172`)
- **Location**: `crates/nif/src/blocks/shader.rs:1624` (`BSEffectShaderProperty::parse` signature has no `block_size` parameter); `crates/nif/src/blocks/mod.rs:576-579` (dispatch call site asymmetry — `BSLightingShaderProperty::parse_with_size(stream, block_size)` immediately above `BSEffectShaderProperty::parse(stream)`)
- **Status**: NEW (corroborates a fresh corpus signal; not covered by the CLOSED `#1721`, which fixed only the Starfield stub-discriminator gate on this same block type, not a tail capture — verified `#1721`'s regression test `parse_bs_effect_starfield_hashpath_name_stubs` only exercises the 12-byte stub path, never a full-body Starfield parse)
- **Description**: A fresh `nif_stats --drift-histogram` sweep over Starfield's `Meshes01.ba2` (this audit's corpus run) shows exactly 665 `BSEffectShaderProperty` blocks under-reading by a uniform **+32 bytes** (`declared block_size − consumed = +32` for all 665 instances, no variance) — the identical shape of problem `BSLightingShaderProperty` had before `#1606` added its `starfield_tail: Vec<u8>` capture. Unlike `BSLightingShaderProperty`, `BSEffectShaderProperty::parse` takes no `block_size` parameter at all — the dispatch site calls it via plain `parse(stream)`, so there is no plumbing (not even unused) through which a future tail-capture fix could reach `block_size` without first changing the function signature and its one call site.
- **Evidence**:
  ```rust
  // blocks/mod.rs — the asymmetry
  "BSLightingShaderProperty" => Ok(Box::new(BSLightingShaderProperty::parse_with_size(
      stream, block_size,
  )?)),
  "BSEffectShaderProperty" => Ok(Box::new(BSEffectShaderProperty::parse(stream)?)),
  ```
  `shader.rs:747` (`pub starfield_tail: Vec<u8>`) + `:756-774` (`read_starfield_tail`, consuming exactly `block_size − consumed`) have no counterpart anywhere in `BSEffectShaderProperty`'s struct (`:1523-1584`) or `parse` body (`:1624-1797`, which ends at the `luminance` field with no tail read). A sweep of every other shader-property type in `blocks/mod.rs` for the same "Starfield-reachable but no tail capture" pattern found no other instance: the FO3/FNV/Skyrim-only shader types never reach Starfield content, and the sole other `_with_size`-style consumer (`BhkRagdollTemplateData::parse(stream, block_size)`) already uses its `block_size` correctly to skip (not silently drop via outer recovery) its untyped constraint-array tail.
- **Impact**: 32 bytes of authored Starfield effect-shader data per full-body (non-stub) `BSEffectShaderProperty` block — 665 instances in `Meshes01.ba2` alone — are discarded by the generic `block_sizes` stream-recovery instead of being preserved as an opaque capture the way the sibling type's tail is. Not a crash or cascade (Starfield's `block_sizes` table keeps the stream self-consistent across the drift), but a silent-data-loss case symmetric to the pre-#1606 `BSLightingShaderProperty` gap.
- **Related**: `#1606` (the `BSLightingShaderProperty` precedent this should mirror), `#1721`/`#1510` (the stub-discriminator fix already ported to this type, but without a tail capture), prior audit's `NIF-D5-01` (fixed the stub gate on this same type, closed as `#1721`)
- **Suggested Fix**: Thread `block_size: Option<u32>` (or a required `u32`, matching `BSLightingShaderProperty`'s convention) through `BSEffectShaderProperty::parse` — add a `parse_with_size(stream, block_size)` entry point, capture `block_start` before parsing, and reuse (or extract to a shared helper) `read_starfield_tail` after the body parse. Update the `blocks/mod.rs:579` call site to pass `block_size`, matching the line immediately above it. Byte-audit the actual 32-byte tail content against a real Starfield `Meshes01.ba2` sample before committing to a fixed-width capture, per the project's no-guessing policy — confirm it's fixed-width and byte-identical across instances the way the BSLSP 38-byte tail was.

#### NIF-D1-01: Pre-4.1 NIF `bool` fields read as 1 byte where the wire format is 32-bit — cascading drift on every real v ≤ 4.0.0.2 (Morrowind-era) NIF
- **Severity**: LOW *(graded per the project's #982 out-of-tier precedent — Morrowind/NetImmerse v≤4.0.0.2 is outside the Oblivion→Starfield target matrix; inside a support tier this would be CRITICAL-class, since the format ships no `block_sizes` table and the drift has no recovery anchor)*
- **Dimension**: Stream Position
- **Game Affected**: `NifVariant::Morrowind` and NetImmerse content at NIF v ≤ 4.0.0.2 (bsver = 0). No content in the target matrix sits in this band — Oblivion's oldest live band is v10.0.1.0; Oblivion's one pre-Gamebryo file (`marker_radius.nif`, closed `#698`) is corrupt-by-design and truncates regardless.
- **Location**: `crates/nif/src/blocks/base.rs:279` (`has_bounding_volume` via `read_u8`); `crates/nif/src/blocks/tri_shape/ni_tri_shape.rs:326,349,379` (`has_vertices`/`has_normals`/`has_vertex_colors` via `read_byte_bool`); `ni_tri_shape.rs:421-422` (`has_uv` — this branch is *exclusively* v ≤ 4.0.0.2, so it is wrong on 100% of the band); `crates/nif/src/blocks/texture.rs:913` (`enable_plane` via `read_u8`)
- **Status**: NEW (no matching open issue; the closed `#982` bundle covered a different field's drift; `#210`'s closure claim of "full Morrowind v4.0.0.2 support" did not cover bool widths)
- **Description**: nif.xml's `bool` basic type is documented as 32-bit up to and including v4.0.0.2, 8-bit from v4.1.0.1 on. `NifStream::read_bool` (`stream.rs:203-211`) implements exactly this switch, but the listed sites bypass it with fixed 1-byte reads, under-reading 3 bytes per field on a real v4.0.0.2 file.
- **Evidence**: OpenMW's canonical Morrowind reimplementation (`components/nif/nifstream.cpp:170-177`) reads `bool` as `int32_t` when `version < 4.1.0.0`, exercised by its `NiAVObject::read` and `NiGeometryData::read`. The project's own synthetic Morrowind fixture (`tests/synthetic_fixtures.rs:434`) writes `has_bounding_volume` as a single byte — self-consistent with the parser bug, meaning the test suite cannot catch this without a real v4.0.0.2 sample.
- **Impact**: Latent today (out-of-tier), but every entry point the band needs (v4.0.0.2 header parse, inline RTTI names, the v4 extra-data path) already exists, so the parser *claims* the band while every real file in it would cascade at the first `NiAVObject` bool with no recovery anchor (no `block_sizes` at this version).
- **Related**: `#210` (closed, Morrowind support), `#982` (closed, out-of-tier precedent), `read_bool` version doctrine (`stream.rs`)
- **Suggested Fix**: Replace the fixed-width reads with `stream.read_bool()` at the listed sites; fix the synthetic fixture to write 4-byte bools at v4.0.0.2; correct the `read_byte_bool` doc comment ("1 byte in all versions **≥ 4.1.0.1**"); validate against one real Morrowind NIF before re-claiming the band.

- **Severity**: LOW *(same out-of-tier grading as NIF-D1-01; same cascade class)*
- **Dimension**: Stream Position
- **Game Affected**: NetImmerse v ≤ 4.0.0.2 (`NiLight` / `NiTextureEffect` — the latter ubiquitous in Morrowind armor/weapon env-map meshes)

#### NIF-D1-03: `read_starfield_tail`'s opaque capture makes the drift histogram blind to future real drift on Starfield `BSLightingShaderProperty`
- **Severity**: LOW (observability gap, not a current defect)
- **Dimension**: Stream Position
- **Game Affected**: Starfield (`bsver >= 172`) only
- **Location**: `crates/nif/src/blocks/shader.rs:756-774` (`read_starfield_tail`), wired at `:870`
- **Status**: NEW (side effect of the closed `#1606` fix operating exactly as designed — not a regression)
- **Description**: By design, `read_starfield_tail` consumes whatever remains up to `block_size` into an opaque `Vec<u8>`, so `consumed == block_size` always holds for Starfield `BSLightingShaderProperty` and the outer drift detector can never fire for this type again. A future edit that mis-reads a mid-block field would have its error silently folded into the tail capture with zero telemetry signal.
- **Evidence**: The dispatch loop only histograms when `consumed != size`; `read_starfield_tail`'s contract guarantees equality by construction.
- **Impact**: None today (guard tests + corpus were green at `#1606` closure); reduces future regression detectability for this one type/game pair.
- **Related**: `#1606`, `#1510`
- **Suggested Fix**: Emit a debug-level counter (or a dedicated histogram keyed on tail length) when `starfield_tail.len()` deviates from the corpus-common value, so tail-length drift stays observable.
