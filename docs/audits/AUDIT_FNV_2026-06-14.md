# Fallout: New Vegas Compatibility Audit — 2026-06-14

**Scope**: All 8 dimensions. FNV is the **reference title** — the most-validated end-to-end
path in the engine and the reference realization for the canonical translation layers (NIFAL
material/physics, PHYSAL ragdoll). This audit is a regression guard first, a coverage check second.

**Orchestration**: 8 dimension agents (cell loading, NIFAL translation, RT lighting, ESM records,
NIF parser, animation/skinning/particles, PHYSAL ragdoll, real-data validation). Dimension 8 ran
the **offline** NIF-parse and ESM-record validators against real on-disk FNV game data; the GPU
bench-of-record could not be exercised in the headless audit environment.

**HEAD audited**: `435e265d`. **Delta since prior FNV audit (2026-05-30, `04cd20a1`)**: 234 commits,
including the PHYSAL ragdoll feature (just shipped 2026-06-14) and a large NIF/renderer hardening
sweep.

## Executive Summary

**No regression against a validated roadmap number.** The two authoritative offline gates were run
live against real FNV data and both match the ROADMAP baseline **exactly**: NIF parse rate 100%
(14881/14881 clean, 0 truncated, 97 distinct block types) and every FNV ESM record-count baseline
on the nose (items 2643, containers 2478, LVLI 2738, LVLN 365, NPCs 3816, races 22, classes 74,
factions 682, globals 218, settings 648). All four prior-audit HIGH findings (texture/lighting
leaks on cell unload) are **FIXED** (#1340, #1341, #1199-followups, #1520), as are both prior MEDIUM
ESM findings (WEAP DNAM mislabel #1342, XXXX extended-size override). The prior-audit D6-01
(BSPSysSimpleColorModifier colors dropped) is also FIXED (#1345/#1513).

**The real findings are latent runtime-lifecycle and dispatch-gate bugs the validated
startup-load numbers don't exercise** — concentrated in (a) the brand-new PHYSAL ragdoll lifecycle
and (b) a record-type gate that mistakes a Gamebryo-era record for an FO4-only one:

- **5 HIGH (all NEW):**
  - **D4-01 / D8-01 — FNV SCOL records silently dropped.** The `is_fo4_plus` top-level gate skips
    all 98 SCOL (static-collection) records in FalloutNV.esm on the false premise that SCOL is
    "FO4+ only." SCOL is a pre-FO4 Gamebryo-Fallout record (FO3 = 54, FNV = 98). 1084 REFRs across
    FalloutNV.esm reference dropped SCOL bases, and 137 cached `meshes\scol\*.nif` ship in the BSA —
    all rendered as missing exterior geometry. Corroborated by two independent agents via byte-scan.
  - **D1-01 / D7-01 — PHYSAL ragdoll multibody leaks on cell unload.** `PhysicsWorld::remove_ragdoll`
    exists (and is documented "mirrors the #1520 no-leak discipline") but has **zero production
    callers**; the unload walk only releases `RapierHandles`, never the `Ragdoll` component. Every
    activated ragdoll leaks ~10-20 rigid bodies + colliders + multibody joints into the persistent
    Rapier sets per cell-unload — the same slow-corruption tail the project engineers against.
  - **D1-02 — exterior terrain-LOD ring leaks entirely on worldspace-state drain.**
    `drain_streaming_state` drains only `state.loaded`, never `lod_blocks` / `object_lod_blocks`;
    LOD blocks carry no `CellRoot`, so the `unload_cell` victim walk can't reach them either. An
    exterior→interior door-walk leaks the whole resident LOD ring (mesh SSBO ranges + ground
    texture refcounts + ECS rows).

- **2 HIGH→MEDIUM borderline (cell-cycle leak class):** see D1-03 below (LOD ground-texture leak;
  fires on normal play, not just on drain). Filed MEDIUM because it self-caps at the count of
  distinct landscape textures.

- **4 MEDIUM:** D1-03 (LOD ground-texture refcount leak), D7-02 (non-Ragdoll/non-LimitedHinge
  constraints silently dropped from the ragdoll chain with no telemetry), D6-01 (`Existing: #1333`,
  NiParticleSystem local transform discarded — confirmed still real & FNV-reachable), D6-02
  (`Existing: #1445`, planar_angle dropped — confirmed but inert, no downstream consumer).

- **2 LOW:** D7-03 (dynamic trimesh ragdoll body degenerate inertia — advisory), D8-02 (CWD
  archive-resolution gotcha verified unchanged — documentation accurate, no regression).

**Severity tally (NEW only):** 0 CRITICAL · 3 HIGH · 2 MEDIUM · 1 LOW. Plus 2 `Existing` (#1333,
#1445) and 1 advisory/verification LOW (D8-02).

> Per the brief, a regression against a validated number would be CRITICAL. **There are none.** The
> HIGH findings are newly-discovered latent issues: a dispatch-gate misclassification (SCOL) and
> two resource leaks in the just-shipped ragdoll + the LOD-drain lifecycle. The validated

### Dimension 1 — Cell Loading End-to-End: 2 HIGH, 1 MEDIUM (all NEW)

> The cell-load *machinery* is structurally sound (process-lifetime NIF Arc cache, deduped clip
> registry, `[first..last)` ID-span unload key, correct M38 water spawn + `submersion_system`, and

#### D1-01: Ragdoll multibodies leak on cell unload (PHYSAL, new code path)
- **Severity**: HIGH
- **Dimension**: Cell Loading
- **Location**: `byroredux/src/cell_loader/unload.rs:365-388` (release walk — only `RapierHandles`) vs `crates/physics/src/ragdoll.rs:319` (`remove_ragdoll`, zero callers); component at `crates/physics/src/components.rs:154-165`; attach at `byroredux/src/ragdoll.rs:229-236`
- **Status**: NEW (introduced by `0a0bc3ce` / PHYSAL, 2026-06-14; cross-confirmed as D7-01)
- **Description**: `unload_cell` releases Rapier bodies only via `release_victim_rapier_bodies`, which walks the **`RapierHandles`** component (one `{body, collider}` per entity). An activated ragdoll stores its N bodies + joints in a *separate* `Ragdoll` component (`bodies: Vec<(EntityId, RigidBodyHandle)>`, `joints: Vec<MultibodyJointHandle>`), created via direct `pw.bodies.insert(...)` with no `RapierHandles` row. The unload walk never queries `Ragdoll`, so on cell unload the actor entity is despawned while its ~10-20 rigid bodies, colliders, and multibody joints stay resident in `RigidBodySet` / `ColliderSet` / `MultibodyJointSet` and the broad-phase forever.
- **Evidence**: `grep -rn "remove_ragdoll"` returns exactly two hits — the `fn` definition and a doc-comment cross-reference — **no production call site**. The teardown was written ("Mirrors the #1520 no-leak discipline so a cell unload mid-ragdoll doesn't strand bodies") but never wired. `release_victim_rapier_bodies` (`unload.rs:365`) calls only `pw.remove_body(h.body)` over `RapierHandles`.
- **Impact**: Unbounded physics-set leak. Every ragdoll the player triggers (or, once AI death-ragdolls auto-fire, every NPC corpse) leaks its full multibody on the next cell-unload. Compounds per cell-cycle (the exterior `PhysicsWorld` is never reset under radius streaming), degrading solver step cost + broad-phase memory across a session. Worst on the reference title under combat + streaming.
- **Suggested Fix**: In `unload_cell` (or a new `release_victim_ragdolls` mirroring `release_victim_rapier_bodies`), walk victims for `Ragdoll` and call `pw.remove_ragdoll(&ragdoll)` before despawn. Add a regression test alongside `rapier_release_tests.rs` asserting `body_count()` returns to baseline after unloading a ragdoll actor.

#### D1-02: Terrain LOD ring leaks entirely on worldspace-state drain
- **Severity**: HIGH
- **Dimension**: Cell Loading
- **Location**: `byroredux/src/streaming_helpers.rs:19-40` (`drain_streaming_state`), `byroredux/src/streaming.rs:177,183` (`lod_blocks` / `object_lod_blocks`); reclaim fn `byroredux/src/cell_loader/terrain_lod.rs:280-283`
- **Status**: NEW
- **Description**: `drain_streaming_state` drains only `state.loaded` (per-cell roots, via `unload_cell`). It never iterates `state.lod_blocks` or `state.object_lod_blocks`. LOD blocks deliberately carry **no `CellRoot`** (`terrain_lod.rs:277-279`), so `unload_cell`'s `CellRootIndex` victim walk can't reach them either. The only path that ever calls `unload_lod_block` is `stream_lod_blocks`'s own steady-state retain/regen loop. Therefore a worldspace-state drain (exterior→interior door-walk mid-session) frees zero LOD blocks.
- **Evidence**: `unload_lod_block` callers are exclusively `terrain_lod.rs:209/226/237`; `unload_object_lod_block` only `object_lod.rs:136`. The `drain_streaming_state` body has no LOD iteration (`cells: Vec<_> = state.loaded.drain().collect()` only). The drain is invoked on real mid-session transitions (`debug_load.rs`, `main.rs` door-walk). Terrain LOD is not game-gated — it runs for FNV.
- **Impact**: Every exterior→interior transition while the wasteland LOD ring is resident leaks the whole ring (up to ~hundreds of blocks at `LOD_RADIUS_BLOCKS`): each block leaks a global-geometry SSBO range (`MeshHandle`, never `drop_mesh`'d), a base ground `TextureHandle` refcount (D1-03), plus its ECS row. The common "walk into a Mojave building" case; unbounded across a session of transitions.
- **Suggested Fix**: In `drain_streaming_state`, before joining the worker, iterate `state.lod_blocks.drain()` → `terrain_lod::unload_lod_block` and `state.object_lod_blocks.drain()` → `object_lod::unload_object_lod_block`.

#### D1-03: `unload_lod_block` never releases the base ground texture refcount
- **Severity**: MEDIUM
- **Dimension**: Cell Loading
- **Location**: `byroredux/src/cell_loader/terrain_lod.rs:280-283` (reclaim); `LodBlock` struct `byroredux/src/streaming.rs:65-69`; acquire at `terrain_lod.rs:431-435,462`
- **Status**: NEW
- **Description**: `unload_lod_block` calls `ctx.mesh_registry.drop_mesh(block.mesh_handle)` + `world.despawn(block.entity)`, but `LodBlock` stores only `{entity, mesh_handle, hole_mask}` — it doesn't retain the texture handle. The base ground texture is acquired via `resolve_texture` (refcount bump) at `terrain_lod.rs:431/434/435` and attached as a `TextureHandle` (line 462), but `World::despawn` has no GPU side effects (`crates/core/src/ecs/world.rs:121` is pure-ECS), so the refcount is never dropped.
- **Evidence**: `LodBlock` field list confirmed (no texture field); `unload_lod_block` body has no `drop_texture`. Same bug class as the #1338/#1341 handle-leak fixes, missed in the LOD path. Unlike D1-02, this fires during **normal play** (boundary-block retain/regen as the player walks, `terrain_lod.rs:209/227/237`).
- **Impact**: One ground-texture refcount leaked per boundary-block regen; the few distinct landscape textures never reach refcount 0 → VkImages + bindless slots pinned for the session. Self-caps at the count of distinct textures (hence MEDIUM, not HIGH).
- **Suggested Fix**: Add a `texture_handle: u32` field to `LodBlock`, populate at spawn, and have `unload_lod_block` call `ctx.texture_registry.drop_texture(&ctx.device, h)` (skip `0`/fallback), as `collect_victim_gpu_handles` does. (Note: `object_lod.rs:309-320` has the identical omission, but object LOD is Skyrim/FO4-gated — out of FNV scope.)

### Dimension 4 — ESM Record Parser: 1 HIGH (NEW)

#### D4-01: FNV SCOL records (98) silently dropped by the `is_fo4_plus` gate → 1084 unresolvable REFRs
- **Severity**: HIGH
- **Dimension**: ESM Parser
- **Location**: `crates/plugin/src/esm/records/mod.rs:175-242` (gate + `b"SCOL" if is_fo4_plus` arm + skip arm); mislabeled doc `crates/plugin/src/esm/records/scol.rs`; consumer `byroredux/src/cell_loader/refr.rs` (`expand_scol_placements`)
- **Status**: NEW (effectively a regression of pre-#1277-task3 behaviour: the gate comment at `mod.rs:178` notes "Pre-#1277-task3 these GRUPs were parsed unconditionally")
- **Description**: The `is_fo4_plus` gate (`mod.rs:185-188`) is `Fallout4 | Fallout76 | Starfield`. The `b"SCOL"` arm parses only when `is_fo4_plus`; for FNV (`GameKind::Fallout3NV`) the fallback arm logs "SCOL is FO4+ only — skipping" and calls `reader.skip_group`. The gate comment claims SCOL "were all introduced in Fallout 4 and don't exist in vanilla Oblivion / FO3 / FNV / Skyrim masters." **This is false for SCOL.** PKIN/MOVS/MSWP genuinely are FO4+ (confirmed absent from FalloutNV.esm by byte-scan); SCOL is a Gamebryo-Fallout record (Oblivion 0, FO3 54, FNV 98, Skyrim 0).
- **Evidence** (real data): FalloutNV.esm carries 1 SCOL group with **98 SCOL records**; **1084 REFRs** reference one of those 98 base form IDs (`NAME` subrecord scan); `Fallout - Meshes.bsa` ships **137 cached `meshes\scol\*.nif`** static-collection meshes. The first FNV SCOL (`0017B667`) DATA blocks are byte-identical to the FO4 layout `parse_scol_group` already decodes — only the gate blocks it. Both `index.scols` and the cached-`CM*.NIF` `index.statics` SCOL entries are empty for FNV, so both resolution paths fail. The 2026-05-30 audit recorded this as a clean guard but only checked dispatch-stealing, missing that FNV legitimately ships SCOL.
- **Impact**: 1084 SCOL-based placements (static collections — road segments, guardrails, pylons, ruined-house/debris LOD clusters) render as nothing. Predominantly **exterior** worldspace content — the interior Prospector Saloon bench-of-record is unaffected (which is why it never showed in interior smoke tests). Not a parse crash; visible missing world geometry on the reference title (FO3 equally affected with 54 SCOL).
- **Suggested Fix**: Widen the SCOL gate to the Gamebryo-Fallout era: parse SCOL when `is_fo4_plus || matches!(game, GameKind::Fallout3NV)`. Keep PKIN/MOVS/MSWP FO4+-gated. Correct the `mod.rs:175-177` and `scol.rs` doc comments. Add an FNV SCOL-count assertion (`index.scols.len() == 98`) to `parse_real_esm.rs` as a regression guard.

#### D6-01: NiParticleSystem local transform discarded — emitter ignores host-relative offset
- **Severity**: MEDIUM
- **Dimension**: Particles
- **Location**: `crates/nif/src/blocks/particle.rs:994-1001` (`parse_particle_system` binds `_av` then drops the transform), `:989` (struct keeps only `original_type` + `modifier_refs`); spawn `byroredux/src/scene/nif_loader.rs:476-534`
- **Status**: Existing: #1333 (confirmed still real & FNV-reachable)
- **Description**: `parse_particle_system` parses `NiAVObjectData` (incl. local TRS) and discards it; the emitter preset is inserted directly onto the parent NiNode's entity, and spawn reads the host `GlobalTransform` as origin. Any non-identity local offset of the NiParticleSystem block relative to its host node is lost.
- **Impact**: FNV FX NIFs that offset the emitter from its node (flame above a torch mount, multi-emitter FX) spawn particles at the wrong position. Visible mis-placement, not a crash; single-emitter-at-node-origin content (common torch/candle case) is unaffected.
- **Suggested Fix**: Capture `_av.transform` into `NiParticleSystem`, surface it on `ImportedParticleEmitter`, and compose it onto the host `GlobalTransform` at spawn.

#### D6-02: extract_emitter_params omits planar_angle from the authored sweep
- **Severity**: MEDIUM (effectively inert — no downstream consumer)
- **Dimension**: Particles
- **Location**: `crates/nif/src/import/walk/mod.rs:737-748` (`extract_emitter_params`), `crates/nif/src/import/types.rs:1226` (`ImportedEmitterParams` has no planar field)
- **Status**: Existing: #1445 (confirmed; impact lower than the issue title implies)
- **Description**: `EmitterBaseParams` parses `planar_angle` + `planar_angle_variation` (`blocks/particle.rs:60-61,95-96`) but `extract_emitter_params` never copies them out, so they drop at the import boundary. A full-tree grep shows `planar_angle` has **zero consumers** outside the parser — `ParticleEmitter` has no azimuth field and the spawn cone uses a fully-random azimuth `phi = rng() * TAU` (`systems/particle.rs`). Even if captured, planar_angle has nowhere to go today.
- **Impact**: Authored azimuth bias dropped → omnidirectional cone. For dominant FNV flame/smoke FX, declination (cone half-angle, correctly applied) governs visible spread, so the delta is rarely perceptible until a directional-spray FX surfaces. No crash, no missing content.
- **Suggested Fix**: Either wire planar_angle end-to-end (`ImportedEmitterParams` → `ParticleEmitter` → bias `phi` in the spawn loop) or close #1445 won't-fix with a note that the cone model intentionally randomizes azimuth.

#### D7-02: Non-Ragdoll/non-LimitedHinge constraints are dropped silently — can fragment a chain with no telemetry
- **Severity**: MEDIUM (escalates to HIGH if a real FNV biped/creature ragdoll authors one of these joint types between load-bearing bones — unverifiable here without on-disk skeleton data)
- **Dimension**: PHYSAL Ragdoll
- **Location**: `crates/nif/src/import/collision.rs:316-320` (the `BhkConstraintData::Other => continue` arm in `extract_ragdoll`)
- **Status**: NEW
- **Description**: `extract_ragdoll` only surfaces constraints decoded as `Ragdoll` or `LimitedHinge`. Any `bhkHingeConstraint`, `bhkBallAndSocketConstraint`, `bhkPrismaticConstraint`, or `bhkStiffSpringConstraint` — all decoded as `BhkConstraintData::Other` by the constraint parser (`constraints.rs:379-386`) — hits `BhkConstraintData::Other => continue` and the joint is **dropped with no log line** (unlike the surrounding FO4-NP / phantom stubs in the same file, which `log::debug!` their drops; see `collision.rs:113,196,206,239,247`). The two bodies it linked stay in `bodies`; the edge vanishes.
- **Evidence**: Match arm is literally `BhkConstraintData::Other => continue,` with no `log::*` (verified `collision.rs:319`). Downstream, `orient_tree` (`crates/physics/src/ragdoll.rs:176`) BFS-walks surviving edges; a dropped sole-link edge produces a disconnected forest, and `build_ragdoll` builds each component as an independent free-floating multibody. The detached limb then free-falls with no warning.
- **Impact**: Visually broken ragdoll (limb detaches / falls through the world) with **no diagnostic**. The silent-drop is the dangerous part — every other block-drop in this file logs. FNV humanoid ragdolls are Ragdoll+LimitedHinge-dominated (so vanilla bipeds are *probably* fine), but FNV creatures (radscorpion, cazador, deathclaw) and modded skeletons mix in plain hinge / ball-and-socket joints.
- **Suggested Fix**: At minimum, `log::warn!` when a constraint linking two ragdoll bodies is dropped as `Other` (name the type + bone names). Better: after `orient_tree`, log/assert when the result is a forest (a humanoid ragdoll should be a single tree). Long-term: decode plain hinge / ball-and-socket into the canonical joint set (a limitless hinge is `LimitedHinge { min: -PI, max: PI }`).

#### D7-03: Dynamic trimesh ragdoll body has degenerate inertia (advisory)
- **Severity**: LOW
- **Dimension**: PHYSAL Ragdoll
- **Location**: `crates/physics/src/convert.rs:156-176` + `crates/physics/src/ragdoll.rs:110-124`
- **Status**: NEW (note only)
- **Description**: If a ragdoll body's shape resolves to `CollisionShape::TriMesh` (possible when a bone hosts a `bhkPackedNiTriStripsShape`/compressed mesh rather than a primitive), `build_ragdoll` attaches a trimesh collider to a **dynamic** body. Rapier's mass/inertia for a possibly-open trimesh is unreliable; the `.mass(part_mass)` override sets total mass but not a sane inertia tensor, so the link can spin pathologically.
- **Impact**: Potential jitter/spin on the rare trimesh-shaped ragdoll bone (mods, creatures). Vanilla FNV ragdoll bones almost always author capsules/boxes, so this likely doesn't fire on stock content — hence advisory.
- **Suggested Fix**: For ragdoll bodies, prefer a convex-hull or bounding-capsule substitution when the resolved shape is a `TriMesh`, or set a principal-inertia tensor from the collider AABB.
