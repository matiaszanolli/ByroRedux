## Findings by Dimension

### Dimension 1 — NIF Parser (FNV regression guard)

#### FNV-D1-01: `bhkBreakableConstraint` FO3+ silently zeroes `threshold` and `remove_when_broken` (MEDIUM)
- **Location**: [crates/nif/src/blocks/collision.rs:1721-1733](crates/nif/src/blocks/collision.rs#L1721-L1733)
- **Description**: `BhkBreakableConstraint::parse` only reads `threshold` and `remove_when_broken` on the Oblivion path. On FO3+ (every FNV instance) it returns zeros. nif.xml line 7027 marks both fields unconditional and trailing the wrapped CInfo on every Bethesda version. The Oblivion `wrapped_payload_size` table at lines 1678-1692 is correct for FNV too in the common case (motor type 0), so the values are derivable but currently dropped. `block_size` recovery keeps the stream aligned (parse-rate gate stays green) but the parsed values are wrong.
- **Fix**: drop the `is_oblivion` gate; use `wrapped_payload_size` on FO3+ for all wrapped types where the payload size is derivable, then read the trailer fields directly. Same pattern applies to `BhkConstraint::parse` at `collision.rs:1346-1431` which short-stubs on FO3+ and loses the entire CInfo payload.

#### FNV-D1-02: Wrapped CInfo size table is Oblivion-only but presented as version-agnostic (LOW)
- **Location**: [collision.rs:1681](crates/nif/src/blocks/collision.rs#L1681) (Hinge=80) and [collision.rs:1358](crates/nif/src/blocks/collision.rs#L1358) (parallel table).
- **Description**: `bhkHingeConstraintCInfo` is 80 B `until 20.0.0.5` but **128 B `since 20.2.0.7`** per nif.xml (extra Axis A, Perp Axis In B1/B2). `bhkLimitedHingeConstraintCInfo` 124 B Oblivion vs 140 B FNV. `bhkRagdollConstraintCInfo` adds Motor A + Motor B on FNV (152 B vs 120 B in our table). Currently dormant — only consulted on the Oblivion branch — but becomes a real bug the moment FNV-D1-01 lifts the `is_oblivion` gate.
- **Fix**: make `wrapped_payload_size` and the parallel table version-aware (key off `stream.version()`); sizes derive mechanically from nif.xml.

### Dimension 2 — ESM Record Parser

#### FNV-D2-01: ENCH enchantments dropped at the catch-all skip (MEDIUM)
- **Location**: no `b"ENCH"` arm in [crates/plugin/src/esm/records/mod.rs:265-438](crates/plugin/src/esm/records/mod.rs#L265-L438).
- **Description**: every `EITM` FormID on WEAP/AMMO/ARMO references an `ENCH` record (Pulse Gun "Pulse" enchantment, This Machine's Marksman Carbine charge effect, Holorifle). With ENCH undispatched, all of these dangle — weapon special effects silently no-op once a consumer wires up.
- **Fix**: add `extract_records(..., b"ENCH", ...)` arm capturing EDID + EFID/EFIT effect chain (same shape as SPEL — can route through `parse_spel`).

#### FNV-D2-02: FLST FormID lists dropped (MEDIUM)
- **Location**: no `b"FLST"` arm in `records/mod.rs`.
- **Description**: FLST is a flat FormID list referenced from PERK entry-point conditions, COBJ recipe ingredients, and quest objective filters. Every condition that runs `IsInList <flst>` returns "not in list" because the list is empty at lookup. Affects ~50 PERK conditions and the Caravan deck (CCRD/CDCK) gameplay.
- **Fix**: add `FlstRecord { form_id, editor_id, entries: Vec<u32> }` parser (EDID + LNAM array) and dispatch.

#### FNV-D2-03: INFO dialogue children never walked (MEDIUM)
- **Location**: [records/mod.rs:127-129, 405-412](crates/plugin/src/esm/records/mod.rs#L127-L129) — code comment explicitly flags this as "follow-up".
- **Description**: `DIAL` records dispatch (line 410) and land in `index.dialogues`, but the nested GRUP under each DIAL containing the actual `INFO` topic responses is silently skipped because `extract_records()` filters on a single `expected_type` (line 489-510). DIAL records exist as empty shells; every NPC response, branch, and script trigger payload is lost.
- **Fix**: specialise `extract_records` to accept a per-type closure map, OR add a dedicated `extract_dial_with_info()` that recurses into the DIAL sub-GRUP and pushes INFO records onto `dialogue.infos`.

#### FNV-D2-06: `EsmIndex.total()` undercount drift (LOW)
- **Location**: [records/mod.rs:153-189](crates/plugin/src/esm/records/mod.rs#L153) vs [441-472](crates/plugin/src/esm/records/mod.rs#L441-L472).
- **Description**: `EsmIndex.total()` and the log line at lines 441-472 are independent — adding a new category (e.g. fixing #519/AVIF) requires updating both. Already missed once: `total()` doesn't double-count `cells.statics` against the `activators`/`terminals` overlap. The log message is correct; `total()` is not authoritative.
- **Fix**: drive `total()` from a `categories()` iterator over a `(&'static str, fn(&Self) -> usize)` table; fold the log message through the same table.

#### FNV-D2-04 (LOW, **DUPLICATE of #624**): IMGS / IMAD dropped — same gap as Skyrim SK-D6-LOW filed today. Skip.

#### FNV-D2-05: confirms #519 (AVIF dispatch) still open. No new issue.

### Dimension 3 — Cell Loading

#### FNV-D3-01: SkyParamsRes cloud + sun textures leak on every cell unload (HIGH)
- **Locations**:
  - Acquire: [byroredux/src/scene.rs:235, 278, 321, 344, 377](byroredux/src/scene.rs#L235) — five `texture_registry.load_dds` calls each bump refcount 0→1 (per [texture_registry.rs:331-335](crates/renderer/src/texture_registry.rs#L331-L335)).
  - Storage: `SkyParamsRes` (Resource, not an ECS component).
  - Release path: [cell_loader::unload_cell](byroredux/src/cell_loader.rs#L189-L321) only walks ECS components (`MeshHandle`, `TextureHandle`, `NormalMapHandle`, `DarkMapHandle`, `ExtraTextureMaps`, `TerrainTileSlot`). `SkyParamsRes` is never touched.
- **Description**: every WastelandNV reload leaks 4 cloud layers + 1 sun sprite (~5 textures, 4-8 MB DDS each). M40 doorwalking will hemorrhage VRAM at the cell-cell boundary.
- **Fix**: in `unload_cell`, after victims are despawned, read `world.try_resource::<SkyParamsRes>()` and call `drop_texture` on each non-zero `cloud_texture_index{,_1,_2,_3}` + `sun_texture_index`; then `world.remove_resource::<SkyParamsRes>()`. Same pattern for `CellLightingRes` / `WeatherDataRes` / `WeatherTransitionRes` (state, not VRAM).

#### FNV-D3-02: Terrain splat-layer texture refcounts never released (HIGH)
- **Locations**:
  - Acquire: `build_cell_splat_layers` at [cell_loader.rs:780](byroredux/src/cell_loader.rs#L780) → `resolve_texture` → `acquire_by_path` (refcount bumped per layer) → `allocate_terrain_tile(indices_arr)` at [cell_loader.rs:1024-1032](byroredux/src/cell_loader.rs#L1024-L1032).
  - Release: `unload_cell` calls `ctx.free_terrain_tile(slot)` at [cell_loader.rs:271-273](byroredux/src/cell_loader.rs#L271-L273); `free_terrain_tile` at [vulkan/context/resources.rs:51-60](crates/renderer/src/vulkan/context/resources.rs#L51-L60) only resets the slot to `None` — does **not** drop the per-layer texture refcounts.
- **Description**: a 7×7 grid with avg 3 splat layers per LAND leaks ~150 texture refcounts per reload — those textures stay pinned in the bindless slab and will eventually saturate `check_slot_available()`.
- **Fix**: have `free_terrain_tile` return the previous `[u32; 8]` (or expose via a peek) so `unload_cell` can drop each non-zero layer index via `texture_registry`. Alternatively pass `&mut TextureRegistry` into `free_terrain_tile`.

#### FNV-D3-03: LIGH refs whose NIF carries a zero-color placeholder spawn no LightSource (MEDIUM)
- **Locations**:
  - Inner filter: [cell_loader.rs:1874-1879](byroredux/src/cell_loader.rs#L1874-L1879) — skips every `nif_light` whose `color[0]+color[1]+color[2] < 1e-4` via `continue`.
  - ESM-fallback gate: [cell_loader.rs:2223-2234](byroredux/src/cell_loader.rs#L2223-L2234) — only attaches the ESM `light_data` LightSource when `nif_lights.is_empty()`.
- **Description**: a NIF that authored a placeholder zero-color NiPointLight (very common in FNV light-bulb meshes) still has `nif_lights.len() == 1` even after the inner loop drops it — so the ESM authoritative LightSource never spawns. Prospector Saloon's "25 point lights" baseline is brittle on this edge.
- **Fix**: replace `nif_lights.is_empty()` with a counter (e.g. `spawned_nif_lights` from the inner loop); attach the ESM light when zero NIF lights actually emitted.

#### FNV-D3-05: `unload_cell` doesn't shrink `NifImportRegistry` (LOW, design gap)
- **Location**: [cell_loader.rs:188-321](byroredux/src/cell_loader.rs#L188-L321) — drops mesh/BLAS/texture GPU resources but the process-lifetime `NifImportRegistry` `Arc<CachedNifImport>` entries persist. Code comment at lines 73-77 acknowledges "unbounded for now".
- **Description**: memory bound across a long playthrough is unbounded; ~14k FNV NIFs at hundreds of MB. Mesh GPU memory is reclaimed (good), but the parsed CPU-side scene graph (vertex arrays, indices, collision shapes, particle emitters) survives. Belongs in M40 doorwalking work.
- **Fix**: either clear the registry inside `unload_cell` on a hard-reset signal, or add an LRU cap surfaced via the `mesh.cache` debug command (already mentioned in the doc comment as a follow-up).

#### FNV-D3-06: PKIN expansion is single-level; nested PKIN/SCOL/LVLI silently miss (LOW, FO4-only impact)
- **Location**: [cell_loader.rs:1188-1228](byroredux/src/cell_loader.rs#L1188-L1228).
- **Description**: `expand_pkin_placements` returns leaf `child_form_id`s looked up via `index.statics.get()`. If the child is itself a PKIN, SCOL, or LVLI, the lookup misses and the placement is dropped at line 1226 with only a `log::debug!`. PKIN is FO4+; FNV has zero impact today, but this is a forward-looking gap.
- **Fix**: hoist PKIN/SCOL/LVLI dispatch into a recursive helper bounded by depth (≤ 4) so PKIN-of-PKIN and SCOL-of-PKIN cases fan out properly. Defer until FO4 cell load needs it.

#### FNV-D3-04 (LOW, **DUPLICATE of #544**): embedded_clip never spawned. Skip.

### Dimension 4 — RT Lighting Pipeline

#### FNV-D4-01: cluster_cull `FAR = 10000` smaller than FNV exterior `fog_far` (MEDIUM)
- **Location**: [crates/renderer/shaders/cluster_cull.comp:18](crates/renderer/shaders/cluster_cull.comp#L18) — `const float FAR = 10000.0`.
- **Description**: `FAR` drives the exponential `sliceDepth(slice)` used to build cluster AABBs. FNV exterior weather/CLMT routinely pushes `fog_far` to 30000-80000 units. Lights past 10000 fall outside every slice's AABB and get culled silently. Indoors, fog ends most lights' visible contribution before this matters; exterior dawn/dusk lights at long range (lampposts down a road, beacons) are the regression surface. Directional lights are flagged "always-affect" and unaffected.
- **Fix**: push `FAR` (and ideally `NEAR`) as a UBO field sourced from `screen.zw` (already carries `fog_near`/`fog_far`) or from a dedicated `cluster_params` push constant. The shader already reads `screen.w` as `fog_far` for other consumers.

#### FNV-D4-02: cluster `MAX_LIGHTS_PER_CLUSTER = 32` with no overflow telemetry (LOW)
- **Location**: [cluster_cull.comp:16, 138](crates/renderer/shaders/cluster_cull.comp#L16) — loop terminates on `count < MAX_LIGHTS_PER_CLUSTER` with no overflow flag.
- **Description**: FNV interiors with dense xenon racks (Lucky 38, Ultra-Luxe) can put 40+ small-radius lights in a single near-camera cluster. The 33rd+ light in that cluster is dropped without diagnostic.
- **Fix**: either widen to 64 (cost: 2× cluster index SSBO, ~256 KB at 16×9×24) or add an atomic overflow counter to surface the case in `sys.stats`. The atomic counter is cheap and high-signal — start there.

### Dimension 5 — Real-Data Validation

(Live measurements above. All findings LOW — observability / docs polish.)

#### FNV-D5-01: ROADMAP "Exterior 3×3" copy is ambiguous (LOW)
- **Location**: [ROADMAP.md:33, 73](ROADMAP.md#L33).
- **Description**: ROADMAP says "Exterior 3×3" (and "3×3 grids from FNV WastelandNV"), but `--grid 0,0 --radius 3` logs `Found 49/49 cells in 7x7 grid`. Either the ROADMAP wording is wrong or the radius semantics are wrong.
- **Fix**: change "3×3" to "7×7 (radius 3)" in ROADMAP.md, OR rename the flag to `--side 3` if intent was radius=1. The number that ships in logs is 7×7; update docs to match.

#### FNV-D5-02: `texture_count` in `stats` is registry-wide, not cell-scoped (LOW)
- **Location**: [byroredux/src/commands.rs:51](byroredux/src/commands.rs#L51) + [crates/core/src/ecs/resources.rs:86](crates/core/src/ecs/resources.rs#L86).
- **Description**: `stats` reports global TextureRegistry size, not "textures referenced by entities in the current scene". For a single-cell session this is fine; at M40 (world streaming) the value will not drop when cells unload, masking texture-leak regressions like FNV-D3-01/D3-02.
- **Fix**: expose two fields — `texture_count` (registry, unchanged) and `textures_in_use` (count of unique handles reachable from `TextureHandle` query). ~10 lines.

#### FNV-D5-03: `--cmd` headless mode has no scene access (LOW)
- **Location**: [byroredux/src/main.rs:84-101](byroredux/src/main.rs#L84-L101).
- **Description**: the `--cmd` short-circuit creates an empty `World` with only `DebugStats::default()`. So `--cmd "stats"` always returns FPS 0, entities 0, draws 0, even with `--esm` flags also passed. The headless path doesn't load any cell content. Blocks CI regression checks against entity counts without X11.
- **Fix**: either reject `--cmd` when `--esm`/`--bsa` is also present (clear error), or wire the cell loader into the `--cmd` path so it can answer `entities` against a loaded cell without a window. The latter unblocks CI baseline checks.

#### FNV-D5-04: 29 unresolved REFR bases in Prospector (LOW)
- **Location**: live `dump_cell_refs` reports 432/461 resolved, 29 unknown; cell_loader logs `5 base forms not found in statics table (sample: 0015D9B9, 0010521D, 0016AD05, 000AE8B3, 000C79EC)` at cell_loader.rs:1223-area.
- **Description**: the two numbers disagree (29 vs 5) — spawn-side filter dedupes; dump_cell_refs counts placements. Both nonzero means either (a) ESM dispatch is missing record kinds those bases land in (CONT? PROJ? IDLM?) or (b) those forms live in plugin overrides the single-master loader doesn't follow.
- **Fix**: investigate the 5 unique unresolved base FormIDs. If CONT/IDLM/MSTT, file a record-dispatch follow-up. If plugin references, that's a `FormIdRemap` test gap (#445).
