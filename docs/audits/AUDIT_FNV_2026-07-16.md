# Fallout: New Vegas Compatibility Audit — 2026-07-16

Reference-title audit. FNV is the most-validated end-to-end path in
ByroRedux and the reference realization for NIFAL (material/collision
translation) and PHYSAL (ragdoll slice 1). 9 dimensions run in parallel

## Executive Summary

**9 NEW findings, 0 CRITICAL.** The reference path stayed structurally
sound: cell loading, RT lighting, NIFAL material/collision translation,
NIF parsing, animation/skinning/particles, and the seven AI package
runtimes all independently re-verified clean against their known guard
regressions (#1520, #1538, #1654, #1652, #1269, #772, #790, #793, #1718,
#1539, #1540, #1772, and yesterday's #1996/DIM9-01 fix). The two HIGH
findings both extend the same root cause as the just-closed #1996: the
ESM FormID-remap fix for multi-plugin loads landed for `parse_npc`'s
"classic" fields but missed three sibling parsers (OTFT/LVLI-LVLN/CONT)
and `parse_npc`'s own FaceGen-recipe fields (HNAM/ENAM/PNAM/FMRI) — on
any real multi-plugin FNV load (base + any DLC, which is the *normal*
case for this title), DLC NPCs whose gear/hair/eyes are defined in the
same DLC plugin can silently spawn naked/bald/mismatched. No blast
radius beyond that FormID-remap family; nothing rises to CRITICAL because
nothing corrupts GPU/AS state, crashes, or causes data loss — the failure
mode throughout is silent-but-wrong content, which is the ceiling severity
for a parser/gameplay-logic gap under this project's scale.

One dimension (Real-Data Validation) surfaced a process-accuracy finding
rather than a code bug: a live, uncontrolled bench sample at current HEAD
came in **95.7% faster** than the ROADMAP's 613-commit-stale bench-of-record
on Prospector Saloon, suggesting the open `R6a-stale-15` "fence recovery
still pending, uninvestigated" framing is materially out of date and the
gap may already be ~90% closed. This is not asserted as a corrected
baseline (single sample, uncontrolled) but is flagged to prioritize the
already-planned formal re-bench.

## Dimension Findings

### Dimension 1 — Cell Loading End-to-End

#### FNV-D1-02: `NifImportRegistry` cache key has no plugin/archive-set discriminant — stale cross-load reuse via debug `cell.load`
- **Severity**: MEDIUM
- **Location**: `byroredux/src/cell_loader/nif_import_registry.rs:136-180`, `byroredux/src/cell_loader/references/mod.rs:610-617`, `byroredux/src/boot.rs:359`, `byroredux/src/debug_load.rs:206-266,268-360`
- **Status**: NEW
- **Description**: `NifImportRegistry` is a process-lifetime cache keyed only by lowercased model path — no field records which BSA/BA2/ESM/load-order resolved that path. Safe under the normal CLI launch (archive set fixed for the whole run) but broken under the debug `cell.load` console command, which can synthesize an arbitrary `--bsa`/`--esm`/`--master` set per request against the *same* singleton registry. A model-path collision across two such requests (e.g. comparing vanilla FNV against a mod's overriding BSA in one debug session) silently serves the first-loaded content for the second. Same pattern exists in `TextureRegistry::path_map`.
- **Impact**: Wrong-but-plausible geometry/collision/texture silently substituted during exactly the inspect/compare workflow the debug tool exists for; easy to misattribute to a content/parser bug. Not reachable via the normal single-launch CLI path, so it doesn't affect bench-of-record numbers.
- **Suggested Fix**: Fold an archive-set identity/generation counter into the cache key, or have `exec_load_interior`/`exec_load_exterior` call `NifImportRegistry::clear()` (+ `TextureRegistry` equivalent) whenever the requested archive set differs from the previously-loaded one.

Regression guards independently re-verified this dimension: XCLL→LTMP→engine-default fallback (FNV-D1-01, fixed `3077dcb0`, test added), `NiAlphaProperty` decal routing, cell-unload hygiene (#1520/#1531, 19/19 tests pass), M38 water/submersion hysteresis (#1450), terrain LAND/LTEX/TXST splat cap (#470), WRLD→CLMT→WTHR resolution (#476 mod-sentinel exclusion), `_far.nif` LOD radius gating (#1866/#1871).

### Dimension 4 — ESM Record Parser (Coverage & Accuracy)

#### FNV-D4-01: `parse_otft` / `parse_leveled_list` (LVLI/LVLN) / `parse_container` (CONT) never remap embedded FormIDs — same bug class as the just-closed #1996, not covered by its fix
- **Severity**: HIGH
- **Location**: `crates/plugin/src/esm/records/outfit.rs:44-61` (INAM), `crates/plugin/src/esm/records/container.rs:80-136` (CNTO), `:139-169` (LVLO); call sites `crates/plugin/src/esm/records/mod.rs:452-480,743-745`; consumer `crates/plugin/src/equip.rs:304-373`
- **Status**: NEW (same root cause as closed #1996/DIM9-01, different un-fixed parsers)
- **Description**: #1996 (closed 2026-07-15) added FormID remapping to `parse_npc` because embedded FormID fields inside a record's sub-records are stored plugin-local and need explicit remapping via `reader.get_form_id_remap()` — only the record's own key is auto-remapped. The fix touched `parse_npc` only; three sibling parsers still store raw, unremapped FormIDs and are never given a `FormIdRemap` at their `mod.rs` call sites (contrast with the PACK/QUST/PERK/AVIF arms nearby, which do obtain one).
- **Impact**: On any FNV load with more than one plugin (the *normal* case — this engine's own multi-master CLI, or any real playthrough with a DLC master), OTFT/LVLI/LVLN/CONT records authored in a non-base plugin whose FormID references point at content defined in that same plugin resolve against the wrong (global-keyed) map and silently miss. `expand_leveled_form_id` treats an unresolvable FormID as "not an item, not a list" and pushes nothing — affected NPCs spawn naked/gearless, affected loot containers spawn empty, no error logged. Undetected by existing tests because a standalone-master load has an identity remap.
- **Suggested Fix**: Add `remap: &Option<FormIdRemap>` to all three parsers, thread `reader.get_form_id_remap()` through at their `mod.rs` call sites (mirroring the PACK/QUST/PERK/AVIF/NPC_ pattern), remap `INAM`/`LVLO.form_id`/`CNTO.item_form_id`.

#### FNV-D4-02: `parse_npc`'s FaceGen-recipe FormID fields (HNAM/ENAM/PNAM-eyebrow/FMRI/FO4 PNAM-head_parts) still unremapped after #1996
- **Severity**: HIGH
- **Location**: `crates/plugin/src/esm/records/actor.rs:691-712,754-763`; consumers `byroredux/src/npc_spawn.rs:1083-1165`
- **Status**: NEW (follow-on gap in the closed #1996 fix)
- **Description**: #1996 added the `remap` parameter to `parse_npc` and applied it to RNAM/CNAM/VTCK/SCRI/SNAM/CNTO/PKID/DOFT/INAM/TPLT/PRPS/PRKR — but the same function's FaceGen-recipe arms (HNAM/ENAM/FNV-PNAM-eyebrow/FMRI/FO4-PNAM-head_parts) still read raw `u32_or_default()` with no `remap_fid()` wrapper, despite `remap` being in scope for the whole function.
- **Impact**: On a multi-plugin load, an NPC defined in a non-base plugin whose hair/eyes/eyebrow reference points at content defined in that same plugin resolves the wrong or no `index.hair`/`index.eyes`/`index.head_parts` entry — silently bald/browless, or (on FormID collision across plugins) wrong hair/eye texture attached. No crash, no log.
- **Suggested Fix**: Wrap HNAM/ENAM/PNAM(both arms)/FMRI reads in the existing local `remap_fid(raw, remap)` helper, exactly like the PRPS/PRKR arms in the same function.

### Dimension 6 — Animation, Skinning & Particles

#### FNV-D6-01: Text-key events fire the wrong set on `CycleType::Reverse` (ping-pong) backward legs
- **Severity**: LOW
- **Location**: `crates/core/src/animation/text_events.rs:20-46`, reached via `stack.rs:222` and `byroredux/src/systems/animation.rs`
- **Status**: NEW
- **Description**: `visit_text_key_events` infers "loop wrap-around" purely from `curr_time < prev_time`. Ping-pong `CycleType::Reverse` clips also produce `prev_time > curr_time` on every backward leg with no actual wrap (both times in-range). The wrap-handling branch then fires the complement of the keys actually crossed on that backward leg.
- **Impact**: Anim-driven events (footsteps, hit frames, IK hints) misfire on the backward half of Reverse-cycle clips carrying text keys. LOW: Reverse + authored text keys is a rare combination in vanilla FNV content (idle/gameplay clips are overwhelmingly Loop/Clamp). No crash or visual corruption.
- **Suggested Fix**: Thread `reverse_direction`/cycle type into `visit_text_key_events`; on a backward leg fire the closed interval `(curr_time, prev_time]` instead of the wrap complement. Add a ping-pong regression test.

Verified clean this dimension: `.kf`/registry/interpolator/text-key/cycle-type/interning (#340, zero per-frame StringPool locks) foundation, NiSkinData sparse-weight parsing + GPU palette, B-spline `FLT_MAX` pose-fallback sentinel (#772, all TRS axes gated), `AnimationClipRegistry` dedup (#790) + cell-streaming LRU twin (#863), NPC hand-mesh triple-load (#793, hands always load even when torso is armor-covered), typed-emitter particle pin (authored values override preset through the single `apply_emitter_overlays` boundary shared by both spawn paths).

### Dimension 7 — PHYSAL Ragdoll (FNV Reference Slice)

#### FNV-D7-01: `activate_ragdoll` has no re-activation guard — second trigger leaks the first ragdoll's Rapier bodies and double-simulates
- **Severity**: MEDIUM (elevation risk to HIGH once gameplay-triggered re-hit reaches this path)
- **Location**: `byroredux/src/ragdoll.rs:204-318`; trigger `byroredux/src/commands/scene.rs:690-703`; cleanup `byroredux/src/cell_loader/unload.rs:418`
- **Status**: NEW
- **Description**: `activate_ragdoll` never checks whether the actor is already `RagdollActive`. A second activation re-seeds from current (simulated) bone poses, builds a fresh full set of Rapier bodies/colliders/multibody joints, and `insert`s a new `Ragdoll` component that overwrites (drops) the first one's handles without calling `PhysicsWorld::remove_ragdoll` — which is only ever invoked from cell unload over *live* components. The orphaned first set (~18 bodies + ~17 joints for a humanoid) is never freed and keeps simulating at the same bone positions, fighting the live solver.
- **Impact**: Today reachable only via the console `ragdoll <id>` command (slice-1 scope, caps at MEDIUM). Elevation risk: PHYSAL rollout step 4 (death/hit-react AI triggers) would make a second hit on an already-ragdolled actor a per-event leak, escalating to HIGH.
- **Suggested Fix**: Make activation idempotent — precheck `RagdollActive`/existing `Ragdoll` and either early-return or call `remove_ragdoll` on the existing component before rebuilding. Add a unit test asserting body count doesn't grow on double-activation.

Verified clean this dimension: end-to-end `ImportedRagdoll`→`activate_ragdoll`→`ragdoll_writeback_system` flow; all four D7 audit-guard regressions hold (#1718 dropped-body/constraint telemetry, #1539 dropped-`Other`-constraint warn, #1540 trimesh→convex-hull inertia substitution, #1772 keyframed-follower teardown); the only per-game seam remains the constraint CInfo decode (`bsver` gate in `constraints.rs`) — no per-game branch in `ragdoll.rs`/`crates/physics/`; FNV's dominant `bhkMalleableConstraint`-wrapping-Ragdoll form decodes to a real jointed body, not a rigid blob; writeback vs. GPU bone palette confirmed race-free (single-threaded staged scheduler, `Stage::Late` exclusive writeback, palette read post-scheduler).
