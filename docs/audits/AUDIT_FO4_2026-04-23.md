# AUDIT — Fallout 4 Compatibility — 2026-04-23

**Scope**: Full 6-dimension sweep of ByroRedux readiness for Fallout 4 content.
**Tree state**: `main` @ `cd959cf` ("docs: session 16 closeout — NIF audit bug-bash + ESM REFR/TXST expansion").
**Method**: Per `.claude/commands/_audit-common.md`. Each dimension ran as its own task; this document is the merged orchestrator output. Severity per `_audit-severity.md`.

## Executive Summary

Fallout 4 compatibility has advanced from "parse-only" to "interior geometry renders at 90 FPS" (MedTekResearch01, 7434 entities, 2026-04-23). The NIF parser is 100% recoverable on 34,995 vanilla `Fallout4 - Meshes.ba2` NIFs, the BA2 reader is production-correct for v1/v7/v8 GNRL and DX10, the BGSM/BGEM parser crate has shipped (`crates/bgsm/`, ~1944 lines), and the ESM dispatch now covers SCOL/TXST architecture records.

The remaining gaps are **shading accuracy** and **placement-record completeness**, not "can I load a cell":

1. **BGSM scalar PBR fields are parsed but dropped at merge time.** `merge_bgsm_into_mesh` forwards the six texture-path `Option<String>` slots and drops everything else — `emittance_color`, `emittance_mult`, `specular_mult/color`, `smoothness`, `fresnel_power`, `alpha_test_ref`, wetness suite, translucency suite, rim/backlight. Every FO4 emissive mesh (lamps, terminal screens, signage) lights with NIF-fallback brightness; every metallic surface (power armor, workbench props) uses placeholder specular.
2. **TXST.MNAM material-path override never resolved.** 37% of vanilla FO4 TXSTs (140/382) are MNAM-only and rely on the BGSM resolver; `cell_loader.rs` never reads `EsmCellIndex.texture_sets`. Compounds with zero REFR-override sub-record parsing (XTNM/XTXR/XATO/XMSP/XLIG all unparsed).
3. **SCOL placements not expanded at cell load.** Parser-side (#405) ships; runtime expander is missing. Vanilla works because 2616/2617 SCOLs ship cached `CM*.NIF` combined meshes — mod-added SCOLs and previsibine-bypass loadouts render as nothing.
4. **BSSubIndexTriShape segmentation still consumed via `block_size`.** Geometry renders but dismemberment/armor-slot routing is impossible. Tracked as #404 (open).
5. **Sanity-check follow-ups**: OOM-hazard `Vec::with_capacity(u32)` class in BA2 reader (not covered by #388 sweep); zero integration tests for the BA2 reader itself; `fo4_slsf1/slsf2` bitflag modules dead in production (shared-bit aliasing via Skyrim modules happens to work).

**Critical path to "FO4 MedTek renders as the shipped game does"**: (1) forward BGSM scalars in `merge_bgsm_into_mesh` [≈20 LOC, ~80% of the visual delta]; (2) wire TXST.MNAM + REFR override sub-records into cell_loader [M]; (3) add SCOL runtime expander [S]. Nothing else blocks initial rendering.

## Dimension 1 — NIF BSVER 130 + Half-Float Vertices

Parse side is correct. `VertexDesc` gate, `half_to_f32` decoder, BSVER-gated trailing fields (subsurface, rim, backlight, fresnel, wetness incl. `unknown_1`), Next-Gen BSVER 131 flag-pair absence, and BSVER >= 132 CRC32 arrays all match nif.xml and are validated against 34,995 vanilla FO4 NIFs. One known gap — segmentation — remains.

### FO4-DIM1-01: `BSSubIndexTriShape` segmentation consumed via `block_size`, dismemberment impossible
- **Severity**: MEDIUM
- **Location**: `crates/nif/src/blocks/tri_shape.rs:1698-1722`
- **Status**: Existing: **#404** (OPEN)
- **Description**: `BSSubIndexTriShape` shares the `BsTriShape` parser with `BSMeshLODTriShape` / `BSTriShape` and does NOT read the FO4-specific segmentation tail (`num_segments` → per-segment `start_index` + `num_primitives` + `parent_array` + `SSE_segment_data`). A regression test locks the "consume via block_size" behavior explicitly.
- **Impact**: Per-segment material routing (dismemberment, armor slot swap, facial morph masks) impossible for FO4 NPCs / creatures / armor. Geometry renders correctly; segment-keyed sub-regions cannot be selected.
- **Suggested Fix**: Extend `BsTriShape` with `Option<Vec<SegmentData>>` populated only when `kind == SubIndex`. Iterate segments up to but not including `SSE_segment_data` (bookkeeping blob). Surface through `ImportedMesh` so ECS `Material` carries a segment index.

### Verified
- **VF_FULL_PRECISION** = `0x400` (bit 10) at `tri_shape.rs:333`. Dispatch `stream.bsver() < 130 || vertex_attrs & VF_FULL_PRECISION != 0` at `tri_shape.rs:441` — FO4 with bit clear goes to half-float.
- **`half_to_f32`** at `tri_shape.rs:664-691` is hand-rolled IEEE 754 binary16: handles signed zero, subnormals (normalize loop), Inf/NaN (payload preserved), normal values. No `half`-crate / intrinsic dependency.
- **u32-pair shader flags** at `shader.rs:604-608`, gated `bsver <= 130`. BSVER 131 returns `(0, 0)` per closed #409. BSVER >= 132 → `sf1_crcs`/`sf2_crcs` CRC32 arrays at `shader.rs:620-635` (num_sf2 gated on `bsver >= 152`).
- **FO4 common trailing fields** at `shader.rs:682-696`: `subsurface_rolloff`/`rimlight_power`/`backlight_power` gated `(130..=139)`. `backlight` conditional on `rimlight < 3.0e38` FLT_MAX sentinel.
- **`unknown_1` wetness field** widened to `bsver >= 130` (closed #403, 1.88M under-reads found in 226k-NIF sweep). Verified at `shader.rs:730`.
- **Packed skinned vertices**: `tri_shape.rs:652-655` reads 4×u16 weights via `half_to_f32` (FO4 uses f16 weights).
- **BGSM material-reference stub**: `shader.rs:520-554` + `586-592` — `bsver >= 155` + non-empty `net.name` returns early-return stub. FO4 (BSVER 130) doesn't take this path; full body parsed and BGSM path surfaced via `NiObjectNETData.name`.

## Dimension 2 — BA2 Archive Reader (GNRL + DX10)

Production-correct for v1/v7/v8 on FO4. 12 checklist items pass. Two actionable findings (OOM class + missing integration tests); rest are DDS-polish / doc-drift items carried over unfiled from the 2026-04-17 audit.

### FO4-DIM2-01: Unchecked `file_count` / `unpacked_size` / `packed_size` allocations (OOM hazard class)
- **Severity**: MEDIUM
- **Location**: `crates/bsa/src/ba2.rs:185, 206, 300, 331, 356, 399, 431, 435, 456, 460`
- **Status**: NEW (related class to closed #388, but BA2 reader was not included in that sweep)
- **Description**: Three unchecked allocations from untrusted archive bytes:
  - `Vec::with_capacity(file_count)` / `HashMap::with_capacity(file_count)` where `file_count` = raw u32 from header offset 12.
  - `vec![0u8; packed_size as usize]` / `vec![0u8; unpacked_size as usize]` during extraction.
  - `Vec::with_capacity(unpacked_size)` inside the zlib arm of `decompress_chunk`.
- **Impact**: DoS on a corrupted or malicious BA2 (`file_count = 0xFFFF_FFFF` → 4B-entry allocation; `unpacked_size = 0xFFFF_FFFF` → 4 GB allocation). Same hazard class as #388. Zero vendor archives trigger it today.
- **Suggested Fix**: Cap `file_count` at ~10M (vanilla max ≈600K in `MeshesExtra.ba2`); cap `packed/unpacked_size` at 256 MB per chunk (vanilla max ~8 MB). Use an `allocate_vec`-style helper.

### FO4-DIM2-05: Zero integration tests for BA2 reader
- **Severity**: MEDIUM
- **Location**: `crates/bsa/` — no `tests/` directory; only 11 synthetic `#[cfg(test)] mod tests` in `ba2.rs`
- **Status**: NEW
- **Description**: `crates/bsa/` has zero real-archive tests committed. Real-data exercise happens in downstream crates (`crates/nif/tests/common/mod.rs`, `crates/bgsm/tests/parse_all.rs`) but never asserts extracted-byte correctness. Session-7's 458,617-file sweep was external — not a CI guard. Uncompressed GNRL (`packed_size == 0`) branch has no vanilla coverage anywhere.

### FO4-DIM2-02: `arraySize = 1` for cubemaps in synthesized DX10 header
- **Severity**: LOW
- **Location**: `crates/bsa/src/ba2.rs:572`
- **Status**: Carry-forward (prior L3, never filed)
- **Description**: `arraySize` hard-coded to `1u32`; DDS spec requires `6 × cube_count` when `miscFlag & D3D10_RESOURCE_MISC_TEXTURECUBE`. 28 cubemaps in vanilla Textures1.ba2 emit technically-invalid DDS. The in-engine dds.rs parser is lenient; external tools (texconv, DirectXTex) reject.
- **Suggested Fix**: `let array_size: u32 = if is_cubemap { 6 } else { 1 };` + test.

### FO4-DIM2-03: `DDSD_LINEARSIZE` set unconditionally for non-BC formats
- **Severity**: LOW
- **Location**: `crates/bsa/src/ba2.rs:514, 582-604`
- **Status**: Carry-forward (prior L1, never filed)
- **Description**: `dwFlags` always includes `DDSD_LINEARSIZE`; correct flag for uncompressed formats is `DDSD_PITCH` with `pitch = width * bpp`. `linear_size_for` only covers BC1-BC7. Harmless for the engine (DXGI extended header is authoritative); rejected by strict validators.
- **Suggested Fix**: Extend `linear_size_for` for uncompressed formats (87, 91, 28, 61, 56) and OR `DDSD_PITCH` instead.

- **Status**: Carry-forward (prior L2, never filed)

### FO4-DIM2-07: `num_mips = 0` silently clamps to 1
- **Severity**: LOW

### FO4-DIM3-01: `fo4_slsf1` / `fo4_slsf2` constants are dead in production code
- **Severity**: LOW
- **Location**: `crates/nif/src/shader_flags.rs:91-193` (defs) ; `crates/nif/src/import/material.rs:30-66` (consumers)
- **Status**: NEW
- **Description**: `pub mod fo4_slsf1` + `pub mod fo4_slsf2` define ~40 named bit constants introduced by closed #414 / FO4-D3-M1. The only references outside the definition file are test assertions inside `shader_flags.rs` itself (lines 223-243) that prove cross-game bit equality (`fo4_slsf1::DECAL == skyrim_slsf1::DECAL`, etc.). Production consumers read FO4 shader flags via Skyrim-module aliases (`DECAL_SINGLE_PASS` / `DYNAMIC_DECAL` from `skyrim_slsf1`, `SF2_DOUBLE_SIDED` from `skyrim_slsf2`, `ALPHA_DECAL_F2` from `fo3nv_f2`). This works today by accident — the bits happen to share positions.
- **Impact**: None today — the bits align. Any future FO4-specific flag addition, or a drift when a newer format reshuffles bits, would silently route through a Skyrim-labelled alias. Adds review load for new shader-flag work.
- **Suggested Fix**: Pick one: (1) add `const FO4_SF2_DOUBLE_SIDED: u32 = fo4_slsf2::DOUBLE_SIDED;` next to `SF2_DOUBLE_SIDED` in `material.rs:86` and route FO4 call sites through FO4 aliases; or (2) delete the FO4 modules and replace with a comment plus dedicated cross-game equality tests.

### Verified
- **Two-u32 read** at `shader.rs:604-608`, gated `bsver <= 130`.
- **BGSM/BGEM stopcond** at `material.rs:533-538` — case-insensitive suffix match on `.bgsm` / `.bgem`; runs regardless of BSVER inside the Skyrim+ `shader_property_ref` branch.
- **Parser-side FO76+ material-reference stub** at `shader.rs:586-592` gated `bsver >= 155` + non-empty name; FO4 NIFs don't take this path.
- **FO4 decal detection** at `material.rs:62-65` correctly ignores flags2 (FO4 bit 21 is `Anisotropic_Lighting`, Skyrim's is `Cloud_LOD`). Legacy path at `material.rs:50-52` preserves FO3/FNV `ALPHA_DECAL_F2` (flag2 bit 21).
- **`alpha_test`** sourced from `NiAlphaProperty.flags & 0x1` (`material.rs:1168-1171`), not shader_flags_2 — correct for FO4 which ships NiAlphaProperty alongside BSLSP.
- **`material_path` plumbing** verified via closed #491 / #492 / #494; MedTekResearch01 end-to-end render (`682b8f4`, 2026-04-23) exercises the path.

### FO4-DIM4-01: Cell loader never expands SCOL placements — mod-added SCOLs disappear
**(Same finding as FO4-DIM6-03 — canonical home here for the "parsed but not consumed" angle; see Dim 6 for the forward-chain view.)**
- **Severity**: HIGH
- **Location**: `byroredux/src/cell_loader.rs:1152-1210` (REFR base resolution); `crates/plugin/src/esm/cell.rs:426` (`EsmCellIndex.scols` populated but unread)
- **Status**: NEW (prior C2 closed the parser side as #405; cell-loader side untouched)
- **Description**: `parse_esm_cells_with_load_order` populates `EsmCellIndex.scols: HashMap<u32, ScolRecord>` with every `(ONAM, DATA)` child placement. `rg -ni scol byroredux/src/` returns zero hits — the main binary never queries this map. `records/scol.rs:26-29` references a `scol_parts` field on `StaticObject` that does not exist.
- **Impact**: Vanilla works (2616/2617 SCOLs ship `CM*.NIF` cached-combined meshes registered via the `parse_scol_group` MODL fallback). DLC + mod-added SCOLs without previsibine CM files render as nothing. Compounds once a user loads ENB or previsibine-bypass mods.
- **Suggested Fix**: In `cell_loader.rs::load_references`, after the REFR base lookup: (1) query `index.scols.get(&base_form_id)`; (2) if hit and `statics[base].model_path` is empty, iterate `scol.parts[*].placements[*]` composing `ref_transform × placement_transform`; (3) resolve each child `onam_form_id` through `index.statics` as a synthetic placement; (4) use `euler_zup_to_quat_yup` (same convention as REFR rotation at `cell_loader.rs:1182`); (5) multiply scales.

## Dimension 6 — Forward Blockers + BGSM Roadmap

BGSM/BGEM parser and template resolver have shipped (`crates/bgsm/`, ~1944 lines, v1–v22 + BGEM v1–v22, child-first walk with LRU). Integration via `MaterialProvider` at `byroredux/src/asset_provider.rs:240-344`. The merge step drops every scalar PBR field — that's the single highest-leverage gap.

### FO4-DIM6-01: BGSM scalar PBR fields parsed but never forwarded
- **Severity**: HIGH
- **Location**: `byroredux/src/asset_provider.rs:356-403` (`merge_bgsm_into_mesh`)
- **Status**: NEW
- **Description**: `merge_bgsm_into_mesh` fills the six texture-path `Option<String>` slots (`texture_path`, `normal_map`, `glow_map`, `gloss_map`, `env_map`, `parallax_map` for BGSM; `texture_path`, `normal_map`, `glow_map`, `env_map`, `env_mask` for BGEM). Every scalar PBR field the parser decodes — `emittance_color`, `emittance_mult`, `emit_enabled`, `specular_color`, `specular_mult`, `smoothness`, `fresnel_power`, `alpha`, `alpha_test_ref`, `rim_power`, `back_light_power`, wetness suite, translucency suite, hair/skin tint, refraction, SSR, mask-writes — is dropped. NIF-side scalars default to concrete values (e.g., `emissive_mult = 0.0`), so the fill predicate `if slot.is_none()` never fires for scalars.
- **Impact**: Every FO4 emissive mesh lights with wrong brightness (NIF placeholder, usually zero); metallic surfaces use NIF-fallback specular; wetness / rim / translucency paths never activate. FO4 MedTek geometry renders at 90 FPS but shading is incorrect — the "interior renders" claim holds only for geometry and base diffuse.
- **Suggested Fix**: See sketch below. Extend `merge_bgsm_into_mesh` to overwrite scalar fields on `ImportedMesh` when BGSM authored them; BGSM is authoritative per Material-Editor reference. Child-first walk already handles template inheritance. Track an `bgsm_overrides` bitset so non-FO4 meshes keep reading NIF scalars. `fresnel_power` has no `GpuInstance` slot — defer to a #344-style shader-struct-sync follow-up across triangle.vert/frag and ui.vert.

### FO4-DIM6-02: TXST.MNAM parsed but never resolved at REFR time
- **Severity**: HIGH
- **Location**: Parser at `crates/plugin/src/esm/cell.rs:1618-1624` (TXST.MNAM → `TextureSet.material_path`); storage at `cell.rs:414`; no consumer in `byroredux/`.
- **Status**: NEW
- **Description**: 37% of vanilla Fallout4.esm TXSTs (140/382) are MNAM-only — they carry no TX00 at all and rely on the BGSM resolver. Those records land in `EsmCellIndex.texture_sets` but `cell_loader.rs` never reads that map. Compounds with REFR override sub-records (XTNM/XTXR/XATO/XMSP/XEMI/XLIG/XSRF) that aren't parsed at all — `cell_loader.rs` reads only base_form_id and six geometric fields from `PlacedRef`.
- **Impact**: Weapons / armor / signage placed via REFR with a TXST override render the base mesh's textures. The 140 MNAM-only TXSTs route nowhere. XLIG alone would fix FO4 directional spotlights (vault floods).
- **Suggested Fix**: Two-stage: (1) extend `PlacedRef` with the six rendering-relevant REFR overrides (XTNM, XTXR, XATO, XMSP, XEMI, XLIG); (2) thread `EsmCellIndex.texture_sets` + `MaterialProvider` into the spawn path — resolve TXST FormID, route `material_path` through MaterialProvider, overlay 8 slot paths onto `ImportedMesh`.

### FO4-DIM6-03: SCOL runtime expander
See **FO4-DIM4-01** above — same finding. Cross-reference only.

### FO4-DIM6-05: COBJ / OMOD / MSWP / CMPO / FLST records not parsed
- **Severity**: MEDIUM
- **Location**: `RecordType::COBJ/OMOD/MSWP/CMPO/FLST` constants defined at `crates/plugin/src/record.rs:158-164`; no dispatch in `esm/cell.rs` or `esm/records/mod.rs`; `EsmIndex` has no fields.
- **Status**: NEW
- **Description**:
  - **COBJ** (Constructible Object) — settlement crafting recipes.
  - **OMOD** (Object Modification) — weapon/armor mods; ~2000 vanilla OMODs. A combat rifle with Short Barrel + Reflex Sight OMODs renders as bare.

  - **CMPO** (Component) — crafting ingredients.
  - **FLST** (Form List) — flat FormID list used by every "accepts any of N" slot.
- **Impact**: MSWP is the highest-value gap for interior rendering accuracy after BGSM scalars.
- **Suggested Fix**: Start with MSWP (single-purpose: FormID array of TextureSet pairs). OMOD/COBJ/CMPO/FLST follow the existing `items.rs` pattern.

### FO4-DIM6-06: NPC_ face-morph data unextracted
- **Severity**: MEDIUM
- **Location**: `crates/plugin/src/esm/records/actor.rs:parse_npc`. No hits for FMRI / FMRS / MSDK / MSDV / NAM9 / `face_morph`.
- **Status**: NEW
- **Description**: FO4 NPC_ carries face-morph across FMRI (morph-range index), FMRS (f32 slider), MSDK/MSDV (slider key/value pairs), NAM9 (19×f32 nose/cheek/jaw scalars), QNAM/NAMA/FTSM (face texture sets), BCLF (body color). - **Impact**: Companions (Nick, Piper, Cait) and all named NPCs look wrong; generic settlers likely OK.
- **Suggested Fix**: Extend `parse_npc` with FMRI/FMRS/NAM9 arrays; add `NpcFaceMorphs { sliders: Vec<(i32, f32)>, nose_cheek_jaw: [f32; 19] }` to `NpcRecord`. Actual morph-target application is downstream of the skinning pipeline (#178) + HDPT mesh linking.

### Fix sketch — FO4-DIM6-01 (scalar forwarding)

```rust
// byroredux/src/asset_provider.rs:356 — extension inside the BGSM arm
if lower.ends_with(".bgsm") {
    let Some(resolved) = provider.resolve_bgsm(&path) else { return false; };

    let mut set_emissive = false;
    let mut set_specular = false;
    let mut set_glossiness = false;

    for step in resolved.walk() {
        let bgsm = &step.file;

        // existing texture fills stay
        fill(&mut mesh.texture_path, &bgsm.diffuse_texture);
        // ... other texture fills ...

        // NEW: scalar overrides, child-first (first-authored wins)
        if !set_emissive && bgsm.emit_enabled {
            mesh.emissive_mult = bgsm.emittance_mult;
            mesh.emissive_color = bgsm.emittance_color;
            set_emissive = true;
        }
        if !set_specular {
            mesh.specular_strength = bgsm.specular_mult;
            mesh.specular_color = bgsm.specular_color;
            set_specular = true;
        }
        if !set_glossiness {
            mesh.glossiness = bgsm.smoothness;
            set_glossiness = true;
        }
        // mat_alpha: BGSM wins when NIF had default 1.0
        if (mesh.mat_alpha - 1.0).abs() < 1e-6 {
            mesh.mat_alpha = bgsm.base.alpha;
        }
        // UV transform: BGSM wins when NIF has identity
        if mesh.uv_offset == [0.0, 0.0] && mesh.uv_scale == [1.0, 1.0] {
            mesh.uv_offset = [bgsm.base.u_offset, bgsm.base.v_offset];
            mesh.uv_scale  = [bgsm.base.u_scale,  bgsm.base.v_scale];
        }
        mesh.two_sided |= bgsm.base.two_sided;
        mesh.is_decal  |= bgsm.base.decal;
        if bgsm.base.alpha_test {
            mesh.alpha_test = true;
            mesh.alpha_threshold = f32::from(bgsm.base.alpha_test_ref) / 255.0;
        }
    }
}
```
