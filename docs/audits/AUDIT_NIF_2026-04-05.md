## CRITICAL

### NIF-01: NiMorphData missing legacy float keys per morph frame
- **Severity**: CRITICAL
- **Dimension**: Block Parsing
- **Location**: `crates/nif/src/blocks/controller.rs:628-655`
- **Game Affected**: All games with morph targets (facial animation)
- **Status**: NEW
- **Description**: Per nif.xml, each `MorphData` frame serializes a `KeyGroup<float>` (legacy animation keys) between the frame name and the vertex delta vectors. Format: `num_keys: u32`, then if > 0, `interpolation: u32` + key array. The parser reads `name` then immediately reads vertex deltas, skipping the keys entirely. Even when `num_keys == 0`, the `num_keys` u32 field itself must be consumed (4 bytes per morph frame under-read).
- **Impact**: Every NIF with morph targets (character head meshes) will desync by 4*N bytes where N = number of morph frames. With block_size correction, only this block is wrong; subsequent blocks are safe.
- **Suggested Fix**: After reading name, read `num_keys = stream.read_u32_le()?`. If > 0, read interpolation type + float keys (same logic as KeyGroup::parse). Discard the result since interpolators drive morph weights now.

### NIF-02: Oblivion v20.0.0.5 has no block_sizes — parse errors are unrecoverable
- **Severity**: CRITICAL
- **Dimension**: Stream Position / Version Handling
- **Location**: `crates/nif/src/header.rs:114-121`, `crates/nif/src/lib.rs:71-83`
- **Game Affected**: Oblivion
- **Status**: NEW
- **Description**: Block sizes are only present for `version >= V20_2_0_7`. Oblivion's v20.0.0.5 has no block sizes. The block_size correction mechanism (the parser's safety net) cannot function. Any parse error in any block cascades into all subsequent blocks with no recovery. Combined with NIF-03 (wrong bool size) and NIF-04 (string palette format), Oblivion NIF parsing is fundamentally broken.
- **Impact**: Blocker for N23.3 (Oblivion milestone). Every Oblivion NIF with Havok collision or NiGeometry fails hard.
- **Suggested Fix**: Architectural prerequisite for Oblivion: either (a) implement byte-perfect parsers for all Oblivion block types, or (b) add a block boundary scanner using RTTI markers from the header.

## HIGH

### NIF-03: read_bool() returns u8 for Oblivion v20.0.0.5, should be u32
- **Severity**: HIGH
- **Dimension**: Version Handling
- **Location**: `crates/nif/src/stream.rs:101-109`
- **Game Affected**: Oblivion
- **Status**: NEW
- **Description**: `read_bool()` uses threshold `version >= V20_2_0_7` for u32 vs u8. Oblivion (v20.0.0.5 < V20_2_0_7) takes the u8 path. Per nif.xml, NiBool became u32 at version 4.1.0.1 — all Bethesda NIFs use u32 bools. The Gamebryo 2.3 source defines NiBool as `unsigned char`, but Bethesda's serialization uses u32.
- **Impact**: NiGeometry `has_shader` field at tri_shape.rs:125 reads 1 byte instead of 4, desyncing by 3 bytes. All subsequent fields (shader_name, implementation) are garbage.
- **Suggested Fix**: Change threshold from `V20_2_0_7` to `NifVersion(0x04010001)` (4.1.0.1) to match nif.xml. Or simply always use u32 for all target game versions.

### NIF-04: BSBehaviorGraphExtraData reads controls_base_skeleton as u32, should be u8
- **Severity**: HIGH
- **Dimension**: Block Parsing
- **Location**: `crates/nif/src/blocks/extra_data.rs:202`
- **Game Affected**: Skyrim (LE and SE)
- **Status**: NEW
- **Description**: Per nif.xml, `BSBehaviorGraphExtraData.Controls Base Skeleton` is type `byte` (1 byte). Parser reads `stream.read_u32_le()? != 0`, consuming 4 bytes instead of 1. Desyncs by 3 bytes. Block_size correction saves subsequent blocks but this block's data is wrong.
- **Impact**: Skyrim character NIFs (skeletons with Havok behavior) will have incorrect BSBehaviorGraphExtraData values. Stream position corrected by block_size.
- **Suggested Fix**: Change to `stream.read_u8()? != 0` (or `stream.read_byte_bool()?`).

### NIF-05: NiControllerSequence assumes string-table format, fails on Oblivion string-palette
- **Severity**: HIGH
- **Dimension**: Block Parsing
- **Location**: `crates/nif/src/blocks/controller.rs:311-385`
- **Game Affected**: Oblivion (.kf files)
- **Status**: NEW
- **Description**: For Oblivion (v < 20.1.0.3), ControlledBlock entries use NiStringPalette byte offsets instead of string table indices. The parser unconditionally calls `stream.read_string()` for 5 fields, which reads string table indices from a file that may have no string table. Known gap for N23.3.
- **Suggested Fix**: Add version branch: for `stream.version() < 0x14010003`, read string_palette_ref + 5 u32 offsets and resolve via NiStringPalette post-link.

## MEDIUM

### NIF-06: BsConnectPointChildren skinned flag reads u32, should be u8
- **Severity**: MEDIUM
- **Dimension**: Block Parsing
- **Location**: `crates/nif/src/blocks/extra_data.rs:361`
- **Game Affected**: Fallout 4
- **Status**: NEW
- **Description**: Per nif.xml, `BSConnectPoint::Children.Skinned` is type `byte`. Parser reads `stream.read_u32_le()? != 0`, over-reading by 3 bytes.
- **Suggested Fix**: Change to `stream.read_u8()? != 0`.

### NIF-07: BSLightingShaderProperty FO76/Starfield skips flags but continues parsing
- **Severity**: MEDIUM
- **Dimension**: Stream Position
- **Location**: `crates/nif/src/blocks/shader.rs:313-318`
- **Game Affected**: Fallout 76, Starfield
- **Status**: NEW
- **Description**: When `uses_fo76_shader_flags()` is true, shader flags are set to (0,0) without reading any bytes. FO76/Starfield use a variable-length flag array. The parser then continues reading UV offset/scale, texture_set_ref, etc. at the wrong position. All fields in this block are garbage, though block_size correction saves the next block.
- **Suggested Fix**: Either implement FO76 flag array parsing, or skip the entire block via block_size (like Havok blocks do). Current approach — read zero bytes for flags but continue parsing — is the worst option.

- **Game Affected**: Fallout 76, Starfield

- **Suggested Fix**: Same as NIF-07.

### NIF-09: BSLightingShaderProperty wetness params missing FO76+ Unknown1/Unknown2 fields
- **Severity**: MEDIUM
- **Dimension**: Stream Position
- **Location**: `crates/nif/src/blocks/shader.rs:395-396`
- **Game Affected**: Fallout 76, Starfield
- **Status**: NEW
- **Description**: BSVER > 130 has additional Unknown1 field; BSVER == 155 has Unknown2. These are not read. Combined with NIF-07, the entire BSLightingShaderProperty parse for FO76 is deeply desynchronized within the block.
- **Suggested Fix**: Covered by NIF-07 fix — skip entire block for FO76 until full support is added.

### NIF-10: NiSkinPartition is a stub — reads 4 bytes, skips rest via block_size
- **Severity**: MEDIUM
- **Dimension**: Stream Position
- **Location**: `crates/nif/src/blocks/skin.rs:167-170`
- **Game Affected**: All (skinned meshes)
- **Status**: NEW
- **Description**: Only reads `num_partitions` (4 bytes), relies on block_size correction for the rest. Partition data (bone indices, vertex maps, weights) is discarded. Acceptable for static geometry rendering but needed for GPU skinning.
- **Suggested Fix**: Acceptable for current milestone. Full parse needed for skeletal animation rendering.
