## Findings (by severity)

### NIF-D1-10: `BsPackedGeomDataCombined.Transform` reads Translation-first, spec requires Rotation-first

- **Severity**: CRITICAL
- **Dimension**: Block Parsing — Field Order
- **Game Affected**: FO4, FO76, Starfield (every terrain mesh using `BSPackedCombinedGeomDataExtra`)
- **Location**: [`crates/nif/src/blocks/extra_data.rs:662`](../../crates/nif/src/blocks/extra_data.rs#L662)
- **Status**: NEW (structural parallel to today's NiSkinData fix at `8ec6a69`)

#### Description

`BsPackedGeomDataCombined::parse()` calls `stream.read_ni_transform()` (Translation-first per nif.xml's `NiAVObject` inline layout). But nif.xml line 8377 declares the field as `type="NiTransform"` — the STRUCT, not the inline NiAVObject layout. `<struct name="NiTransform" size="52">` at nif.xml line 1808 specifies **Rotation → Translation → Scale**.

```rust
// extra_data.rs:662 — WRONG
let transform = stream.read_ni_transform()?;          // Translation-first

// Correct:
let transform = stream.read_ni_transform_struct()?;   // Rotation-first
```

This is the same bug class as the NiSkinData scrambling fixed today. Same Rust type, same wrong reader, different on-disk byte layout.

#### Why it survived previous audits

The existing test `bs_packed_combined_geom_data_extra_fully_parses_variable_tail()` (dispatch_tests.rs:1494–1590) uses identity rotation (1,0,0,0,1,0,0,0,1) + modest translation (10,20,30) + scale 1. When read with the wrong field order, the bytes happen to be semantically plausible — and the bounding sphere following immediately keeps stream position correct. Parse succeeds; values are scrambled silently.

#### Impact

Latent. M41.0 has no terrain LOD import path yet. Will manifest on M41.1+ when terrain mesh LOD lands, AND on any user mod with custom terrain using `BSPackedCombinedGeomDataExtra`. Symptom would be terrain LOD instances rendering at scrambled positions/orientations (rotation reading as translation).

#### Suggested Fix

Two-line change:
```rust
// extra_data.rs:662
let transform = stream.read_ni_transform_struct()?;
```

Plus a value-validation unit test that uses non-identity rotation and verifies the parsed struct fields match what was written. The existing round-trip test passes both pre- and post-fix because the test data was constructed identity-symmetric.

### NIF-D3-13: Unbounded inner-loop allocation in `bs_geometry.rs::parse_packed_skin_data`

- **Severity**: MEDIUM
- **Dimension**: Stream Position — Allocation Safety
- **Game Affected**: Starfield (BSGeometry skinning payload)
- **Location**: [`crates/nif/src/blocks/bs_geometry.rs:450`](../../crates/nif/src/blocks/bs_geometry.rs#L450)
- **Status**: NEW

#### Description

```rust
let weights_per_vert = stream.read_u32_le()?;         // unbounded — file-driven
for _ in 0..outer_len {                                // outer bounded via allocate_vec
    let mut row = Vec::with_capacity(weights_per_vert as usize);  // INNER UNBOUNDED
    for _ in 0..weights_per_vert {
        row.push(stream.read_f32_le()?);
    }
}
```

The outer `outer_len` is bounded through `allocate_vec` (preventing infinite loops), but `weights_per_vert` flows direct from the file to `Vec::with_capacity`. A malformed NIF with `weights_per_vert = 0xFFFFFFFF` allocates 16 GB of capacity per row — instant OOM panic before any stream read fails.

#### Impact

A corrupt or hostile NIF triggers a panic on OOM with no diagnostic — this is the same DoS vector that #764's `allocate_vec` budget guard was designed to prevent. The outer loop's guard catches one half; this inner allocation slips through.

#### Suggested Fix

Either:
1. Route through `allocate_vec`:
   ```rust
   stream.allocate_vec::<f32>(weights_per_vert as u32)?;  // budget-checked
   ```
2. Or use push-based growth (no `with_capacity`), letting `Vec::push` allocate organically — slightly slower but bound by the total stream-size limit.

~5 LOC change; companion fix to #764.

### NIF-DIM2-01: `until=` boundary uses ≤ instead of < (off-by-one in 2 sites)

- **Severity**: LOW
- **Dimension**: Version Handling
- **Game Affected**: None of the 7 targeted titles (Bethesda content never lands at the boundary versions)
- **Location**:
  - [`crates/nif/src/blocks/interpolator.rs:294`](../../crates/nif/src/blocks/interpolator.rs#L294) — `NiTransformData.Order`
  - [`crates/nif/src/blocks/properties.rs:233`](../../crates/nif/src/blocks/properties.rs#L233) — `NiTexturingProperty.Apply Mode`
- **Status**: NEW (instances of #765's pattern)

#### Description

Per niftools convention, `until="X"` means *exclusive*: the field is absent at version X exactly. Both call sites use `<=` (inclusive), reading the field at the exact boundary version. Vanilla content unaffected (Bethesda jumps don't land on these specific versions), but any boundary-exact NIF mis-parses by 4 bytes here.

#### Suggested Fix

`<=` → `<` in both sites. Add boundary unit tests.

### NIF-DIM2-02: Dead arm in `has_shader_emissive_color` includes Fallout3

- **Severity**: LOW (cosmetic)
- **Dimension**: Version Handling — Dead Code
- **Game Affected**: None
- **Location**: [`crates/nif/src/version.rs:184-189`](../../crates/nif/src/version.rs#L184-L189)
- **Status**: NEW (instance of #724's pattern)

#### Description

`has_shader_emissive_color()` includes `Self::Fallout3` in its match arm, but FO3 ships BSVER=21. The emissive-color field on `BSShaderPPLightingProperty` is gated `vercond="#BS_GT_FO3#"` (bsver > 34) per nif.xml line 6250. FO3 at 21 < 34: the field never appears in FO3 NIFs, so the predicate's FO3 arm is unreachable in practice.

#### Suggested Fix

Remove `Self::Fallout3` from the match arm; document that the real guard is `stream.bsver() > 34` and the variant predicate misleads.
