(unlike the salvaged 2026-04-22 synthesis). Git HEAD: `09dbcfc`.

FNV, Skyrim SE, FO4, FO76, Starfield. The 2026-04-22 audit deferred

parser bugs from the 2026-04-22 audit are **closed at HEAD** — the new

### Key shifts since 2026-04-22

- **All 8 prior-audit critical undispatched types are now in `parse_block`** and
  parse cleanly on the corpus they target: `bhkRigidBody`/`T`,
  `NiBoolTimelineInterpolator`, `NiAdditionalGeometryData`, `bhkBlendController`,
  `NiFloatExtraData` + `Controller`, `BSNiAlphaPropertyTestRefController`,
  `NiPathInterpolator`. Verified by issues #546–#553 + grep on the dispatch arms.
- **Stream drift is essentially eliminated** on FO3 / FNV / Skyrim SE Meshes0:
  zero `block_sizes` recovery warnings across 44,732 NIFs. The 2026-04-22
  audit's "drift is widespread" framing no longer holds.
- **The new gap surface is FO4+ era**, in two flavours:
  1. *Coverage gaps* — Starfield's `BSGeometry`, `SkinAttach`, `BoneTranslations`
     and the FO4/FO76 `BSPositionData` / `BSEyeCenterExtraData`.
  2. *Importer regressions on FO4+ data* — the parser captures CRC32 shader
     flag arrays (`sf1_crcs` / `sf2_crcs`) but no import path consumes them,
     so `is_decal` and `two_sided` silently fail on every FO76 / Starfield mesh.
- **Two regression-class bugs found during importer review** that are not in
  the issue tracker: `walk_node_lights` and `walk_node_particle_emitters_flat`
  skip `NiSwitchNode` / `NiLODNode` subtrees entirely (D4-02), and
  BSEffectShaderProperty FO4+ env-map textures are captured into the
  `effect_shader` field but never copied to `MaterialInfo.env_map` (D4-03).

## Findings — by dimension

### Dimension 1: Block Parsing Correctness

#### NIF-D1-01: NiTransformData drops legacy `Order` float for XYZ rotations on pre-10.1 NIFs
- **Severity**: MEDIUM
- **Game Affected**: Pre-Oblivion (Morrowind hybrid / NetImmerse v < 10.1.0.0)
- **Location**: [crates/nif/src/blocks/interpolator.rs:280-336](../../crates/nif/src/blocks/interpolator.rs#L280-L336)
- **Status**: NEW (related to but not the same as #697)
- **Description**: nif.xml line 4333 — `<field name="Order" type="float" cond="Rotation Type == 4" until="10.1.0.0" />`. Parser jumps from `Rotation Type` straight to the three `XYZ Rotations` `KeyGroup<float>` reads, under-reading by 4 bytes when `RotationType == XyzRotation` on pre-10.1 NIFs.
- **Suggested Fix**: `if rt == KeyType::XyzRotation && stream.version() <= NifVersion(0x0A010000) { let _order = stream.read_f32_le()?; }` before the three KeyGroup reads.

#### NIF-D1-02: NiSourceTexture missing `Use Internal` byte for pre-10.0.1.4 internal-pixel-data path
- **Severity**: MEDIUM
- **Game Affected**: Pre-Oblivion only
- **Location**: [crates/nif/src/blocks/texture.rs:38-95](../../crates/nif/src/blocks/texture.rs#L38-L95)
- **Status**: NEW
- **Description**: nif.xml line 5117 — `<field name="Use Internal" type="byte" default="1" cond="Use External == 0" until="10.0.1.3" />`. Parser misses the 1-byte `Use Internal` flag between `Use External` and the optional file-name / pixel-data ref.

#### NIF-D3-01: `BSSkyShaderProperty` + `BSWaterShaderProperty` Skyrim+ shaders aliased to FO3 PPLighting parser
- **Severity**: HIGH
- **Game Affected**: Skyrim SE, FO4
- **Location**: [crates/nif/src/blocks/mod.rs:293-300](../../crates/nif/src/blocks/mod.rs#L293-L300)
- **Status**: NEW (sibling to closed #550 — that fix only covered the FO3 `SkyShaderProperty`, not the Skyrim BS-prefixed variants)

#### NIF-D4-01: FO4+/FO76/Starfield CRC32 shader flag arrays parsed but never consumed by importer
- **Severity**: HIGH
- **Game Affected**: FO4 (BSVER 130-131 partial), FO76 (BSVER 152-155), Starfield (BSVER 172+)
- **Location**: [crates/nif/src/import/material/walker.rs:131-143](../../crates/nif/src/import/material/walker.rs#L131-L143), [walker.rs:200-213](../../crates/nif/src/import/material/walker.rs#L200-L213); [crates/nif/src/blocks/shader.rs:604-635](../../crates/nif/src/blocks/shader.rs#L604-L635)
- **Status**: NEW
- **Description**: For BSVER > 130 the parser stores `shader_flags_1` and `shader_flags_2` as **literal zeros** (`if bsver <= 130 { read } else { (0, 0) }`) and populates `sf1_crcs` / `sf2_crcs` with CRC32-hashed flag identifiers. The importer's `is_decal_from_modern_shader_flags` and `shader_flags_2 & SF2_DOUBLE_SIDED` checks read the zeroed fields, so `info.is_decal` and `info.two_sided` are **always false** on FO76/Starfield meshes.
- **Impact**: Every Starfield content NIF has backface culling on and depth-bias off regardless of authoring. FO76/SF blood splats, posters, ground decals, foliage, banners, hair all affected.
- **Suggested Fix**: Build a CRC32 → flag-bit lookup (the SLSF1 / F4SF1 / Starfield enums in [shader_flags.rs](../../crates/nif/src/shader_flags.rs) are the reference set; CRC32 over the bit name strings yields the wire values per nif.xml). Test the lookup result for `Decal`, `Dynamic_Decal`, `Double_Sided` on the BSVER >= 132 path. Could fold into the existing `is_decal_from_modern_shader_flags` helper.

#### NIF-D4-02: `walk_node_lights` and `walk_node_particle_emitters_flat` skip NiSwitchNode/NiLODNode subtrees
- **Severity**: MEDIUM
- **Game Affected**: All games (Oblivion → Starfield)
- **Location**: [crates/nif/src/import/walk.rs:485-571](../../crates/nif/src/import/walk.rs#L485-L571), [walk.rs:579-635](../../crates/nif/src/import/walk.rs#L579-L635)
- **Status**: NEW
- **Description**: `walk_node_hierarchical` (line 127) and `walk_node_flat` (line 343) explicitly call `switch_active_children` BEFORE `as_ni_node` (which returns `None` for switch/LOD nodes). `walk_node_lights` and `walk_node_particle_emitters_flat` go straight to `as_ni_node`, so any `NiPointLight` / `NiSpotLight` / `NiAmbientLight` / `NiDirectionalLight` or particle emitter parented under a `NiSwitchNode` / `NiLODNode` silently disappears.
- **Impact**: Lights and particle emitters under destruction-stage / weapon-sheath / LOD-gated subtrees drop. Lighting goes dim where the mesh visibly carries a torch; magic effect emitters under LOD nodes vanish past the first LOD switch.
- **Suggested Fix**: Mirror the `switch_active_children` block from `walk_node_flat:343-378` into both functions with appropriate transform composition.

#### NIF-D4-03: BSEffectShaderProperty FO4+ env_map / env_mask textures never reach `MaterialInfo.env_map`
- **Severity**: MEDIUM
- **Game Affected**: FO4, FO76, Starfield
- **Location**: [crates/nif/src/import/material/walker.rs:158-263](../../crates/nif/src/import/material/walker.rs#L158-L263)
- **Status**: NEW
- **Description**: Parser populates `shader.env_map_texture` / `env_mask_texture` on FO4+ BSEffectShader meshes (BSVER >= 130 fields). The importer captures them into `info.effect_shader.env_map_texture` via `capture_effect_shader_data`, but the `MaterialInfo.env_map` / `env_mask` slots — which the renderer routes through GPU instances for the standard env-map path — are not similarly forwarded. - **Impact**: FO4+ shielded / energy / chrome effect surfaces lose env-map reflection. Visible regression on Power Armor frames, energy weapons, force-field VFX.
- **Suggested Fix**: After `info.env_map_scale = shader.env_map_scale;` add `if !shader.env_map_texture.is_empty() && info.env_map.is_none() { info.env_map = Some(shader.env_map_texture.clone()); }` and the parallel `env_mask_texture → info.env_mask` copy.
