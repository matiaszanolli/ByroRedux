# Fallout 4 Compatibility Audit — 2026-04-17

**Scope**: Readiness to load and render content from Fallout 4 (NIF BSVER 130, BA2 v1/v7/v8, TES5/FO4 ESM with SCOL/PKIN/TXST records, BGSM/BGEM external materials).
**Method**: Six parallel dimension audits against the code tree and the vanilla FO4 install at `/mnt/data/SteamLibrary/steamapps/common/Fallout 4/Data/` — including all DLCs (Coast, NukaWorld, Robot, workshop01-03) and the UltraHighResolution texture pack.
**Verdict**: Parse layer is solid (226,009 NIFs at 100%, 458,617 BA2 files extracted). Two quality defects hide behind the 100% metric: every FO4 lit mesh loses a 4-byte shader field (~1.88M blocks affected), and the same unchecked-`Vec::with_capacity` OOM class Oblivion hit is latent here (73 sites). Cell loading has session-10-added record dispatch but no actual parsers, and BGSM parsing doesn't exist.

### Top blockers (priority order)

1. **[CRITICAL / Dim 1 C-1 + Dim 3 H-01 + Dim 5 H-1]** `wetness.unknown_1` BSVER gate excludes 130 (`bsver > 130` should be `bsver >= 130 && bsver < 155`). Root cause of the 1.88M 4-byte under-reads on every FO4 lit mesh. One-line fix at [shader.rs:528](crates/nif/src/blocks/shader.rs#L528).
2. **[CRITICAL / Dim 4 C1]** `cell.rs:233` has `unreachable_patterns` — `b"TXST"` duplicates line 224. Issue #378 is still OPEN. Delete `| b"TXST"` from line 233.
3. **[CRITICAL / Dim 4 C2]** SCOL records parsed as STAT — all 15,878 ONAM/DATA placement entries silently discarded. Works for vanilla only because 2616/2617 SCOLs ship a cached CM*.NIF; mods are dead.
4. **[CRITICAL / Dim 4 C3]** TXST records with MNAM→BGSM path (140/382 = 37% of vanilla TXST) silently dropped — `parse_txst_group` reads only `TX00`.
5. **[CRITICAL / Dim 1 C-2]** `BSSubIndexTriShape` segment table skipped wholesale via `block_size`. Dismemberment impossible; fragile if `block_size` ever absent.
6. **[HIGH / Dim 5 H-3, cross-ref Oblivion #388]** 73 unchecked `Vec::with_capacity(file_u32)` sites across 12 files bypass `check_alloc`. Fix by routing through `stream.check_alloc_for::<T>(count)` helper.
7. **[HIGH / Dim 5 H-2]** `NiParticleSystem` parser over-reads by ~75× (165-byte block → 12,495 bytes consumed), corrupting downstream stream state before `block_size` recovery. This is the exact vector that would turn a malformed NIF into an H-3 OOM.
8. **[HIGH / Dim 1 H-1]** Flag-pair gate at `shader.rs:411` (`bsver < 130 || bsver == 130`) vs CRC-count gate at `:427` (`bsver >= 132`) leaves BSVER 131 in no-man's-land — Next-Gen patch NIFs silently misalign.
9. **[HIGH / Dim 6]** BGSM/BGEM parser does not exist. Materials orphan-drop `rootmaterial_path` inheritance, UV offset/scale, material alpha, subsurface, rimlight, etc.
10. **[HIGH / Dim 4 H4 + Dim 6]** `crates/plugin/src/legacy/fo4.rs` is `todo!()` with zero callers — any `LegacyLoadOrder` call on `Fallout4.esm` panics.

## Dimension Findings

### Dim 1 — NIF BSVER 130 Parser Correctness

**Critical**:
- **C-1** — `wetness.unknown_1` gate at [shader.rs:528](crates/nif/src/blocks/shader.rs#L528) reads `if bsver > 130` — excludes BSVER 130 (vanilla FO4). Per nif.xml `#BS_FO4#` (BSVER ≥ 130 && BSVER < 155) the f32 is present for the whole 130–139 range. This is the 4-byte desync affecting 1.88M lit meshes per Dim 5. One-line fix: `if bsver >= 130 && bsver < 155`.
- **C-2** — `BSSubIndexTriShape` segment data wholesale-skipped via `block_size` at [blocks/mod.rs:226-236](crates/nif/src/blocks/mod.rs#L226-L236). No segment/subsegment/partition decode. Dismemberment impossible. Fragile against any future `block_size = None` path.

**High**:
- **H-1** — Flag-pair gate (`<= 130`) at [shader.rs:411-415](crates/nif/src/blocks/shader.rs#L411-L415) vs CRC-count gate (`>= 132`) at `:427` leaves BSVER 131 reading neither pair nor counts — 4-8 byte gap on Next-Gen patch NIFs.
- **H-2** — No fixture coverage for Next-Gen BSVER 131–139. Claimed 100% parse rate is error-suppressed recovery, not structural correctness.
- **H-3** — BGSM stopcond at `shader.rs:400` gates correctly at `bsver >= 155` (FO76+), but no negative test locks BSVER = 130 out of the stopcond. Loosening it by accident would turn every FO4 mesh into a material stub.

**Medium / Low**: Half-float subnormal branch untested (M-1); `VF_FULL_PRECISION` polarity correct (SET = full) but unprotected by fixture (M-3); `material_reference_stub` default-construction duplicated between BSLightingShaderProperty and BSEffectShaderProperty (L-2).

### Dim 2 — BA2 Reader

**No Critical or High findings.** Live sweep: 458,617 files across 46 archives; 185,100 brute-force extracted with zero failures. Version coverage verified on v1/v7/v8 across GNRL + DX10 variants. DDS reconstruction is byte-exact for BC1-BC7 (analytic mip-sum match). Cubemap detection via `flags & 0x1` is correct.

**Low**:
- **L1** — `linear_size_for()` at [ba2.rs:571-593](crates/bsa/src/ba2.rs#L571-L593) reports whole-payload size for non-BC formats (dxgi 87/91/28/61/56 uncovered). Harmless with DX10 extended header but fails strict validators.
- **L2** — Comment at `ba2.rs:334` says `0x0800 = cubemap?` but actual cubemap bit is `0x0001`. Code correct, comment misleading.
- **L3** — `arraySize = 1` for cubemaps in DX10 extended header; spec says 6. One-line fix.
- **L4** — Uncompressed GNRL path (`packed_size == 0`) has zero real-world test coverage (all vanilla FO4 BA2s are 100% compressed).
- **L5** — `packed / unpacked` ratio of 3.0 max on Meshes.ba2 is nonsensical for zlib; zero failures so likely a tiny-file layout quirk worth investigating.
- **L6** — Module docstring claims per-archive compression flag; actually per-chunk via `packed_size != 0`.

### Dim 3 — FO4 Shader Flags & BGSM Material Reference

**High**:
- **D3-H-01** — Cross-link to Dim 5 H-1 / Dim 1 C-1. BSLightingShaderProperty under-reads by 4 bytes on every FO4 lit mesh. Root cause: Dim 1 C-1 (`wetness.unknown_1` BSVER gate).

**Medium**:
- **D3-M-01** — No named `SLSF1_` / `SLSF2_` bitflags enum; decal / alpha-test / skinned / glow bits addressed by raw hex literals at [material.rs:280, :452](crates/nif/src/import/material.rs). FO4 flag bit positions differ from Skyrim; currently only a single decal bit is tested, so skinned / glow / alpha-tested meshes fall through the wrong shading branch.
- **D3-M-02** — BGSM stopcond at [shader.rs:400-406](crates/nif/src/blocks/shader.rs#L400-L406) correctly gates at `bsver >= 155` (FO76+). FO4 (BSVER 130) reads the full body AND exposes BGSM path as a diagnostic via `net.name` — correct design, undocumented at call-site.
- **D3-M-03** — `material_path` plumbing verified end-to-end: NIF `net.name` → `MaterialInfo.material_path` → `ImportedMesh.material_path` → `Material.material_path` → debug CLI. Chain is intact.

**Low**:
- **D3-L-01** — `mesh.info <entity_id>` debug command at [commands.rs:186](byroredux/src/commands.rs#L186) surfaces `material_path` and the FALLBACK marker for `TextureHandle(0)`. Good diagnostic pattern.
- **D3-L-02** — `TextureHandle(0)` checkerboard fallback at [texture_registry.rs:173-178](crates/renderer/src/texture_registry.rs#L173-L178). Unresolved BGSM meshes render as checkerboard-magenta, visibly flagged without crashing.

### Dim 4 — ESM Architecture Records

**Critical**:
- **C1** — `cell.rs:233` has `unreachable_patterns`: `b"TXST"` at [:224](crates/plugin/src/esm/cell.rs#L224) shadows the same literal at [:233](crates/plugin/src/esm/cell.rs#L233). Issue #378 is still OPEN. Delete `| b"TXST"` from line 233.
- **C2** — SCOL records routed to `parse_modl_group` which only reads EDID+MODL. **15,878 ONAM/DATA placement entries across 2617 vanilla SCOLs silently discarded.** Saved today by 2616/2617 shipping a cached CM*.NIF (via BSPackedCombined, #365). Mods without precomputed CM NIFs silently disappear.
- **C3** — `parse_txst_group` at [:997-1008](crates/plugin/src/esm/cell.rs) reads only `TX00`. Of 382 vanilla TXST records, 140 (37%) use `MNAM`→BGSM path only and are silently skipped.

**High**:
- **H1** — MOVS match arm is dead code — 0 MOVS records in vanilla `Fallout4.esm`. Appears only in DLCs.
- **H2** — PKIN (872 records) has no MODL subrecord; `parse_modl_group` produces nothing. CNAM→LVLI/CONT resolution not implemented.
- **H3** — No `records/scol.rs`, `records/movs.rs`, `records/pkin.rs`, `records/txst.rs` files. Session 10's "records/ module with … SCOL/MOVS/PKIN/TXST" is just a match-arm addition.
- **H4** — `crates/plugin/src/legacy/fo4.rs` still `todo!()` with zero callers. Same shape as [tes4.rs (#390)](https://github.com/matiaszanolli/ByroRedux/issues/390).

**Medium**:
- **M1** — `StaticObject` has no `scol_parts: Option<Vec<ScolPart>>` field — any future SCOL expander has nowhere to put decoded data.
- **M2** — `RecordType` enum missing `SCOL`, `MOVS` constants; walker matches raw bytes but DataStore consumers drop them.
- **M3** — Cell loader has no SCOL-expansion fallback path; grep `SCOL` in `cell_loader.rs` returns nothing.

### Dim 5 — Real-Data Validation

**Parse rate**: 226,009 / 226,009 (100.00%) across all 8 FO4 main + DLC mesh archives. 119 MB peak RSS. No OOMs, panics, or kills. Subprocess-isolated per archive so one OOM couldn't poison the sweep.

**High**:
- **H-1** — **1,876,931 `BSLightingShaderProperty` blocks silently under-read by exactly 4 bytes** across the FO4 corpus. `block_size` recovery hides it. Two "expected" sizes (140 and 146) plus exact-4-byte deltas point at one conditional u32 field — this is the same as Dim 1 C-1 (`wetness.unknown_1`). **Single highest-impact fix for FO4 rendering.**
- **H-2** — `NiParticleSystem` parser over-reads by up to 75× (12,495 bytes consumed for a 165-byte block). 1,345 occurrences in Meshes.ba2. All 10,648 `NiUnknown` recoveries in the main archive trace here. Particle FX silently dropped.
- **H-3** — **Same OOM class Oblivion hit (issue #388) is alive on FO4.** 73 unchecked `Vec::with_capacity(u32_from_file)` sites across 12 files: interpolator.rs (6), collision.rs (17), skin.rs (13), legacy_particle.rs (7), tri_shape.rs (3), plus kfm.rs, extra_data.rs, texture.rs, controller.rs, palette.rs, shader.rs, anim.rs. FO4 vendor content doesn't trip them; malformed mod or fuzzer input would.

**Medium**:
- **M-1** — `BSBehaviorGraphExtraData` over-reads by 3 bytes on every FO4 occurrence (736 warnings). Likely `controls_base_skeleton` is a u8 bool on BSVER 130+, not u32. Strings themselves parse correctly.
- **M-2** — 8,811 non-BSLSP block_size mismatches: BSEffectShaderPropertyFloatController (5,264), NiPSysRotationModifier (836), NiPSysBoxEmitter (629), NiPSysMeshEmitter (405), BSClothExtraData (309), NiStringsExtraData (285), NiPointLight (166), BSProceduralLightningController (64), BSLagBoneController (60), BSSkyShaderProperty (19), BSWaterShaderProperty (12). Each is a missing field or stolen downstream bytes.
- **M-3** — `truncated` flag never fires on FO4 (Oblivion-only path); instead 12,580 blocks across 226K NIFs silently become `NiUnknown` via `block_size` recovery. Per-NIF "clean %" metric hides this; add a per-block recovery counter to `nif_stats`.

**Low**:
- **L-1** — 8 representative probes (workshop lightbox/bell, deathclaw, supermutant torso, PA T60 body/helmet, 10mm pistol + reflex sight) trace cleanly through `import_nif_scene`: correct BGSM material paths, skin up to 30 bones, BSConnectPoint::Parents/Children extraction works.
- **L-2** — FO4-specific block types seen: BSTriShape, BSSubIndexTriShape, BSDynamicTriShape, BSPackedCombinedSharedGeomDataExtra (721K on MeshesExtra), bhkNPCollisionObject, bhkPhysicsSystem, BSSkin::BoneData/Instance, BSConnectPoint::Parents/Children. All parse without hard failure.
- **L-3** — `Fallout4 - MeshesExtra.ba2` (124,871 NIFs) is the batched-statics / LOD pack; 99.6% of H-1 under-reads are here. Fixing H-1 is the single highest-impact parser change for FO4 world rendering.

### Dim 6 — BGSM Roadmap & Forward Blockers

**BGSM/BGEM format verified** from 6,899 live samples (6616 BGSM + 283 BGEM):
- All binary. Magic `"BGSM"` or `"BGEM"` + `u32 version` (FO4=2, FO76=20, Starfield=21/22).
- 9 texture slots for BGSM (diffuse, normal, specular `_s.dds`, greyscale, envmap, glowmap, env_mask, unused×2); BGEM shares prefix then diverges to base_color + falloff.
- **`rootmaterial_path` template inheritance is mandatory** — every creature BGSM points at `template/CreatureTemplate_Wet.bgsm`. Must recurse + merge parent fields bottom-up with an LRU cache.

**Parser placement recommendation**: New `crates/bgsm/` crate. NOT `plugin::fo4` (BGSM is not an ESM record), NOT `crates/nif` (independent format/version matrix). Mirror `crates/bsa` factoring.

**BGSM Readiness Table** (selection):

| BGSM field | Current status | GpuInstance support |
|---|---|---|
| diffuse / normal / alpha_blend / alpha_test / two_sided / z_test/write / decal | NIF-covered (fallback) | Reachable |
| emissive / specular / smoothness | Reachable (via NIF Phong) | Reachable |
| **UV offset / scale** | Orphaned — Material stores it, no GPU slot | **Missing** |
| **material alpha** | Orphaned — Material stores it, no GPU slot | **Missing** |
| **specular (`_s.dds`) / glowmap / greyscale / envmap / env_mask** | Orphaned | **Missing** |
| **subsurface / rimlight / backlight / fresnel / wetness** | Orphaned | Missing |
| **hair/skin/grayscale tint** | Orphaned | Missing |
| **rootmaterial_path (inheritance)** | **None — BGSM-only** | **Missing** |

**Takeaway**: Parsing BGSM alone lands ~30% of fields on the GPU. Without parallel `GpuInstance` expansion + fragment-shader lockstep, BGSM parsing will not visibly improve FO4 renders.

**FO4 ESM coverage beyond SCOL/MOVS/PKIN/TXST**:

| Record | Missing fields | Render impact |
|---|---|---|
| REFR | XESP (enable parent), XTEL (teleport destination), XPRM (primitive bounds), XLKR (linked refs), XPOD/XRMR (portal/room) | **Doors are dead without XTEL**; quest-gated refs always visible |
| SCOL | Body (ONAM+DATA placements) | See Dim 4 C2 |
| LIGH | DATA tail (48+ bytes): fade, FOV, flicker period, intensity amplitude | Flickering torches render static |
| CONT / OMOD / MSWP | PTRN+TNAM; weapon/armor mods; material swaps | Weapon receivers/barrels render as default variant |
| NPC_ | Inline FaceGen (FMRI/FMRS) + BOD2 | Blocks NPC spawning (M41) |
| ACHR | XAPD / XLCN / XESP / XLKR | Actor linkage |

**RecordType enum gaps**: SCOL, MOVS, OMOD, MSWP, CMPO, FLST, ADDN, BNDS, TACT, TERM, IDLM all missing as constants at [record.rs](crates/plugin/src/record.rs). Cell walker matches raw bytes so parsing works, but DataStore consumers silently drop typed records.

**Havok Next-Gen**: Verified **parse-only**. BsBehaviorGraphExtraData, BSClothExtraData, BSConnectPoint::Parents/Children have no downstream consumers. Correct state — no action required for FO4 cell render.

## Regression Guard List

Patterns verified correct by this audit that must not regress:

1. **BA2 v7/v8 fall through to v1 layout** — same 24-byte base header, no `v == 2 || v == 3` extension. Confirmed on Fallout4 - Meshes.ba2 (v8) and Textures1 (v7).
2. **Cubemap detection via `flags & 0x1`** — 28 cubemaps in Textures1.ba2 correctly distinguished from 3523 non-cubemap records. `0x0800` is an always-set DX10 marker, not the cubemap bit.
3. **DDS reconstruction is 148 bytes** — DDS magic(4) + 124-byte header + 20-byte DX10 extension. Mip chain byte-exact: 512×512 BC3 10-mip extracts to 349,700 bytes = 148 + 349,552 analytic mip-sum.
4. **shader_flags_1 / shader_flags_2 u32 PAIR for BSVER 130** — two distinct 32-bit reads at [shader.rs:411-415](crates/nif/src/blocks/shader.rs#L411-L415). Skyrim LE/SE + FO4 v130 share this layout; BSVER 131+ switches to CRC32 arrays.
5. **BGSM stopcond at `bsver >= 155`** — FO76+ only. FO4 (BSVER 130) reads full body. Add negative test locking BSVER=130 + non-empty name does NOT produce `material_reference: true`.
6. **VF_FULL_PRECISION polarity** — bit SET = full f32, bit CLEAR = half f16. Matches nif.xml BSVertexDesc bit 10.
7. **material_path plumbing** — NIF `net.name` → `MaterialInfo.material_path` → `ImportedMesh.material_path` → `Material.material_path` → debug CLI. Survives both loose-NIF and cell-load paths.
8. **TextureHandle(0) checkerboard fallback** — unresolved BGSM meshes render magenta-style, visibly flagged, never crash the pipeline.
9. **BsBehaviorGraphExtraData / BSClothExtraData / BSConnectPoint parse-only** — zero downstream consumers. Intentional; do not add ECS components or Havok integration on the FO4 cell-render path.
