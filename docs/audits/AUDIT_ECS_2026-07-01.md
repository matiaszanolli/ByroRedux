# ECS Core Audit — 2026-07-01

Deep audit of the ByroRedux ECS (`crates/core/src/ecs/`) plus its cross-cutting
lifecycle, hot-path, and NIFAL-material consumers, across all 9 `/audit-ecs`
dimensions. Method per `_audit-common.md`: green-test baseline gate, then each
dimension verified top-down (lock ordering first) against **current code**, with
every candidate finding's premise re-verified before inclusion (stale-finding
hygiene). Baseline for delta focus: the clean 59/59 pass in
`AUDIT_ECS_2026-06-27.md` — extra scrutiny went to code changed since then
(the CHARAL arc, the `ActorValues` component, condition/console commands,
particle `radius_variation` forwarding).

## Executive summary

**3 findings: 0 CRITICAL · 0 HIGH · 1 MEDIUM · 2 LOW.**

The ECS core's high-blast-radius guarantees all hold: lock ordering / deadlock
protection (dim 1), storage correctness (dim 2), query borrow safety (dim 3),
scheduler access declarations (dim 5b), the transform-propagation change-
detection fast path (dim 8), and the NIFAL plain-`f32` material contract
(dim 9) are intact with no regressions. The one MEDIUM is a lifecycle wiring
gap on the **new** (post-last-audit) `ActorValues` component: it is mutable,
player-visible game state that the M45 save registry does not capture, so a
save→load cycle silently reverts runtime actor-value edits. The two LOWs are
poison-handling diagnostics deviations in `World` (`clear_entities`,
`insert_resource`) relative to the documented #466 loud-poison policy.

**Test baseline:** `cargo test -p byroredux-core` → **465 passed, 0 failed**;
`cargo test -p byroredux` → **521 passed, 0 failed** (GPU/game-data suites
ignored as designed). No pinning test regressed.

## Findings

### ECS-2026-07-01-01: `ActorValues` is mutable game state but is absent from the M45 save registry — save→load silently reverts actor-value edits
- **Severity**: MEDIUM
- **Dimension**: 7 — Component Lifecycles (save/load)
- **Location**: `byroredux/src/save_io.rs:82-90` (`MUTABLE_DELTA_COLUMNS`), `byroredux/src/save_io.rs:151-177` (`build_save_registry`), `crates/core/src/ecs/components/actor_values.rs:41-143`
- **Status**: NEW (component landed 2026-06-28 in `49d3745a`, after the last ECS audit)
- **Description**: The production `ActorValues` component (#1663) is stamped on
  every NPC placement root at spawn (`byroredux/src/npc_spawn.rs:83-101`) and is
  **mutated at runtime** by the new console commands (`setav` → `set_base`,
  `modav` → `mod_permanent`; `byroredux/src/commands/actor_value.rs:86-87`). It
  is the sanctioned composition target for perks / magic effects /
  `ModActorValue` per its module docs. But it is registered neither in
  `build_save_registry()` (11 components: Transform, Name, Parent, Children,
  Inventory, EquipmentSlots, LightSource, LightFlicker, AnimationPlayer,
  AnimationStack, ScriptTimer) nor in `MUTABLE_DELTA_COLUMNS` (the M45.1
  live-load overlay set), and it carries no serde derives — so
  `SaveRegistry::register_component` (which bounds `T: Component + Serialize +
  DeserializeOwned`, `crates/save/src/registry.rs:73-75`) cannot even accept it
  today.
- **Evidence**: `#[derive(Debug, Clone, Default)]` only on `ActorValues`
  (`actor_values.rs:66-69`); no `ActorValues` match in `save_io.rs`; runtime
  mutation path `edit_av → avs.set_base / avs.mod_permanent`
  (`commands/actor_value.rs:55-88`) operates on live entities via
  `world.query_mut`.
- **Impact**: A save→load cycle drops every permanent/temporary/damage layer
  and any console-edited base; the reloaded cell re-derives only the class
  auto-calc base at NPC respawn. Today the only mutation sources are the
  console commands, so blast radius is small — but this is the store perks and
  magic effects are specified to compose into, so the silent-revert gap widens
  with every CHARAL/gameplay system that lands. Note the component *does*
  satisfy the delta-column session-stability contract (keys are global
  load-order-space `u32` AVIF FormIDs — same rationale as `ItemStack.instance`;
  values are plain `f32` layers; no `EntityId`, no `FixedString`, no session
  handles), so registration is straightforward.
- **Related**: #1663 (component introduction); M45 save architecture
  (`docs/audits/AUDIT_SAVE_*`); `delta_columns_carry_only_session_stable_fields`
  tripwire test.
- **Suggested Fix**: Add `#[cfg_attr(feature = "save", derive(serde::Serialize,
  serde::Deserialize))]` (matching `ScriptTimer`) to `ActorValue`/`ActorValues`,
  register it in `build_save_registry()`, and add `"ActorValues"` to
  `MUTABLE_DELTA_COLUMNS` — then extend the tripwire test's pinned set.

### ECS-2026-07-01-02: `clear_entities` poison panic drops the component type name (bypasses the #466 side-table)
- **Severity**: LOW
- **Dimension**: 1 — Lock Ordering & Deadlock (poison-on-panic resolution)
- **Location**: `crates/core/src/ecs/world.rs:227-233`
- **Status**: NEW
- **Description**: `World::despawn` resolves type-erased poison through the
  `type_names` side-table so the panic names the offending component (#466,
  `world.rs:121-136`). `clear_entities` — added later for the M45 restore path
  and run on every full snapshot load — iterates `self.storages.values_mut()`
  and panics with the generic message `"storage lock poisoned during
  clear_entities"`, discarding the TypeId key it could use to resolve the name.
  It still fails loud (the policy's core guarantee holds), but loses the
  diagnostic the side-table was built to provide.
- **Evidence**:
  ```rust
  for lock in self.storages.values_mut() {
      lock.get_mut()
          .unwrap_or_else(|_| panic!("storage lock poisoned during clear_entities"))
          .clear_erased();
  }
  ```
  vs. `despawn`'s `self.storages.iter_mut()` + `self.type_names.get(type_id)` +
  `storage_lock_poisoned_erased(type_name)`.
- **Impact**: On a poisoned-lock cascade during a save load, the panic cannot
  say *which* storage poisoned — the exact "10× harder bisect" #466 fixed for
  `despawn`. Post-panic-only path, so no runtime correctness impact.
- **Related**: #466; `despawn_poisoned_lock_panics_with_type_name`
  (`world_tests.rs:1256`).
- **Suggested Fix**: Iterate `self.storages.iter_mut()`, resolve the name via
  `self.type_names.get(type_id)`, and call `storage_lock_poisoned_erased(name)`
  — a three-line change mirroring `despawn`.

### ECS-2026-07-01-03: `insert_resource` silently swallows a poisoned prior-value lock instead of re-panicking loud
- **Severity**: LOW
- **Dimension**: 4 — Resource Lifetimes (poison-on-panic resolution)
- **Location**: `crates/core/src/ecs/world.rs:541-552`
- **Status**: NEW
- **Description**: The documented poison policy (dim 1 checklist; pinned by
  `poisoned_resource_lock_panics_with_type_name`) is that a post-panic lock
  access re-panics loud with the type name. `remove_resource` follows it
  (`resource_lock_poisoned::<R>()`, `world.rs:555-561`). `insert_resource`
  does not: when replacing an existing resource whose lock was poisoned, the
  chain `old.and_then(|lock| lock.into_inner().ok().and_then(|boxed|
  boxed.downcast::<R>().ok()).map(|b| *b))` maps the `PoisonError` to `None` —
  indistinguishable from "no prior value existed". The `.downcast::<R>().ok()`
  also swallows a type mismatch that `remove_resource` `expect`s on (unreachable
  in practice since the map is TypeId-keyed, but inconsistent defense).
- **Evidence**: `world.rs:542-552` vs. `remove_resource` at `world.rs:555-561`.
- **Impact**: Post-panic-only. No torn-state read occurs (the old value is
  dropped, not dereferenced, and the replacement installs a fresh un-poisoned
  `RwLock`), but a recovery path that replaces a resource after a caught panic
  gets a silent `None` where the sibling API panics loud — masking the original
  failure. Purely a diagnostics/consistency gap under the fail-fast policy.
- **Related**: #466 (loud-poison policy), TS-08/#1412 (fail-fast); finding
  ECS-2026-07-01-02.
- **Suggested Fix**: Resolve the old lock via
  `.unwrap_or_else(|_| resource_lock_poisoned::<R>())` and the downcast via
  `.expect("resource type mismatch")`, matching `remove_resource`.
