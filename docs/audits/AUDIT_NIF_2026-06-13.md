## Executive Summary

The NIF parser is in strong shape. **Block-type dispatch coverage is 100% for all seven games** — there are zero genuinely-missing dispatch arms. Five of the seven dimensions that own runtime correctness came back essentially clean; all regression-guard pins (Session 34/35 splits, allocation hygiene, NIFAL boundary, collision translation) hold.

**Corrected brief premises** (verification discipline caught both):
1. The "Oblivion `NiStringExtraData` 65 undispatched" reading was wrong — `NiStringExtraData` **is** dispatched (`blocks/mod.rs`, #1073); the "65" is the *recovered* count, an artifact of how `nif_stats` buckets runtime-size-cache recoveries.
2. The Oblivion truncation is therefore **not** a coverage gap. It is byte-stride drift inside three *dispatched* parser families on Oblivion's pre-20.x sub-versions (10.1.x / 10.2.0.0), correctly caught by the alloc-cap / plausibility guards but unrecoverable in the sizeless format.

**The one real correctness theme** is a tightly-coupled cluster: the parser has no version-predicate surface for the `10.1.0.104 .. 10.2.0.0` file-version sub-band, so a handful of `since/until`-gated fields (the `NiInterpController.Manager Controlled` bool, plus particle/interpolator/morph stride fields) are either never read or can't be gated correctly. This truncates **~56 of 8,032 vanilla Oblivion mesh files** — a narrow but real blast radius (first-person skeleton, lockpicking/menu animation rigs, particle FX, creature facial morphs). No corruption or OOM (the guards hold); the affected files keep their head and drop the tail.

A second, previously-unsurfaced signal: the full-data-dir baseline records **1,036 `NiUnknown` on the Starfield corpus (0.13%)** from three *dispatched* parsers failing on Starfield-specific layouts (`BSEffectShaderProperty`, `BSClothExtraData`, `NiPointLight`). Root cause is not yet isolated — flagged for a follow-up byte-audit.

## Findings

### HIGH

#### NIF-NEW-01: `NiInterpController.Manager Controlled` bool never read on v10.1.0.104–108 → controller-chain drift on the first-person skeleton
- **Severity**: HIGH (drops the entire animation-controller chain of affected files; narrow band)
- **Dimension**: Block Parsing + Version Handling (coupled — field-read fix is blocked on the version-predicate plumbing)
- **Game Affected**: Oblivion (the v10.1.0.x sub-corpus). Vanilla hit: `meshes\characters\_1stperson\skeleton.nif` (file-version 10.1.0.106). Any NIF in `[10.1.0.104, 10.1.0.108]` carrying NiInterpController-derived controllers.
- **Location**: field read — `crates/nif/src/blocks/controller/mod.rs:227-241` (`NiSingleInterpController::parse`) and every NiInterpController descendant (`NiMultiTargetTransformController` in `controller/sequence.rs`, `NiGeomMorpherController` in `controller/morph.rs`); version surface — `crates/nif/src/version.rs` (no `V10_1_0_108` constant, no 104–108 range predicate).
- **Status**: NEW (absorbs agent IDs NIF-D1-NEW-01 + NIF-D2-NEW-01 — same logical fix in two layers)
- **Description**: nif.xml:3615 defines `NiInterpController."Manager Controlled" : bool` `since="10.1.0.104" until="10.1.0.108"`. The parser collapses NiInterpController into `NiSingleInterpController` and reads only the `interpolator_ref` (gated `>= V10_1_0_104`); the preceding 1-byte bool is never read. version.rs jumps `V10_1_0_104 → V10_1_0_106 → V10_1_0_113` with **no `0x0A01006C` (108)** constant, so even a corrected field-read has no comparator to express the `until=10.1.0.108` upper bound.
- **Evidence**: Byte-trace of block 5 `NiTransformController` in `_1stperson\skeleton.nif` (v10.1.0.106): base 26 B ends at target@1362=2, then **Manager Controlled bool@1366=0 [UNREAD]**, interpolator@1367. Parser ends at 1370; correct end 1371. The next block's `name_len@1375=14` ("Bip01 NonAccum") confirms the exact 1-byte shortfall. A per-file sweep showed `_1stperson\skeleton.nif` is the ONLY Oblivion mesh with `recovered>0` (135) — the DIGEST's "NiNode=70 / NiStringExtraData=65 / NiMaterialProperty=1 / NiTexturingProperty=1" recoveries are all this one file's controller-drift cascade tail.
- **Impact**: 1-byte/controller drift accumulates across the controller-dense skeleton until a downstream block reads a garbage length and recovers; 135 blocks become opaque `NiUnknown` — the first-person view-model skeleton loses every animation-controller linkage. Confined to the 104–108 band (1 vanilla file today; widens for modded / early-Gamebryo content).
- **Suggested Fix**: (1) Add `pub const V10_1_0_108: Self = Self(0x0A01006C);` to version.rs. (2) Add a range predicate `has_interp_controller_manager_controlled(self) -> bool { self >= V10_1_0_104 && self <= V10_1_0_108 }`. (3) Read the 1-byte bool through that helper in a shared `NiInterpController` base layer used by all descendants. The version-gate plumbing must land first.

#### NIF-NEW-02: NiPSysData / particle-emitter stride drift on v10.2.0.0 — ~27 truncated Oblivion FX scenes
- **Severity**: HIGH (drops the particle subtree + everything after it)
- **Dimension**: Stream Position (field-read/stride) + Version Handling (comparator surface)
- **Game Affected**: Oblivion only (v10.2.0.0, sizeless format).
- **Location**: `crates/nif/src/blocks/particle.rs::parse_particles_data` (and the shared `parse_geometry_data_base` / `parse_psys_geometry_data_base`); surfaces as truncation in `crates/nif/src/lib.rs::parse_nif` Err-branch. Comparator surface: `crates/nif/src/version.rs` (missing `V10_1_0_108/110/111`).
- **Status**: NEW (absorbs NIF-D3-NEW-08 + the v10.2 half of NIF-D2-NEW-02)
- **Description**: On Oblivion particle NIFs, `NiPSysData` over/under-consumes, leaving the stream misaligned; the following `NiPSys*Emitter` reads a string-length off the misaligned offset, reinterprets ASCII as a u32 count, trips the 268 MB alloc cap, and returns Err. Sizeless format → no realignment → truncate.
- **Evidence**: `meshes\effects\metalsparks.nif` (v10.2.0.0): block 30 `NiPSysData` consumed 4900 B, block 32 `NiPSysBoxEmitter` then requested a **1,752,457,573-byte** allocation = `0x68746165` = `"eath"` (tail of an `...AgeDeath` name) — DISCARDING 40 blocks. A static walk from the believed offset yields a clean empty-name emitter, confirming the drift entered upstream at block 30/31.
- **Impact**: ~27 of 56 truncated Oblivion scenes (23 on `NiPSysBoxEmitter`, +3 other emitter types, +1 `NiPSysPositionModifier`). Concentrated in `effects/`, `magiceffects/`, dungeon FX.
- **Suggested Fix**: Byte-audit `parse_particles_data` + the shared geometry-data base against nif.xml for the **v10.2.0.0 / non-BS202** path specifically (existing gates were tuned for Oblivion 20.0.0.4 and FNV+ 20.2.0.7; the 10.x sub-versions are under-tested). `NiParticleInfo` stride (40 B ≤10.4.0.1) is correct per nif.xml:2263 — the drift is upstream in the geometry-data base or a has-flag array. Add the missing version constants (NIF-NEW-01 shares `V10_1_0_108`) and a v10.2 NiPSysData round-trip fixture.

#### NIF-NEW-03: Interpolator / NiControllerSequence stride drift on v10.1.0.x — ~13 truncated Oblivion menu/animation scenes
- **Severity**: HIGH (drops the animation-controller subtree)
- **Dimension**: Stream Position + Version Handling
- **Game Affected**: Oblivion only (v10.1.0.106 and siblings, sizeless format).
- **Location**: `crates/nif/src/blocks/interpolator.rs` (NiBlendTransformInterpolator / controller-manager interpolator path); surfaces at the subsequent `NiControllerSequence` in `lib.rs::parse_nif`. Comparator surface: `version.rs` (`V10_1_0_108/110/111`; nif.xml:1928-1936 NiControllerSequence string set `since=10.1.0.104 until=10.1.0.113`, :3327-3335 NiBlendInterpolator `Single Interpolator since=10.1.0.108 until=10.1.0.111`).
- **Status**: NEW (absorbs NIF-D3-NEW-09 + the v10.1 half of NIF-D2-NEW-02)
- **Description**: An interpolator block in the `NiControllerManager` chain under/over-consumes; the next `NiControllerSequence` reads `name` length off a misaligned offset, gets garbage, trips the alloc cap, truncates.
- **Evidence**: `meshes\menus\lockpicking\pickold.nif` (v10.1.0.106): block 5 `NiBlendTransformInterpolator` reported consumed=64, block 6 `NiControllerSequence` then requested **4,294,934,527** bytes — DISCARDING 42 blocks. Static walk: the first real `0xFFFF7FFF` is ~640 bytes downstream of the believed block-6 start, confirming block 5 under-consumed.
- **Impact**: 13 of 56 truncated scenes, concentrated in `menus/lockpicking/` (tumbler/bolt/pick rigs) and `marker_*` UI meshes.
- **Suggested Fix**: Byte-audit the `NiControllerManager → NiTransformController → NiBlendTransformInterpolator → NiControllerSequence` chain against nif.xml for **v10.1.0.x** (the interpolator base and NiControllerSequence header carry version-gated fields — `Managed Controllers`, accumulation flags, weighted/cycle ordering — that shift between 10.1.x and 20.x). Shares the version-constant prerequisite with NIF-NEW-01/02. Add a v10.1.0.106 controller-chain fixture.

### MEDIUM

#### NIF-NEW-04: NiMorphData / NiGeomMorpherController stride drift — ~9 truncated Oblivion creature-head scenes
- **Severity**: MEDIUM (drops morph-animation blocks; smaller cluster)
- **Dimension**: Stream Position
- **Game Affected**: Oblivion only.
- **Location**: `crates/nif/src/blocks` NiMorphData / NiGeomMorpherController parsers (Oblivion path); surfaces in `lib.rs::parse_nif`.
- **Status**: NEW (NIF-D3-NEW-10)
- **Description**: `NiMorphData` trips its own num_morphs/num_vertices plausibility guard, or `NiGeomMorpherController` fails "failed to fill whole buffer" — stream drift caught by sanity guards rather than the alloc cap. Sizeless → truncate.
- **Evidence**: `sweep_oblivion.err`: `NiMorphData: implausible num_morphs=1535393654 num_vertices=568017444 — DISCARDING 10 blocks`; `NiGeomMorpherController (consumed 3366): failed to fill whole buffer — DISCARDING 12 blocks`. 8 morph-data + 1 morpher-controller failures.
- **Impact**: ~9 of 56 truncated scenes — creature facial morph rigs (goblinhead, minotaur head, doghead, mountainlion). The plausibility guard is doing its job; the fix is upstream stride correctness.
- **Suggested Fix**: Byte-audit `NiMorphData` (morph-key array stride, per-frame key layout) and `NiGeomMorpherController` against nif.xml for the Oblivion version band.

#### NIF-NEW-05: Starfield — 1,036 NiUnknown (0.13%) from dispatched parsers failing on Starfield layouts
- **Severity**: MEDIUM (Starfield is a stretch-target game, not a primary; but 665 effect-shader + 298 cloth + 18 light blocks drop)
- **Dimension**: Block Parsing (root cause not yet isolated — flagged for follow-up byte-audit)
- **Game Affected**: Starfield (bsver 172).
- **Location**: `crates/nif/src/blocks/shader.rs` (`BSEffectShaderProperty`), `crates/nif/src/blocks/extra_data.rs` (`BSClothExtraData`), `crates/nif/src/blocks/node.rs` (`NiPointLight`) — all dispatched, all failing mid-parse on the Starfield field layout.
- **Status**: NEW (surfaced empirically by `block_coverage_baselines.rs`; no dimension agent root-caused it — Dims 1/3 focused on Oblivion)
- **Description**: The full-data-dir baseline records 1,036 `NiUnknown` on the Starfield corpus: `BSEffectShaderProperty` 83 parsed / **665 unknown**, `BSClothExtraData` 0 / **298**, `NiPointLight` 8 / **18**. These are dispatched types whose parsers don't match the Starfield (bsver-172) layout, so they fail and fall to `NiUnknown`. This is the single largest block of parse failures in the entire corpus.
- **Evidence**: `crates/nif/tests/block_coverage_baselines.rs` Starfield row (770,322 blocks, 1,036 NiUnknown, 99.8655%).
- **Impact**: Starfield effect-shader surfaces (the bulk of the 665) render without their authored material; cloth and point-light data dropped. Bounded because Starfield isn't a primary compat target yet, but it's the highest-count correctness gap in the corpus.
- **Suggested Fix**: Byte-audit `BSEffectShaderProperty::parse` for the bsver-172 band first (highest count), then `BSClothExtraData` and the Starfield `NiPointLight` layout, against nif.xml. Likely a missing FO76→Starfield field-band split mirroring the existing `parse_fo76_plus` pattern in `shader.rs`. Add a Starfield fixture or extend the baseline assertion downward as each is fixed.

### LOW

#### NIF-NEW-06: Six NifVariant feature-flag helpers have zero production call sites (architectural foot-gun)
- **Severity**: LOW (no mis-parse today; maintenance/regression-surface risk)
- **Dimension**: Version Handling
- **Location**: `crates/nif/src/version.rs` — `has_dedicated_shader_refs` (:503), `uses_bs_lighting_shader` (:518), `uses_fo4_shader_flags` (:588), `uses_fo76_shader_flags` (:594), `has_dynamic_effect_fields` (:635), `has_legacy_binary_extra_data` (:653).
- **Status**: NEW (NIF-D2-NEW-03)
- **Description**: These six helpers are exercised only by unit tests; every real parser queries `stream.bsver()` directly (deliberately — the helpers key off the coarse variant and mis-classify transitional `v20.2.0.7/bsver≤26` and `bsver==0` exports). The #938 cleanup already deleted three zero-call-site helpers for this exact reason. Worse, `uses_fo4_shader_flags()` returns true for the entire `Fallout4` variant including in-file bsver 132/139, while the real parser (`shader.rs:411`) switches to CRC arrays at `bsver >= 132` — adopting the helper would invert the FO4-DLC shader-flag layout.
- **Evidence**: `grep -rn '.uses_fo4_shader_flags()' crates/` → only the `fn` def + tests. `shader.rs:411 if bsver < bsver::FO4_CRC_FLAGS` is the authority; the helper disagrees for bsver 132/139.
- **Suggested Fix**: Delete the six unused helpers (matching the #938 precedent), or annotate each loudly as "NO production call site — parsers MUST use stream.bsver()". Do NOT add call sites for `uses_fo4_shader_flags` / `uses_fo76_shader_flags` — they'd mis-gate the 132 CRC boundary.

#### NIF-NEW-07: Ten residual bare-decimal BSVER literals where named `bsver::*` constants exist

- **Dimension**: Version Handling
- **Location**: `particle.rs:350,971,974,1024,1156,1165` and `tri_shape/bs_tri_shape.rs:263` (7 substantive); `controller/mod.rs:440`, `ni_tri_shape.rs:408`, `particle.rs:1095` (3 `> 0` / `== 0` "is-Bethesda?" sentinels).

- **Evidence**: `grep -rnE 'bsver\(\)\s*[<>=!]+\s*[0-9]+' crates/nif/src/blocks/` → 10 hits.
- **Suggested Fix**: Mechanical substitution to the named constants. Closes the #1336 residual.

#### NIF-NEW-08: `d5_coverage` example `DISPATCH_KEYS` snapshot is stale (over-reports uncovered types)
- **Severity**: LOW (audit tooling, not parser)
