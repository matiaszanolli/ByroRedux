## Regression Check: `#1823` (prior HIGH, FO4-D2-01)

The prior pass's HIGH finding (`FO4-D2-01`) was that `#1651`'s
`gl_to_gamebryo_blend` 0↔1 swap corrupted the two real authored non-trivial
BGSM/BGEM blend modes (Additive, Multiplicative), based on a false "GL-style
enum" premise. That finding was filed and fixed same-day as commit `27334481`
("Fix #1823: remove wrong 0/1 blend-factor swap..."), which lands in the
`1b4e8e84..8498e559` range this pass covers. Re-verified this pass:

- `byroredux/src/asset_provider/material.rs:495-513` — the function is now
  named `bgsm_blend_to_gamebryo` (renamed from `gl_to_gamebryo_blend`) and is
  a pure `raw as u8` narrowing cast with **no swap**. The doc comment
  explicitly derives the three authored tuples from the reference parser
  (`Material-Editor/MaterialLib/BaseMaterialFile.cs::ConvertAlphaBlendMode`):
  Standard `(6,7)`, Additive `(6,0)`, Multiplicative `(4,1)`, and explains why
  the swap was wrong and must not be reintroduced.
- Call sites at `material.rs:947-948` (BGSM) and `:1043-1044` (BGEM) both call
  `bgsm_blend_to_gamebryo` directly on the parsed `src_blend`/`dst_blend`.
- `byroredux/src/asset_provider/tests.rs:838-946` — the additive/multiplicative
  test fixtures were replaced with the real authored tuples
  (`function=1, src=6, dst=0` for Additive; `function=1, src=4, dst=1` for
  Multiplicative), each asserting the correct resolved `vk::BlendFactor`
  (`ONE` / `ZERO` respectively) instead of the synthetic `(2,1,1)` tuple that
  previously green-lit the wrong translation.
- `grep -rn "gl_to_gamebryo_blend"` across the tree returns zero hits — the
  old name (the regression signature the skill calls out to watch for) does
  not exist anywhere.

**Verdict: fixed, correctly, with test coverage. Not re-listed as a finding.**
