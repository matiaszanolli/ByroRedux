## D7-RAPIER-LEAK: Cell unload leaks Rapier bodies/colliders (despawn frees the ECS row, never the physics body)

- **Severity**: HIGH
- **Dimension**: 7 — Streaming, Scripting & New Component Lifecycles
- **Location**: leak site `byroredux/src/cell_loader/unload.rs:179-182`; origin
  `byroredux/src/cell_loader/spawn.rs:93` + `:1062-1081` (static-trimesh ghost
  spawn); registration `crates/physics/src/sync.rs:262,285,330-332`; missing API
  `crates/physics/src/world.rs` (no removal method).
- **Status**: NEW (no physics/rapier leak in open issues or prior audits)
- **Description**: The streamed-cell path creates physics entities carrying
  `CollisionShape::TriMesh` + `RigidBodyData` + `GlobalTransform`
  (`synthesize_static_trimesh`, used for static architecture meshes that didn't
  get a bhk collider). `physics_sync_system` (registered in `byroredux/src/main.rs:821`,
  runs every frame) registers each such newcomer into `PhysicsWorld.bodies`
  (`RigidBodySet`) and `PhysicsWorld.colliders` (`ColliderSet`), attaching a
  `RapierHandles` component. On cell unload, `unload_cell` despawns the victim
  entities with `world.despawn(eid)`, which removes only the ECS component rows
  — including `RapierHandles`, whose handle is dropped **without being used**.
  Nothing removes the body/collider from the Rapier sets.
- **Evidence**:
  - `PhysicsWorld` public API has no removal method (`new`, `body_count`,
    `awake_counts`, `wake`, `step`, `update_query_pipeline`, `cast_ray_down`,
    `static_colliders_aabb`, `move_character` — verified `crates/physics/src/world.rs`).
  - Repo-wide grep for `remove_rigid_body` / `bodies.remove` / `colliders.remove`
    / `remove_collider` over `crates/physics` + `byroredux/src/cell_loader` +
    `streaming.rs` → **zero** matches.
  - `unload.rs:180-182`: `for eid in victims { world.despawn(eid); }` — no
    physics release pass (unlike the mesh/texture/terrain-tile/skin-slot/
    item-instance release passes that precede it in the same function).
  - Ghost entities are spawned inside the cell's `first..last` span and so
    receive `CellRoot`/`CellRootIndex` membership → they ARE collected as unload
    victims and their ECS rows are freed.
- **Impact**: Every cell crossing leaks one Rapier `RigidBody` + its colliders
  per static trimesh ghost (plus any bhk-collider / keyframed bodies). Cadence
  is per-cell-crossing, not per-frame, but it is monotonic and unbounded over a
  session — most damaging for exterior radius streaming, which incrementally
  unloads cells with no full `PhysicsWorld` reset. Beyond memory, the dead fixed
  bodies remain in the broad-phase / `QueryPipeline` BVH, so the per-frame
  physics step + query-pipeline rebuild cost climbs with the total number of
  cells ever visited — a session-length CPU regression (CPU bottleneck = bug,
  per project constraints). This is the one external resource `unload_cell`
  forgot to release.
- **Related**: parallels the GPU mesh/texture (#637) and skin-slot release
  passes already in `unload_cell`; lifecycle dimension explicitly lists
  "verify no orphaned CharacterController/RapierHandles after a cell load/unload
  cycle." The ECS row is not orphaned — the external Rapier resource it pointed
  to is.
- **Suggested Fix**: Add `PhysicsWorld::remove_body(&mut self, handle)` that
  calls `RigidBodySet::remove(body, &mut islands, &mut colliders,
  &mut impulse_joints, &mut multibody_joints, /*wake_dependents=*/true)`
  (Rapier cascades attached colliders, so removing the body suffices). In
  `unload_cell`, before the despawn loop, read each victim's `RapierHandles`
  and call it, gated on `try_resource_mut::<PhysicsWorld>()` (matches the
  optional-resource pattern used for the item-instance pool). Add a regression
  test mirroring the skin-cleanup test: assert `pw.body_count()` returns to
  baseline after a load→unload cycle. The character capsule
  (`scene.rs`, `CharacterKinematic`) is World-scoped and correctly excluded —
  it has no `CellRoot`, so it is never an unload victim.

## ECS-DOC-01: dimension-5b / memory claim of `AccessConflict::Parallel` is stale — undeclared pairs are `Unknown`

- **Severity**: LOW
- **Dimension**: 5b — M27 parallel scheduler access declarations
- **Location**: `.claude/commands/audit-ecs/SKILL.md` (dimension 5b text);
  code of record `crates/core/src/ecs/access.rs:139-152` + `scheduler.rs:660-666`.
- **Status**: NEW (doc rot)
- **Description**: Dimension 5b (and the mirrored project-memory note) states
  that under #1394 systems with undeclared access are "classified as
  `AccessConflict::Parallel` — NOT `Unknown`" and that `unknown_pair_count()`
  "will return 0 when all undeclared pairs are now tagged Parallel." This does
  not match the code: the `AccessConflict` enum has only `None`, `Unknown`, and
  `Conflict` — there is **no** `Parallel` variant. `analyze_pair` returns
  `Unknown { left_undeclared, right_undeclared }` for any undeclared side, and
  the test `undeclared_closure_pairs_show_as_unknown` (`scheduler.rs:1100`)
  asserts two undeclared closures produce `unknown_pair_count() == 1`.
- **Evidence**: `git show a7e1502b` (the #1394 commit) added the
  `undeclared_parallel_count()` accessor — that is the actual "undeclared-parallel
  guard," not a reclassification of pairs to `Parallel`. No `Parallel` token
  exists in `access.rs`/`scheduler.rs`.
- **Impact**: None to runtime. A future audit trusting the 5b wording would look
  for a nonexistent variant and could misread the conflict report. The
  underlying behavior (undeclared pairs remain visible as `Unknown`) is correct
  and safe.
- **Suggested Fix**: Edit dimension 5b (and the `command_system` / scheduler
  memory) to say undeclared pairs classify as `AccessConflict::Unknown`, and that
  #1394 added `undeclared_parallel_count()` as the migration KPI driven toward 0.

## MAT-DOC-01: Starfield `.mat` arm comment references a deleted shader-side `classify_pbr` fallback

- **Severity**: LOW
- **Dimension**: 9 — NIFAL canonical material in the component layer
- **Location**: `byroredux/src/asset_provider.rs:1020-1022`
- **Status**: NEW (doc rot)
- **Description**: The Starfield `.mat` arm returns `true` without setting
  `metalness_override`/`roughness_override` (unlike the BGSM arm), and its
  comment says "the shader's legacy `classify_pbr` fallback handles the missing
  override gracefully." There is no shader-side / render-side `classify_pbr`
  fallback anymore — it was removed when the canonical resolve-once contract
  landed.
- **Evidence**: render path reads `m.roughness`/`m.metalness` directly
  (`render/static_meshes.rs:314-315`); every `classify_pbr` token there is in a
  comment documenting the removed fallback. The actual gap-fill is parse-time:
  unset override → `None` → `f32::NAN` → `Material::resolve_pbr`'s NaN-sentinel
  classifier arm at the `translate_material` boundary.
- **Impact**: None to runtime — Starfield `.mat` content still routes correctly
  through `translate_material`/`resolve_pbr`. Only the comment misdescribes
  *where* the gap-fill happens.
- **Suggested Fix**: Reword the comment to point at the `resolve_pbr` NaN-sentinel
  backstop at the translation boundary. No code change.
