## Dimension 1: Storage Correctness

### E-04: PackedStorage insert/remove is O(n)
- **Severity**: LOW
- **Location**: `crates/core/src/ecs/packed.rs:38`, `:47`
- **Status**: NEW
- **Description**: `Vec::insert(idx, ..)` / `Vec::remove(idx)` shift every
  element after `idx`. For Transform with thousands of entities, each
  insert/remove is O(n). Sort order is maintained correctly — this is a perf
  hardening note, not a correctness bug.
- **Impact**: Measurable only during bulk scene load (BSA/NIF import).
  Steady-state frames do not hit this.
- **Suggested Fix**: If profiling shows a hotspot, add a batched
  `extend` + `sort_unstable_by_key` path for bulk import. Do NOT switch to
  swap-remove — that would break the sort invariant.

**Other storage checks — confirmed correct:**
- `SparseSetStorage::remove` fixes the **moved** entity's sparse pointer
  (`sparse_set.rs:75`).
- `SparseSetStorage::insert` overwrites existing rows in place
  (`sparse_set.rs:40-41`), no duplicates.
- `SparseSetStorage::remove` of the last slot short-circuits without swap
  (`sparse_set.rs:60-64`).
- `PackedStorage::insert` binary-searches and inserts at the sorted position;
  overwrite is in place (`packed.rs:34`).

## Dimension 2: Query Safety

All paths sound. Specifically verified:

- `QueryRead<'w, T>` owns `RwLockReadGuard<'w, Box<dyn DynStorage>>`; the
  `'w` lifetime keeps the wrapper from outliving `&World`.
- `QueryWrite<'w, T>` holds `RwLockWriteGuard` — exclusive by construction.
- `query_2_mut` / `query_2_mut_mut` lock in **TypeId-ascending order**
  regardless of generic-parameter order (`world.rs:332-363`, `:394-424`).
  Both tracker scopes AND real locks follow the same order (the #313 fix).
- Same-type double-lock panics at the `assert_ne!` at the top of both
  methods (`world.rs:315`, `:381`); never reaches the lock site.
- Same-thread reentrant deadlock (read-then-write on the same type) panics
  via `lock_tracker` in debug AND release builds (the thread-local tracker
  at `lock_tracker.rs:54` is unconditional).
- `Deref`/`DerefMut` on all query/ref wrappers is sound: each returns a
  reference borrowed from `&self` / `&mut self`, so the Rust borrow checker
  prevents the unsoundness of the pre-#35 raw-pointer `World::get`.
- `ComponentRef::new` pre-checks `contains(entity)` (`query.rs:225`), so the
  deref `expect` on `get(entity)` is infallible while the read lock is held.

## Dimension 3: World Integrity

### E-03: despawn erased-poison panic loses type name
- **Severity**: LOW
- **Location**: `crates/core/src/ecs/world.rs:113-115`
- **Status**: NEW
- **Description**: Inside `despawn`, the poisoned-lock helper receives
  `"<unknown>"` even though the loop variable holds the `TypeId`. Every other
  poison path in the file surfaces the concrete type name via
  `std::any::type_name::<T>()`.
- **Impact**: Harder diagnostics if a prior system panic poisons a storage
  lock AND a subsequent despawn trips the helper. Rare compound failure.
- **Suggested Fix**: Plumb a `TypeId → &'static str` side-table through
  `register`/`storage_write`, or at minimum panic with the `TypeId` value so
  the user can grep for it.

**Other world checks — confirmed correct:**
- `storage_write<T>` downcasts via `as_any_mut().downcast_mut::<T::Storage>()`
  with explicit `expect` (`world.rs:620-621`); cannot mistype because the
  key is always `TypeId::of::<T>()`.
- Lazy init: first `insert<T>` creates the storage; pre-insert `query<T>` /
  `query_mut<T>` return `None` without creating empty storage.
- `spawn()` uses `checked_add(1)` with explicit panic-on-overflow
  (`world.rs:82-83`). IDs are never reused (#36, #372).
- `find_by_name` resolves through `StringPool`, then compares `Name(Symbol)`
  by integer equality — no string comparison in the scan (`world.rs:233-239`).

## Dimension 4: Resource Safety

### E-02: resource_2_mut has no `try_` variant
- **Severity**: LOW
- **Location**: `crates/core/src/ecs/world.rs:514-571`
- **Status**: NEW
- **Description**: `resource<R>()` and `resource_mut<R>()` each have `try_`
  Option-returning siblings. `resource_2_mut<A,B>()` does not — it panics on
  missing resource. Systems wanting two resources conditionally must fall
  back to sequential `try_resource_mut` + `try_resource_mut`, which loses
  the TypeId-sorted ordering guarantee (the ABBA hazard #313 guards against).
- **Impact**: None at present — all callers always have both resources
  installed at bootstrap. Future optional-resource pairings would have to
  choose between a panic and losing the ordered-lock guarantee.
- **Suggested Fix**: Add `try_resource_2_mut<A, B>()` mirroring the existing
  TypeId-sorted body; return `None` if either lookup fails before the first
  lock is acquired.

**Other resource checks — confirmed correct:**
- `resource::<T>()` panics with the type name via
  `std::any::type_name::<R>()` (`world.rs:462-467`).
- `try_resource::<T>()` returns `None` on missing lookup.
- `ResourceRead` / `ResourceWrite` `Deref` impls downcast through `Any` with
  explicit `expect`; since `TypeId` is the map key, downcast cannot fail in
  practice.
- Resources are usable from `&World` systems (interior mutability via
  per-resource `RwLock`).

## Dimension 5: System & Scheduler

### E-01: BFS doc mismatch — queue.pop is LIFO (DFS)
- **Severity**: LOW
- **Location**: `crates/core/src/ecs/systems.rs:92, :109`
- **Status**: NEW
- **Description**: `make_transform_propagation_system` is documented as
  "breadth-first walk" (line 27, 48, 92). The implementation uses
  `Vec::pop()` at `:109`, which is LIFO — this is actually DFS. The walk is
  still **functionally correct** (each child's parent is written in phase 1b
  or earlier in the same DFS branch before the child pops), but the docs and
  the variable name `queue` are misleading.
- **Impact**: Docs only. Anyone reading this and assuming level-order
  traversal for a future optimization (e.g. parallel per-level dispatch)
  will get a surprise. No functional bug.
- **Suggested Fix**: Either (a) switch to `VecDeque<EntityId>` + `pop_front`
  for true BFS (keeps the docs honest, near-zero cost), or (b) update
  comments to say "DFS walk" and rename `queue` → `stack`. Option (a) is
  preferable because it makes future parallel per-level dispatch trivial.

**Other scheduler checks — confirmed correct:**
- `impl<F: FnMut(&World, f32) + Send + Sync> System for F` at `system.rs:32`
  is correct. `FnMut` allows captured mutable state; `Send + Sync` gates
  parallel dispatch.
- Mutations across stages are visible: each stage releases all guards before
  the next stage begins, so there is no stale-read hazard
  (test `mutation_visible_across_stages` at `scheduler.rs:336`).
- Empty scheduler runs cleanly (test at `scheduler.rs:441`).
- `system_names()` preserves stage order via BTreeMap iteration.
- Duplicate-name detection via `try_add_to` / `try_add_exclusive` (#312,
  already closed).
