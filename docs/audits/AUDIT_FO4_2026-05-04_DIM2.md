### Findings

#### FO4-D2-NEW-01: Unknown BA2 versions silently fall through to v1 layout instead of being rejected

**Severity**: MEDIUM
**Location**: `crates/bsa/src/ba2.rs:173-197`
**Status**: NEW

**Description**: Version dispatch is two cascading `if`s gated on
`version == 2 || version == 3` and `version == 3`. Any version not in `{2, 3}`
falls through to `read_general_records` / `read_dx10_records` as if it were the
24-byte v1 layout, with no validation that the version is in the supported set
`{1, 2, 3, 7, 8}`. A future v4 / corrupted / fuzzer version is accepted
silently and parses garbage offsets. The module docstring (lines 20-26)
explicitly commits to the `{1, 2, 3, 7, 8}` allowlist; the implementation is
out of sync with its documented contract. openMW's reference reader rejects
unknown versions with `fail("Unrecognized")`.

**Why it's wrong**: The BA2 lineage shows Bethesda is willing to break header
layout between major releases (v2 added 8 bytes, v3 added 12 bytes). Assuming
future versions revert to v1's 24-byte header is empirically falsifiable
within this very codebase.

**Repro**: Hand-craft a BA2 with `version = 5` and `file_count = 0`.
`Ba2Archive::open()` succeeds with `version() == 5`. With non-zero file count
the reader will eat offset bytes from the v1 layout and either error
confusingly at extract time or return corrupted bytes.

**Fix sketch**: Replace the cascading `if`s with a `match version` enumerating
{1, 2, 3, 7, 8} and emitting `InvalidData` on the default arm. Add a
regression test (`unknown_version_rejected`) mirroring
`v3_unknown_compression_method_rejected`.

#### FO4-D2-NEW-02: Zlib decompressed-size mismatch is logged at `debug` but the chunk is still returned

**Severity**: LOW
**Location**: `crates/bsa/src/ba2.rs:466-472`
**Status**: NEW (related to open #598)

**Description**: `decompress_chunk`'s zlib branch uses `read_to_end` and
emits only a `log::debug!` when the actual size disagrees with the declared
`unpacked_size`. The LZ4-block branch immediately below hard-errors on the
same mismatch (via `lz4_flex::block::decompress(packed, unpacked_size)`'s
inherent size check). Asymmetric strictness — zlib silently passes a
short buffer to downstream parsers (NIF, DDS), LZ4 fails fast.

**Why it's wrong**: The asymmetry is invisible to callers. A malicious or
truncated archive can feed downstream parsers a buffer shorter than declared
and pivot to a different bug class. openMW's reference uses `::uncompress(...)`
which fails with a non-`Z_OK` return code on declared-size mismatch.

**Repro**: Hand-craft a BA2 with a GNRL record claiming `unpacked_size = 100`
but a deflate stream that emits 20 bytes before its end-of-stream marker.
ByroRedux returns a 20-byte buffer with a debug log; openMW errors.

**Fix sketch**: Promote `log::debug!` → `log::warn!` (lower-effort), or
introduce `Ba2Strictness::{Lenient, Strict}` and have `Strict` mode return
`InvalidData` on size mismatch. Resolving the open #598 (the Meshes.ba2
`packed_size > unpacked_size` anomaly) is the natural prerequisite for
tightening the default to `Strict`.

### Verified correct (no finding)

- DX10 chunk concatenation order: largest mip first, matches DDS spec

  chunk[0] = 524288 bytes = mip 0).

  FO4-observed DXGI formats (matches openMW switch coverage).
- DDS header reconstruction is byte-exact 148 bytes (124 base + 4 magic +
  20 DX10 ext) and passes structural validation.
