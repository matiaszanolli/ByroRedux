# NIF Parser Audit — 2026-07-16

Deep audit of the NIF binary-format parser (`crates/nif/src/`) across six
dimensions: Stream Position Integrity, Version Gating, Block Dispatch
Coverage, Geometry Extraction & Import Handoff, Collision & Shader Block
Parsing, and Allocation Hygiene. Each dimension ran as an independent agent
pass, cross-referencing `docs/legacy/nif.xml`, `docs/engine/nif-parser.md`,
and the checked-in per-game baseline TSVs; several passes additionally
re-ran real-game-data sweeps (~207,000 files across 6 of 7 supported games)
to validate dispatch coverage and heap budgets against live code rather than
cached figures.

## Executive Summary

- **10 NEW findings**: 0 CRITICAL, 3 HIGH, 4 MEDIUM, 3 LOW.
- **0 regressions** of any previously-fixed bug — every regression guard
  cited in the audit brief (spanning #688, #698, #1301, #1302, #1310,
  #1329, #1337, #1509, #1510, #1552, #1592, #1604–#1609, #1652, #1721,
  #1843 (still open, not regressed), #1881, #1884, plus the NIFAL
  collision-shape-translation and `CollisionAuthoring` discriminator, plus
  every Dim-6 allocation-hygiene pin #408/#831/#832/#833/#1245/#1246/
  #1261/#1262/#1263/#1265/#877) was independently re-verified against
  current source and found intact.
- **1 existing issue reconfirmed, not regressed**: #1897 (`ShaderFlags`/
  `has_shader_property_fo3_fields`/`NifStream::variant()` dead-code chain).
- **Clean/recoverable rate**: matches `docs/engine/nif-parser.md`'s cited
  figures — 100% recoverable across all seven games, 100% clean on
  FO3/FNV/Skyrim SE/FO4/FO76, 99.93% Oblivion, 99.64% aggregate Starfield.
  No dimension found evidence this has regressed. Block dispatch coverage
  was freshly re-measured at **100%** on every one of 6 games checked
  (~207,000 files, ~5.9M block instances) — zero `NiUnknown` fallbacks on
  any block type that actually appears in shipped content.
- **Notable pattern**: the two most consequential findings (NIF-D1-02,
  NIF-D1-03) and the one Dim-5 finding (NIF-D5-01) are all instances of the
  *same* bug class already fixed elsewhere in the codebase (#1329/#1337's
  "old Oblivion no-`block_sizes`" fix, and #1552's `FO4_DLC_UPPER` gate) —
  a version-gate fix applied to one parser in a module but not to its
  siblings. This is a good signal for where to look next: any parser
  sharing a version band with an already-fixed sibling is worth a targeted
  grep for the same gate.
- **3 of the 10 findings are documentation-only** (NIF-D2-A, NIF-D3-01,
  NIF-D3-02) — `docs/engine/nif-parser.md` has drifted from the code it
  documents in three places: a reverted version-gating pattern still shown
  as current guidance, a stale dispatch-arm count, and a mischaracterized
  `NiFogProperty` "gap" that is actually a downstream material-walker
  concern, not a dispatch-table one.

## Findings

### HIGH

#### NIF-D1-02: BhkSimpleShapePhantom / BhkAabbPhantom miss the bhkWorldObject "Unknown Int" for v10.0.1.0–10.0.1.2 files
- **Severity**: HIGH (this version band ships with no `block_sizes` table — the same "old Oblivion" band #1329/#1337 fixed elsewhere in this exact module)
- **Dimension**: Stream Position Integrity
- **Location**: `crates/nif/src/blocks/collision/phantom_action.rs:21-46` (`BhkSimpleShapePhantom::parse`), `:63-89` (`BhkAabbPhantom::parse`)
- **Game Affected**: `NifVariant::Oblivion` pre-release content at file version `[10.0.1.0, 10.0.1.2]` — the identical band `BhkRigidBody::parse_oblivion_old` (`rigid_body.rs:230-266`) was fixed for under #1329.
- **Status**: NEW
- **Description**: Per nif.xml, `bhkWorldObject` (the shared base of every `bhk*Shape`-holding collision block, including phantoms) is `Shape(Ref) + Unknown Int(uint, until="10.0.1.2") + HavokFilter + WorldObjectInfo`. `BhkRigidBody::parse_oblivion_old` correctly reads this 4-byte field; the two phantom parsers in the same module read `shape_ref` then jump straight to `havok_filter`, never consuming it. The file's own doc comment on `BhkAabbPhantom` asserts the field is never present for any Bethesda game — the exact premise #1329/#1337 already disproved for the sibling Havok blocks in this same file range.
- **Evidence**:
  ```rust
  // phantom_action.rs:22-26 — BhkSimpleShapePhantom::parse
  let shape_ref = stream.read_block_ref()?;
  let havok_filter = stream.read_u32_le()?;   // reads the phantom "Unknown Int" as HavokFilter on v10.0.1.0-.2
  stream.skip(20)?; // bhkWorldObjectCInfo
  ```
  vs. the sibling fix already in the same module (`rigid_body.rs:232-234`):
  ```rust
  let shape_ref = stream.read_block_ref()?;
  stream.skip(4)?; // Unknown — only present `<= VER_OB_OLD` (10.0.1.x)
  let havok_filter = stream.read_u32_le()?;
  ```
- **Impact**: If a `bhkSimpleShapePhantom` or `bhkAabbPhantom` block appears in a v10.0.1.0–10.0.1.2 file (no `block_sizes` table), the parser misreads the offset and the stream ends 4 bytes short, cascading unrecoverably into every following block.
- **Related**: #1329, #1337 (same version band, same module, different block types)
- **Suggested Fix**: Add the same `stream.version() <= NifVersion::V10_0_1_2 { stream.skip(4)? }` gate (or a shared `read_bhk_world_object_prefix` helper) to both parsers, and correct the stale doc comment.

#### NIF-D1-03: NiGeomMorpherController reads Morpher Flags / Num Interpolators without their nif.xml version gates
- **Severity**: HIGH by impact (same no-`block_sizes` v10.0.1.x band as NIF-D1-02), very narrow trigger window
- **Dimension**: Stream Position Integrity
- **Location**: `crates/nif/src/blocks/controller/morph.rs:30-36`
- **Game Affected**: `NifVariant::Oblivion`/pre-release NetImmerse content at file version `< 10.0.1.2` (Morpher Flags) and `< 10.1.0.106` (Num Interpolators)
- **Status**: NEW
- **Description**: nif.xml gates `Morpher Flags` `since="10.0.1.2"` and `Num Interpolators`/`Interpolators` `since="10.1.0.106"`. Both are read unconditionally right after the `NiInterpController` base — mirroring exactly the bug class #1329/#1337 fixed for `BhkRigidBody`, `bhkMoppBvTreeShape`, `bhkNiTriStripsShape`, and `NiSkinData` in the identical version window, but that work never touched `NiGeomMorpherController`. No dispatch test pins either boundary (only the already-fixed #1509 `bsver > 9` gate is tested).
- **Evidence**:
  ```rust
  let base = parse_interp_controller_base(stream)?;
  let morpher_flags = stream.read_u16_le()?;      // no since=10.0.1.2 gate
  let data_ref = stream.read_block_ref()?;
  let always_update = stream.read_u8()?;
  let num_interpolators = stream.read_u32_le()?;  // no since=10.1.0.106 gate
  ```
- **Impact**: On a file version below either threshold (a real, if rare, "old Oblivion" pre-release band), the phantom read misaligns the stream in a band with no `block_sizes` table, cascading unrecoverably.
- **Related**: #1329, #1337, #1509 (same controller, different gate already fixed)
- **Suggested Fix**: Gate `morpher_flags` behind `version() >= V10_0_1_2` (default 0 below) and `num_interpolators`/the ref loop behind `version() >= V10_1_0_106` (default empty), matching the pattern already used a few lines below for the trailing `Num Unknown Ints` gate.

#### NIF-D1-01: NiPersistentSrcTextureRendererData aliased to NiPixelData's parser — missing Pad Num Pixels + Platform fields
- **Severity**: HIGH on the Oblivion path (no `block_sizes` — unrecoverable cascade) / MEDIUM on FO3+ (`block_sizes` masks the drift into a silently-wrong block)
- **Dimension**: Stream Position Integrity
- **Location**: `crates/nif/src/blocks/mod.rs:604-606` (dispatch alias), `crates/nif/src/blocks/texture.rs:236-306` (`NiPixelData::parse`)
- **Game Affected**: any `NifVariant` where a `NiSourceTexture` sets `Persist Render Data = true` and its `Pixel Data` ref resolves to a literal `NiPersistentSrcTextureRendererData` block — Oblivion through Starfield, the niobject carries no version gate in nif.xml.
- **Status**: NEW
- **Description**: `blocks/mod.rs` dispatches `"NiPixelData" | "NiPersistentSrcTextureRendererData"` to the same `NiPixelData::parse`, but the two types diverge after the shared `NiPixelFormat` prelude: `NiPersistentSrcTextureRendererData` additionally reads `Pad Num Pixels` (since 20.2.0.6) and an unconditional 4-byte `Platform` field that `NiPixelData` doesn't have.
- **Evidence**:
  ```rust
  // texture.rs:286-289 — correct only for NiPixelData
  let num_pixels = stream.read_u32_le()? as usize;
  let num_faces = stream.read_u32_le()?;   // actually "Pad Num Pixels" on FO3+ NiPersistentSrcTextureRendererData
  let total_bytes = num_pixels * num_faces as usize;  // wrong length
  let pixel_data = stream.read_bytes(total_bytes)?;   // Platform (4 B) never read
  ```
- **Impact**: Wherever this block type actually appears, the aliased parser misreads `Pad Num Pixels` as a byte-length multiplier and drops the mandatory `Platform` field — the stream ends up short (4 bytes on old-format files, 8 on FO3+). On Oblivion (no `block_sizes`) this cascades unrecoverably; on FO3+ it's masked by the `block_sizes` reconciliation but leaves a corrupted `pixel_data` buffer silently in the scene.
- **Related**: none found in existing issues.
- **Suggested Fix**: Give `NiPersistentSrcTextureRendererData` its own parser (or a shared prelude + two-way tail split, matching the existing `NiPixelData` old/new-layout split), reading `Num Pixels` → `Pad Num Pixels` (since 20.2.0.6) → `Num Faces` → `Platform`/`Renderer` → `Pixel Data`.

### MEDIUM

#### NIF-D5-01: BSLightingShaderProperty::parse_fo4 reads the pre-Name Shader Type field unconditionally across BSVER 130–154, but nif.xml gates it to BSVER ≤ 139
- **Severity**: MEDIUM
- **Dimension**: Collision & Shader Block Parsing
- **Location**: `crates/nif/src/blocks/shader.rs:976-978` (`parse_fo4`)
- **Game Affected**: `NifVariant::Fallout4`/`Fallout76` — any file with `bsver` in `[140, 154]`
- **Status**: NEW
- **Description**: nif.xml gates the pre-`Name` `Shader Type` field with `vercond="#BS_GTE_SKY# #AND# #NI_BS_LTE_FO4#"` (`bsver` in `[83, 139]`). `parse_fo4` handles the wider BSVER 130-154 band but reads `shader_type` unconditionally as the very first field, before `NiObjectNETData::parse`. The sibling function `parse_shader_type_data_fo4` already has the correct `bsver < FO4_DLC_UPPER` gate for its own trailing field (fixed under #1552), but that fix was never applied to this earlier, more consequential read — misaligning `Name` and every subsequent field for the block, not just one tail.
- **Evidence**:
  ```rust
  // shader.rs:976-978 — unconditional across the whole 130-154 band parse_fo4 handles
  fn parse_fo4(stream: &mut NifStream, bsver: u32) -> io::Result<Self> {
      let shader_type = stream.read_u32_le()?;
      let net = NiObjectNETData::parse(stream)?;
  ```
  vs. the already-fixed sibling one call deeper (`shader.rs:1392-1398`, #1552):
  ```rust
  let env_map_scale = if bsver < crate::version::bsver::FO4_DLC_UPPER {
      stream.read_f32_le()?
  } else {
      1.0
  };
  ```
  No test fixture exercises `bsver` in `[140, 154]` (only 131/132 are pinned).
- **Impact**: For any block at `bsver` in `[140, 154]`, `Name` and every subsequent field are corrupted. FO3+'s `block_sizes` table means this does not cascade to later blocks, but the affected block is silently wrong. `#1552`'s own comment calls 140-154 "a dead band, no shipping game," so vanilla-content impact is unlikely, but the `FO76_SF2_CRCS = 152` special-case elsewhere in the same function shows the authors did find it necessary to handle part of this band — residual risk on dev-stream/modded content is non-zero.
- **Related**: #1552 / SK-D2-01 (same gate, sibling field, already fixed there).
- **Suggested Fix**: Gate `shader_type` the same way #1552 gated `env_map_scale`: read only if `bsver < FO4_DLC_UPPER`, default 0 otherwise; add a `bsver=145`-style test fixture.

#### NIF-D1-04: NiShadeProperty.Flags read unconditionally, but nif.xml gates it to bsver ≤ FO3
- **Severity**: MEDIUM (Skyrim+ has `block_sizes`, which masks the drift)
- **Dimension**: Stream Position Integrity
- **Location**: `crates/nif/src/blocks/properties.rs:547-556` (`NiFlagProperty::parse`), dispatch at `crates/nif/src/blocks/mod.rs:592-601`
- **Game Affected**: any `NifVariant` at bsver > FO3 containing a literal `NiShadeProperty` block
- **Status**: NEW
- **Description**: nif.xml gates `NiShadeProperty.Flags` with `vercond="#NI_BS_LTE_FO3#"`; the other three types sharing this parser (`NiDitherProperty`/`NiSpecularProperty`/`NiWireframeProperty`) have no such gate. `NiFlagProperty::parse` treats all four identically, reading `flags` unconditionally.
- **Impact**: On a bsver > FO3 file with a literal `NiShadeProperty`, the parser reads 2 bytes belonging to the next field as `flags`. No confirmed corpus regression today (the import walker's own comment scopes `NiShadeProperty` usage to "a handful of Oblivion architectural pieces") — a latent gap, not an observed failure.
- **Related**: none found.
- **Suggested Fix**: Split `NiShadeProperty` into its own thin parser gated on `bsver <= FO3`, or branch inside `NiFlagProperty::parse` on `type_name == "NiShadeProperty" && bsver > FO3` to skip the read and default `flags`.

#### NIF-D1-05: NiTexturingProperty decal-slot count has no upper bound against nif.xml's fixed 4 slots
- **Severity**: MEDIUM (defense-in-depth gap, requires an anomalous `Texture Count`)
- **Dimension**: Stream Position Integrity
- **Location**: `crates/nif/src/blocks/properties.rs:300-313`
- **Game Affected**: any `NifVariant`, contingent on a corrupt/anomalous `Texture Count`
- **Status**: NEW
- **Description**: nif.xml defines exactly 4 decal texture slots; the parser computes an open-ended `num_decals = texture_count.saturating_sub(8 or 6)` with no cap. `allocate_vec` bounds the allocation against remaining file bytes (OOM guard only), not against the format's known maximum of 4.
- **Impact**: A corrupt/malformed `texture_count` above 12 (or 10) causes the loop to read `TexDesc`s the format doesn't define, consuming bytes that belong to later fields — recoverable via `block_size` on FO3+, unrecoverable on Oblivion.
- **Related**: #429, #450 (adjacent version-gate fixes in the same function)
- **Suggested Fix**: Clamp `num_decals` to a hard maximum of 4 and treat any file claiming more as a parse error.

#### NIF-D2-A: nif-parser.md's "GameVariant trait pattern" example reintroduces the exact bug class #1838/#1839 fixed
- **Severity**: MEDIUM
- **Dimension**: Version Gating
- **Location**: `docs/engine/nif-parser.md:180-200` (code block at lines 184-191)
- **Game Affected**: All — this is the parser's own canonical reference doc
- **Status**: NEW
- **Description**: The doc's example shows six `stream.variant().has_*()` calls (`has_properties_list`, `avobject_flags_u32`, `has_material_crc`, `uses_bs_lighting_shader`, `uses_bs_tri_shape`, `uses_fo76_shader_flags`), none of which exist on `NifVariant` any more — `#1840` deleted the last call-site-less predicates precisely because a `#1277` migration onto them broke real files (transitional `v20.2.0.7, uv=11, bsver≤26` content misdetected via `NifVariant` and slipped the stream by 2-4 bytes, #160/#1331), and `#160`/`#1331`/`#1838`/`#1839` reverted the migration. The current doctrine — stated in ~10 in-code comments — is to gate on raw `stream.bsver()` against named `crate::version::bsver` constants, not `NifVariant` helpers. The doc was never updated after the revert.
- **Evidence**: `grep -rn "has_properties_list\|avobject_flags_u32\|has_material_crc\|uses_bs_lighting_shader\|uses_bs_tri_shape\|uses_fo76_shader_flags" crates/nif/src` → zero non-comment hits. Revert rationale verbatim at `crates/nif/src/blocks/base.rs:78-81`. Meta-evidence: this very audit's own Dimension 2 checklist names several of the same six methods as "candidates to check," suggesting the checklist itself traces back to this stale doc section rather than live code.
- **Impact**: A contributor following the doc's literal example can't compile; more realistically, they may add a *new* `NifVariant`-keyed helper for a new feature and reintroduce the exact `Unknown`/hybrid-corner mis-parse class already paid down, this time uncaught by any compile error.
- **Related**: #1897 (the code-level instance of the same foot-gun — `ShaderFlags`/`has_shader_property_fo3_fields`/`variant()`), #1840, #1841, #1511, #1838, #1839, #160, #1331.
- **Suggested Fix**: Replace the doc's example with the actual current doctrine (`stream.bsver()` vs. named `bsver` constants) and add a short note on why `NifVariant`-method gating was tried and reverted.

#### NIF-D3-02: nif-parser.md mischaracterizes NiFogProperty as a dispatch-table gap; it is dispatched
- **Severity**: LOW
- **Dimension**: Block Dispatch Coverage
- **Location**: `docs/engine/nif-parser.md:359-360`; `crates/nif/src/blocks/mod.rs:590`
- **Game Affected**: Oblivion, Fallout 3 (only games with observed `NiFogProperty` occurrences)
- **Status**: NEW
- **Description**: The doc's "Block coverage" section states `NiFogProperty` is "a deliberate non-dispatch (#1224)" — but `"NiFogProperty" => Ok(Box::new(NiFogProperty::parse(stream)?))` is a live, working dispatch arm. The actual #1224 gap is one layer downstream: the import material walker never calls `scene.get_as::<NiFogProperty>()`, so parsed fog data never reaches `MaterialInfo`. Confirmed via `git show` of the #1224 closeout commit (touches only `properties.rs` and `import/material/walker.rs`, not `blocks/mod.rs`) and the checked-in baselines (`NiFogProperty 11 0` Oblivion, `1 0` FO3 — parses cleanly, zero `NiUnknown`).
- **Impact**: Low, but a future auditor trusting this doc section could file a duplicate issue against the wrong layer.
- **Related**: #1224 (closed, correctly scoped to the material walker — no regression).
- **Suggested Fix**: Reword to note `NiFogProperty` parses cleanly but is a deliberate non-consumption at the import/material-walker layer (#1224) — cross-link NIFAL/Dimension 4 rather than implying a dispatch gap.

#### NIF-D4-01: BsOrderedNodeData.alpha_sort_bound is the only node-level bound field left in raw Gamebryo Z-up space
- **Severity**: LOW
- **Dimension**: Geometry Extraction & Import Handoff
- **Location**: `crates/nif/src/import/walk/mod.rs:1567-1575` (`extract_bs_ordered_node`), `crates/nif/src/import/types.rs:252-261`, `crates/nif/src/blocks/node.rs:132-166`
- **Game Affected**: FO3/FNV (`BSOrderedNode`, transparent-object draw-order sorting); theoretically any game that authors the block
- **Status**: NEW
- **Description**: Every other position/bound field on `ImportedNode` and siblings (`translation`/`rotation`, `LodGroupData.center`, `ImportedScene.bs_bound`, attach points, furniture markers) is explicitly converted Z-up → Y-up and documented as such. `BsOrderedNodeData.alpha_sort_bound` (`[x, y, z, radius]`) is copied verbatim with no conversion and no handedness note in its doc comment.
- **Evidence**: `walk/mod.rs:1567-1575` copies `n.alpha_sort_bound` with no `zup_to_yup` call. Confirmed via grep that no consumer reads `ImportedNode.bs_ordered_node` anywhere outside the `nif` crate today — the field is currently inert.
- **Impact**: None today (unconsumed field; the renderer-side consumption is documented future work). Latent risk: a future consumer wiring up alpha-sort occlusion would very plausibly compose it as Y-up like every sibling field, silently producing a wrong bound center (axis swap + sign flip; radius unaffected).
- **Related**: None — distinct from `#625` (added the field's presence, not its coordinate contract).
- **Suggested Fix**: Either convert via `zup_point_to_yup` at extraction (matching `bs_bound`'s treatment) and document as Y-up, or explicitly document it as intentionally NOT converted and why. Zero behavior risk either way given no current consumer.

## Prioritized Fix Order

1. **NIF-D1-02** (HIGH) — `BhkSimpleShapePhantom`/`BhkAabbPhantom` missing bhkWorldObject prefix skip. Same fix pattern as #1329/#1337, small and mechanical.
2. **NIF-D1-03** (HIGH) — `NiGeomMorpherController` missing two version gates. Same fix pattern, small and mechanical.
3. **NIF-D1-01** (HIGH/MEDIUM) — `NiPersistentSrcTextureRendererData` mis-aliased to `NiPixelData`'s parser. Needs a dedicated tail parser — slightly larger but self-contained.
4. **NIF-D5-01** (MEDIUM) — `BSLightingShaderProperty::parse_fo4` unconditional `shader_type` read in the 140-154 dead band. Mirrors the already-landed #1552 pattern one call away.
5. **NIF-D1-04, NIF-D1-05** (MEDIUM) — `NiShadeProperty` flags gate, `NiTexturingProperty` decal-slot cap. Both defense-in-depth, low urgency given no observed corpus trigger.
6. **NIF-D2-A** (MEDIUM, doc) — Refresh `nif-parser.md`'s version-gating example to the current `bsver`-constant doctrine; prevents a future contributor from reintroducing #1838/#1839's bug class.
7. **NIF-D3-01, NIF-D3-02, NIF-D4-01** (LOW) — doc citation refresh, doc mischaracterization fix, coordinate-space doc note on an inert field. Bundle into routine doc maintenance.

Per-game **material** classification at the NIFAL `translate_material` boundary is out of scope for this audit — see `/audit-nifal`.
