## Findings

### NIF-D5-01: `BSEffectShaderProperty` Starfield material-reference stopcond missed the `#1510` `!name.is_empty()` discriminator (sibling-parser divergence)
- **Severity**: MEDIUM
- **Dimension**: Collision/Shader Parsing (stream position)
- **Game Affected**: Starfield (`bsver >= STARFIELD = 172`)
- **Location**: `crates/nif/src/blocks/shader.rs` — `BSEffectShaderProperty::parse` (the `bsver >= FO76` stopcond block, ~lines 1632–1638)
- **Status**: NEW
- **Description**: `BSEffectShaderProperty::parse` decides whether a block is a
  material-reference stub (body absent, real data in the sidecar `.bgem`/`.mat`
  file) by testing `is_material_reference(name)` for **all** `bsver >= FO76`,
  including Starfield. Its sibling `BSLightingShaderProperty::parse_fo76_plus`
  was explicitly fixed under `#1510` to use a *different* discriminator for
  Starfield:
  ```rust
  // BSLightingShaderProperty::parse_fo76_plus (current)
  let is_ref = if bsver >= crate::version::bsver::STARFIELD {
      !name.is_empty()            // Starfield: content-hash path, no suffix
  } else {
      is_material_reference(name) // FO76: suffix-aware (#749 editor-label split)
  };
  ```
  The `#1510` commit body documents *why*: "Starfield material references are
  content-hash paths (`<hex>\<hex>`) with NO `.mat`/`.bgsm` suffix, so the #749
  `is_material_reference` stopcond missed them … In Starfield a full-body block
  carries an EMPTY name, so `!name.is_empty()` is the correct stub gate."
  `BSEffectShaderProperty` is the same `NiObjectNET`-derived `BSShaderProperty`
  subclass with the identical stopcond pattern, so the same reasoning applies —
  but it never received the fix. A Starfield `BSEffectShaderProperty` whose
  `net.name` is a suffix-less content-hash material reference therefore takes
  the **full-body parse path** instead of the 12-byte stub, reading the inline
  body fields off a block that on disk carries none.
- **Evidence**:
  - `BSLightingShaderProperty::parse_fo76_plus` has the `bsver >= STARFIELD ⇒ !name.is_empty()` branch (and a regression test `parse_bs_lighting_starfield_hashpath_name_stubs`).
  - `BSEffectShaderProperty::parse` has only `if is_material_reference(name)`.
  - No test in `shader_tests.rs` covers a Starfield `BSEffectShaderProperty` with a hash-path name (grep for `Starfield`+`effect` returns nothing; the only Starfield effect-shader coverage is the `#746`/`#747` `refraction_power` / FO76-tail value-gate tests, all using full bodies).
  - The `#1510` commit body asserts the BSEffect/cloth/point-light NiUnknown were "already fixed since the audit HEAD" — i.e. fixed for the *over-read symptom* (the `refraction_power` / FO76-tail `>=` value-gates), **not** by adding the Starfield stub discriminator.
- **Impact**: For any Starfield `.bgem`/`.mat`-referenced effect shader stored
  as a content-hash name, the parser reads `shader_flags`/CRC arrays/UV/
  source-texture/falloff/etc. from bytes that belong to the next block's
  padding or the file footer. The damage is bounded by two things, which is why
  the checked-in Starfield baseline still shows `748 / 0`: (a) `block_size`
  reconciliation in `lib.rs` realigns the outer stream so following blocks are
  unaffected, and (b) the run-off either lands inside `block_size` (recovered as
  silent drift, still counted "parsed") or hits EOF mid-block and recovers to
  `NiUnknown`. Per `_audit-severity.md` ("NIF parse mismatch the `block_size`
  reconciliation covers" = MEDIUM) this is MEDIUM: the *struct* lands with
  garbage source-texture / base-color / falloff fields on the affected blocks,
  which then flow through `import/material` into the effect-shader material — a
  silent-wrong-material risk, not a crash. Blast radius is Starfield effect
  shaders only (a stretch target); the corpus count of *affected* blocks is
  unknown from a static read (the baseline can't separate clean-parse from
  drift-recovered).
- **Related**: `#1510` (the BSLightingShaderProperty fix this mirrors), `#749`
  (the FO76 suffix-aware editor-label split this must preserve), `#746`/`#747`
  (the BSEffect Starfield value-gate fixes that closed the *over-read* half).
- **Suggested Fix**: Mirror the `parse_fo76_plus` discriminator in
  `BSEffectShaderProperty::parse`:
  ```rust
  if bsver >= crate::version::bsver::FO76 {
      if let Some(name) = net.name.as_deref() {
          let is_ref = if bsver >= crate::version::bsver::STARFIELD {
              !name.is_empty()
          } else {
              is_material_reference(name)
          };
          if is_ref { return Ok(Self::material_reference_stub(net)); }
      }
  }
  ```
  Add a `parse_bs_effect_starfield_hashpath_name_stubs` test paralleling the
  BLSP one. Consider also routing `BSEffectShaderProperty` through a
  `parse_with_size` so any Starfield-specific trailing block (the BLSP
  `starfield_tail` analog, `#1606`) is captured opaquely rather than left to
  block_size recovery — but only if a corpus walk surfaces a non-zero BSEffect
  drift; do not add the tail speculatively.
