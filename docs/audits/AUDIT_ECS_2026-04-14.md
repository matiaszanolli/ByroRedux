## Dimension 2: Query Safety — 1 LOW

### D6-01: Constructor doc comments still reference raw `track_read`/`track_write`
- **Severity**: LOW
- **Dimension**: Query Safety
- **Location**:
  - `crates/core/src/ecs/query.rs:30-31` (`QueryRead::new`)
  - `crates/core/src/ecs/query.rs:83-84` (`QueryWrite::new`)
  - `crates/core/src/ecs/query.rs:201-204` (`ComponentRef::new`)
  - `crates/core/src/ecs/resource.rs:24-25` (`ResourceRead::new`)
  - `crates/core/src/ecs/resource.rs:68-69` (`ResourceWrite::new`)
- **Status**: NEW
- **Description**: Five `pub(crate)` constructor doc comments state:
  > "Caller must have already called `lock_tracker::track_read` before acquiring the
  > RwLock guard."
  This was accurate under the pre-#137 pattern, but every internal call site in
  `world.rs` now uses the RAII `TrackedRead::new(...)` / `TrackedWrite::new(...)`
  scope guards and calls `defuse()` after the lock is acquired. A new contributor
  reading the constructor docs would be misdirected toward the raw `track_*` API
  (still present but no longer the intended public-of-crate path).

  In addition, the `ComponentRef::new` doc comment on line 204 includes an out-of-date
  instruction: "Returns `None` if the entity doesn't have this component — caller must
  call `lock_tracker::untrack_read` in that case." That obligation is now discharged by
  the caller's `TrackedRead` scope guard dropping naturally when `ComponentRef::new`
  returns `None` (see `World::get`, `world.rs:122-140`). The instruction is misleading:
  following it literally would double-untrack.
- **Evidence**:
  ```rust
  // query.rs:29-33
  impl<'w, T: Component> QueryRead<'w, T> {
      /// Create a new read query. Caller must have already called
      /// `lock_tracker::track_read` before acquiring the RwLock guard.
      pub(crate) fn new(
          guard: RwLockReadGuard<'w, Box<dyn Any + Send + Sync>>,
          type_id: TypeId,
      ) -> Self { ... }

  // world.rs:210-217 (actual call site — uses TrackedRead, not raw track_read)
  pub fn query<T: Component>(&self) -> Option<QueryRead<'_, T>> {
      let type_id = TypeId::of::<T>();
      let lock = self.storages.get(&type_id)?;
      let scope = lock_tracker::TrackedRead::new(type_id, std::any::type_name::<T>());
      let guard = lock.read().unwrap_or_else(|_| storage_lock_poisoned::<T>());
      scope.defuse();
      Some(QueryRead::new(guard, type_id))
  }
  ```
- **Impact**: Documentation only. No runtime effect. Slight risk for future
  contributors adding a new tracked entry point: copying from the stale comments could
  recreate the #137 leak pattern.
- **Related**: #137 (closed), E26-05 (closed in this cycle)
- **Suggested Fix**: Replace the five stale comments with:
  ```
  /// Create a new <read|write> query. Caller owns a `TrackedRead`/`TrackedWrite`
  /// scope guard for `type_id`; the matching `Drop` impl on this type will untrack
  /// it. Ensure the caller has called `scope.defuse()` after successful lock
  /// acquisition. (See #137.)
  ```
  Remove the "caller must call untrack_read" sentence from `ComponentRef::new` — with
  RAII the caller's still-armed `TrackedRead` drops naturally on the `None` return.

Prior fixes re-verified:
- **E26-05 (prior audit) — FIXED**: `World::get()` (`world.rs:122-140`) now uses

- **#137**: All 9 tracked entry points (`query`, `query_mut`, `query_2_mut`,

- TypeId-sorted lock acquisition in `query_2_mut`, `query_2_mut_mut`, `resource_2_mut`. ✓

## Dimension 5: System & Scheduler — CLEAN (open issues already filed)

Stage-based parallel scheduler re-verified. `Scheduler::run` iterates `stages: BTreeMap<Stage, StageData>` in stage order, dispatches `parallel` via `rayon::par_iter_mut`, then runs `exclusive` sequentially. Empty scheduler is a no-op. System names are returned in stage order (parallel-then-exclusive within each stage).

- `System` blanket impl is `FnMut(&World, f32) + Send + Sync` (system.rs:32-36), compatible with stateful closures and rayon. ✓
- Mutations across stages are sequentially visible (no cross-stage parallelism). ✓
- Within a stage, `RwLock` per storage + per resource serialises conflicting accesses. ✓
- `has_system` is O(systems) on every `add` — acceptable given typical N ≤ 50 systems. (Not a finding.)
- `system_names()` correctly flat-maps over `stages.values()` → `(parallel.chain(exclusive))` — deterministic order. ✓
