## Findings

### FO4-D2-01: `#1651` blend-factor swap corrupts the real authored Additive + Multiplicative BGSM/BGEM modes
- **Severity**: HIGH
- **Dimension**: 2 ‚Äî BGSM/BGEM Consumption
- **Location**: `byroredux/src/asset_provider/material.rs:497-503` (`gl_to_gamebryo_blend`), applied at `:936-937` (BGSM) and `:1032-1033` (BGEM)
- **Status**: NEW (regression of the *intent* of `#1651`, commit `ada75ee3`)
- **Description**: `#1651` added `gl_to_gamebryo_blend`, which swaps `0‚Üî1` on
  the BGSM/BGEM `alpha_blend_mode.src_blend`/`dst_blend` before writing
  `mesh.src_blend_mode`/`dst_blend_mode`, on the premise that the file stores a
  "GL-style enum (Zero=0, One=1)" that is inverted from the renderer's Gamebryo
  nibble. That premise is false. The authoritative reference parser
  (`Material-Editor/MaterialLib/BaseMaterialFile.cs:363-387`, `ConvertAlphaBlendMode`)
  enumerates the only authored `(function, src, dst)` tuples Bethesda writes:
  - None = `(0, 0, 0)`
  - Standard = `(1, 6, 7)`
  - **Additive = `(1, 6, 0)`**
  - **Multiplicative = `(1, 4, 1)`**

  The `(One, One) = (1, 1)` value the `#1651` commit message and code comment
  claim additive cards author **does not exist** in the format. The stored
  `src`/`dst` numbers are already Gamebryo `NiAlphaProperty` ordinals identical
  to the renderer's `gamebryo_to_vk_blend_factor` table (`ONE=0, ZERO=1,
  DstColor=4, SrcAlpha=6, InvSrcAlpha=7`), not GL constants (real GL blend
  enums are large hex values, e.g. `GL_SRC_ALPHA = 0x0302`, not `6`). Read
  directly through the renderer table, every authored mode is already correct:
  - Standard `(6,7)` ‚Üí `SRC_ALPHA, INV_SRC_ALPHA` ‚úì
  - Additive `(6,0)` ‚Üí `SRC_ALPHA, ONE` ‚úì (correct additive)
  - Multiplicative `(4,1)` ‚Üí `DST_COLOR, ZERO` ‚úì (correct multiply)

  Applying the `0‚Üî1` swap breaks the two that use `0` or `1`:
  - Additive dst `0` ‚Üí `gl_to_gamebryo_blend(0)=1` ‚Üí `gamebryo_to_vk_blend_factor(1)=ZERO`
    ‚Üí `srcŒ±¬∑src + 0¬∑dst` = **alpha-weighted opaque overwrite, not additive**.
  - Multiplicative dst `1` ‚Üí `gl_to_gamebryo_blend(1)=0` ‚Üí `gamebryo_to_vk_blend_factor(0)=ONE`
    ‚Üí `DstColor¬∑src + 1¬∑dst` = **destination leaks through, wrong multiply**.

  Standard `(6,7)` is a fixed point of the swap (6‚Üí6, 7‚Üí7), so glass and decal
  content is unaffected ‚Äî masking the regression.
- **Evidence**:
  - Reference authored tuples: `BaseMaterialFile.cs:377-384`
    (`a=1,b=6,c=0` = Additive; `a=1,b=4,c=1` = Multiplicative).
  - `gl_to_gamebryo_blend`: `0 => 1`, `1 => 0`, else `as u8`
    (`material.rs:497-503`).
  - Renderer table: `0 => ONE`, `1 => ZERO`, `4 => DST_COLOR`, `6 => SRC_ALPHA`,
    `7 => ONE_MINUS_SRC_ALPHA` (`crates/renderer/src/vulkan/pipeline.rs:162-177`).
  - `#1651` commit `ada75ee3` message asserts additive cards author `(1,1)` ‚Äî
    the value that does not appear in `ConvertAlphaBlendMode`.
  - The only additive test (`byroredux/src/asset_provider/tests.rs`, additive
    case) uses a synthetic `(function=2, src=1, dst=1)` tuple that matches no
    real Bethesda material, so it green-lights the wrong translation.
- **Impact**: FO4 additive-glow / energy-effect BGEM+BGSM cards (muzzle flashes,
  energy-weapon beams, glow decals, force-field overlays) render as
  alpha-weighted opaque instead of additive; multiplicative cards render with a
  wrong destination term. Wrong `Material` out of the parser‚Üí`Material`
  boundary ‚Üí HIGH per the severity guide (all-FO4 blast radius, blend state is
  invisible to `cargo test`).
- **Related**: `#1651`, `#1649` (particle additive twin).
- **Suggested Fix**: Re-derive the mapping from the four authoritative
  `ConvertAlphaBlendMode` tuples end-to-end (author intent ‚Üí `vk::BlendFactor`)
  rather than from the unverified "GL enum" premise. The evidence points to the
  raw values already being Gamebryo ordinals, i.e. the pre-`#1651` verbatim
  forward was correct ‚Äî but do not blind-revert: verify each of Standard
  `(6,7)`, Additive `(6,0)`, Multiplicative `(4,1)`, None `(0,0,0)` produces its
  correct `vk` factor pair, replace the synthetic `(1,1)` test with the real
  Additive `(function=1, src=6, dst=0)` + Multiplicative `(4,1)` fixtures, and
  confirm with a RenderDoc capture of an FO4 additive card before committing.

### FO4-D2-02: `gl_to_gamebryo_blend` truncates u32 src/dst via `as u8` with no range guard
- **Severity**: LOW
- **Dimension**: 2 ‚Äî BGSM/BGEM Consumption
- **Location**: `byroredux/src/asset_provider/material.rs:497-503`
- **Status**: NEW
- **Description**: `src_blend`/`dst_blend` are parsed as `u32`
  (`crates/bgsm/src/base.rs:51-52`) and passed to `gl_to_gamebryo_blend(gl: u32)
  -> u8`, whose fallthrough does `other as u8`. A malformed/modded factor ‚â• 256
  wraps silently (`256 ‚Üí 0`), and values in `11..=255` land in the renderer's
  `_ => SRC_ALPHA` catch-all with no diagnostic. Not a vanilla-content problem
  (authored values are 0/1/4/6/7); a latent footgun for corrupt/modded input.
  (This function is the subject of FO4-D2-01; any fix there should also address
  the guard.)
- **Evidence**: `material.rs:501` `other => other as u8`; renderer
  `pipeline.rs:175` `_ => SRC_ALPHA`.
- **Impact**: Corrupt/out-of-spec modded blend factors map to an arbitrary vk
  factor with no log. Cosmetic-only, vanilla-safe.
- **Related**: FO4-D2-01.
- **Suggested Fix**: Clamp/validate to the known `0..=10` domain and `log::warn!`
  once on out-of-range input, mirroring the magic-vs-extension warn already in
  the same file.

### FO4-D3-01: BA2 DX10 `chunk_hdr_len` read but never used to advance the reader
- **Severity**: LOW
- **Dimension**: 3 ‚Äî BA2 Reader
- **Location**: `crates/bsa/src/ba2.rs:538-551, 580-582`
- **Status**: NEW (defense-in-depth gap; documented trade-off, not a live bug)
- **Description**: `read_dx10_records` reads `chunk_hdr_len` from `base[14..16]`
  and `debug_assert_eq!`s it to 24, but the chunk loop unconditionally reads a
  fixed `[0u8; 24]` per chunk. In release the assert compiles out, so a
  (hypothetical, third-party) archive declaring `chunk_hdr_len != 24` would
  misparse every following chunk with no telemetry. The inline comment already
  acknowledges the tolerant-in-release trade-off. Zero impact on vanilla FO4
  (all DX10 records set 24, confirmed via successful live extraction of
  `Fallout4 - Textures1.ba2` v7 and `DLCCoast - Textures.ba2` v1).
- **Evidence**: `chunk_hdr_len` (`ba2.rs:539`) feeds only the `debug_assert`
  (`:546`); the read uses the literal `[0u8; 24]` (`:581`).
- **Impact**: None on vanilla content; silent misparse only on a malformed /
  future DX10 archive with a wider chunk header.
- **Suggested Fix**: For parity with the `num_mips==0` (`ba2.rs:568`) and
  non-monotonic (`:628`) siblings, add a release-path `log::warn!` when
  `chunk_hdr_len != 24` (keep the tolerant clamp-to-24).

### FO4-D4-01: `parse_sub_index` recovery can desync the stream when `block_size` is `None`
- **Severity**: LOW
- **Dimension**: 4 ‚Äî NIF BSVER 130
- **Location**: `crates/nif/src/blocks/tri_shape/bs_tri_shape.rs:639-654`
- **Status**: NEW (latent, unreachable on current dispatch)
- **Description**: When `BsSubIndexTriShapeData::parse` errors AND the caller
  passed `block_size == None`, the recovery only does
  `stream.set_position(segmentation_start)` (`:653`) ‚Äî a bare rewind with no
  compensating forward skip, so a subsequent block read could start inside this
  block's segmentation payload. The `Some(size)` arm (`:639-651`) computes a
  proper skip. The live dispatcher (`blocks/mod.rs:466-478`) always passes
  `Some(_)` (block sizes appear in the header since V20.2.0.5), so the `None`
  path is currently unreachable.
- **Evidence**: `Some` arm computes a skip target; the `else` arm rewinds only.
- **Impact**: None today (unreachable). If a future size-less caller hits it on a
  decode failure, the outer block-loop's block_size resync still fires, so parse
  rate is unaffected ‚Äî but a debug breadcrumb would point at a misleading offset.
- **Suggested Fix**: If `parse_sub_index` ever gains a size-less caller, make the
  `None` arm return `Err` so the outer loop's block_size recovery is the single
  source of truth.
