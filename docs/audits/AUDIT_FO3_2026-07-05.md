## Executive Summary

**Compatibility level: HIGH — parity with FNV, no new correctness defects.** FO3
rides the FNV path end-to-end and the audit confirms that ride is clean: retail FO3
ships `bsver = 34` and `NifVariant::detect` resolves it to the same `FalloutNV`
variant / `GameKind::Fallout3NV` enum that FNV uses, so FO3 executes the *identical*
shared code with essentially no surface for an FNV-only assumption to diverge. Every
divergence the skill flagged as a risk (inline-shader-only material universe, Capital
Wasteland worldspace, FO3-era particle/collision authoring, pre-FNV record subforms)
was traced and found already handled and regression-pinned.

**Delta vs FNV**: The only *functional* FO3-distinctive gap is the **Obscript
scripting runtime** (1 257 SCPT records parse; nothing executes them) — a known,
tracked blocker, not a bug. Everything else FO3 needs is shared with FNV and green.

**Findings: 4, all LOW, all documentation/log-noise** (3 NEW, 1 premise-correction on
an existing issue). Zero CRITICAL / HIGH / MEDIUM. No behavioral change is required
for FO3 content to render at FNV parity.

## Dimension Findings

### Dimension 1 — FO3 Rendering Path (Inline Shaders) · 1 LOW

#### FO3-D1-001: Stale execution-order comment in `MATERIAL_KIND_NO_LIGHTING` shader branch
- **Severity**: LOW (documentation)
- **Dimension**: FO3 Rendering — Inline Shaders
- **Location**: `crates/renderer/shaders/triangle.frag` — the `mat.materialKind == MATERIAL_KIND_NO_LIGHTING` branch
- **Status**: NEW
- **Game Affected**: FO3 (bsver 34) — also FNV (shared 102 path)
- **Description**: The comment above the NO_LIGHTING early-return states the alpha-test
  discard "already ran (~line 1100)". The alpha-test discards (`if (!pass) discard;`
  and the implicit fully-transparent discard) actually run well *above* the NO_LIGHTING
  return; the cited "~line 1100" points *past* the return, inverting the apparent order.
  Behavior is correct — this is a stale line reference.
- **Evidence**: `discard` for alpha-test lives in the per-instance alpha-test block far
  above `mat.materialKind == MATERIAL_KIND_NO_LIGHTING`; "~line 1100" is greater than the
  return's own location.
- **Impact**: None at runtime. Maintenance risk only: a future edit to the FO3
  terminal/HUD/neon/blood-splat NO_LIGHTING path could "reinstate" a discard believing it
  was skipped.
- **Related**: `MATERIAL_KIND_NO_LIGHTING = 102u` (`include/shader_constants.glsl`); 102 set in `import/material/walker.rs` NoLighting arm.
- **Suggested Fix**: Replace "(~line 1100)" with a symbol reference ("the per-instance
  alpha-test discard above") so it survives line drift. Doc-only; no cargo-test visibility.

### Dimension 3 — ESM Record Coverage (Fallout3.esm) · 2 LOW

#### FO3-D3-001: XATO texture-overlay path is inapplicable to FO3 (FONV-only subrecord); benign FNV misparse
- **Severity**: LOW
- **Dimension**: ESM Record Coverage
- **Location**: `crates/plugin/src/esm/cell/walkers.rs::parse_refr_group` (the `b"XATO"` arm) → `byroredux/src/cell_loader/refr.rs::build_refr_texture_overlay`
- **Status**: NEW
- **Game Affected**: FO3 (premise refutation); FNV (benign misparse, shared parser)
- **Description**: The audit premise that FO3 REFRs carry XATO/XTNM/XTXR texture overrides
  (#584) is **incorrect**. openmw's authoritative `esm4/loadrefr.cpp` tags `XATO` as
  `// FONV` (New-Vegas-only) and *skips* it — it is the Activation-Prompt subrecord, not a
  TXST FormID. FO3 REFRs never carry XATO. The ByroRedux `b"XATO"` arm has no game gate and
  the comment claiming it holds "140 MNAM-only vanilla FO4 TXSTs" is unsourced.
- **Evidence**: `reference/openmw/components/esm4/loadrefr.cpp` — `case ESM::fourCC("XATO"): // FONV` (skip list); the FO3-tagged REFR subrecords carry no XATO. `walkers.rs` reads `XATO` as a u32 FormID unconditionally.
- **Impact**: **FO3**: zero (arm never fires). **FNV**: a REFR's XATO activation-prompt
  string has its first 4 bytes read as a FormID, near-certainly missing `texture_sets`,
  yielding a spurious empty `Some(overlay)` + wasted per-REFR allocation. Functionally
  inert (base-mesh textures ride through).
- **Related**: #584, #1654. Overlay tests (`refr_texture_overlay_tests.rs`) are entirely FO4-authored — no FO3/FNV fixture exercises the arm.
- **Suggested Fix**: Either (a) game-gate the XATO/XTNM/XTXR arms to FO4+ so FNV
  activation-prompt strings aren't mis-read, or (b) at minimum correct the `walkers.rs`
  comment to cite the real provenance (XATO = FONV REFR subrecord, not "FO4 TXST").

#### FO3-D3-002: FO3 baseline comment in `parse_rate_fo3_esm` is self-contradictory and stale
- **Severity**: LOW (documentation)
- **Dimension**: ESM Record Coverage
- **Location**: `crates/plugin/tests/parse_real_esm.rs::parse_rate_fo3_esm` (+ `FO3_TOTAL_FLOOR`)
- **Status**: NEW
- **Game Affected**: FO3
- **Description**: The test asserts `index.total() >= FO3_TOTAL_FLOOR` with
  `FO3_TOTAL_FLOOR = 30_000`, but the explanatory comment says "18,007 records observed …
  FO3_TOTAL_FLOOR sits slightly below." 30 000 is 12k *above* 18 007 — mutually
  inconsistent. The "18,007" figure is a stale leftover (from AUDIT_FO3_2026-04-19, before
  the index grew to ~95 typed categories). `index.total()` sums the typed category maps (a
  subset of structured records), which is also not the file's raw 37 459-structured /
  44 657-total baseline — three unreconciled numbers in one place.
- **Evidence**: `FO3_TOTAL_FLOOR: usize = 30_000` vs the "18,007 … slightly below" comment; `index.rs::total()` sums `Self::categories()` (typed-map records only).
- **Impact**: No runtime effect (assertion passes on live data). Documentation rot that
  would mislead a future auditor about the FO3 parse yield.
- **Related**: The brief's "do NOT use any older 13,684 structured figure" note — same stale-baseline hazard class.
- **Suggested Fix**: Update the comment to state what `index.total()` measures and cite the
  re-verified 2026-05-26 figures (44 657 total = 37 459 structured + 7 198 NAVM) for the
  *file*, distinguished from `total()`.

## FO3-Distinctive Gaps

1. **Obscript scripting runtime (the one real functional gap — known blocker, not a bug).**
   1 257 FO3 SCPT records parse but no runtime executes them. **Accuracy correction to the
   audit premise**: the shipped M47.2 vertical slice is a **Papyrus `.pex`** decompiler,
   whereas FO3 scripts are **Obscript (SCDA bytecode / SCTX source)**. ROADMAP M47.2 lists
   "Obscript via SCTX" as *Remaining*, so the M47.x progress does **not** yet close the FO3
   loop — FO3 quest/world interactivity stays blocked until an Obscript recognizer lands.
   Runtime owned by `/audit-scripting`; recorded here, not filed.
2. **Inline-shader-only material universe.** Not a gap so much as a constraint the renderer
   already honors: FO3 authors no BGSM, so the Disney-BSDF lobe is unreachable and every
   material resolves through the legacy classifier. Verified clean (Dim 1).
3. **Capital Wasteland worldspace.** Distinct WRLD form ID — handled by the game-agnostic
   worldspace-selection + grid-containment path; no FNV-hardcoding (Dim 4). Open GPU render
   bench (R6a-stale-15) is measurement, not a feature gap.
