FNV is the reference title and remains in excellent shape. All eight dimensions were run inline against
the live tree, and the reference-data paths were exercised on-device (interior bench-of-record + exterior
5×5 grid + skinning e2e + real-ESM/real-NIF parse). **No regressions were found** against any shipped
foundation; every regression-guard checked still holds.

Headline real-data results (all on `2d4c350d`, RTX 4070 Ti):

- **FNV NIF parse rate: 100.00% (14881/14881 clean, 0 truncated, 0 failures)** — exact match to ROADMAP.
- **FNV ESM parse: clean** (`parse_rate_fnv_esm` passes; 77 825 structured + 5 625 long-tail records).
- **Prospector Saloon interior bench: 159.6 wall FPS / 6.26 ms / fence=4.31 ms / 3626 ent / 1224 draws** —
  ~2.1× faster than the (332-commit-stale) ROADMAP baseline of 76.2 FPS / 13.11 ms / fence=11.12 / 3516 ent.
- **`tex.missing` = 0** ("No missing textures — all entities have resolved textures", 0 fallback entities).
- **FNV exterior 5×5 grid (radius 2): 25 cells, LAND + splat + LOD ring, 5920 ent @ 246 FPS, no errors.**

Findings are 1 MEDIUM (a telemetry/silent-drop gap in the newly-shipped PHYSAL ragdoll body-resolution path,

(ROADMAP already flags this as R6a-stale-15).

## Findings

### FNV-D7-01: Ragdoll body + dependent constraints dropped silently on bone-name miss (no telemetry)
- **Severity**: MEDIUM
- **Dimension**: 7 — PHYSAL Ragdoll
- **Location**: `byroredux/src/ragdoll.rs:88-90` (body skip) and `byroredux/src/ragdoll.rs:108-111` (constraint skip)
- **Status**: Related to OPEN #1539 (D7-02) — same telemetry gap, body-resolution branch
- **Description**: `resolve_template` skips any `ImportedRagdoll` body whose `bone_name` is absent from the
  skeleton's `name→EntityId` map (`let Some(&bone) = skel_map.get(&b.bone_name) else { continue };`) with no
  log line. Any constraint referencing a dropped body is then also silently skipped
  (`(Some(a), Some(b)) = ... else { continue }`). The only downstream signal is the `< 2 bodies` / empty-constraint
  early-`None` returns, which collapse a partially-resolved ragdoll to "no ragdoll" with zero diagnostic about
  *why*. #1539 already flags the constraint-*kind* drop at import; this is the body-*name*-resolution sibling.
- **Evidence**:
  ```rust
  let Some(&bone) = skel_map.get(&b.bone_name) else {
      continue;            // dropped body — no warn
  };
  ...
  let (Some(a), Some(b)) = (old_to_new[c.body_a], old_to_new[c.body_b]) else {
      continue;            // dropped constraint — no warn
  };
  ```
- **Impact**: On a real FNV creature/NPC skeleton whose bone-naming diverges from the ragdoll's authored bone
  names (skeleton variant, renamed bone, importer name-canonicalisation mismatch), the ragdoll degrades or
  vanishes with no breadcrumb. The verified reference path (Doc Mitchell, 18 bodies) is unaffected; the risk is
  the long tail of FNV skeletons. Defense-in-depth / observability gap, not a correctness break on the reference
  content — hence MEDIUM.
- **Related**: #1539 (D7-02), #1540 (D7-03)
- **Suggested Fix**: Emit a `log::warn!` (rate-limited / `Once`-gated per skeleton) listing the count and names of
  dropped bodies and dependent constraints, mirroring the skinning path's "N unresolved bones" telemetry. Consider
  folding into #1539's fix so the ragdoll-fragmentation diagnostic is unified across both drop sites.

### FNV-D5-LOW-01: V20_0_0_4 ambiguous-routing WARN still fires on FNV mesh data (closed-but-unresolved #1219)
- **Severity**: LOW
- **Dimension**: 5 — NIF Parser
- **Location**: `crates/nif/src/version.rs` (`NifVariant::detect` ambiguity warn)
- **Status**: Existing: #1219 (CLOSED "needs sample data") — diagnostic still active on the reference title
- **Description**: Parsing `Fallout - Meshes.bsa` emits the `ambiguous tuple (V20_0_0_4, user_version=11,
  user_version_2=11) routed as Oblivion` WARN. #1219 was closed without resolution (awaiting sample-data
  disambiguation), so the warning persists for FNV content that shares the v20.0.0.4 / user 11 tuple with FO3/Oblivion.
- **Impact**: None observed — FNV still parses **100.00% clean** with this routing, so no data is lost. It is log
  noise on the reference title and a latent risk should an FNV-specific field-layout divergence ever ride that tuple.
- **Related**: #1219
- **Suggested Fix**: Capture a representative FNV v20.0.0.4/user-11 sample and flip the routing (or downgrade the
  warn to `debug!` once confirmed benign for FNV), closing out the #1219 follow-up.

### FNV-D4-LOW-02: SCHR header doc comment lists 16 bytes but says "20-byte"
- **Severity**: LOW
- **Dimension**: 4 — ESM Record Parser
- **Location**: `crates/plugin/src/esm/records/script.rs:5-6`
- **Status**: NEW (doc-rot only; parser is correct)
- **Description**: The module doc says `SCHR — 20-byte header: numRefs u32, compiled_size u32, var_count u32,
  script_type u16, flags u16`, but those five fields sum to 16 bytes. The actual 20-byte layout (correctly read at
  `script.rs:122-144`) has a leading unused/pad `u32` before `numRefs` that the doc omits.
- **Impact**: Documentation only — the parser reads the right 20 bytes. A future maintainer reconciling the doc
  against the code could misjudge the field offsets.
- **Suggested Fix**: Add the leading `pad u32` to the doc field list so the byte count matches the "20-byte" claim.

### FNV-D1-LOW-03: NifImportRegistry::insert doc claims default cache mode is unlimited (it is 2048)
- **Severity**: LOW
- **Dimension**: 1 — Cell Loading
- **Location**: `byroredux/src/cell_loader/nif_import_registry.rs:294-296`
- **Status**: NEW (doc-rot only)
- **Description**: The `insert` doc says "Empty Vec on the no-eviction path (the default `BYRO_NIF_CACHE_MAX=0`
  mode)", but the constructor (`new()`, line 170-173) defaults to `unwrap_or(2048)` — `BYRO_NIF_CACHE_MAX=0`
  is the *opt-in unlimited* mode, not the default. With the real default of 2048, eviction *can* fire (and the
  returned clip-handle Vec is the mechanism that must be released).
- **Impact**: Documentation only — the LRU code and `#[must_use]` contract are correct. The doc inverts which mode
  is the default, which could mislead a reader into thinking eviction never runs by default.
- **Suggested Fix**: Reword to "the no-eviction path (`BYRO_NIF_CACHE_MAX=0`, opt-in unlimited)"; the default 2048
  cap *does* evict.
