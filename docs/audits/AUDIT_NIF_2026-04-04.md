## Findings — CRITICAL

### NIF-009: NiBooleanExtraData reads u32 instead of u8
- **Severity**: CRITICAL
- **Dimension**: Block Parsing
- **Location**: `crates/nif/src/blocks/extra_data.rs:46-48`
- **Game Affected**: FNV, FO3, all games
- **Status**: NEW
- **Description**: NiBooleanExtraData.boolean_data is type `byte` in nif.xml (1 byte) but the parser reads `u32` (4 bytes), consuming 3 extra bytes. The block_size guard silently adjusts, but the value is wrong (reads garbage from adjacent bytes).
- **Evidence**: `integer_value: Some(stream.read_u32_le()?)` should be `integer_value: Some(stream.read_u8()? as u32)`
- **Impact**: Every NiBooleanExtraData block over-reads 3 bytes. Masked by block_size adjustment.
- **Suggested Fix**: Change to `stream.read_u8()? as u32`.

## Findings — HIGH

### NIF-008: BSShaderPPLightingProperty missing refraction/parallax fields
- **Severity**: HIGH
- **Dimension**: Block Parsing + Version Handling
- **Location**: `crates/nif/src/blocks/shader.rs:42-54`, `crates/nif/src/version.rs:115-117`
- **Game Affected**: FNV (bsver=34)
- **Status**: NEW
- **Description**: BSShaderPPLightingProperty should read 4 fields after texture_set_ref for bsver >= 15: Refraction Strength (f32), Refraction Fire Period (i32), Parallax Max Passes (f32), Parallax Scale (f32) = 16 bytes. Instead, it reads Emissive Color (4xf32 = 16 bytes) under condition `has_shader_emissive_color()` which checks bsver >= 34. The emissive color field is from a DIFFERENT inheritance path (BSShaderPPLightingProperty does NOT have emissive color in nif.xml — that's BSLightingShaderProperty). By coincidence, both are 16 bytes, so stream position is correct but values are completely wrong.
- **Evidence**: `has_shader_emissive_color()` returns true for FNV (bsver=34 >= 34), reading 4 floats labeled "emissive" that are actually refraction_strength, refraction_fire_period, parallax_max_passes, parallax_scale.
- **Impact**: Shader refraction and parallax parameters are garbage values. Not currently rendered, but will be wrong when N23.2 adds shader completeness.
- **Suggested Fix**: Replace emissive_color read with the 4 correct fields from nif.xml. Remove `has_shader_emissive_color` or fix its condition.

### NIF-301: NiTexturingProperty missing bump map extra fields
- **Severity**: HIGH
- **Dimension**: Stream Position
- **Location**: `crates/nif/src/blocks/properties.rs:115-164`
- **Game Affected**: FNV, FO3, Oblivion
- **Status**: NEW
- **Description**: When a NiTexturingProperty has a bump texture (slot 5), nif.xml specifies 3 additional fields AFTER the TexDesc: Bump Map Luma Scale (f32), Bump Map Luma Offset (f32), Bump Map Matrix (2x2 f32 = 4 floats). Total: 24 extra bytes. The parser reads only the TexDesc and skips these. This is the root cause of the consistent "expected N bytes, consumed N-1" warnings.
- **Evidence**: nif.xml `NiTexturingProperty` bump texture section has `Bump Map Luma Scale`, `Bump Map Luma Offset`, `Bump Map Matrix` fields after the TexDesc.
- **Impact**: Stream position off by 24 bytes for any NIF with bump-mapped textures. Masked by block_size guard.
- **Suggested Fix**: After reading bump TexDesc, read the 3 additional fields (or skip 24 bytes if not stored).

### NIF-302: NiBooleanExtraData reads u32 instead of u8 (duplicate of NIF-009)
- **Severity**: HIGH (duplicate — same as NIF-009)
- **Note**: Found independently by both dim 1+2 and dim 3 agents. Confirms the bug.

## Findings — MEDIUM

### NIF-010: NiGeometryData uses NiGeometryDataFlags instead of BSGeometryDataFlags
- **Severity**: MEDIUM
- **Dimension**: Block Parsing
- **Location**: `crates/nif/src/blocks/tri_shape.rs:403,448`
- **Game Affected**: FNV, FO3
- **Status**: NEW
- **Description**: The data flags field in NiGeometryData is interpreted using NiGeometryDataFlags bit layout, but for Bethesda streams (bsver > 0) it should use BSGeometryDataFlags. Works in practice because the UV count bits overlap, but tangent/bitangent flags differ.
- **Impact**: Tangent generation may be skipped when it shouldn't be, or generated when not needed.
- **Suggested Fix**: Check bsver and use the Bethesda flag interpretation when > 0.

### NIF-011: NiTexturingProperty texture slot conditionals may skip reads
- **Severity**: MEDIUM
- **Dimension**: Block Parsing
- **Location**: `crates/nif/src/blocks/properties.rs:126-139`
- **Game Affected**: FNV, FO3, Oblivion
- **Status**: NEW
- **Description**: The parser skips reading `has_X_texture` booleans for slots 1-6 when `texture_count` is below the slot number. But nif.xml may require reading the boolean unconditionally, with the actual TexDesc conditional on the boolean value. The current code is safe when texture_count accurately reflects the present slots, which it does in practice.
- **Impact**: Minor — would only matter if a NIF had texture_count=2 but still serialized has_texture booleans for later slots.
- **Suggested Fix**: Consider always reading the has_texture boolean for each slot up to some maximum, and only reading TexDesc if true.

### NIF-303: NiTexturingProperty missing parallax offset float
- **Severity**: MEDIUM
- **Dimension**: Stream Position
- **Location**: `crates/nif/src/blocks/properties.rs:134-139`
- **Game Affected**: FNV, FO3
- **Status**: NEW
- **Description**: When parallax texture (slot 7) is present, nif.xml specifies a `Parallax Offset` float after the TexDesc. The parser doesn't read it.
- **Impact**: 4 bytes under-read when parallax texture present. Masked by block_size guard.
- **Suggested Fix**: Read an f32 after the parallax TexDesc.

### NIF-404: NiStencilProperty two-sided detection uses fragile byte heuristic
- **Severity**: MEDIUM
- **Dimension**: Import Pipeline
- **Location**: `crates/nif/src/import.rs:602-624`
- **Game Affected**: FNV, FO3, Oblivion
- **Status**: NEW
- **Description**: NiStencilProperty is parsed as NiUnknown (no dedicated parser), and find_two_sided() reads the last 4 bytes of the raw data to extract draw_mode. This is fragile and depends on block layout assumptions.
- **Impact**: Two-sided detection may fail on NIFs with unusual NiStencilProperty sizes.
- **Suggested Fix**: Add a proper NiStencilProperty parser to the dispatch table (N23.3 scope).

### NIF-501: NiStencilProperty missing from parser dispatch
- **Severity**: MEDIUM
- **Dimension**: Coverage
- **Location**: `crates/nif/src/blocks/mod.rs`
- **Game Affected**: FNV, FO3, Oblivion
- **Status**: NEW
- **Description**: NiStencilProperty is not in the parse_block dispatch table. Falls through to NiUnknown with raw byte capture. This forces the fragile heuristic in NIF-404.
- **Suggested Fix**: Add NiStencilProperty parser (NiObjectNETData + flags u16 + stencil fields + draw_mode).
