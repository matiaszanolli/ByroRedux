## Findings

### ECS-D7-NEW-01: dead code in `mg07_on_activate_system`
- **Severity**: LOW
- **Dimension**: 7 (script lifecycles / R5 demo)
- **Location**: [`crates/scripting/src/papyrus_demo/mg07_door.rs:303-321, 394-395`](../../crates/scripting/src/papyrus_demo/mg07_door.rs#L303-L321)
- **Status**: NEW
- **Description**: Two unrelated dead-code blocks left from
  iterative development of the MG07 R5 follow-up translation
  (commit `58fe3ce4`).

  Block 1 (lines 303-321) acquires a `QuestStageState` resource
  read lock, performs a useless `get_stage(QuestFormId(0))` lookup,
  assigns `stage_state_done_10 = true` to a `bool` that's never
  read, and then `let _ = stage_state_done_10;` to suppress an
  unused-variable warning. The real per-door stage resolution
  happens inside the loop at line 367. The pre-loop block is a
  half-finished "hoist the stage check" optimisation that was
  abandoned when the per-door quest FormID requirement was
  discovered, but the dead code was never cleaned up.

  Block 2 (lines 394-395) declares `let mut
  pending_my_door_activates: Vec<EntityId> = Vec::new();` and then
  immediately suppresses an unused-mut warning via `let _ = &mut
  pending_my_door_activates;`. The Vec is never written or read —
  the actual cross-reference activation is emitted by the sibling
  `mg07_tick_system`, not by `mg07_on_activate_system`.
- **Evidence**:
  ```rust
  // mg07_door.rs:303-321 — dead block 1
  pub fn mg07_on_activate_system(world: &World) {
      let player = world.resource::<PlayerEntity>().0;
      let stage_state_done_10: bool;
      {
          let stage_state = world.resource::<QuestStageState>();
          // ...
          let _ = stage_state.get_stage(QuestFormId(0));
          stage_state_done_10 = true; // unused; re-resolved per door
      }
      let _ = stage_state_done_10;
      // ...
  }
  ```
  ```rust
  // mg07_door.rs:394-395 — dead block 2
  let mut pending_my_door_activates: Vec<EntityId> = Vec::new();
  let _ = &mut pending_my_door_activates; // suppress unused-write warning
  ```
- **Impact**: Code quality only. Pre-loop block does an extra
  resource-read lock acquire + drop + useless HashMap lookup
  every system invocation (negligible cost — single read lock,
  single hash miss). The `pending_my_door_activates` Vec is a
  one-time zero-capacity allocation per system invocation, also
  negligible. The real cost is **future-reader confusion** — both
  blocks look load-bearing at a glance and will distract anyone
  auditing the system in 6 months. R5 documentation reviewers
  (this one included) are the first such readers.
- **Related**: None — fresh in the R5 follow-up 3 commit
  (`58fe3ce4`). 12 tests in `papyrus_demo/mg07_door/tests.rs`
  pass independently of these blocks.
- **Suggested Fix**: Delete both blocks. Phase 1 of the system
  then reads as:
  ```rust
  pub fn mg07_on_activate_system(world: &World) {
      let player = world.resource::<PlayerEntity>().0;

      // Two-phase: collect (read), then apply (write).
      enum Outcome { /* ... */ }
      let mut outcomes: Vec<Outcome> = Vec::new();
      {
          // ... existing read-phase body ...
      }
      // ...
  }
  ```

## Verified Invariants (Dims 1-7)

### Dim 1 — Storage Correctness

- `SparseSetStorage::insert` overwrites in place when entity
  already has the component (sparse_set.rs:39-48); never duplicates.
  Tests: `overwrite` at sparse_set.rs:144.
- `SparseSetStorage::remove` correctly fixes up the sparse pointer
  for the swap-moved entity (sparse_set.rs:75). Tests:
  `swap_remove` at sparse_set.rs:153.
- `PackedStorage::insert` maintains sort invariant via
  `binary_search` (packed.rs:41-51); overwrites in-place on Ok,
  inserts at sorted position on Err.
- `PackedStorage::remove` shifts via `Vec::remove` on the matched
  index, preserving sort order (packed.rs:54-62).
- `insert_bulk` correctly handles dedup-keep-last (packed.rs:140-162).
- **New components** (`AttachPoints`, `ChildAttachConnections`,
  R5's `RumbleOnActivate` / `RecurringUpdate` / `OnUpdateEvent` /
  `QuestStageAdvanced` / `MG07LabyrinthianDoor` / `KeystoneInventory`
  / `UiMessageCommand` / `Dlc2Ttr4aPlayerScript` / `ActorStats` /
  `QuestAdvanceOnActivate` / `CameraShakeCommand` /
  `ControllerRumbleCommand`) all impl `Component` with
  `SparseSetStorage<Self>` — the standard pattern.

### Dim 2 — Query Safety

- `query_2_mut` and `query_2_mut_mut` enforce TypeId-sorted lock
  acquisition (`world.rs:410-440` / `world.rs:472-499`). Same-type
  call panics with explicit message via `assert_ne!`.
- `QueryRead` / `QueryWrite` Deref impls (`query.rs`) gate access
  through the `RwLockReadGuard` / `RwLockWriteGuard` they hold,
  with no aliasing escape.
- Debug-build `lock_tracker` (`lock_tracker.rs`) records every
  read / write acquisition per-thread, panics on same-thread
  conflict with explicit message ("ECS deadlock detected:
  attempted write lock on `T` while ...").

### Dim 3 — World Integrity

- TypeMap downcasts: `storage_write` at `world.rs:726-745` uses
  `.downcast_mut::<T::Storage>().expect("storage type mismatch
  (bug in World)")`. Lazy `entry().or_insert_with(...)`
  guarantees the TypeId-keyed slot was created with `T::Storage`,
  so downcast can never fail in practice; the panic is for the
  hypothetical TypeMap-key-collision case (which Rust's TypeId
  guarantees against).
- `find_by_name` at `world.rs:316`: `names.iter().find(|(_, n)|
  n.0 == sym)` — both operands are `FixedString` symbols
  (integer equality). No string compare in the hot path.
- Entity ID monotonicity: `World::spawn` increments `next_entity`
  on every call; no reuse path. Overflow at `u32::MAX` documented
  as a known limitation (effectively unreachable — would require
  ~4 G spawns in one process).

### Dim 4 — Resource Safety

- `resource<R>` panics with `panic!("Resource `{}` not found —
  call world.insert_resource() first", std::any::type_name::<R>())`
  on miss (`world.rs:540-545`). Type name included for fast triage.
- `try_resource<R>` and `try_resource_mut<R>` return `Option`
  (`world.rs:658-683`).
- `try_resource_2_mut<A, B>` performs both existence checks before
  acquiring either lock (`world.rs:714-720`) — pre-#465 the first
  lock could be held while the second check failed, leaking a
  half-acquired pair.
- New debug-server resources (`ScreenshotBridge`,
  `DebugShutdownSignal`, etc.) in `resources.rs` follow the
  standard `impl Resource for X {}` pattern; no novel patterns
  introduced.
- R5 resources `PlayerEntity` and `QuestStageState` both impl
  `Resource` with no field-visibility leakage (PlayerEntity is
  `Copy`, QuestStageState owns its inner HashMap).

### Dim 5 — System & Scheduler

- Scheduler blanket impl for `Fn(&World, f32)` works against the
  R5 systems (verified by `cargo test -p byroredux-scripting`'s
  successful test runs invoking the systems directly with
  `&world` references).
- R5 systems all follow the two-phase collect-then-apply pattern
  used elsewhere (`rumble_on_activate_system`,
  `quest_advance_on_activate_system`, `dlc2_ttr4a_on_update_system`,
  `mg07_on_activate_system`). No held-borrow-across-mutation
  violations.

### Dim 6 — Unsafe Code Review

- **Zero `unsafe` blocks** in `crates/core/src/ecs/` and
  `crates/core/src/animation/` (`grep -rn 'unsafe '` returns no
  hits in the audit scope). The Dim 6 review item about
  `World::get()` using raw pointer to extend lifetime is **no
  longer applicable** — the lifetime is handled by the
  `ComponentRef<'_, T>` wrapper holding an `RwLockReadGuard`

## Notes on the R5 Follow-up Campaign

`8d054a9d`, `58fe3ce4`) added substantial new surface area to
`crates/scripting/`:

in `mg07_on_activate_system`) — code-quality only, no behavioral
or perf impact. Every other R5-added construct conforms to the
existing ECS patterns and is covered by tests (47 new in
byroredux-scripting alone).

`mg07_on_activate_system` (`beenOpened` is intentionally not
flipped to mirror the Papyrus author's `==`-not-`=` typo, with
test `beenopened_is_never_flipped_due_to_source_typo` enforcing
the preservation) is **deliberate per the R5 contract** and not
a finding.
