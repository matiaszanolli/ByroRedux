# NIF Parser Audit ‚ÄĒ 2026-07-05

**Scope:** byte-accurate correctness of the NIF binary-format parser
(`crates/nif/src/`) across the Oblivion ‚Üí Starfield version span. The recurring
failure mode is **stream-position drift** ‚ÄĒ a block over-/under-reads its
payload, consumed bytes diverge from the header `block_sizes` entry, and either
the reconciliation masks it (silent corruption) or, on Oblivion-era files with
no `block_sizes` table, every following block misaligns and the scene truncates.

**Method:** 6 dimension agents, each verifying live code by symbol-grep (not line
numbers) against nif.xml (`/mnt/data/src/reference/nifxml/nif.xml`) + the nifly
reference, with empirical `nif_stats --drift-histogram` sweeps over real mesh
archives (FNV / Skyrim SE / FO4 / Starfield on disk). Engine not launched.
Reference: `docs/engine/nif-parser.md`.

---

## Executive Summary

**9 findings: 3 MEDIUM, 6 LOW. Dimension 4 (geometry handoff) fully clean.** The
parser is byte-accurate on every `block_sizes`-bearing game empirically swept
(FNV / Skyrim SE / FO4 = **0 drift events**). All actionable correctness findings
are **Starfield-only** ‚ÄĒ the one game with a size table *and* undocumented
trailing fields the parser doesn't yet capture. The remaining findings are
dead-abstraction hygiene (Dim 2), test-harness measurement blind spots (Dim 3),
a doc mis-attribution (Dim 5), and one alloc-idiom consistency nit (Dim 6).

## Findings

### NIF-D1-001: `BSEffectShaderProperty` drops a 32-byte Starfield trailing tail (silently absorbed)
- **Severity**: MEDIUM
- **Dimension**: Stream Position
- **Game Affected**: Starfield only (`NifVariant::Starfield`, bsver ‚Č• `bsver::STARFIELD` = 172). FO76/FO4 unaffected.
- **Location**: `crates/nif/src/blocks/shader.rs` (`BSEffectShaderProperty::parse`, dispatched in `crates/nif/src/blocks/mod.rs`)
- **Status**: NEW. The missed sibling of the resolved `BSLightingShaderProperty` tail, #1606.
- **Description**: After the FO76+ trailing fields, `BSEffectShaderProperty::parse` returns. Every retail-Starfield instance carries ~32 additional undocumented bytes the parser never consumes; `block_size` reconciliation seeks the stream to `start + block_size` so the file parses "clean". This is exactly #1606's situation for `BSLightingShaderProperty`, which was given `parse_with_size(stream, block_size)` + `read_starfield_tail(...)` capturing `block_size - consumed` opaque bytes. `BSEffectShaderProperty` was never given the treatment ‚ÄĒ its dispatch arm calls `parse(stream)` with no `block_size` argument, so it structurally cannot capture the tail.
- **Evidence**: Dispatch asymmetry in `blocks/mod.rs` ‚ÄĒ `"BSLightingShaderProperty" => ‚Ä¶parse_with_size(‚Ä¶block_size‚Ä¶)` vs `"BSEffectShaderProperty" => ‚Ä¶parse(stream)` (no size, no tail hook). `read_starfield_tail` + `starfield_tail: Vec<u8>` exist ONLY on `BSLightingShaderProperty`. Drift histogram: `BSEffectShaderProperty drift=+32` on 48/48 (LODMeshes) and 118/118 (MeshesPatch) instances ‚ÄĒ perfectly systematic.
- **Impact**: 32 undocumented Starfield effect-shader bytes silently discarded; no cascade (reconciliation realigns), but invisible to `per_block_baselines.rs` (which gates parsed-vs-unknown, not drift). A future consumer has nothing to read.
- **Related**: #1606 (BLSP starfield_tail), #746 (BSEffect FO76 `>=` gate), #1510 (BSEffect Starfield empty-name stopcond).
- **Suggested Fix**: Mirror #1606 exactly ‚ÄĒ add `BSEffectShaderProperty::parse_with_size(stream, block_size)` that runs the existing body then `read_starfield_tail(...)` into a new `starfield_tail: Vec<u8>` field; switch the `blocks/mod.rs` arm to `parse_with_size`. Add `parse_bs_effect_starfield_captures_trailing_tail` paralleling the BLSP guard test.

### NIF-D1-002: `BSWeakReferenceNode` systematically under-consumes 2 bytes on Starfield (silently absorbed)
- **Severity**: MEDIUM
- **Dimension**: Stream Position
- **Game Affected**: Starfield only (Starfield-introduced block; bsver ‚Č• 172).
- **Location**: `crates/nif/src/blocks/node.rs` (`BsWeakReferenceNode::parse`, dispatched in `crates/nif/src/blocks/mod.rs`)
- **Status**: NEW.
- **Description**: `BsWeakReferenceNode::parse` reads the `NiNode` base then the weak-ref / water-ref lists and returns, stopping a constant **2 bytes short of `block_size`** on 6894 instances (8 short on 13). The parser's own doc comment claims it "reads and discards the trailing data so block alignment is maintained" ‚ÄĒ the residual drift means that contract is not met. The nifly reference matches the top-level list layout and the per-ref `formID` gate (`Stream()>=173` == `bsver::SF_FORM_ID`), so the missing 2 bytes are neither. The constant +2 (independent of ref counts, present even with empty lists) points to either a 2-byte trailing/pad field or a NiNode-level Starfield field ‚ÄĒ `version.rs` documents that `bsver >= SF_FORM_ID` content "carries a form_id field in some blocks (e.g. NiNode)", yet `NiNode::parse` reads no such field.
- **Evidence**: `node.rs` `BsWeakReferenceNode::parse` returns immediately after the water-ref loop, no trailing read, no `block_size` awareness. Doc comment contradicted by the +2 residual. `version.rs` `SF_FORM_ID` comment vs `NiNode::parse` having no bsver‚Č•173 branch. Drift histogram (MeshesPatch): `+2 √ó6894`, `+8 √ó13`.
- **Impact**: 2 B/node dropped on effectively every Starfield packin/composite-LOD reference node (6907 in one patch archive). Discard-only today (LOD-streaming payload unconsumed pending M35+), no visible artifact, but an unmodelled field that will bite once the data is consumed. If the culprit is a NiNode-level bsver‚Č•173 field, the correct fix in `NiNode::parse` would protect every Starfield NiNode subclass.
- **Related**: #754 (BSWeakReferenceNode introduction), `bsver::SF_FORM_ID`.
- **Suggested Fix**: Byte-dump one Starfield `BSWeakReferenceNode` (compare `block_size` vs consumed) to localize the 2-byte field. If NiNode-level, add the `bsver >= SF_FORM_ID` branch in `NiNode::parse`; if block-local, read it in `BsWeakReferenceNode::parse`. Add a `dispatch_tests` case asserting exact consumption.

### NIF-D2-001: Three NifVariant feature helpers are dead AND documented-as-incorrect by their own sibling parsers
- **Severity**: MEDIUM
- **Dimension**: Version Gating
- **Game Affected**: All (latent regression risk on Unknown-variant / transitional-export content ‚ÄĒ pre-Bethesda Gamebryo, tool exports at v20.2.0.7 uv=11 bsver‚Č§26)
- **Location**: `crates/nif/src/version.rs` (`NifVariant::has_properties_list`, `::has_effects_list`, `::avobject_flags_u32`)
- **Status**: NEW (instance of the #938/#1511 dead-helper anti-pattern)
- **Description**: Dim-2 checklist item 2 blesses these as the named helper surface parsers should route through, but they have **zero production consumers workspace-wide** ‚ÄĒ the only non-test references are the version.rs unit-test asserts. The production parse paths deliberately use raw `stream.bsver()` instead, and their comments explicitly document the helper as *wrong* for the edge cases: `has_properties_list` returns `false` for the `Unknown` variant (would 4-byte-misalign non-Bethesda Gamebryo files, #160); `has_effects_list` similarly (#160); `avobject_flags_u32` says u32 for a transitional v20.2.0.7/bsver‚Č§26 export that detects as `Fallout3`, reading 4 bytes where 2 exist and slipping the stream +2 (#1331).
- **Evidence**: `base.rs` uses `stream.bsver() > FLAGS_U32_THRESHOLD` with a comment "not variant().avobject_flags_u32() ‚Ä¶ would slip the stream +2. See #1331"; `base.rs` `properties` uses `stream.bsver() <= FO3_FNV` citing #160; `node.rs` `effects` uses `stream.bsver() < FALLOUT4` citing #160. Grep for the three helper calls excluding fn/comment/assert ‚Üí no production hits.
- **Impact**: No current runtime bug ‚ÄĒ the raw-bsver paths are correct. Latent: the helpers survive in the "blessed" surface and are still asserted correct by tests, so a future contributor "cleaning up raw literals into the named helper" would re-adopt them and reintroduce the exact #1331 / #160 corruption. This is precisely the foot-gun #938/#1511 deleted nine other helpers to eliminate.
- **Related**: #938, #1511 (deleted call-site-less helpers); #1331, #160 (the bugs the raw-bsver paths fix).
- **Suggested Fix**: Delete the three helpers + their unit tests, OR reimplement them to honour the same Unknown/transitional semantics as the raw-bsver paths and migrate the three call sites onto them (single authority). Do not leave them as unconsumed "approved" helpers that disagree with the live parse path.

### NIF-D2-002: `uses_bs_tri_shape` and the entire `ShaderFlags` typed view are unadopted dead abstractions
- **Severity**: LOW
- **Dimension**: Version Gating
- **Game Affected**: None (dead code)
- **Location**: `crates/nif/src/version.rs` (`NifVariant::uses_bs_tri_shape`, `::has_shader_property_fo3_fields`); `crates/nif/src/shader_flags.rs` (`ShaderFlags` `classify`/`is_decal`/`is_two_sided`/`is_crc32`)
- **Status**: NEW (instance of the #1511 dead-helper anti-pattern)
- **Description**: `uses_bs_tri_shape` has zero references workspace-wide ‚ÄĒ BSTriShape is dispatched by block-type string, so a variant predicate is never needed. The `ShaderFlags` typed view (#1277) has no Rust consumer outside its own module; `import/material` uses the free functions `is_decal_from_legacy/modern_shader_flags` + the raw constant modules directly. Because `ShaderFlags::classify` is the only caller of `has_shader_property_fo3_fields`, that helper is transitively dead too.
- **Evidence**: `grep uses_bs_tri_shape` ‚Üí only the `fn` def. `ShaderFlags::classify` / `.is_decal()` / `.is_two_sided()` ‚Üí no Rust hits outside `shader_flags.rs` (the `walker.rs` `stencil.is_two_sided()` is a different type). `has_shader_property_fo3_fields` sole caller is inside `ShaderFlags::classify`.
- **Impact**: None on correctness; an abstraction built to prevent cross-game bit-vocabulary mistakes is not adopted, so its safety isn't in force and it sits as maintenance weight.
- **Related**: #1277 (added ShaderFlags), #1511/#938.
- **Suggested Fix**: Prefer adoption ‚ÄĒ replace the ad-hoc `is_decal_from_legacy/modern` split at the `import/material` call sites with `ShaderFlags` (gives the type a consumer, makes `has_shader_property_fo3_fields` live). Otherwise delete the unused view + `uses_bs_tri_shape` + `has_shader_property_fo3_fields`.

### NIF-D3-001: Per-block baseline collapses ~30 distinct header types into single alias buckets
- **Severity**: LOW
- **Dimension**: Block Dispatch Coverage
- **Game Affected**: All (most visible on particle-heavy FO76/FO4/FNV/FO3)
- **Location**: `crates/nif/tests/common/mod.rs` (`PerBlockHistogram::record_scene_blocks`); alias structs in `crates/nif/src/blocks/particle.rs`
- **Status**: NEW (mechanism behind the project-memory note "FO76 can sit silently RED on NiPSysBlock")
- **Description**: `PerBlockHistogram`'s doc-comment says every block is keyed by its header-advertised type name and that parsed/unknown counts share a key. The code honours this only for `NiUnknown` (keyed on `unknown.type_name`); the parsed branch keys on `block.block_type_name()` ‚ÄĒ the parsed struct's name. `impl_ni_object!(NiPSysBlock, ‚Ä¶)` implicit form returns `"NiPSysBlock"` for **28** distinct header modifier types (and `"NiPSysEmitter"` for 5), collapsing them into one bucket that never appears in any NIF header table.
- **Evidence**: `record_scene_blocks` unknown branch ‚Üí `unknown.type_name`; parsed branch ‚Üí `block.block_type_name()`. Checked-in `per_block_baselines/fallout_76.tsv` has a single `NiPSysBlock 129380 0` row and no individual `NiPSys*Modifier` rows.
- **Impact**: Per-type regression *resolution* is lost for collapsed families (the "silently RED on NiPSysBlock" volatility). Aggregate regressions are still normally caught (a member regressing to NiUnknown shrinks the bucket ‚Üí `ParsedShrank`). LOW.
- **Related**: NIF-D3-002 (the two combine into a real blind spot); project memory "FO76 silently RED on NiPSysBlock".
- **Suggested Fix**: (a) key the parsed branch on the header-advertised name (restores per-type resolution, closes NIF-D3-002), or (b) correct the doc-comment to state alias families aggregate. (a) preferred.

### NIF-D3-002: `compare_histograms` only iterates baseline keys ‚ÄĒ a brand-new NiUnknown type is invisible to the per-block gate
- **Severity**: LOW
- **Dimension**: Block Dispatch Coverage
- **Game Affected**: All
- **Location**: `crates/nif/tests/common/mod.rs` (`compare_histograms`)
- **Status**: NEW
- **Description**: `compare_histograms` loops `for (name, base_counts) in &baseline.counts`. A block type absent from the baseline that starts landing in `NiUnknown` has no baseline key, so neither `UnknownGrew` nor `ParsedShrank` is evaluated for it.
- **Evidence**: The only iteration is over `baseline.counts`; current-only keys are never examined.
- **Impact**: In isolation compensated by `block_coverage_baselines.rs`'s `total_unknown()` ceiling. **But combined with NIF-D3-001** an offsetting alias-family swap (member A regresses to a *new* NiUnknown key, member B newly dispatches into the same bucket) leaves `total_unknown` flat and the bucket `parsed` flat, slipping both gates. Narrow but real.
- **Related**: NIF-D3-001.
- **Suggested Fix**: Iterate the **union** of baseline and current keys (missing baseline entry ‚Üí `{parsed:0, unknown:0}`), flagging any newly-appearing NiUnknown type directly.

### NIF-D3-003: Coverage regression baselines guard only the primary mesh archive per game
- **Severity**: LOW
- **Dimension**: Block Dispatch Coverage
- **Game Affected**: FO4 (FaceGen tail), FO76, Starfield, Skyrim SE (animation/FaceGen/DLC archives)
- **Location**: `crates/nif/tests/common/mod.rs` (`Game::mesh_archive` / `open_mesh_archive`)
- **Status**: NEW (scope note)
- **Description**: `open_mesh_archive` walks exactly one archive per game. Animation (`.kf`), FaceGen/geometry, texture, and DLC/patch archives are never swept by the coverage gates.
- **Evidence**: `mesh_archive()` returns a single `&'static str` per game; no sibling enumeration (contrast the engine's runtime `<stem>N.bsa` auto-load).
- **Impact**: The documented sub-100% *clean* rates (FO4 96.46%, FO76 97.34%, Starfield 98.6%) are file-level trailing-truncation counts, several living outside the swept archive ‚ÄĒ not regression-protected. Also explains the apparent non-divergence: the checked-in ceilings read `unknown_blocks 0` while the doc matrix reads <100% clean (block dispatch vs file trailing-byte truncation ‚ÄĒ different metrics). The FO4 FaceGen-morph fix (#1073) is in the swept archive and reads clean, so the doc's 96.46% may itself be stale.
- **Related**: NIF-D3-001; `docs/engine/nif-parser.md` dispatch figure (254/310 vs live 315).
- **Suggested Fix**: Optionally extend `open_mesh_archive` to include the sibling archives carrying the known tails. Lower urgency than 001/002. At minimum refresh the doc dispatch figure (‚Üí live 315) and re-verify the FO4 clean rate post-#1073.

### NIF-D5-001: `RIGID_BODY_EXTRA_FLOATS` docstring mis-attributes the owning block
- **Severity**: LOW
- **Dimension**: Collision/Shader Parsing
- **Game Affected**: pre-mainline Oblivion dev content (bsver < 9); all others unaffected
- **Location**: `crates/nif/src/version.rs` (`bsver::RIGID_BODY_EXTRA_FLOATS`)
- **Status**: NEW (documentation only ‚ÄĒ NOT a parse divergence)
- **Description**: The const's docstring states `bhkRigidBody` carries two trailing `Unknown Float 1/2` fields on `bsver < RIGID_BODY_EXTRA_FLOATS`. Its sole consumer is `BhkCollisionObject::parse` (`crates/nif/src/blocks/collision/collision_object.rs`), which gates the two floats on the **bhkBlendCollisionObject** path (`is_blend`), per nif.xml 3428-3429. The bytes are correct; only the prose names the wrong block.
- **Evidence**: `collision_object.rs` reads the floats inside `if is_blend { if stream.bsver() < RIGID_BODY_EXTRA_FLOATS { ‚Ä¶ } }`, whereas the version.rs docstring attributes them to `bhkRigidBody`.
- **Impact**: None at runtime. Risk is a future maintainer searching for the field in the rigid-body path and reintroducing a duplicate/mis-gated read.
- **Related**: #549 (the fix that added the gate, correctly, in collision_object.rs).
- **Suggested Fix**: Reword the docstring to reference `bhkBlendCollisionObject` `Unknown Float 1/2` (nif.xml 3428-3429). No code change.

### NIF-D6-001: Interpolator legacy-blend array uses raw `Vec::with_capacity` on a file-driven count instead of `allocate_vec`
- **Severity**: LOW
- **Dimension**: Allocation Hygiene
- **Game Affected**: Pre-Gamebryo / old NetImmerse content hitting `NiBlendInterpolator::parse_legacy` (10.1.0.x bands) ‚ÄĒ reachable on Oblivion-era and older streams.
- **Location**: `crates/nif/src/blocks/interpolator.rs` (`parse_legacy`, the `Vec::with_capacity(array_size as usize)` site)
- **Status**: NEW
- **Description**: Every other file-driven count in the crate routes its pre-allocation through `allocate_vec` / `read_pod_vec`, which bounds the claimed count against remaining stream bytes. `parse_legacy` calls `Vec::with_capacity` directly on a count read straight from the stream.
- **Evidence**: `array_size` is read as u16 (int-priority) or u8, then `let mut items = Vec::with_capacity(array_size as usize);`.
- **Impact**: Bounded and low ‚ÄĒ `array_size` is u16/u8 (‚Č§65535), can't express a `0xFFFFFFFF` over-commit; each per-element read is EOF-checked so a corrupt count runs to EOF ‚Üí block-size recovery, not OOM. Pattern-consistency divergence, not a live vulnerability.
- **Related**: #388/#764/#768 (the allocate_vec sweep this site was missed by).
- **Suggested Fix**: Replace with `stream.allocate_vec::<_>(array_size as u32)?` so the legacy blend path shares the one bound-check idiom. One-line change.

## Prioritized Fix Order

Rendering-blocking first, then animation, then collision, then hygiene:

1. **NIF-D1-001** (MEDIUM) ‚ÄĒ mirror #1606 onto `BSEffectShaderProperty` (structural, well-templated; closes a systematic Starfield drop). Rendering-adjacent.
2. **NIF-D1-002** (MEDIUM) ‚ÄĒ localize the 2-byte Starfield weak-ref/NiNode field; if NiNode-level, one fix protects all Starfield NiNode subclasses. Needs a byte-dump first.
3. **NIF-D2-001** (MEDIUM) ‚ÄĒ delete or correctly-reimplement the 3 dead-and-wrong helpers before someone re-adopts them into a corruption. Cheap, high latent-risk payoff.
4. **NIF-D3-001 + NIF-D3-002** (LOW, do together) ‚ÄĒ union-key `compare_histograms` + key the parsed histogram branch on the header name; together they close the offsetting-alias blind spot and restore per-type resolution.
5. **NIF-D6-001, NIF-D5-001, NIF-D2-002, NIF-D3-003** (LOW) ‚ÄĒ one-line alloc idiom, a docstring reword, dead-code deletion/adoption, and a test-scope/doc-figure refresh. Batchable.
