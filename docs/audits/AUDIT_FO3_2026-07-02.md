#### FO3-D1-01: FO3 `WaterShaderProperty.water_shader_flags` dead-ends at MaterialInfo
- **Severity**: LOW
- **Dimension**: FO3 Rendering Path
- **Location**: `crates/nif/src/import/material/walker.rs:583-591,962-964`; `crates/nif/src/import/material/mod.rs:730-741`; `crates/nif/src/import/types.rs`
- **Status**: NEW
- **Description**: `MaterialInfo.water_shader_flags` is captured from
  `BSWaterShaderProperty` (`walker.rs:590`), but the FO3/FNV non-BS
  `WaterShaderProperty` branch (`walker.rs:962-964`) forwards only `env_map_scale`
  and does not even set `water_shader_flags`. More broadly, `water_shader_flags`
  is a field on `MaterialInfo` but NOT on `ImportedMesh` nor the ECS `Material`,
  so the captured value is dropped at the `MaterialInfo â†’ ImportedMesh` boundary.
- **Evidence**: `grep -rln water_shader_flags` returns only the parser + walker +
  its own tests â€” zero consumers past `MaterialInfo`. `water_shader_legacy_tests.rs`
  asserts only `env_map_scale` reaches `MaterialInfo` for the FO3/FNV
  `WaterShaderProperty`.
- **Impact**: Minimal for FO3. FO3 mesh-driven water (`meshes/water/*.nif`)
  renders as a `material_kind = 0` lit surface with the authored `env_map_scale`;
  the authored reflection/refraction/cubemap intent in `water_shader_flags` is
  silently unused. This is **NOT** a dedup collapse (water â‰  glass by
  `material_kind`) and **NOT** a divergent `Material` out of the boundary â€” it is
  a capture-with-no-consumer gap. There is no visual regression today because the
  renderer has no legacy-mesh-water route for the flag word to land in.
- **Related**: WATAL water-layer work (`docs/engine/watal.md`); adjacent to but
  distinct from #1243 (`WaterShaderProperty` â†’ distinct `GpuMaterial`).
- **Suggested Fix**: When a legacy-mesh-water render route lands, plumb
  `MaterialInfo.water_shader_flags` through `ImportedMesh` and set it on the
  FO3/FNV `WaterShaderProperty` branch too (currently only the Skyrim+
  `BSWaterShaderProperty` branch populates it). Until then, no action required â€”
  documented so a future audit does not refile it as a `translate_material`
  divergence (it is not; the value never reaches `translate_material`).

#### FO3-D2-INFO-01: NiControllerSequence FO3-era singular Anim-Notes ref (informational â€” correctly implemented)
- **Severity**: LOW (informational â€” NOT a defect)
- **Dimension**: NIF v20.2.0.7 Parser

- **Status**: NEW (informational)
- **Description**: nif.xml splits `NiControllerSequence` anim-notes by BSVER: the

  early-dev/beta/mod content.
- **Impact**: None on retail FO3 or FNV. Recorded so a future audit does not
  re-flag the divergence as a missing branch â€” it is present and correct (#432).
- **Suggested Fix**: None.
