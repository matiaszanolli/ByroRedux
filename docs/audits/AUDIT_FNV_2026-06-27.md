## Dimension Findings

### Dimension 1 — Cell Loading End-to-End  ·  1 MEDIUM

#### FNV-D1-NEW-01: Sky-texture handles leak on worldspace transition into a climateless worldspace
- **Severity**: MEDIUM
- **Location**: `byroredux/src/scene/world_setup.rs:349-362` (the `else` branch of `apply_worldspace_weather`); triggered from `byroredux/src/main.rs:1498`
- **Status**: NEW (adjacent to closed #1339 / #1199; not a regression of either)
- **Description**: `apply_worldspace_weather` releases the prior worldspace's ≤5 sky-texture handles (4 WTHR cloud layers + 1 CLMT sun sprite) **only inside the `if let Some(ref wthr) = wctx.default_weather` branch**. When a transition lands on a worldspace whose climate/weather does not resolve (`default_weather == None`), control takes the `else` branch, which calls `insert_procedural_fallback_resources` and overwrites `SkyParamsRes` **without capturing or releasing the prior handles**. The ≤5 bindless slots + VkImages pin for the rest of the session.
- **Impact**: Up to 5 leaked textures per transition into a climateless worldspace (corrupt/partial ESM, mod worldspace with no CLMT, synthetic cell). Bounded per-event (not per-cell, not unbounded) → MEDIUM. Does **not** fire on vanilla FNV WastelandNV (resolves a real climate→weather). Note: closed #1339's INVESTIGATION premise ("procedural fallback runs only for the no-plugin synthetic scene") is incorrect — the `else` branch is live for any climateless transition.
- **Suggested Fix**: Hoist the `prev_sky_textures = world.try_resource::<SkyParamsRes>().map(|s| s.texture_indices())` capture above the `if/else` and run the `sky_textures_to_release` drop loop on **both** branches. The `else` branch installs an all-zero-index `SkyParamsRes`, so `sky_textures_to_release(prev, fallback)` already filters `0`/fallback correctly.

### Dimension 4 — ESM Record Parser  ·  2 LOW

#### FNV-D4-NEW-01: `trees` typed map excluded from `EsmIndex::categories()` / `total()`
- **Severity**: LOW
- **Location**: `crates/plugin/src/esm/records/index.rs:382-489` (`categories()` table); map populated at `crates/plugin/src/esm/records/mod.rs:343-351`
- **Status**: NEW
- **Description**: `categories()` is the documented single source of truth for `total()` / `category_breakdown()` (the #634/#817 invariant). 85 of 86 typed maps are listed — including the FO4-architecture `EsmCellIndex` maps added by #817 *specifically* to stop a category-wipe passing CI silently. `trees` (TREE) is the lone populated map missing from the table.
- **Impact**: Cosmetic on the count (FNV TREE=3), but a real guard gap: a regression emptying `index.trees` would not move `total()` below the `parse_rate_*` floors nor surface in the breakdown — the exact failure mode #817 guards against. Widens on FO3/Oblivion (SpeedTree-heavy).
- **Suggested Fix**: Add `("trees", |s| s.trees.len()),` to the `categories()` array. Optionally assert `categories().len()` tracks the map count.

#### FNV-D4-NEW-02: `parse_scol` docstring claims SCOL is FO4-and-later
- **Severity**: LOW (doc-rot; behavior correct)
- **Location**: `crates/plugin/src/esm/records/scol.rs:132`
- **Status**: NEW
- **Description**: The `parse_scol` doc comment says *"Wire format is FO4-and-later; earlier games don't emit SCOL"* — false (FalloutNV.esm ships 98 SCOL / 423 ONAM-DATA pairs, all DATA ≡ 0 mod 28) and it contradicts both the module doc (`scol.rs:8-9`) and the correct #1538 dispatch gate. The parser handles FNV SCOL perfectly; only the comment is stale — and it is exactly the kind of claim that justified the #1538 gate regression.
- **Suggested Fix**: Reword to "FO3/FNV and FO4+ (byte-identical 28-byte DATA layout); Oblivion/Skyrim don't emit SCOL."

### Dimension 6 — Animation, Skinning & Particles  ·  1 MEDIUM, 1 LOW

#### FNV-D6-NEW-01: Authored birth-rate of exactly 0.0 silently kills the emitter (no preset fallback)
- **Severity**: MEDIUM
- **Location**: `crates/nif/src/import/walk/mod.rs:775-778` (`extract_emitter_rate::sane`); consumed at `byroredux/src/systems/particle.rs:75` (`apply_emitter_overlays`) + gated at `byroredux/src/systems/particle.rs:385` (`em.rate > 0.0`)
- **Status**: NEW
- **Description**: `sane()` is `(r.is_finite() && (0.0..3.0e38).contains(&r)).then_some(r)` — the inclusive lower bound **accepts 0.0**. When the authored rate (first key of the `NiPSysEmitterCtlr` interpolator's `NiFloatData`) resolves to exactly 0.0, `extract_emitter_rate` returns `Some(0.0)`, `apply_emitter_overlays` unconditionally writes `preset.rate = 0.0`, and the spawn guard `em.rate > 0.0` is then false forever — the emitter never spawns a particle. FLT_MAX/negative/NaN all correctly reject to `None` (→ preset); 0.0 is the one finite-but-degenerate value that slips through.
- **Impact**: FNV/FO3 ship many ramp-up emitters whose authored rate *first key* is 0.0 (spell-cast bursts, geyser/steam ramps, ignition FX). Pre-overlay these fell back to a non-zero heuristic preset and emitted something; now they go fully invisible. Visible content loss scoped to rate-at-t=0 == 0.0.
- **Related**: scope-limitation #1402 (rate animation deferred — single representative scalar). Not a dup of the FLT_MAX/NaN guards (those return `None`; this returns `Some(0.0)`).
- **Suggested Fix**: Make the range exclusive-low (`0.0 < r && r < 3.0e38`) so a zero first-key returns `None` and keeps the preset rate, matching the no-controller path. (Fuller fix: sample the rate curve over time per #1402.)

#### FNV-D6-NEW-02: `EmitterBaseParams.radius_variation` parsed but dropped at the ImportedEmitterParams handoff
- **Severity**: LOW
- **Location**: `crates/nif/src/blocks/particle.rs:64,112-116,128` (parsed) vs. `crates/nif/src/import/types.rs:1236` (`ImportedEmitterParams` — field absent); handoff at `crates/nif/src/import/walk/mod.rs:744-754`
- **Status**: NEW
- **Description**: `read_emitter_base` parses `radius_variation` (version-gated `>= 10.4.0.1`, satisfied by all FNV) but `ImportedEmitterParams` has no field for it, so `extract_emitter_params` never forwards it and every particle spawns at exactly `initial_radius × base_scale` with zero radius jitter. Same drop-at-handoff family as #1445 (planar_angle), #1610 (tint_map), #1580 (BGEM bool).
- **Impact**: Low — FNV FX that authored a per-particle radius spread render with uniform-size particles. No crash, byte is consumed (no stream desync). Visual-fidelity only.
- **Suggested Fix**: Add `radius_variation` to `ImportedEmitterParams`, forward it in `extract_emitter_params` (in the finite sweep), and consume a size-jitter at spawn. Lowest-risk first step: forward the value so it isn't lost.

### Dimension 7 — PHYSAL Ragdoll (FNV reference slice)  ·  1 MEDIUM

#### FNV-D7-NEW-01: Keyframed bone collision bodies never torn down on ragdoll activation; no collision groups separate them from the simulated ragdoll bodies
- **Severity**: MEDIUM
- **Location**: `byroredux/src/ragdoll.rs:170-239` (`activate_ragdoll` — no teardown) · `byroredux/src/npc_spawn.rs:101-112` (`keyframe_live_ragdoll_bones`) · `crates/physics/src/sync.rs:497-522` (`push_kinematic`, no `RagdollActive` gate) · `crates/physics/src/ragdoll.rs:131-166` (`build_ragdoll` — no `InteractionGroups`/`solver_groups`)
- **Status**: NEW (distinct from #1718 — that is a parse/resolve silent-drop; this is a runtime solver double-body conflict on bones that *did* resolve)
- **Description**: At spawn, each ragdoll bone NiNode carrying a `bhkRigidBody` gets a `RigidBodyData`, flipped to `Keyframed` by `keyframe_live_ragdoll_bones` and registered with Rapier. When `ragdoll <id>` runs, `activate_ragdoll` builds ~18 *new* dynamic bodies via `build_ragdoll` but does NOT remove/disable/re-type the pre-existing keyframed bone bodies. From that frame the bone carries TWO live Rapier bodies. `ragdoll_writeback_system` writes the simulated pose onto the bone `GlobalTransform`; the next frame `push_kinematic` (gated only on `motion_type == Keyframed`, no `RagdollActive` check) drives the keyframed body to chase it. With no `InteractionGroups` (default collide-with-all), the kinematic followers generate dynamic-vs-kinematic contact forces that fight the multibody solver the writeback then reads back.
- **Impact**: Manifests on the documented `ragdoll <id>` debug/smoke path (`docs/smoke-tests/m41-ragdoll.sh`, Doc Mitchell). ~18 redundant kinematic bodies per ragdolled actor in the broad-phase; contacts can jitter the crumple, push the ragdoll off its rest pose, or twitch limbs. Not a crash (joints keep the chain connected) → MEDIUM. Bounded: only console/smoke-ragdolled actors; the production death-trigger is not wired yet (physal.md §6).
- **Suggested Fix**: In `activate_ragdoll`, after a successful `build_ragdoll`, tear down or kinematic-disable the bone entities' pre-existing keyframed bodies (`pw.remove_body` per ragdolled bone, or skip them in `push_kinematic` when the owner is `RagdollActive`). Alternatively give the dynamic ragdoll bodies a self-collision-excluding `InteractionGroups`.

### Dimension 8 — Real-Data Validation & Bench-of-Record  ·  1 LOW

#### FNV-D8-NEW-01: No aggregate "zero archives opened" guard — wrong-CWD bench loads near-empty and reports a spurious FPS with only a per-archive WARN
- **Severity**: LOW
- **Location**: `byroredux/src/asset_provider/archive.rs:306-331` (`open_with_numeric_siblings`); `byroredux/src/asset_provider/texture.rs:122-146` (`build_texture_provider`); call sites `byroredux/src/scene.rs:198,236`, `byroredux/src/main.rs:1430,1487`
- **Status**: NEW (delta on prior AUDIT_FNV_2026-06-14 §D8-02, which verified the gotcha and proposed but never filed the `--esm`-parent fallback)
- **Description**: A bare/relative `--bsa`/`--textures-bsa` that fails to resolve against CWD produces only a per-archive `log::warn!`; the engine proceeds with empty archive vectors and the load succeeds with placeholder textures. No aggregate check distinguishes "0 mesh archives opened" (misconfig) from a legitimate load, so a bench prints a spurious FPS (~36 entities / ~1792 FPS). The `--game <key>` profile path (joins against the resolved Data dir) is immune; smoke scripts use absolute paths and are immune.
- **Impact**: UX trap / masked-failure-as-data, not a correctness defect. Any manual (non-`--game`) repro run from the wrong directory or with a mistyped name.
- **Suggested Fix**: (a) Retry a bare archive name against the `--esm` parent directory before giving up; and/or (b) after `build_texture_provider`, if a cell load was requested but `mesh_archives.is_empty()`, emit one `log::error!`. Neither changes the happy path.
