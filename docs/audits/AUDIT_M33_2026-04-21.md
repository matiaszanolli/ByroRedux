# M33 Sky & Atmosphere Correctness Audit — 2026-04-21

Post-merge correctness audit of the M33 Sky & Atmosphere milestone plus
its follow-on fixes (#463 CLMT TNAM, #468 texture-path normalisation, #476
WLST signed chance, #478 CLMT FNAM sun sprite, #513 fog empty-depth, #531
`--radius`). Scope: WTHR + CLMT parsers, WRLD→CLMT→WTHR resolution,
`weather_system` TOD interpolation, composite-pass sky branch, procedural
fallback, cross-game parity. Tuning / feature-request items are out of
scope; see the scope fence at end.

Git HEAD at audit time: `b69608c` (`Fix #494: consume BGSM uv_offset /
uv_scale / material_alpha in triangle.frag`).

**Every finding was verified against code + byte-level evidence from the
shipped masters (FalloutNV.esm, Fallout3.esm, Oblivion.esm). A throwaway
scratch harness at `crates/plugin/tests/m33_audit_dump.rs` produced the
evidence; it is not committed and should be deleted before the fix
session commits land.**

## Executive Summary

**Compatibility level**: M33 looked done on the surface but byte-level
verification against the three reference masters reveals **four CRITICAL
parser defects** that mean the milestone is, in practice, far less
functional than the roadmap claims:

- **[M33-01](#m33-01-nam0-parse-gate-rejects-fo3--oblivion)** — NAM0 size
  gate is `>= 240` but FO3 + Oblivion NAM0 is **160 bytes** (10 groups × 4
  TOD slots × 4 B, not 6 slots). **All 27 Fallout 3 weathers and all 37
  Oblivion weathers silently parse with zero sky colours.** Every FO3 and
  Oblivion exterior cell falls back to the `WeatherRecord::default()`
  all-zero NAM0 — meaning the sky gradient is black-on-black for those
  games, exactly what M33 was supposed to fix.
- **[M33-02](#m33-02-cloud-texture-sub-record-fourccs-never-match)** —
  Parser looks for `00TX / 10TX / 20TX / 30TX` sub-records for cloud
  layers. Actual FourCCs are **`DNAM / CNAM / ANAM / BNAM`** for FNV + FO3
  (and `CNAM / DNAM` for Oblivion). **Zero of 127 shipped weathers have
  any cloud texture parsed.** Every exterior cell renders without clouds.
- **[M33-03](#m33-03-dnam-misinterpreted-as-cloud-speeds-its-a-texture-path)** —
  `DNAM` is a cloud-texture-path zstring, **not a `[u8; 4]` of cloud
  speeds**. The parser reads the first 4 bytes (`"sky\"` = `0x73 0x6b
  0x79 0x5c` = `[115, 107, 121, 92]`) as cloud speeds. `weather_system`
  feeds this garbage at `systems.rs:1277` into the cloud-scroll rate,
  driving the single visible cloud layer to scroll at a speed that has
  no correlation with the weather.
- **[M33-04](#m33-04-fnam-arm-body-is-empty--fnvfo3-fog-never-parses)** —
  FNAM arm at `weather.rs:158` matches but its body is empty (only a
  comment). **FNV and FO3 don't have HNAM — they have FNAM**, 24 bytes of
  fog data. `HNAM` only appears in Oblivion (and is 56 B, not 16 B).
  Result: **every FNV and FO3 weather returns `fog_day_far = 10000.0`**
  (the default), and Oblivion's HNAM parse reads the wrong 4 f32s within
  a 56-byte payload. No game's fog distance is sourced from WTHR today.

In short: for FO3 and Oblivion, the entire WTHR→rendering chain is a
no-op — the sky is a hardcoded default. For FNV, ~80% of weathers get
correct sky colours but **no cloud textures, no scroll rate, no fog
distance.** The "M33 closeout" and "Phase 3" commits landed against
synthetic test fixtures that happen to use FNV's 240-byte NAM0 layout and
the `00TX/...` FourCCs — conditions that no real ESM actually produces.

**One HIGH finding**: `classification` byte offset at DATA[13] is
inconsistent with the raw bytes for FNV's `NVWastelandClear`, which has
`0x01` at byte 11 (WTHR_PLEASANT) and `0x00` at byte 13 — the parser
returns classification=0 for what should be a pleasant weather. See
[M33-06](#m33-06-classification-byte-offset-likely-off-by-two).

**Three MEDIUM findings**: Skyrim routing without a GameKind gate
([M33-07](#m33-07-skyrim-wthr-not-gated-by-gamekind)), latent WLST
12-vs-8-byte autodetect collision at 3/6/9-entry Oblivion CLMTs
([M33-08](#m33-08-wlst-autodetect-collision-at-multiples-of-three-oblivion-entries)),
and four NAM0 colour groups (SKY_STARS / SKY_LOWER / SKY_CLOUDS_LOWER /
SKY_CLOUDS_UPPER) parsed but never consumed
([M33-09](#m33-09-four-nam0-colour-groups-parsed-but-never-read)).

**Two LOW findings**: procedural fallback sun never sets
([M33-10](#m33-10-procedural-fallback-sun-never-sets)), and DATA comment
block contradicts its own field offsets
([M33-11](#m33-11-data-comment-block-contradicts-field-offsets)).

**Regression guard**: all eleven M33 + follow-on commits are still
attached to their call sites — the defects above are original bugs that
have been present since M33's initial landing (8b1c514), not regressions.

**Stale docs**: `ROADMAP.md` Tier-1 row still lists M33 as open (fine — it
*is* open in practice, despite the "M33 closeout" commit label). CLAUDE.md
"Next: M33 sky/atmosphere" line (line ~299) should stay pointing at M33
until the CRITICALs close. `AUDIT_FNV_2026-04-21.md` exec summary notes
"sky+cloud compositing verified correct" at end of §Exec — that claim
was verified against the synthetic shader state only and should be
narrowed to "sky branch compiles and runs" pending the parser fixes.

**Dim 3 — WRLD→CLMT→WTHR resolution chain** (`byroredux/src/cell_loader.rs:567-605`):

## Findings

### M33-01: NAM0 parse gate rejects FO3 + Oblivion

**Severity**: CRITICAL · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/weather.rs:133](crates/plugin/src/esm/records/weather.rs#L133)

**Description**: The NAM0 arm gates on
`sub.data.len() >= SKY_COLOR_GROUPS * SKY_TIME_SLOTS * 4 = 10 × 6 × 4 = 240`.
Byte-level inspection of vanilla masters shows:

| Master | NAM0 size observed | Parse outcome |
|---|---|---|
| `Fallout3.esm` (27 WTHRs) | **160 bytes** on 27/27 records | 27/27 fail gate → all defaults (zero) |
| `Oblivion.esm` (37 WTHRs) | **160 bytes** on 37/37 records | 37/37 fail gate → all defaults (zero) |
| `FalloutNV.esm` (63 WTHRs) | 240 bytes on 51/63 records | 51/63 parse; 12/63 fail (same 160-byte layout) |

**Evidence** (from `crates/plugin/tests/m33_audit_dump.rs` raw scan):
```
FO3 — WTHR #0 "MegatonFalloutDe...":  NAM0 size=160 peek=9d ac a4 00 c8 db d1 00 ...
OBL — WTHR #0 "SEClearTrans":          NAM0 size=160 peek=6c 9d 9d 00 6c 9d 9d 00 ...
FNV — WTHR #1 "NVWastelandClear*":     NAM0 size=240 peek=32 34 89 00 32 47 87 00 ...
```
Structured dump confirms consequence: `zero SKY_UPPER/DAY (unparsed
NAM0): 27/27 FO3, 37/37 Oblivion, 12/63 FNV`.

**Impact**: **Every FO3 and Oblivion exterior cell has a black sky
dome, no ambient, no sun colour, no fog colour sourced from WTHR**, and
12 FNV weathers (the 160-byte variant ones — likely DLC or older format
records) are equally black. The sky gradient and lighting fall back to
`WeatherRecord::default()` → zero RGB everywhere. Only the sun arc
(computed from `GameTimeRes.hour` independently of NAM0) still varies.

This is the single highest-impact M33 defect and directly invalidates
the milestone description's promise of a working sky for the "all three
Gamebryo-era titles" target. The bug is silent: no warning, no error,
no log — the sky just stays black.

**Interpretation**: The 160-byte layout is 10 groups × **4** TOD slots ×
4 B (Oblivion predates HIGH_NOON + MIDNIGHT as separate slots; UESP
Oblivion-WTHR references match). FNV weathers with a 160-byte NAM0 are
likely DLC or older-format records using the Oblivion stride. A correct
parser needs a per-format stride dispatch.

### M33-02: Cloud-texture sub-record FourCCs never match

**Severity**: CRITICAL · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/weather.rs:184-196](crates/plugin/src/esm/records/weather.rs#L184-L196)

**Description**: The cloud-texture arms match on FourCC byte patterns
`[b'0', b'0', b'T', b'X']` through `[b'3', b'0', b'T', b'X']`. Byte-level
scan of vanilla masters:

| Master | Count of `00TX/10TX/20TX/30TX` occurrences across all WTHRs |
|---|---|
| `Fallout3.esm` | **0** |
| `FalloutNV.esm` | **0** |
| `Oblivion.esm` | **0** |

The actual FourCCs that appear in every WTHR record are (FNV histogram,
all 63 records):
```
  DNAM=63, CNAM=61, ANAM=62, BNAM=61    (plus LNAM, ONAM, PNAM, INAM, NAM0, FNAM, DATA, IAD×N)
```
Oblivion schema is simpler (only CNAM + DNAM cloud layers — Oblivion
has 2 layers, not 4).

**Evidence** (`NVWastelandClear*` WTHR #0 sub-record peeks):
```
DNAM size=14  peek=73 6b 79 5c 61 6c 70 68 61 2e 64 64 73 00   ("sky\alpha.dds\0")
CNAM size=14  peek=73 6b 79 5c 61 6c 70 68 61 2e 64 64 73 00   ("sky\alpha.dds\0")
ANAM size=14  peek=73 6b 79 5c 61 6c 70 68 61 2e 64 64 73 00   ("sky\alpha.dds\0")
BNAM size=21  peek=73 6b 79 5c 4e 56 43 6c 6f 75 64 6c 69 67 68 74  ("sky\NVCloudlight...")
```

**Impact**: **Zero of 127 vanilla weather records have any cloud texture
parsed.** The shader-side cloud sampling at `composite.frag:106-117`
gates on `cloud_tile_scale > 0.0`, which in `scene.rs:216-253` stays
at 0 whenever `wthr.cloud_textures[0].is_none()` — and that is always
the case because the parser never populates that field. Every exterior
cell across FNV / FO3 / Oblivion is cloudless, independent of weather
type. `NVWastelandClear` → cloudless. `NVWastelandStorm` → cloudless.

### M33-03: DNAM misinterpreted as cloud speeds — it's a texture path

**Severity**: CRITICAL · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/weather.rs:176-182](crates/plugin/src/esm/records/weather.rs#L176-L182)

**Description**: The DNAM arm reads `sub.data[0..4]` as
`[u8; 4]` cloud speeds per layer. Byte-level inspection shows DNAM is
actually a cloud-layer texture path (variable-length zstring).

**Evidence**: For `NVWastelandClear*` (FNV WTHR #0, byte dump):
```
DNAM size=14  bytes = "sky\alpha.dds\0"   (0x73 0x6b 0x79 0x5c 0x61 ...)
```
Structured dump returned `cloud_speeds = [115, 107, 121, 92]` — which
is `[0x73, 0x6b, 0x79, 0x5c]`, i.e. the literal ASCII bytes of `"sky\"`.

**Impact**: `weather_system` reads `wd.cloud_speeds[0]` at
[byroredux/src/systems.rs:1277](byroredux/src/systems.rs#L1277) and
normalises it as `cloud_speed_01 = wd.cloud_speeds[0] as f32 / 128.0`,
then multiplies into `cloud_scroll_rate = 0.02 * cloud_speed_01`. With
the garbage-in (115 → 0.898 scale-factor from `"s"`), the scroll rate
on every FNV exterior is identically `0.898 * 0.02 = 0.018` UV/sec —
coincidentally close to the authored target, which is why the bug went
undetected visually. But it is not sourced from any weather data — it's
the ASCII ordinal of the lower-case letter `s`.

Once M33-02 lands and `DNAM` is reinterpreted as a cloud texture path,
M33-03 is implicitly fixed (the `cloud_speeds` field disappears from
the struct). They should be fixed together.

### M33-04: FNAM arm body is empty — FNV/FO3 fog never parses

**Severity**: CRITICAL · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/weather.rs:157-161](crates/plugin/src/esm/records/weather.rs#L157-L161)

**Description**: The FNAM arm has a match guard but an empty body (only
a comment: `// Only used as fallback when HNAM is absent`). Byte-level
inspection disproves the premise:

| Master | HNAM presence | FNAM presence |
|---|---|---|
| `Fallout3.esm` (27 WTHRs) | **0 records** have HNAM | 27/27 records have FNAM (size 24) |
| `FalloutNV.esm` (63 WTHRs) | 0 records have HNAM | 63/63 records have FNAM (size 24) |
| `Oblivion.esm` (37 WTHRs) | 37/37 records have HNAM (size 56) | 37/37 records have FNAM (size 16) |

**Evidence** (`NVWastelandClear*` WTHR #0):
```
FNAM size=24 peek=00 00 fa c3 00 50 c3 47 00 00 fa c3 00 50 c3 47
                 (24 B = 6 × f32: fog distances, day/night/interior?)
```

**Impact**: **No FNV or FO3 weather has any fog distance parsed.**
`WeatherRecord::default()` returns `fog_day_far = 10000.0` at
[weather.rs:108-110](crates/plugin/src/esm/records/weather.rs#L108-L110),
so every FNV / FO3 exterior renders with a hardcoded fog distance of
`0..=10000` game units — independent of weather type, independent of
storm vs clear. `CompositeParams.fog_params.y` (fog-far) is therefore
constant at 10000 on the composite pass. The exterior landscape in
Prospector's FO3 equivalent has no fog falloff whatsoever.

For Oblivion, HNAM *does* match the `>= 16` gate but Oblivion HNAM is
56 bytes of different semantic fields (see
[M33-05](#m33-05-oblivion-hnam-semantic-offsets-dont-match-the-56-byte-layout))
— the 4 f32s at offsets 0/4/8/12 are not `day_near/day_far/night_near/night_far`.

### M33-05: Oblivion HNAM semantic offsets don't match the 56-byte layout

**Severity**: HIGH · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/weather.rs:148-154](crates/plugin/src/esm/records/weather.rs#L148-L154)

**Description**: HNAM exists only in Oblivion (not FNV / FO3). The
parser gates on `>= 16` and reads 4 f32s as `fog_day_near /
fog_day_far / fog_night_near / fog_night_far`. Oblivion HNAM is
56 bytes of different fields.

**Evidence** (Oblivion WTHR #0 `SEClearTrans`):
```
HNAM size=56
peek (first 16 B) = 33 33 33 3f 00 00 80 40 00 00 00 40 00 00 80 3f
                   = f32 { 0.7 }, { 4.0 }, { 2.0 }, { 1.0 }
```
UESP Oblivion:WTHR documents HNAM as 14 × f32 covering eye adapt speed,
blend-in/out distances, lighting effects, sun glare, sun damage, and
several other lighting-model parameters — **no direct fog-distance
fields**.

**Impact**: For every Oblivion weather the four f32s read out as
`fog_day_near = 0.7`, `fog_day_far = 4.0`, `fog_night_near = 2.0`,
`fog_night_far = 1.0` (sample values from `SEClearTrans`). At world
scale ≈ 4096 units per cell, `fog_far = 4.0` means fog saturation
within 4 units of the camera — the entire scene is solid
`fog_color` on Oblivion exterior cells.

**Reality check**: `parse_real_fo3_esm` and equivalent tests assert
record-count floors only, never per-field values. No regression test
would catch a fully-green fog field.

### M33-06: `classification` byte offset likely off by two

**Severity**: HIGH · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/weather.rs:163-174](crates/plugin/src/esm/records/weather.rs#L163-L174)

**Description**: DATA byte 13 is read as `classification`. Byte-level
inspection of `NVWastelandClear*` DATA:
```
DATA size=15 bytes = 32 2d 2d ff 36 00 00 00 00 00 ff 01 00 00 00
                    [ 0] [ 1] [ 2] [ 3] [ 4] [ 5] [ 6] ... [11] [12] [13] [14]
```
Byte 13 = `0x00` (parser returns `classification = 0`). Byte 11 = `0x01`
— which is `WTHR_PLEASANT`. The semantic meaning of this WTHR's
EDID ("Wasteland**Clear**") aligns with byte 11 being the classification
flag, not byte 13.

Cross-check with histograms:
- FNV: `{0: 51, 215: 1, 255: 11}` — 51 out of 63 return classification 0
- FO3: `{0: 15, 255: 12}` — 12 return 0xFF (all-flags-set, nonsensical)
- Oblivion: `{192: 1, 240: 1, 248: 2, 254: 1, 255: 32}` — 32 out of 37 return 0xFF

The `0xFF` cluster for Oblivion and FO3 strongly suggests the parser is
reading into the trailing padding / lightning-color bytes, not the
classification slot. The in-code comment at `weather.rs:170-173` even
says "bytes 13-14: lightning color (partial)" while *also* reading
`classification` from byte 13 — the two sentences contradict.

**Impact**: Low consumer blast today — a grep across the codebase
finds `classification` used only inside the parser's own unit test
(`weather.rs:276`). No downstream consumer reads the flag, so the
wrong value never drives rendering. But any future feature that
branches on weather type (transitions, precipitation, music selection)
will be wrong.

**Partial cross-check needed**: I did not pin the correct byte offset
against UESP-authoritative docs inside this audit (byte 11 is a
hypothesis from the single-record evidence). A fix session should
first byte-sample 3–5 WTHRs with known classifications (e.g.
`NVWastelandClear`=PLEASANT, `NVWastelandStorm`=RAINY,
`NVWastelandSnow`=SNOW) before choosing the offset. This is why the
severity is HIGH rather than CRITICAL — the parser is wrong, but I
haven't byte-confirmed the correct offset.

### M33-07: Skyrim WTHR not gated by GameKind

**Severity**: MEDIUM · **Dimension**: Cross-game · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/mod.rs:340](crates/plugin/src/esm/records/mod.rs#L340)

**Description**: `parse_wthr(fid, subs)` takes no `GameKind` parameter.
Every WTHR in any master runs through the FNV-assuming parser. Skyrim
WTHR has a substantially different schema (expanded NAM0 for additional
TOD variants, different sub-record set including lighting template
ref). The 240-byte NAM0 gate will silently accept a larger Skyrim NAM0
and read the first 240 B as if they were FNV colours.

**Evidence**: `items.rs:150, 253, 274` and sibling parsers all take
`game: GameKind` and branch per-game. `weather.rs:122` takes no such
argument.

**Impact**: Latent — today's build doesn't route `Skyrim - Objects.esm`
through `parse_esm` for cell loading, so the bad parse never fires.
But M32.5 (Skyrim + FO4 cell-loader parity — see ROADMAP Tier 1)
will flip this from latent to live. Any fix for M33-01 (per-format
NAM0 stride) should also add the GameKind parameter for future-proofing.

### M33-08: WLST autodetect collision at multiples-of-three Oblivion entries

**Severity**: MEDIUM · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/climate.rs:70-89](crates/plugin/src/esm/records/climate.rs#L70-L89)

**Description**: The WLST length-based autodetect picks 12-byte
entries when `sub.data.len() % 12 == 0`, else 8-byte entries. For
Oblivion (which uses 8-byte WLST entries), any CLMT with `N` entries
where `N × 8 % 12 == 0` (i.e. `N ∈ {3, 6, 9, 12, …}`) will misparse as
`N × 2/3` 12-byte entries — producing wrong FormIDs and wrong chances.

**Evidence** (`parse_esm` output on Oblivion.esm, entry-count histogram):
```
  wlst entry-count histogram: {0: 1, 1: 14, 2: 2, 5: 1, 7: 1}
```
Vanilla Oblivion.esm has zero 3/6/9-entry CLMTs → not tripped by vanilla
content. But Shivering-Isles-style DLC and OBSE-mod climates frequently
add weather variants; a climate with 6 entries is plausible and would
parse as 4 entries with garbage FormIDs in the even slots.

**Impact**: Latent for vanilla Oblivion; moddable. FNV / FO3 use
12-byte WLST unconditionally, so `% 12 == 0` always and the path
is correct there.

**Self-verification log**: Per the audit plan's self-verification step,
this was flagged as the HIGH candidate. The vanilla histogram
verification disqualifies it as HIGH and drops it to MEDIUM (latent,
moddable only).

### M33-09: Four NAM0 colour groups parsed but never read

**Severity**: MEDIUM · **Dimension**: Shader · **Status**: NEW

**Location**: [byroredux/src/systems.rs:1197-1226](byroredux/src/systems.rs#L1197-L1226)

**Description**: `weather_system` reads 6 of the 10 `SKY_*` colour
groups from `WeatherDataRes.sky_colors`:
```
SKY_UPPER, SKY_HORIZON, SKY_SUN, SKY_AMBIENT, SKY_SUNLIGHT, SKY_FOG  (consumed)
SKY_STARS, SKY_LOWER, SKY_CLOUDS_LOWER, SKY_CLOUDS_UPPER             (ignored)
```
`grep -n SKY_STARS|SKY_LOWER|SKY_CLOUDS_LOWER|SKY_CLOUDS_UPPER` confirms
zero consumer references outside of `weather.rs` constant declarations
and test fixtures.

**Impact**: Stars never render at night (SKY_STARS is a black-dome
overlay colour). The lower hemisphere keeps the same gradient as
upper (SKY_LOWER would drive the ground-plane approximation —
currently faked with `horizon * 0.3` at `composite.frag:91`). Cloud
colour modulation (`SKY_CLOUDS_LOWER/UPPER` are per-TOD tint RGBAs
applied to the cloud texture) is absent, so clouds render as
white/grey regardless of sunrise/sunset tint.

These four groups are parsed correctly (their bytes live in the
NAM0 block once M33-01 is fixed for FO3/Oblivion); they simply
have no shader consumer.

### M33-10: Procedural fallback sun never sets

**Severity**: LOW · **Dimension**: Fallback · **Status**: NEW

**Location**: [byroredux/src/scene.rs:351-373](byroredux/src/scene.rs#L351-L373)

**Description**: When `load_exterior_cells` returns `result.weather ==
None` (no climate / no weather resolved), `scene.rs:351-373` inserts a
hardcoded `SkyParamsRes` + `CellLightingRes` but does **not** insert
`GameTimeRes` + `WeatherDataRes`. Consequently `weather_system` at
`systems.rs:1129-1144` early-returns on the missing resources, and the
fallback sun direction / intensity stays at their hardcoded values
forever (`sun_dir = [-0.4, 0.8, -0.45]`, `sun_intensity = 4.0`).

**Impact**: Cosmetic today — every vanilla FNV / FO3 / Oblivion
exterior cell has a resolved climate + weather (though the NAM0 /
cloud / fog data are wrong, per M33-01..M33-04), so the fallback
path isn't hit. But once M33-01..M33-04 ship and a real record
fails its now-correct gates (corrupt ESM, a broken plugin, a bespoke
test case), the fallback sky will appear frozen at a fixed time of
day — confusing for debugging.

### M33-11: DATA comment block contradicts field offsets

**Severity**: LOW · **Dimension**: Parser · **Status**: NEW

**Location**: [crates/plugin/src/esm/records/weather.rs:163-174](crates/plugin/src/esm/records/weather.rs#L163-L174)

**Description**: The comment block inside the DATA arm says:
```
// bytes 6-11: precipitation/thunder fade params
// byte 12: thunder frequency
record.classification = sub.data[13];
// bytes 13-14: lightning color (partial)
```
Bytes 13 cannot simultaneously be both `classification` and the first
byte of "lightning color (partial)". Doc-drift from the field offset
chosen in [M33-06](#m33-06-classification-byte-offset-likely-off-by-two).

**Impact**: Readers of the parser will walk away with the wrong mental
model. Ties directly to M33-06 — fix both in the same change.
