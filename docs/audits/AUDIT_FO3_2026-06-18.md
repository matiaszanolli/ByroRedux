## Executive Summary

**Compatibility level: HIGH.** Fallout 3 continues to ride the FNV code path end-to-end.
The prior FO3 audit (2026-06-16) was thorough; this delta audit focused on (a) the shared-path
commits that landed since then (DXT1 1-bit alpha `2aac5351`, additive-particle batching
`067a8354`, soft-particle depth fade, ReSTIR attachment removal) and (b) corners the prior
sweep treated as clean.

The headline result is **one NEW HIGH finding**: the bhk `motion_type` → engine `MotionType`
mapping in `crates/nif/src/import/collision.rs:155-159` is **wrong against the canonical Havok
`hkMotionType` enum** — it mislabels the most common dynamic body type (BOX_INERTIA) as
Keyframed, and silently drops KEYFRAMED to Static. This is FNV-shared (all Havok-physics games)
but reaches FO3 clutter/crates/doors. The two shared-path render commits since 2026-06-16 were
verified **benign for FO3**.

**Totals: 3 findings — 0 CRITICAL · 1 HIGH · 0 MEDIUM · 2 LOW.**
(The 2 LOW are re-confirmations of still-open prior findings, carried for tracking; the HIGH is new.)

## Findings by Severity

### HIGH (1)

#### D5-1: bhk `motion_type` → `MotionType` mapping is wrong vs canonical Havok enum
- **Severity**: HIGH (physics correctness; silently wrong; FNV-shared, reaches FO3)
- **Dimension**: 5 — Collision (Havok → CollisionShape)
- **Location**: `crates/nif/src/import/collision.rs:155-159`
- **Status**: NEW
- **Description**: The extractor maps the raw `bhkRigidBody.motion_type` byte to the engine
  `MotionType` with:
  ```rust
  let motion_type = match body.motion_type {
      1..=3 => MotionType::Dynamic,
      4 => MotionType::Keyframed,
      _ => MotionType::Static,
  };
  ```
  This does not match the authoritative Havok `hkMotionType` enum (nif.xml `<enum
  name="hkMotionType">`, lines 1109-1121; `hkpMotion::MotionType`):

  | Value | Havok name | Correct intent | Current mapping | Wrong? |
  |-------|-----------|----------------|-----------------|--------|
  | 1 | MO_SYS_DYNAMIC | Dynamic | Dynamic | ok |
  | 2 | MO_SYS_SPHERE_INERTIA | Dynamic | Dynamic | ok |
  | 3 | MO_SYS_SPHERE_STABILIZED | Dynamic | Dynamic | ok |
  | **4** | **MO_SYS_BOX_INERTIA** | **Dynamic** (box inertia tensor — fully simulated) | **Keyframed** | **YES** |
  | **5** | **MO_SYS_BOX_STABILIZED** | **Dynamic** | **Static** (`_` arm) | **YES** |
  | **6** | **MO_SYS_KEYFRAMED** | **Keyframed** | **Static** (`_` arm) | **YES** |
  | 7 | MO_SYS_FIXED | Static | Static | ok |
  | **8** | **MO_SYS_THIN_BOX** | **Dynamic** | **Static** (`_` arm) | **YES** |
  | 9 | MO_SYS_CHARACTER | (character) | Static | edge |

  The code comments on the value-1 arm aside (`bhk_rigid_body_tests.rs:95` documents
  `1 = MO_SYS_DYNAMIC`), the `4 => Keyframed` / `_ => Static` collapse is the structural
  error: **BOX_INERTIA (4) is the single most common dynamic-body motion type** for box-shaped
  clutter (crates, ammo boxes, debris), and **KEYFRAMED (6) — doors, platforms, scripted props,
  creature ragdoll bones — falls into `_` and becomes Static.**
- **Evidence**: Mapping at `collision.rs:155-159`. Canonical enum at
  `/mnt/data/src/reference/nifxml/nif.xml:1109-1121`. The discriminator is consumed end-to-end:
  `byroredux/src/scene/nif_loader.rs:421-431` attaches the extracted `RigidBodyData` to the
  entity, and `crates/physics/src/sync.rs:215-222` (`motion_type_to_rapier`) maps
  `Static → Fixed`, `Keyframed/CharacterKinematic → KinematicPositionBased`, `Dynamic → Dynamic`.
  Consequences:
  - **BOX_INERTIA (4) crate → Keyframed → KinematicPositionBased.** `push_kinematic`
    (`sync.rs:361`) then drives it from its `GlobalTransform` every frame; a statically-placed
    crate has no animated transform, so it freezes in place with infinite mass instead of
    falling/responding to forces. `pull_dynamic` (`sync.rs:411`) never writes it back (gated on
    `== Dynamic`).
  - **KEYFRAMED (6) door/platform → Static → Fixed.** A scripted-moved door becomes permanently
    immovable; `push_kinematic` is gated on `== Keyframed` (`sync.rs:361`) so it will never
    track its transform.
  - **BOX_STABILIZED (5) / THIN_BOX (8) → Static.** Simulated debris becomes immovable.
- **Impact**: Physics-correctness regression across **all Havok-content games** (FNV, FO3,
  Oblivion, Skyrim share this extractor). For FO3 specifically: dynamic clutter that the player
  should be able to knock around, and scripted doors/platforms, behave wrong. Blast radius is
  the full bhk dynamic-body population — not a rare edge. Severity is HIGH (not CRITICAL) because
  it does not crash or corrupt memory and static architecture (FIXED=7) still maps correctly, so
  the bulk of walkable collision is unaffected.
- **Related**: Distinct from #1540 (degenerate trimesh inertia) and #1539 (dropped constraints).
  Not present in the 2026-06-16 FO3 report or the FO4/Oblivion audits (which mention `motion_type`
  only for the unrelated MOVS gap #1598).
- **Suggested Fix**: Map by the full enum: `1..=5 | 8 => Dynamic`, `6 => Keyframed`,
  `7 => Static`, `9 => CharacterKinematic` (or Static if character bodies are handled
  elsewhere), `0 | _ => Static` for INVALID. Add a regression test covering values 4, 5, 6, 8.

### LOW (2) — both re-confirmed STILL OPEN from 2026-06-16 (carried for tracking, not new defects)

#### D3-1 (re-confirmed): SCPT SCHR `flags` read as `u32` where field is `u16`
- **Severity**: LOW
- **Dimension**: 3 — ESM Records
- **Location**: `crates/plugin/src/esm/records/script.rs:136-137`
- **Status**: Re-confirmed open (first reported 2026-06-16 D3-1)
- **Description**: Still `if sub.data.len() >= 20 { record.flags = r.u32().unwrap_or(0); }`.
  After the 16-byte prefix + `type` u16 (cursor@18), only 2 bytes remain in the vanilla 20-byte
  FO3 SCHR, so the strict `u32` read fails and `flags` is always 0. **New supporting evidence
  this pass**: the in-module test at `script.rs:259` (`flags & 0x2`) only passes because its
  fixture is a non-vanilla **22-byte** SCHR — the test masks the bug rather than catching it.
  `flags` is parsed-but-unused, hence LOW.
- **Suggested Fix**: Read `r.u16()`; fix the comment; use a 20-byte fixture in the test.

#### D4-1 (re-confirmed): Worldspace selection runs preferred-default list before grid-containment
- **Severity**: LOW
- **Dimension**: 4 — Cell Loading
- **Location**: `byroredux/src/cell_loader/exterior.rs:113-139`
- **Status**: Re-confirmed open (first reported 2026-06-16 D4-1)
- **Description**: The preferred-default list (`["wastelandnv","wasteland","tamriel","skyrim"]`,
  line 114) still runs in `.or_else()` order *before* the grid-containment check (line 120).
  With `Fallout3.esm` + a DLC master, `--grid X,Y` targeting an FO3 DLC worldspace (Anchorage,
  Point Lookout, Zeta) without `--wrld` silently lands on Capital Wasteland. `--wrld` is the
  workaround. FNV-shared, unchanged.
- **Suggested Fix**: Run grid-containment before the preferred list when `--grid` is supplied.

## Shared-Path Commits Since 2026-06-16 — Verified Benign for FO3

- **`2aac5351` DXT1 1-bit punch-through alpha (BC1_RGB → BC1_RGBA).** FO3 textures are heavily
  DXT1/DXT5. Verified non-regressing: for opaque DXT1 blocks (4-colour mode, `color0 > color1`)
  BC1_RGBA samples `.a == 1.0` everywhere, so opaque FO3 diffuse maps are byte-identical and
  alpha-test never spuriously discards. Only 3-colour-mode blocks expose the punch-through bit —
  which is the intended authoring semantics. No FO3 defect. (`crates/renderer/src/vulkan/dds.rs`
  `map_fourcc` / `map_dxgi_format`.)
- **`067a8354` additive particle instance-batching + blend-enum fix.** FO3 fire/smoke/dust FX
  import via heuristic presets (`byroredux/src/scene/nif_loader.rs:495-519`); the corrected
  additive presets (`dst=0`) and the preserved alpha-over `smoke()` preset (`dst=7`) are
  internally consistent. Additive sort-key reorder also covers non-particle transparent FO3
  static meshes (real NIF blend modes). No FO3 regression.
- **Soft-particle depth fade (`f0c81539`/`1ddeae28`/`f0c…`)** and **ReSTIR attachment removal
  (`218b425b`).** No FO3-reachable regression: the depth fade is render-origin-relative and the
  removed reservoir attachment was write-only dead VRAM (#1583).
