## Findings

### ECS-2026-06-23-01: M47.2 `fragment.rs` declares an external `mod tests` whose file is missing — breaks `cargo test`
- **Severity**: HIGH
- **Dimension**: 7 (Component Lifecycles / scripting) + audit Process step 2 (`cargo test`)
- **Location**: `crates/scripting/src/fragment.rs:244`
- **Status**: NEW
- **Description**: `fragment.rs` ends with
  ```rust
  #[cfg(test)]
  mod tests;
  ```
  an **external** module declaration, but no `crates/scripting/src/fragment/tests.rs`
  (nor `fragment_tests.rs`) exists. The library builds (`cargo build -p
  byroredux-scripting` succeeds because `#[cfg(test)]` excludes the module from a
  non-test build), but **any test build fails to compile**:
  ```
  error[E0583]: file not found for module `tests`
    --> crates/scripting/src/fragment.rs:244:1
     |
  244 | mod tests;
     | ^^^^^^^^^^
  ```
  This contradicts the run brief's note that the M47.2 work "compiles" — it
  compiles as a *library* but not as a *test target*. Every other file in the
  crate that uses the external-tests pattern has its sibling (e.g.
  `recurring_update.rs` → `crates/scripting/src/recurring_update/tests.rs`,
  which exists); `fragment.rs` is the lone dangling declaration. The doc comment
  and the rich in-module test set referenced by the design (the `apply_effects` /
  cascade tests) were evidently authored but the file was never created — note
  `effects.rs` keeps its tests **inline** (`mod tests { ... }`), so the author
  switched conventions mid-module without landing the file.
- **Evidence**:
  ```
  $ cargo test -p byroredux-scripting --no-run
  error[E0583]: file not found for module `tests`  (fragment.rs:244)
  $ test -f crates/scripting/src/fragment/tests.rs && echo EXISTS || echo MISSING
  MISSING crates/scripting/src/fragment/tests.rs
  $ cargo build -p byroredux-scripting   # lib only
  Finished `dev` profile ...
  ```
- **Impact**: `cargo test -p byroredux-scripting` and the whole-workspace
  `cargo test` fail to compile. CI's test job is RED until this lands; any
  contributor running the standard `cargo test` gate before committing is blocked
  by an unrelated module. The fragment-dispatch runtime (`quest_fragment_dispatch_system`,
  `apply_effects`, the MAX_CASCADE backstop) ships **with zero executable test
  coverage** despite the design calling for it — the cascade/cycle guard and the
  `Self`/`OwningQuest`/`Property` resolution are unverified at runtime.
- **Related**: Sibling-tests convention is documented in MEMORY (session34/35
  layout). Closest prior shape: none — this is a fresh M47.2 regression, not a
  reopen.
- **Suggested Fix**: Either create `crates/scripting/src/fragment/tests.rs` with
  the intended dispatch/cascade tests, or convert line 244 to an inline
  `#[cfg(test)] mod tests { ... }` block matching `effects.rs`. Then re-run
  `cargo test -p byroredux-scripting` to confirm green before committing the
  M47.2 slice.

## Verified-Clean (attempted-and-disproved)

These were examined as candidate findings and **disproved** — recorded so a
future audit doesn't re-flag them:

- **`fragment::quest_fragment_dispatch_system` holds three resource locks
  simultaneously** (`QuestStageFragments` read + `QuestStageState` write +
  `QuestObjectiveState` write, acquired in source order, NOT TypeId-sorted). Not
  a finding: the system and every other quest-resource consumer
  (`quest_advance`, `dlc2_ttr4a`, `mg07_door`, `condition` eval) run in the
  **exclusive** phase of `Stage::Update`, which executes serially after the
  parallel batch. No parallel-batch system touches any of these resources, so
  there is no cross-thread hold-overlap and the ABBA graph has no edge to close.
  The thread-local reentrancy tracker also passes because the three types are
  distinct. (If any of these systems were ever moved to a parallel registration,
  this would become a real ABBA risk — flag it then.)
- **`PackedStorage::insert_bulk` marks pre-existing (unchanged) entities dirty**
  via `self.dirty.extend_from_slice(&self.entities)` after the merge. Not a
  finding: it is documented as a conservative all-mark that runs only at
  cell-load boundaries, dups are tolerated by the transform-propagation consumer,
  and the over-marking is bounded by the load (not per-frame).
- **Incremental transform-propagation path reads a dirty child's parent global
  before the parent (also dirty, later in the sorted list) is recomposed.** Not a
  finding: the ancestor, when processed, seeds its children into the BFS queue,
  which re-composes the child — self-healing regardless of dirty-list order (the
  in-file comment and `change_detection_reacts_to_reparent_overwrite` /
  `root_cache_steady_state_still_runs_propagation` tests pin this).
- **`timer_tick_system` could accumulate negative `remaining`.** Not a finding:
  the timer is removed in the same tick it crosses `<= 0.0`, so it never ticks
  again.
