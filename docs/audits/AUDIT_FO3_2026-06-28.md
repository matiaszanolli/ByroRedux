# Fallout 3 Compatibility Audit — 2026-06-28

**Engine HEAD**: `e76c70ba` · **Scope**: 7-dimension `/audit-fo3` sweep of ByroRedux
readiness for Fallout 3 content. **Reference data**: full FO3 GOTY at
`/mnt/data/SteamLibrary/steamapps/common/Fallout 3 goty/Data/` (base + all five DLCs).

FO3 rides the FNV path almost end-to-end (NIF v20.2.0.7 / BSVER 34, BSA v104, the
shared ESM parser, the shared cell loader, RT lighting). This audit hunted the
**divergences** — where FO3 content exercises a path FNV doesn't, where shared code
carries an FNV-only assumption, and where a feature is verified on FNV but only
*assumed* on FO3. Every checklist item was re-read against the live tree, and the
data-bearing dimensions (2, 3, 6) were validated against the real FO3 masters.

### Dimension 1 — FO3 Rendering Path (Inline Shaders) · **CLEAN**

inline-shader material universe is provably the only path taken:
`BSShaderPPLightingProperty` / `BSShaderNoLightingProperty` flow through the FO3/FNV

#### D2-01: BSSegmentedTriShape arm correct but dormant on vanilla FO3
- **Severity**: LOW (informational) · **Status**: Informational
- **Location**: `crates/nif/src/blocks/tri_shape/ni_tri_shape.rs:182-191`, dispatch at `crates/nif/src/blocks/mod.rs:434`
- **Description**: The checklist premise "FO3 uses these for biped parts" does not hold —
  vanilla FO3 ships **zero** BSSegmentedTriShape blocks across all 16 BSAs (FO3
  dismemberment uses NiSkinPartition dismember partitions / `_gore` swaps;
  BSSegmentedTriShape is a Skyrim-LE/FO4 construct). The arm itself is correct for the FO3
  (`#NI_BS_LT_FO4#`) 9-byte-per-segment layout.
- **Impact**: None for vanilla FO3; the arm correctly handles modded Skyrim/FO4-exporter content.
- **Suggested Fix**: None required. Optionally add a `check_alloc(num_segments*9)` guard for
  defense-in-depth parity with the other PSys loops (cosmetic).

#### D2-02: nif.xml "Unknown QQSpeed Floats[2]" trailing field is correctly NOT read
- **Severity**: LOW (informational) · **Status**: Informational
- **Location**: `crates/nif/src/blocks/particle.rs:90-132` (`read_emitter_base`)
- **Description**: nif.xml lists a final unconditional `Unknown QQSpeed Floats[2]` field on
  `NiPSysEmitter` ("Both 1.0 in example nif"). `read_emitter_base` correctly does NOT read
  these 8 bytes — they are a niftools speculative artifact absent from retail streams.
  Reading them would introduce an 8-byte over-read on every emitter (all 422 FO3 emitters
  parse with zero drift without them).
- **Suggested Fix**: None — do **not** add a read for this field (captured so a future audit
  doesn't "fix" it from nif.xml and regress the 100% rate — a no-guessing trap).

#### D3-01: WATR.DATA shallow/deep/reflection colors mis-read on 186-byte records (off-by-4)
- **Severity**: MEDIUM · **Status**: NEW (shared FO3/FNV — surfaced by this FO3 run)
- **Location**: `crates/plugin/src/esm/records/misc/water.rs:135-209` (`decode_data`)
- **Description**: `decode_data` documents FO3/FNV `WATR.DATA` as a "60-byte prefix" with
  `shallow_color`@36 / `deep_color`@40 / `reflection_color`@44. This does not match the
  on-disk data: the DATA sub-record is never 60 bytes — it is either 2 bytes (Damage only,
  42 FO3 / 70 FNV records, colors benignly default OOB) or **186 bytes** (11 FO3 / 8 FNV
  records, full visual params). On the 186-byte records there is an extra f32 (a second fog
  field) at offset 36, so the real color block sits at **40/44/48**, not 36/40/44. The code
  reads shallow@36 = the fog float's bytes (garbage near-pure-blue), and deep/reflection
  each shifted one slot toward shallow; the true reflection color (offset 48) is never read.
- **Evidence**: hex dump of `PPurityWater01Murky` (FO3, 186-byte DATA): offset 36-39 =
  `00 00 da 42` = 109.0 (a float, not a color); real shallow `[36,47,36]`@40, deep
  `[13,13,11]`@44, reflection `[41,48,46]`@48. Second sample `DupontFontWaterType` confirms
  the identical 4-byte shift. The existing "53/53 non-default colors" test passes because
  the misalignment always yields *a* non-zero color — which is why it slipped past prior audits.
- **Impact**: Every FO3/FNV water body with a 186-byte DATA (Project Purity water, Dupont
  fountain, named river/pond types) renders with a wrong shallow tint and one-slot-shifted
  deep/reflection colors. Visible but cosmetic; the fog/wind/wave/fresnel prefix (offsets
  0-35) reads correctly.
- **Suggested Fix**: Move the three color reads to 40/44/48 and add the missing f32 at offset
  36, gated on `data.len() >= 52`; keep the OOB-defaults fallback so 2-byte records are
  unaffected. **Determine the authoritative 186-byte field map from an external reference
  (xEdit `wbDefinitionsFNV` WATR / GECK wiki) — do not guess the full layout** (per the
  no-guessing policy). Correct the `decode_data` doc comment (the "60-byte prefix identical
  across Oblivion/FO3/FNV" claim is false). Add a regression fixture from the
  `PPurityWater01Murky` bytes.

#### D4-01: XCCM per-cell climate override parsed but never consumed
- **Severity**: LOW · **Status**: Existing — #693 / O3-N-05 (tracked in
  `docs/audits/AUDIT_OBLIVION_2026-04-25.md`, `AUDIT_FNV_2026-04-17.md`; cross-game, not FO3-specific)
- **Location**: `byroredux/src/cell_loader/exterior.rs:178-196`; field source
  `crates/plugin/src/esm/cell/wrld.rs:324`
- **Description**: `XCCM` (per-cell CLMT FormID override) is parsed into

- **Suggested Fix**: In `load_one_exterior_cell`, when `cell.climate_override.is_some()`,

### Dimension 5 — Collision Import (Havok → CollisionShape) · **1 LOW (new) + 1 existing**

FO3 collision rides the FNV/Oblivion legacy path with no FO3-specific divergence. Retail
FO3 (NIF v20.2.0.7 / BSVER 34) classifies as `CollisionAuthoring::Classic` (the
discriminator downcasts on struct type, so `bhkSPCollisionObject`/`bhkBlendCollisionObject`
aliases also route to Classic); `havok_motion_type` matches the nif.xml `hkMotionType`
enum byte-for-byte (#1652 — BOX_INERTIA clutter simulates, KEYFRAMED doors aren't frozen);
MultiSphere/ConvexList translate (#1277); every dispatched `bhk*Shape` has a CI-enforced
`resolve_shape` arm. 79 collision tests pass; the motion-type HIGH from
`AUDIT_FO3_2026-06-18.md` is confirmed fixed at HEAD.

#### D5-01: TriMesh collision resolvers lack the #1409 finite guard
- **Severity**: LOW · **Status**: NEW
- **Location**: `crates/nif/src/import/collision.rs` (`resolve_packed_mesh`,
  `resolve_tri_strips_data_refs`, `resolve_compressed_mesh`); consumer
  `crates/physics/src/convert.rs:156-176`
- **Description**: The #1409 hardening added a `finite()`/`finite_vec()` guard to every
  *primitive* shape resolver (sphere/box/capsule/cylinder/convex-hull/multi-sphere) so a
  NaN/±Inf scalar drops the authored shape to the synthesized-trimesh fallback. The three
  *mesh* resolvers were never brought into that sweep — they build `CollisionShape::TriMesh`
  from raw `read_f32_le`/dequantized values with no finite check and bail only on
  `vertices.is_empty()`. The downstream consumer `collision_shape_to_parts` likewise guards
  only `is_empty` before `SharedShape::trimesh_with_flags`, whose default
  `MERGE_DUPLICATE_VERTICES` spatial-hash bucketing on NaN coordinates is undefined.
- **Impact**: FO3 leans heavily on `bhkPackedNiTriStripsShape` (MOPP→PackedNiTriStrips is
  the dominant static-architecture collider), so this is FO3's most-used collision path.
  Real-world reachability is low (vanilla BSA content is finite) — it only fires on
  corrupt/truncated/adversarial NIFs, which is exactly #1409's threat model. When it fires
  the result (a poisoned broadphase / Qbvh with NaN bounds) is worse than the clean
  fallback the primitives get. LOW because it does not drop translatable vanilla content.
- **Suggested Fix**: Mirror the primitive guard in the three mesh builders
  (`if all_verts.iter().any(|v| !v.is_finite()) { return None; }`), and add the same filter
  at `convert.rs:156` as the single choke point all TriMesh sources pass through.

#### D5-02: bhk Hinge/Prismatic/Breakable/StiffSpring constraints dropped in FO3 ragdoll chains
- **Severity**: LOW · **Status**: Existing — #1718 (adjacent OPEN) / #1539; reported as
  D5-FO3-01 in `docs/audits/AUDIT_FO3_2026-06-23.md`. Recorded for completeness; **not new.**
- **Location**: `crates/nif/src/import/collision.rs` (`extract_ragdoll`,
  `BhkConstraintData::Other` arm)
- **Description**: `extract_ragdoll` decodes only `Ragdoll` and `LimitedHinge` CInfo (plus
  the FO3 `bhkMalleableConstraint` unwrap); every other kind decodes to
  `BhkConstraintData::Other` and is `continue`d (with a `log::warn!`). FO3 base meshes carry
  `bhkHingeConstraint`/`bhkPrismaticConstraint`/`bhkBreakableConstraint` in small counts.
- **Impact**: A handful of FO3 articulated objects lose a joint; static architecture
  unaffected (guarded by `bodies.len() >= 2`). Already warned, already tracked.
- **Suggested Fix**: Per #1539 — map a limitless hinge to `LimitedHinge { min: -PI, max: PI }`.

## FO3-Distinctive Gaps

1. **Inline-shader-only material universe.** FO3 authors no BGSM/BGEM and no
   BSLighting/BSEffect shader properties, so the Disney-BSDF lobe and the Effect/Lighting
   emissive sources are unreachable. This is correctly handled (Dim 1) — listed here as the
   defining FO3 material characteristic, not a gap.

2. **Capital Wasteland worldspace.** A distinct WRLD form ID with its own origin/CLMT
   curves. Handled correctly — worldspace selection is record-driven with zero FNV
   hardcode, and the grid→world transform is the universal 4096-unit constant (Dim 4).
   The open item is a fresh GPU exterior bench (R6a), not a code gap.

3. **The Obscript scripting runtime gap (the dominant FO3 interactivity blocker).** FO3
   ships **Obscript** (1,257 SCPT records: SCDA compiled blob + SCTX source). The blob is
   parsed-but-kept-opaque (`crates/plugin/src/esm/records/script.rs`); nothing executes it.
   **Critical distinction:** the M47.x runtime that has shipped (M47.0 event hooks, M47.1
   CTDA condition eval, M47.2 `.pex` recognizer slice) targets **compiled Papyrus**
   (Skyrim+/FO4). FO3 Obscript is a *different* pre-Papyrus bytecode language that the
   `.pex` recognizer does not decode — the ROADMAP explicitly lists "Obscript via SCTX" as
   still **remaining**. Net for FO3: NPCs/objects spawn, equip, animate (kf), and exist,
   but any SCPT-driven behavior (terminals, scripted doors/triggers, quest-stage side
   effects, ScriptEffect magic) is inert. Owned by `/audit-scripting`; characterized here,
   not filed.
