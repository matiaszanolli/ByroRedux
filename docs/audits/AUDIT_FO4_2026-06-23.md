## Findings

### FO4-2026-06-23-L01: #1592 comment claims Glow_Map is OR'd into MaterialInfo, but only MSN + Alpha_Test are
- **Severity**: LOW
- **Dimension**: 5 (FO4 shader flags)
- **Location**: `crates/nif/src/import/material/walker.rs:307-340`
- **Status**: NEW
- **Description**: The #1592 block comment (lines 310-311) states the walker ORs
  "F4SF2 bit 25 (`Alpha_Test`) and bit 6 (`Glow_Map`)" into `MaterialInfo`, but
  the implemented code only consumes `MODEL_SPACE_NORMALS` (F4SF1 bit 12) and
  `ALPHA_TEST` (F4SF2 bit 25). There is no `GLOW_MAP` (F4SF2 bit 6,
  `shader_flags::fo4_slsf2::GLOW_MAP = 0x40`) gate in this band.
- **Evidence**: `grep GLOW_MAP crates/nif/src/import/material/walker.rs` matches
  only the comment, not a flag test. The const exists
  (`shader_flags.rs:177 GLOW_MAP = 0x0000_0040`) but is unused in the walker.
  The `fo4_shader_flag_tests.rs` suite covers MSN + Alpha_Test, not Glow_Map.
- **Impact**: Effectively none for content correctness — the glow *texture* is
  sourced unconditionally from the `BSShaderTextureSet` glow slot
  (`walker.rs:156, 669, 779`) whenever present, so glow rendering does not
  depend on the shader-flag gate. The gap is only on a hypothetical inline /
  modded FO4 NIF that signals glow via the flag bit alone with no glow texture
  in its texture set — a degenerate authoring that produces nothing to glow
  regardless. The finding is a comment-vs-code drift, not a content-loss bug.
- **Related**: #1592 (closed, `f7fbbed5`).
- **Suggested Fix**: Either drop "and bit 6 (`Glow_Map`)" from the comment, or
  add the one-line `info.glow_map`-presence-independent flag (low value). Comment
  trim is the honest fix.

### FO4-2026-06-23-L02: dead `take == 0` break in CsgArchive::read_psg
- **Severity**: LOW
- **Dimension**: 1 (M49 CSG)
- **Location**: `crates/bsa/src/csg.rs:207-213`
- **Status**: NEW
- **Description**: In `read_psg`, after the `local >= chunk.len()` guard
  returns early, `take = remaining.min(chunk.len() - local)` is always `>= 1`
  (both operands positive), so the trailing `if take == 0 { break; }` is
  unreachable dead code.
- **Evidence**: The preceding guard (`csg.rs:198-206`) already returns
  `UnexpectedEof` when `local >= chunk.len()`, guaranteeing
  `chunk.len() - local >= 1`; `remaining > 0` is the loop condition. So `take`
  is never 0.
- **Impact**: None — correctness is unaffected; it is a redundant guard.
- **Related**: M49 CSG reader (`067adc34`).
- **Suggested Fix**: Remove the dead branch, or leave as defensive (the cost is
  zero). Cosmetic.
