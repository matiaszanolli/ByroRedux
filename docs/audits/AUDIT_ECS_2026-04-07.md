## Findings

### E26-01: Lock tracker state can leak if `lock.read/write()` panics after `track_*`
- **Severity**: LOW
- **Dimension**: Query Safety (hardening)
- **Location**: `crates/core/src/ecs/world.rs` — `query` (214-220), `query_mut` (226-234), `query_2_mut` (245-287), `query_2_mut_mut` (297-338), `resource` (369-382), `resource_mut` (389-402), `resource_2_mut` (412-464), `try_resource` (467-475), `try_resource_mut` (478-486)
- **Status**: NEW
- **Description**: `world.rs` calls `lock_tracker::track_read/write()` BEFORE acquiring the `RwLock` guard. This is intentional — it's the only way to detect a same-thread deadlock *before* blocking on the lock. But if `lock.read()` or `lock.write()` panics (e.g. `storage_lock_poisoned::<T>()` fires on a poisoned lock), the `Drop` impl on `QueryRead`/`QueryWrite`/`ResourceRead`/`ResourceWrite` doesn't run because no guard was ever constructed. The thread-local `LockState` row for that `TypeId` now has a stale `read_count` or `has_write` flag.
- **Evidence**:
  ```rust
  pub fn query<T: Component>(&self) -> Option<QueryRead<'_, T>> {
      let type_id = TypeId::of::<T>();
      let lock = self.storages.get(&type_id)?;
      lock_tracker::track_read(type_id, std::any::type_name::<T>());  // tracked
      let guard = lock.read().unwrap_or_else(|_| storage_lock_poisoned::<T>());  // may panic
      Some(QueryRead::new(guard, type_id))
  }
  ```
- **Impact**: If a caller uses `catch_unwind` to recover from a poisoned lock and then tries another query on the same type, the tracker will falsely report a deadlock. In normal operation, a poisoned lock means a system panicked and the process is already broken — this is a hardening concern, not a real-world hazard.
- **Related**: #86 (the lock tracker itself), #95 (type-aware poison panics)
- **Suggested Fix**: Use a small RAII scope guard that untracks on drop, then `std::mem::forget` it just before returning the successful guard. Or perform the `lock.read()/write()` FIRST, then track after acquisition — but that re-introduces the original deadlock problem (the thread blocks before the tracker can panic). The scope-guard approach is the minimally invasive fix.

### E26-02: `has()` and `count()` manually track/untrack without RAII guard
- **Severity**: LOW
- **Dimension**: Query Safety (hardening)
- **Location**: `crates/core/src/ecs/world.rs:137-166`
- **Status**: NEW
- **Description**: `has()` and `count()` bypass the `QueryRead` RAII wrapper and manually call `track_read` / `untrack_read` around a directly-acquired guard. If any operation between track and untrack panics (e.g. the theoretically-unreachable `downcast_ref::<T::Storage>().expect("storage type mismatch")`), the `untrack_read` call is skipped and the tracker leaks state for that type.
- **Evidence**:
  ```rust
  pub fn has<T: Component>(&self, entity: EntityId) -> bool {
      self.storages.get(&TypeId::of::<T>()).is_some_and(|lock| {
          let type_id = TypeId::of::<T>();
          lock_tracker::track_read(type_id, std::any::type_name::<T>());
          let guard = lock.read().unwrap_or_else(|_| storage_lock_poisoned::<T>());
          let result = guard
              .downcast_ref::<T::Storage>()
              .expect("storage type mismatch")  // ← panic here leaks tracker
              .contains(entity);
          drop(guard);
          lock_tracker::untrack_read(type_id);
          result
      })
  }
  ```
- **Impact**: The `expect` is unreachable under current invariants (TypeId key guarantees the downcast succeeds), so in practice this cannot leak. Hardening only.
- **Related**: E26-01, #86
- **Suggested Fix**: Rewrite `has()` / `count()` to call `self.query::<T>()` and use the returned `QueryRead::contains` / `len`. Simpler code and automatic `Drop`-based untrack. Slight extra indirection but negligible cost.

### E26-03: `resource.rs` — `Drop` impls for `ResourceWrite` appear before the struct definition
- **Severity**: LOW
- **Dimension**: Code Quality
- **Location**: `crates/core/src/ecs/resource.rs:44-48` vs `61-65`
- **Status**: NEW
- **Description**: `impl<R: Resource> Drop for ResourceWrite<'_, R>` is defined at lines 44–48, BEFORE the `ResourceWrite` struct itself is defined at lines 61–65. Rust allows forward references within a module so this compiles fine, but it's confusing to read — the reader encounters a `Drop` impl for an undefined type.
- **Evidence**:
  ```rust
  // Line 38: Drop for ResourceRead (expected — ResourceRead defined above)
  impl<R: Resource> Drop for ResourceRead<'_, R> { ... }

  // Line 44: Drop for ResourceWrite (out of order)
  impl<R: Resource> Drop for ResourceWrite<'_, R> { ... }

  // Line 50: Deref for ResourceRead
  impl<R: Resource> Deref for ResourceRead<'_, R> { ... }

  // Line 61: ResourceWrite struct definition (finally)
  pub struct ResourceWrite<'w, R: Resource> { ... }
  ```
- **Impact**: Code readability. No functional effect.
- **Suggested Fix**: Move the `Drop for ResourceWrite` impl to after the struct definition. Keep all impls for a type grouped together.

### E26-04: Public query/resource docs don't mention same-thread deadlock detection
- **Severity**: LOW
- **Dimension**: Documentation
- **Location**: `crates/core/src/ecs/world.rs` — all query/resource methods' doc comments
- **Status**: NEW
- **Description**: The doc comments on `query`, `query_mut`, `resource`, `resource_mut`, and their 2-variant cousins describe return types and same-type panic behavior but don't mention the `lock_tracker`-based panic that fires when a caller on the same thread already holds a conflicting lock on the same type. Users reading the API docs may be surprised by the `"ECS deadlock detected: attempted X lock on Y while ..."` panic.
- **Evidence**: `query_2_mut` at line 236 documents "Panics if `A` and `B` are the same type (would deadlock)" but not "Panics if a read/write lock on `A` or `B` is already held on the same thread."
- **Impact**: Documentation completeness. No behavioral issue.
- **Suggested Fix**: Add a "# Panics (debug only)" section to each query/resource method listing the `lock_tracker` panic cases. Single-paragraph doc update per method.
