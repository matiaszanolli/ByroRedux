The ECS core remains in excellent shape. `cargo test -p byroredux-core`
runs **280 lib tests + 2 doc tests, 0 failed** (3 doc-tests ignored —
intentional `ignore` markers). `cargo test -p byroredux-scripting`

`crates/core/src/ecs/`.

  fixed. `crates/core/src/ecs/systems.rs:43,111` now uses
  `VecDeque::pop_front`. Doc and behaviour aligned.

  `world.rs:687-704` adds the TypeId-sorted Option-returning sibling,
  with four dedicated tests
  (`world.rs:1483-1527`).

- **E-03 (LOW, Dim 3)** [#466] `World::despawn` poisoned-lock panic
  attribution still uses `"<unknown>"`.

This audit's net-new findings are all in **Dimension 7** (lifecycle of
streaming / scripting / animation), with one **HIGH** asset-leak hazard

documentation gap. Storage / query / world / resource / scheduler /

## Dimension 1: Storage Correctness

All checks pass.

- `SparseSetStorage::remove` swap-remove correctly fixes the moved
  entity's sparse pointer (`sparse_set.rs:75`); the last-element case
  short-circuits at `:60-64` without an unnecessary swap.
- `SparseSetStorage::insert` overwrites in place
  (`sparse_set.rs:39-41`) — no duplicate dense rows.
- `PackedStorage::insert` keeps sort order via `binary_search` +
  `Vec::insert` at `packed.rs:41-51`. Overwrite (`Ok(idx)`) is
  in-place; insert (`Err(idx)`) shifts.
- `PackedStorage::remove` matches via `binary_search` + `Vec::remove`
  at `packed.rs:54-62`.
- `insert_bulk` (`packed.rs:93-166`, #467 closeout) provides the
  O(N + M log M) fast path used by `World::insert_batch`.
  Last-writer-wins dedup pinned by
  `insert_bulk_duplicate_ids_last_writer_wins`.

### E-04 (carryover): single-call PackedStorage insert is O(n)

- **Severity**: LOW
- **Location**: `crates/core/src/ecs/packed.rs:48-49`, `:57-58`
- **Status**: KNOWN — documented trade-off in module head comment
  (`packed.rs:1-15`).
- **Description**: One-at-a-time `insert` / `remove` call still uses
  `Vec::insert` / `Vec::remove` at the binary-search position. For
  Transform with thousands of entities, each call shifts every later
  element. The `insert_bulk` path (#467) covers the cell-load hot
  path; the cell-loader's mixed-component pattern
  (Transform + Material + MeshHandle per REFR) doesn't always hit
  bulk and pays the shift.
- **Impact**: Bulk paths are fixed. Steady-state per-frame mutation
  rarely inserts into Packed storage so single-call cost is
  amortised; this is a perf hardening note, not a bug.
- **Suggested Fix**: If profiling shows a hotspot in mixed-component
  cell load, restructure that loader to collect-then-insert_batch
  per type. Do NOT switch to swap-remove — that breaks the sort
  invariant.

## Dimension 2: Query Safety

All checks pass.

- `QueryRead<'w, T>` / `QueryWrite<'w, T>` own their `RwLockReadGuard`
  / `RwLockWriteGuard`; lifetime `'w` ties the wrapper to `&World`
  (`query.rs:23-94`).
- `query_2_mut` / `query_2_mut_mut` both acquire in **TypeId-ascending
  order** regardless of generic-parameter order. Both tracker scopes
  AND real locks follow the same order (`world.rs:393-423`,
  `:455-485`, the #313 fix).
- Same-type double-lock panics with a clear message at the top of all
  three 2-call methods (`world.rs:376-380`, `:442-446`, `:578-582`,
  `:690-694`); never reaches the lock site.
- Same-thread reentrant deadlock panics via `lock_tracker` in BOTH
  debug AND release builds (`lock_tracker.rs:54-126`,
  unconditional). Specifically pinned by
  `query_read_then_write_same_type_panics`,
  `query_write_then_read_same_type_panics`, and the resource
  variants.
- `Deref` / `DerefMut` on every wrapper returns a reference borrowed
  from `&self` / `&mut self`; the borrow checker prevents the
  pre-#35 raw-pointer unsoundness.
- `ComponentRef::new` pre-checks `contains(entity)` (`query.rs:225`);
  the deref `expect` is infallible while the read lock is held.
- The `lock_tracker_is_clean_after_poisoned_panic` test
  (`world.rs:1823-1908`) pins that every poisoned-lock panic path
  unwinds through the RAII `TrackedRead` / `TrackedWrite` scope and
  leaves the thread-local map empty — no false-positive deadlock
  reports on a subsequent recovery.

## Dimension 3: World Integrity

### E-03 (carryover, OPEN as #466): despawn erased-poison panic loses type name

- **Severity**: LOW
- **Location**: `crates/core/src/ecs/world.rs:113-115`
- **Status**: OPEN — tracked at GitHub #466. No code change since
  2026-04-19.
- **Description**: Inside `despawn`, the poisoned-lock helper still
  receives `"<unknown>"`. The loop variable `type_id` is in scope but
  not propagated to the panic message; line 117 explicitly suppresses
  it with `let _ = type_id;`.
- **Impact**: Harder diagnostics if a prior system panic poisons a
  storage lock AND a subsequent `despawn` trips the helper. Rare
  compound failure.
- **Suggested Fix**: Plumb a `TypeId → &'static str` side-table
  through `register` / `storage_write`, or panic with the `TypeId`
  hex value so the user can grep for it.

## Dimension 7: Streaming, Scripting & New Component Lifecycles

### E-N1 (NEW): AnimationClipRegistry is grow-only across cell streaming

- **Severity**: HIGH
- **Location**: `byroredux/src/cell_loader.rs:848-857`,
  `byroredux/src/npc_spawn.rs:175-181`,
  `crates/core/src/animation/registry.rs:19-23`
- **Status**: NEW
- **Description**: `AnimationClipRegistry::add` is monotonic
  (`registry.rs:19-23`) — handle = `self.clips.len() as u32`, no
  removal API, no path-keyed dedup. `cell_loader::load_references`
  calls `npc_spawn::load_idle_clip` ONCE per cell load with NPCs
  (`cell_loader.rs:848`); that helper unconditionally
  `.push`-converts the parsed `idle.kf` and calls `registry.add(clip)`
  (`npc_spawn.rs:179-180`).

  With M40 cell streaming enabled (3×3 to 7×7 grids around the
  player), every cell-crossing that loads a fresh NPC-bearing cell
  appends another full copy of the same `meshes\characters\_male\idle.kf`
  clip. Walking the Wasteland for 50 cells leaves ~50 duplicate
  clips in the registry, each with the full per-channel keyframe
  data. `cell_loader::unload_cell` despawns the
  `AnimationPlayer` / `AnimationStack` entities but NEVER touches
  the registry — clips persist for the process lifetime.
- **Impact**: Memory growth proportional to cells visited × NPC
  count per cell. A long FNV exterior session leaks tens of MB of
  duplicated keyframe data and grows the per-frame `advance_stack`
  scan (each `registry.get(layer.clip_handle)` is O(1) but the
  registry's allocator footprint is unbounded).
- **Suggested Fix**: Either
  1. Cache `idle_clip_handle` on a process-lifetime ECS resource
     keyed by `(GameKind, Gender, kf_path)` and reuse across cell
     loads, or
  2. Add a `register_or_get_by_path(&str, AnimationClip) -> u32`
     method to `AnimationClipRegistry` that dedups by path symbol
     before pushing.
  Option (1) is the smaller change and matches how
  `NifImportRegistry` is already used (process-lifetime cache,
  see #381). The doc comment at `cell_loader.rs:837-841` is
  defending against per-NPC duplication but doesn't notice the
  per-cell-load duplication, so the comment should be updated as
  part of the fix.

### E-N2 (NEW): unload_cell victim collection scans every CellRoot entity

- **Severity**: LOW
- **Location**: `byroredux/src/cell_loader.rs:104-116`
- **Status**: NEW
- **Description**: `unload_cell` filters all `CellRoot` rows
  (`q.iter().filter(|(_, root)| root.0 == cell_root)`) every time a
  cell unloads. With a 3×3 streaming grid plus a current cell
  containing ~1500 entities, an unload at the boundary scans ~13500
  rows to find the ~1500 belonging to the unloading cell.

  `CellRoot` is `SparseSetStorage` so the scan is dense-array
  iteration — fast in absolute terms, but quadratic in cells × cell
  size as the streaming radius grows.
- **Impact**: At radius 1 the scan is invisible. At radius 3 (default
  exterior streaming, 49 cells) it's measurable. The scaling is
  fine for the current target hardware; flagged because future
  radius bumps or streaming + interior coexistence could surface
  this.
- **Suggested Fix**: Maintain a `HashMap<EntityId, Vec<EntityId>>`
  resource keyed by `cell_root → owned entities` populated by
  `stamp_cell_root` (`cell_loader.rs:77-85`). Unload becomes
  `map.remove(&cell_root).into_iter().flatten()`. Memory cost is
  ~8 B per cell-owned entity, dwarfed by the entity's component
  data.

### E-N3 (NIT): stamp_cell_root doc grammar

- **Severity**: NIT
- **Location**: `byroredux/src/cell_loader.rs:80-82`
- **Status**: NEW
- **Description**: The docstring on the per-entity `insert` reads
  "entities that were never given any component never created a
  CellRoot storage entry". This is muddled — what happens is that
  `world.insert(eid, CellRoot(cell_root))` always succeeds for
  spawned `eid`s in the range; entities that were spawned but
  received zero other components still get a `CellRoot` row added,
  which is fine. The comment seems to be defending against a case
  that doesn't apply.
- **Impact**: None — pure doc clarity.
- **Suggested Fix**: Replace with "every spawned entity in the
  range is reachable from this stamp pass; the loop is keyed on
  `EntityId` so entities that received zero other components still
  pick up a `CellRoot` row, which is what unload's filter scans".

### Other Dim-7 checks — confirmed correct

- **`event_cleanup_system`** at `crates/scripting/src/cleanup.rs:15-31`
  drains all four marker types (`ActivateEvent`, `HitEvent`,
  `TimerExpired`, `AnimationTextKeyEvents`) on every Late-stage tick.
  Single-frame lifetime pinned by
  `cleanup_removes_all_event_types`, `cleanup_preserves_non_event_components`.
- **`timer_tick_system`** at `crates/scripting/src/timer.rs:28-58`
  decrements `remaining` by `dt`, collects hits with
  `remaining <= 0.0`, removes the timer, and inserts a
  `TimerExpired` marker. No negative-time accumulation possible:
  the timer is removed in the same frame it triggers. Pinned by
  `timer_fires_after_duration`, `timer_zero_duration_fires_immediately`,
  `multiple_timers_independent`. (One subtle observation: if `dt`
  is huge, `remaining` may go very negative before the check — fine
  because the entity is removed before the next tick.)
- **`AnimationController`** lifecycle (`controller.rs`): the
  controller stores `clip_handle: u32` indices into
  `AnimationClipRegistry`. Despite E-N1's leak, the **handle
  itself never aliases stale data** — the registry is grow-only,
  so a handle issued on cell N still points at the same clip on
  cell N+10. Component lifecycle is sound: when the entity is
  despawned (cell unload), its `AnimationController` /
  `AnimationPlayer` / `AnimationStack` are despawned with it via
  `World::despawn`'s `DynStorage::remove_entity_erased` walk.
  No dangling handle reads possible.
- **NPC spawn idempotency** (`byroredux/src/npc_spawn.rs:291`,
  invoked from `cell_loader.rs:974-986`): each REFR is processed
  once per cell load via the `cell.references` Vec. Cell-load
  re-issues are blocked by `WorldStreamingState::loaded`'s
  `HashMap<(gx, gy), LoadedCell>` keying — `compute_streaming_deltas`
  filters `to_load` by `!loaded.contains_key(coord)`
  (`streaming.rs:362-364`), so a still-loaded cell never re-spawns.
  Within one ESM, a duplicate REFR FormID would double-spawn, but
  that's an ESM authoring error and not an ECS lifecycle concern.
- **`DebugDrainSystem`** (`crates/debug-server/src/system.rs`):
  TCP threads in `listener.rs::handle_client` push commands into
  the shared `Mutex<Vec<PendingCommand>>` queue
  (`listener.rs:103-109`); they NEVER touch `World`. The drain
  system is registered as Late-stage exclusive
  (`system.rs:48-49,141-143`), so it has the entire stage to
  itself and runs `evaluator::evaluate(world, ..., &cmd.request)`
  — read-only on the world. Screenshot bridge is the only path
  that crosses thread boundaries (request flag set, payload taken
  via `ScreenshotBridge::take_result`), and that uses an explicit
  Resource as the rendezvous point, not raw `World` mutation.

  (`sparse_set.rs:75`).
- SparseSet insert overwrites in place — no duplicate row.
- Packed maintains sort order after insert (`binary_search` +
  insert-at-idx) and remove (`binary_search` + remove-at-idx).
- `insert_bulk` is dedup-safe and last-writer-wins
  (#467 closeout).
- Query lifetimes tied to `&World` via guard lifetime `'w`.
- Same-type double-lock panics immediately in `query_2_mut`,
  `query_2_mut_mut`, `resource_2_mut`, `try_resource_2_mut`, AND
  via the `lock_tracker` thread-local check.

  (tests `poisoned_storage_lock_panics_with_type_name`,
  `poisoned_resource_lock_panics_with_type_name`).

- `find_by_name` uses `StringPool` symbol equality, not string
  comparison.

- `try_resource_2_mut` returns `None` without touching locks or
  the tracker when either resource is missing

  Late-stage `event_cleanup_system` drains all four event types.
- Per-client TCP threads do not mutate `World`; all mutation is
  routed through Late-stage exclusive `DebugDrainSystem`.
