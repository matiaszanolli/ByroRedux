## Dimension Findings

### Dimension 1: NIF BSVER 130 + Half-Float Vertices

**Result: 7/7 PASS — No HIGH/MEDIUM findings.**

All checklist items confirmed correct: `VF_FULL_PRECISION` flag resolution, IEEE 754 binary16 decode (including denormals, NaN), BSSubIndexTriShape segment walk, skinned-vertex packed layout + partition remap, BSVER 130 trailing fields on BSLightingShaderProperty, Next-Gen dispatch (version-agnostic by design), and the `BhkMultiSphereShape`/`BhkConvexListShape` collision regression pin.

#### FO4-D1-LOW-01: VF_UVS_2 / VF_LAND_DATA / VF_INSTANCE decoded as opaque skip
**Severity**: LOW | **Location**: `crates/nif/src/blocks/tri_shape/bs_tri_shape.rs:212-242` | **Status**: EXISTING (deliberate deferral #336)  
**Description**: Second UV set, FO4 landscape per-vertex blend data, and GPU-instance data are not field-decoded. Bytes are absorbed by the `consumed < vertex_size_bytes` skip so no stream corruption occurs, but data is discarded. Affects terrain splatting (sourced elsewhere for now).  
**Fix**: Decode VF_LAND_DATA when the FO4 terrain consumer materializes (ROADMAP M40); validate offset via BSVertexDesc nibble against real geometry first.

#### FO4-D1-LOW-02: `extract_skin_bs_geometry` leaves per-vertex weights empty (Starfield only)
**Severity**: LOW | **Location**: `crates/nif/src/import/mesh/skin.rs:225-267` | **Status**: EXISTING (#1203)  
**Description**: Starfield BSGeometry skin extractor builds bone list but returns empty weights/indices. Starfield-specific (BSVER 172) — FO4 uses BSTriShape via `extract_skin_bs_tri_shape` which is complete. Listed to scope it out.

#### FO4-D1-INFO-01: Next-Gen dispatch is version-agnostic by design
**Severity**: INFO | **Location**: `crates/nif/src/blocks/mod.rs:398-445`  
**Description**: Block dispatch is keyed on the block-type string, not BSVER. FO4 Next-Gen update did not introduce new BSVER values. Non-actionable.

### Dimension 2: BA2 Reader — GNRL + DX10 Variants

**Result: 7/7 PASS — No HIGH/MEDIUM findings.**

Version dispatch is an exhaustive `match` over `{1, 2, 3, 7, 8}` with explicit `Err(InvalidData)` wildcard — no silent v1-fallback, pinned by regression test `unknown_version_rejected`. GNRL 36-byte record layout correct, DX10 DDS header synthesis handles all BC formats + cubemaps + zero-mip. Two-axis compression model (archive-wide codec × per-chunk on/off) correct. v2/v3 header extensions distinct from v1. Stale `archive.rs:165-173` comment previously reported: already fixed.

#### FO4-D2-LOW-01: Missing bpp entries for R8G8_UNORM (49), B8G8R8X8_UNORM (88), A8_UNORM (65)
**Severity**: LOW | **Location**: `crates/bsa/src/ba2.rs:907-947` | **Status**: EXISTING  
**Description**: Three uncompressed DXGI formats fall through to the `DDSD_LINEARSIZE` fallback instead of `DDSD_PITCH`. In-engine Vulkan loader reads the DX10 extended header directly and ignores `dwPitchOrLinearSize`, so rendering is unaffected. Exported DDS files fail strict validation in texconv/DirectXTex.  
**Fix**: Add `49 => Some(2), 88 => Some(4), 65 => Some(1)` entries to the `bpp` match arm in `pitch_or_linear_size_for`.

#### FO4-D2-INFO-02/03: Stale internal line references
**Severity**: INFO | **Location**: `crates/bsa/src/ba2.rs:539, 618` | **Status**: EXISTING  
**Description**: Two inline comments reference incorrect line numbers after code was inserted around them (off by ~55–85 lines). No behavioral impact.

#### FO4-D2-INFO-04: `chunk_hdr_len` validation is debug_assert-only
**Severity**: INFO | **Location**: `crates/bsa/src/ba2.rs:544-548` | **Status**: EXISTING  
**Description**: Release builds are tolerant of `chunk_hdr_len != 24` (documented). No current FO4 archive uses a different value. Consider promoting to `log::warn!` for defense-in-depth.

#### FO4-D8-NEW-03: BGSM `fresnel_power` not merged despite `ImportedMesh` slot (NEW)
**Severity**: MEDIUM | **Location**: `byroredux/src/asset_provider.rs:1144-1337` (BGSM chain walk) | **Status**: NEW

#### FO4-D8-NEW-04: BGSM `grayscale_to_palette_scale` not merged despite `ImportedMesh` slot (NEW)
