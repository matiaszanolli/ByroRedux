# Fallout 4 Compatibility Audit — 2026-05-18

**Scope**: All 6 dimensions. Engine HEAD `7d6e2bf` (Session 39 closeout).
Live data: `/mnt/data/SteamLibrary/steamapps/common/Fallout 4/Data/`.
Methodology: per-dimension agents + manual verification; findings cross-checked against `/tmp/audit/issues.json` for dedup.

## Executive Summary

FO4 readiness is the strongest of the six target titles. NIF + BA2 parse at 100% on the full 34,995-NIF vanilla mesh corpus; the ESM-architecture record family (SCOL / MOVS / PKIN / TXST / MSWP) is fully parsed, registered, and routed through `EsmCellIndex` → cell loader → REFR overlay. The BGSM/BGEM parser exists as `crates/bgsm/` (lockstep with Bethesda's Material-Editor reference impl across 22 versions) and forwards 10 texture slots + scalar PBR + the three BGSM-only shader flags (`BGSM_PBR` / `BGSM_TRANSLUCENCY` / `BGSM_MODEL_SPACE_NORMALS`) into `ImportedMesh.material_flags`.

**One new HIGH finding** surfaced in dim 1: the `BSLightingShaderProperty.Backlight Power` presence gate at `shader.rs:882` is logically inverted vs nif.xml + nifly + openmw. The boolean opposite drives a silent 4-byte drift across the rest of the LSP block whenever `rim` is at (or away from) the FLT_MAX sentinel; `block_size` recovery resyncs at the next boundary, so parse-rate stays at 100% but a measurable subset of FO4 LSPs is reading the wrong field at the wrong offset. The matching regression-test fixture (`shader_tests.rs:506`) was hand-authored to match the inverted code, so tests pass for the wrong reason.

**Two LOW BA2 findings** are hardening / doc-rot: `extract_dx10` doesn't assert mip-chunk monotonicity (vanilla always-ordered, so no real-data hit), and `docs/engine/archives.md` lines 230-232 + 248-249 are stale against #593 / #594.

**Dim 4 surfaces five LOW/MEDIUM forward-risk** items on the ESM side — none vanilla regressions, all gating mod-content or DLC coverage. Dim 3, dim 5, and dim 6 contribute no new findings; every gap they touch is already on the issue tracker (#1147, #1148, #1076 follow-up, #973, #972, #1132, M41.x Havok).

**The forward blockers between "MedTek renders" and "FO4 interior content renders as shipped" are no longer parser-side.** They are renderer-side (#1147 PBR shader gate, #1076 GpuMaterial slot expansion for the four BGSM-only maps, #973 per-shape MSWP at spawn) plus the Havok `.hkx` skeleton parser (FO4-D6-008 / M41.x — blocks NPC visuals only).

## Findings — Grouped by Severity

### HIGH

#### FO4-D1-NEW-01 — `BSLightingShaderProperty.Backlight Power` gate is inverted
- **File**: `crates/nif/src/blocks/shader.rs:876-890` (specifically line 882)
- **Observation**: code reads Backlight when `rim < 3.0e38`; nif.xml 6609 / nifly `Shaders.cpp:477` / openmw `property.cpp:335` all read it when `rim == FLT_MAX` (sentinel). Boolean opposite.
- **Why bug**: 4-byte drift across `grayscale_to_palette_scale` / `fresnel_power` / the 7-float `WetnessParams` / shader-type trailing data / FO76 luminance + translucency whenever the wire format and the gate disagree. `block_size` resyncs at the next block boundary so parse-rate stays at 100% — but the misread fields silently default to wrong values. The matching test fixture at `shader_tests.rs:506` was hand-authored to match the inverted code, so existing regression coverage is green for the wrong reason. Matches the ~3.5% silent under-read pattern previously called out as FO4-D1-C1 / #403.
- **Fix**: Flip the gate to `rim >= 3.0e38` (or equivalently `rim.is_finite() && rim >= f32::MAX`). Update the test fixture comment + value to set `rimlight_power = f32::MAX`, keeping the backlight read live. Re-run the FO4 main + DLC NIF sweep to confirm downstream under-read counts on `wetness` / `fresnel` drop.

### MEDIUM

#### FO4-D4-004 — `parse_rate_fo4_esm` is `#[ignore]`-gated; not in default CI
- **File**: `crates/plugin/tests/parse_real_esm.rs:962-963`
- **Observation**: Test runs only when `BYROREDUX_FO4_DATA` is set or the fallback Steam path exists. The five-map regression net designed by #817 only fires on a manual ignored-test run.
- **Why bug**: A future refactor that silently empties one of `texture_sets` / `scols` / `packins` / `movables` / `material_swaps` will not surface in default CI. The parsers work today, but the regression net only catches the regression on opt-in.
- **Fix**: Add a synthetic-bytes Fallout4-shape fixture (a handful of SCOL/PKIN/TXST/MSWP records assembled in-memory) that runs unconditionally and asserts the five categories are non-empty. Keep the real-data run as the secondary gate.

### LOW

#### FO4-D2-NEW-01 — DX10 mip chunks concatenated without `start_mip` monotonicity check
- **File**: `crates/bsa/src/ba2.rs:643-657` (extract), `533-545` (record read)
- **Observation**: per-chunk `start_mip` / `end_mip` u16s are parsed but never consulted by the extract path. No assertion / sort enforces ascending mip order.
- **Why bug**: Vanilla FO4/FO76/Starfield archives are always mip-0-first; brute-force sweep + cubemap test pass. A hand-crafted or third-party-repacked archive with non-canonical chunk ordering would silently produce a DDS whose synthesized header doesn't match the payload.
- **Fix**: `debug_assert!` + release `log::warn!` after the chunk loop in `read_dx10_records`, matching the `num_mips == 0` and `chunk_hdr_len != 24` patterns. Don't auto-sort — that would mask the archive being malformed.

#### FO4-D2-NEW-02 — `docs/engine/archives.md` doc rot vs #593 / #594
- **File**: `docs/engine/archives.md:230-232`, `:248-249`
- **Observation**: Lines describe pre-#593 `arraySize = 1` and pre-#594 unconditional LinearSize fallback. Code is correct (cubemap arraySize=6 + per-format pitch branch); docs are not.
- **Fix**: Refresh both passages, cite `build_dds_header_cubemap_array_size_is_six` and `pitch_rgba8_unorm_matches_row_size_with_pitch_flag`.

#### FO4-D4-001 — SCOL `StaticObject` hardcodes `has_script: false`
- **File**: `crates/plugin/src/esm/cell/support.rs:341-345`; parser at `crates/plugin/src/esm/records/scol.rs:121-219`
- **Observation**: `parse_scol` doesn't inspect VMAD. Call-site comment acknowledges the gap as mod-only.
- **Fix**: Scan `sub.sub_type == b"VMAD"` once in `parse_scol`, plumb `has_script: bool` onto `ScolRecord`.

#### FO4-D4-002 — vanilla Fallout4.esm ships zero MOVS; decode path has no real-data coverage
- **File**: `crates/plugin/tests/parse_real_esm.rs:1065-1071`
- **Observation**: `assert_eq!(index.cells.movables.len(), 0)` is pinned; the first DLC/mod MOVS will trip the assert and any decoder regression today silently ships.
- **Fix**: Gate a `>= 1` assert on `BYROREDUX_FO4_DATA_DLC` or add a synthetic-bytes fixture exercising `parse_movs_group`.

#### FO4-D4-003 — PKIN expansion does not recurse into SCOL or LVLI children
- **File**: `byroredux/src/cell_loader/refr.rs:307-309`
- **Observation**: Docstring states children resolving to SCOL or LVLI stay single-level. LVLI tracked separately as #386.
- **Fix**: Second resolution pass against `scols` / `leveled_items`, bounded by `MAX_PKIN_DEPTH`.

#### FO4-D4-005 — SCOL expansion is single-level (no SCOL-of-SCOL)
- **File**: `byroredux/src/cell_loader/refr.rs:383-386`
- **Observation**: Vanilla FO4 has no SCOL-of-SCOL nesting. Mod-only forward risk.
- **Fix**: Recurse SCOL placement targets against `scols` with a depth cap.
