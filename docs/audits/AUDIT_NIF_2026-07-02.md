## Executive Summary

The parser is in strong shape. No CRITICAL or HIGH defects were found in any dimension. Every
resolved regression guard listed in the audit protocol (the v10.x truncation family, the
Starfield `BSLightingShaderProperty` over-read #1510 + trailing-tail #1606, root-NiNode /

**Per-game coverage** is unchanged from the `nif-parser.md` matrix — all seven games remain at
100% *recoverable*; the checked-in `block_coverage_baselines` unknown ceilings are 0 for all six
sized games, and Oblivion's only sizeless residue is 6 regression-gated pre-Gamebryo
editor-marker files (never rendered) plus 2 instance-level parse recoveries on dispatched types.
Dispatch↔resolve parity for `bhk*` shapes is complete in both directions (automated by
`every_dispatched_bhk_shape_has_resolve_arm`).

**Total findings: 6** — 0 CRITICAL, 0 HIGH, 1 MEDIUM, 5 LOW.

The single MEDIUM (NIF-D2-01) is a **regression of the closed #982**: a #1277 refactor migrated
two `tri_shape` version gates back to `NifVariant` helpers while leaving in place the very
comments that mandate raw-`bsver` reads — a silent undo of a shipped audit fix, affecting only
non-retail hybrid headers today but actively misleading the next maintainer. The remaining five
LOW findings are out-of-tier Morrowind-era bool/tail gaps (latent, no target-matrix content),
dead-helper foot-guns surviving the #938/#1511 prunings, a stale opt-in per-block baseline gate,
and two doc/constant-naming nits.

**One candidate finding was refuted during verification and is NOT reported** (see § Refuted):
a proposed "BSEffectShaderProperty Starfield 32-byte tail under-read" — the parser already reads
the full FO76+/Starfield trailing field set via `bsver >= FO76` gates (closed #746 / SF-D1-04).

## Findings

### NIF-D2-01: #1277 refactor reverted the #982 fix — variant-helper gates re-introduced at both `tri_shape` sites, with the contradicting comments still in place
- **Severity**: MEDIUM
- **Dimension**: Version Gating
- **Game Affected**: no retail title (all seven shipping games classify onto the aligned side);
  divergence hits unclassified/hybrid headers only — `(uv=11, bsver 35..=82)` → `Unknown` →
  helper returns false while nif.xml `#BS_GT_FO3#` (`bsver > 34`) authors the fields (8-byte
  under-read of shader/alpha refs, 4-byte under-read of material CRC); `(uv=12, bsver ≤ 34)` →
  `SkyrimLE` → helper true where the fields are absent (over-read); `(uv ≤ 10, bsver > 34)` →
  `Oblivion` → false (under-read).
- **Location**: `crates/nif/src/blocks/tri_shape/ni_tri_shape.rs:135` (`has_shader_alpha_refs()`),
  `crates/nif/src/blocks/tri_shape/ni_tri_shape.rs:341` (`has_material_crc()`)
- **Status**: Regression of #982 (audit finding NIF-D2-NEW-07, 2026-05-12)
- **Description**: Commit `01957517` (Fix #982) deliberately replaced
  `variant().has_shader_alpha_refs()` / `variant().has_material_crc()` at these two sites with
  raw `stream.bsver() > 34`, adding comments explaining the BSVER 35..=82 `Unknown`-corner
  false-negative. Commit `2bd447d5` (refactor, #1277 Task 5) migrated both sites **back** to the
  variant helpers — listing them as "existing helpers now called from production" — while
  leaving the raw-bsver-mandating comments untouched. The commit applied the Unknown-semantics
  analysis to five *other* sites ("KEPT on raw bsver") but missed that these two carried the
  identical constraint from a shipped audit fix.
- **Evidence**: Current code at `ni_tri_shape.rs:128-135` — comment reads "Query the file's
  actual bsver rather than the routed game variant — `variant().has_shader_alpha_refs()` returns
  false for BSVER in 35..=82 … the field IS authored there … See NIF-D2-NEW-07", immediately
  followed by `if stream.variant().has_shader_alpha_refs() {`. Same contradiction at `:337-341`.
  `git show 01957517` introduces the raw-bsver reads + comments; `git show 2bd447d5` flips both
  back (`- if stream.bsver() > 34` / `+ if stream.variant().has_shader_alpha_refs()`). nif.xml
  gates purely on BSVER (`#BS_GT_FO3#`), never on user_version — raw bsver is the spec-correct read.
- **Impact**: Functional divergence confined to non-retail hybrid headers (the original #982
  rated it "forward-compat hygiene"). Rated MEDIUM because (a) it silently undoes a shipped audit
  fix, (b) the surviving comments actively mislead the next maintainer, and (c) the 2026-06-23
  audit certified this dimension clean over exactly this state.
- **Related**: #982 / NIF-D2-NEW-07, #1277 Task 5 (`2bd447d5`), #160 / #1331 (the raw-bsver
  doctrine at base.rs/node.rs), NIF-D2-02 (same-commit sibling), NIF-D2-03 (fixing this orphans helpers)
- **Suggested Fix**: Restore `stream.bsver() > crate::version::bsver::FO3_FNV` at both sites
  (matching the comments), and add a regression test with a `(uv=11, bsver=50)` synthetic header
  so the next "migrate to helpers" sweep trips a red test instead of silently reverting again.

### NIF-D2-02: Same-commit helper substitutions at the MOPP Build-Type and BsMultiBoundNode culling-mode gates diverge from their nif.xml BSVER gates on hybrid headers
- **Severity**: LOW
- **Dimension**: Version Gating
- **Game Affected**: no retail title; hybrid tuples only — `(uv=11, bsver ≥ 35)` → `Unknown` →
  1-byte under-read of MOPP `Build Type` / 4-byte under-read of `Culling Mode`; `(uv=12, bsver ≤ 34 / < 83)` → over-read
- **Location**: `crates/nif/src/blocks/collision/shape_compound.rs:127`
  (`has_shader_alpha_refs()` gating `bhkMoppBvTreeShape.Build Type`),
  `crates/nif/src/blocks/node.rs:257` (`has_culling_mode()` gating `BSMultiBoundNode.Culling Mode`)
- **Status**: NEW (introduced by `2bd447d5`, pre-baseline; never covered by #982, which fixed only the two tri_shape sites)
- **Description**: nif.xml gates `Build Type` on `#BS_GT_FO3#` (`bsver > 34`, nif.xml:3149) and
  `Culling Mode` on `#BS_GTE_SKY#` (`bsver >= 83`, nif.xml:6914) — pure BSVER predicates. Both
  sites were `stream.bsver()` compares until `2bd447d5` substituted variant helpers, inheriting
  the same Unknown/hybrid-corner divergence as NIF-D2-01. Additionally the MOPP site answers a
  *collision-era* question with a *NiGeometry shader-refs* helper — the exact "can't tell the
  blessed family from the trap family" foot-gun #1511 documented.
- **Evidence**: `git log -L127,129:…/shape_compound.rs` shows `- if stream.bsver() > 34` →
  `+ if stream.variant().has_shader_alpha_refs()` in `2bd447d5`; the in-code comment still says
  "Build Type: only for BSVER > 34".
- **Impact**: Zero on shipping content; doctrine/readability divergence and latent drift on
  unclassified exports (the class #160/#1331/#982 all chose raw bsver for).
- **Related**: NIF-D2-01 (same root commit), #1511 (trap-family doctrine)
- **Suggested Fix**: `stream.bsver() > crate::version::bsver::FO3_FNV` at shape_compound.rs:127;
  `stream.bsver() >= crate::version::bsver::SKYRIM_LE` at node.rs:257. Fixing D2-01/D2-02 fully
  orphans `has_shader_alpha_refs`, `has_material_crc`, `has_culling_mode` — fold into the D2-03 decision.

### NIF-D2-04: `FLAGS_U32_THRESHOLD` doc cites a nonexistent nif.xml token `#BS_GTE_26#` (GTE vs the actual GT gate)
- **Severity**: LOW (doc rot on a boundary constant)
- **Dimension**: Version Gating
- **Game Affected**: none today (code is correct); a future "align to the doc" edit would
  mis-parse any `bsver == 26` content boundary
- **Location**: `crates/nif/src/version.rs:322-325`
- **Status**: NEW
- **Description**: The constant's doc reads "Corresponds to nif.xml's `#BS_GTE_26#` predicate".
  nif.xml defines no such token (verified against the verexpr table + zero grep hits for
  `#GTE# 26`); the actual gate is the inline `vercond="#BSVER# #GT# 26"` on `NiAVObject.Flags`
  (nif.xml:3442). Both call sites use the correct operators (`> 26` flags width at base.rs:82;
  `>= 26` as the negation of the different `#BSVER# #LT# 26` ambient/diffuse gate at
  properties.rs:44), but the doc's "GTE" contradicts the "GT" semantics on the exact boundary
  value the constant exists to pin.
- **Evidence**: `grep -c "GTE_26\|#GTE# 26" nif.xml` → 0; nif.xml:3442.
- **Impact**: Misleading citation on an off-by-one-critical constant.
- **Related**: #1042 (literal-naming sweep)
- **Suggested Fix**: Reword to "nif.xml gates `NiAVObject.Flags` on the inline `#BSVER# #GT# 26`
  (no named token); u16 at `bsver <= 26`."

### NIF-D1-01: Pre-4.1 NIF `bool` fields read as 1 byte where the wire format is 32-bit — full-file cascade on real v ≤ 4.0.0.2 (Morrowind-era) NIFs
- **Severity**: LOW *(graded per the project's #982 "out-of-tier / latent" precedent for
  Morrowind V4; inside a support tier this would be a sizeless-format cascade with no recovery anchor)*
- **Dimension**: Stream Position
- **Game Affected**: `NifVariant::Morrowind` and all NetImmerse content at NIF v ≤ 4.0.0.2
  (bsver = 0). No shipped content of the target matrix (Oblivion→Starfield) sits in this band —
  Oblivion's oldest live band is v10.0.1.0, and its lone pre-Gamebryo file (`marker_radius.nif`,
  #698) is corrupt-by-design and truncates regardless.
- **Location**: `crates/nif/src/blocks/base.rs:279` (has_bounding_volume via `read_u8`),
  `crates/nif/src/blocks/tri_shape/ni_tri_shape.rs:326,349,379,421-422` (has_vertices /
  has_normals / has_vertex_colors / has_uv via `read_byte_bool`), `crates/nif/src/blocks/texture.rs:913`
  (enable_plane via `read_u8`); family likely extends to other `read_byte_bool` sites reachable pre-4.1.
- **Status**: NEW
- **Description**: nif.xml `<basic name="bool">`: "A boolean; 32-bit up to and including 4.0.0.2,
  8-bit from 4.1.0.1 on." `NifStream::read_bool` (stream.rs:203-211) implements exactly this
  switch — but the listed sites bypass it with fixed 1-byte reads. On a real v4.0.0.2 file each
  such bool under-reads 3 bytes; files in this band carry no `block_sizes` table, so the drift
  cascades unrecoverably.
- **Evidence**: OpenMW (`/mnt/data/src/reference/openmw/components/nif/nifstream.cpp:170-177`)
  reads `bool` as `int32_t` when `version < 4.1.0.0`; its `NiAVObject::read` / `NiGeometryData::read`
  route these fields through the version-aware bool. The `read_byte_bool` doc comment
  (stream.rs:213-215) contradicts nif.xml for the pre-4.1 band. The synthetic Morrowind fixture
  (`tests/synthetic_fixtures.rs:434`) writes `has_bounding_volume` as 1 byte, so it is
  self-consistent with the bug and stays green while a real file drifts.
- **Impact**: Latent today (band is out of the compat matrix), but every entry point for the band
  exists and is tested, so the parser *claims* the band. #210's closure claim ("Full Morrowind
  v4.0.0.2 support … verified end-to-end") is false for real files.
- **Related**: #210 (closed), #325 (closed), #982 (out-of-tier precedent), stream.rs `read_bool` doctrine
- **Suggested Fix**: Replace the fixed-width reads with `stream.read_bool()` at the listed sites;
  fix the synthetic fixture to write 4-byte bools at v4.0.0.2; correct the `read_byte_bool` doc
  comment; sweep remaining pre-4.1-reachable `read_byte_bool` sites. Validate against one real
  Morrowind NIF before re-claiming the band. (Ship with the related `NiDynamicEffectData` v≤4.0.0.2
  affected-nodes gap, which is masked by this one and would otherwise be unreachable-correct.)

## Refuted / disproven candidates (not reported as findings)

- **BSEffectShaderProperty Starfield "32-byte tail under-read"** — A Dim-5 candidate proposed
  that `BSEffectShaderProperty::parse` (no `block_size` parameter, unlike
  `BSLightingShaderProperty::parse_with_size`) under-reads Starfield blocks by 32 bytes and
  silently drops the tail, citing a "665-block corpus drift" figure. **Refuted**: the structural
  asymmetry is real (`blocks/mod.rs:579` calls bare `parse(stream)`), but the parser already reads
  the full FO76+/Starfield trailing field set — `refraction_power`, the four sized-string texture
  fields (`reflectance_texture`/`lighting_texture`/`emit_gradient_texture`/env), `emittance_color`,
  and the `luminance` block — all behind `bsver >= FO76` gates (`shader.rs:1698-1762`). This is
  exactly the fix from **closed #746 / SF-D1-04** ("tail-fields gated on `bsver == 155` skip on
  BSVER 172"), whose in-code comments describe the identical symptom the candidate re-derived.
  Unlike `BSLightingShaderProperty` (whose `starfield_tail` captures an *undocumented*, opaque
  post-nif.xml tail), `BSEffectShaderProperty`'s tail is *documented* and fully typed — so no
  opaque-capture path is needed. The "665 / +32" magnitude was attributed to an orchestrator
  corpus pass that was not actually run this cycle and could not be reproduced. No finding.
- **`bsver == FO76` strict-equality gates** (bs_tri_shape.rs:263, particle.rs:1027): nif.xml's
  `#BS_F76#` token is itself `#BSVER# #EQ# 155`; both sites document exactly that. Not a bug.
- **properties.rs:44 `>=` vs base.rs:82 `>` on the `26` constant**: two different nif.xml gates
  (`#LT# 26` negated vs `#GT# 26`); both byte-correct.
- **morph.rs `bsver > 9` / `< 10` raw literals**: no semantically matching named constant exists
  (`RIGID_BODY_EXTRA_FLOATS = 9` gates a different field); both documented + corpus-verified (#1509).
- **`nif-parser.md` block-coverage counts drifted** (254 arms/310 literals vs live 249/313): the
  literal growth is honest and self-disclaimed; arm count is counting-method noise. Folded into
  the doc-rot bucket, not filed separately (the doc instructs downstream audits to recount).
