## Findings

### NIF-D1-01: Three FO76 particle blocks silently drop their `#BS_F76#` fields (block_size absorbs the drift)
- **Severity**: MEDIUM
- **Dimension**: Stream Position
- **Location**: `crates/nif/src/blocks/particle.rs` (`parse_particles_data` `NiPSysData`, `parse_rotation_modifier` `NiPSysRotationModifier`, `BSPSysSimpleColorModifier::parse`)
- **Status**: NEW
- **Game Affected**: Fallout 76 (`bsver 155` / `#BS_F76#`). Not FO4 (bsver 130, corpus-verified 0 drift), not Starfield (bsver ≥ 172 — separate tail), not Oblivion/FO3/FNV (fields absent).
- **Description**: On FO76, nif.xml inserts extra `#BS_F76#`-gated fields into three particle blocks that these parsers do not model. The drift is constant per type and byte-exactly matches the unmodeled fields:
  - `NiPSysData` — `+12 B` (`Unknown Vector`, `Vector3`) × 7 478.
  - `NiPSysRotationModifier` — `+17 B` (`Unknown Vector` `Vector4` + `Unknown Byte`) × 4 571.
  - `BSPSysSimpleColorModifier` — `+52 B` (trailing `Unknown Shorts[26]`) × 7 306. This one is *already documented* in-code (the leading `colors` payload is captured; only the opaque tail is skipped).
- **Evidence**: `nif_stats --drift-histogram` on `Fallout76/Data/SeventySix - Meshes.ba2` reports 19 355 drift events across exactly these three types (`+12×7478`, `+52×7306`, `+17×4571`); each drift cross-checked against nif.xml `#BS_F76#` field additions to the byte.
- **Impact**: **Functionally negligible today.** All three blocks are discarded opaque stubs / payload-captured, so no consumer reads the dropped fields; FO76 ships a `block_sizes` table, so the reconciliation loop seeks to the exact block end and the *next* block is never misaligned (no cascade). Real cost is (a) ~19 k drift-histogram events/archive raising the noise floor the opt-in `NIF_STATS_MAX_DRIFT_EVENTS` gate must clear to catch a net-new FO76 drift regression, and (b) a latent trap: if `NiPSysData` / `NiPSysRotationModifier` are ever promoted to typed consumption (as the NIFAL particle slice did for emitters), the post-`#BS_F76#` fields would be read from misaligned positions and yield garbage on FO76.
- **Related**: `#1606` / `#1881` / `#1882` (the opaque-tail-capture pattern this could adopt); the in-code precedent already annotating `BSPSysSimpleColorModifier`.
- **Suggested Fix**: Read the `#BS_F76#` fields under a `bsver >= FO76` gate (12 B / 17 B, fixed) so `consumed == block_size` and the drift histogram is clean, matching the `read_starfield_tail` doctrine — this also removes the future-typed-consumption trap. Lowest-effort alternative: extend the `BSPSysSimpleColorModifier` explanatory comment to the other two parsers so the accepted drift is documented.

### NIF-D2-05: `ShaderFlags` typed view + `has_shader_property_fo3_fields` are transitively dead in production
- **Severity**: LOW
- **Dimension**: Version Gating
- **Location**: `crates/nif/src/shader_flags.rs` (`ShaderFlags` + `classify` / `is_decal` / `is_two_sided`), `crates/nif/src/version.rs` (`has_shader_property_fo3_fields`), `crates/nif/src/stream.rs` (`variant()` accessor)
- **Status**: NEW
- **Game Affected**: All (`NifVariant` feature-flag surface).
- **Description**: `version.rs` justifies keeping `has_shader_property_fo3_fields` — the sole surviving `NifVariant` feature predicate after the `#938`/`#1511`/`#1840` prunes — on the grounds that it "still has a live consumer (`shader_flags.rs`)". That consumer is `ShaderFlags::classify`, but `ShaderFlags` itself has **zero production callers**: every `classify` / `is_decal` / `is_two_sided` invocation is inside `#[cfg(test)]`. `NifStream::variant()` has no production call site either. The production material importer reads the raw `flags1`/`flags2`/`sf1_crcs`/`sf2_crcs` fields directly. Because all three are `pub`, `cargo build` emits no dead-code warning — which is how the chain survived the `#1840` sweep (the sweep counted the test-only `classify` caller as live).
- **Evidence**: `grep -rn "ShaderFlags::classify" crates/ byroredux/ tools/` → only `shader_flags.rs` tests; `grep -rn "\.variant()" crates/nif/src` → no production call site; `grep -rn "has_shader_property_fo3_fields" crates/` → only `shader_flags.rs` `classify` + the `version.rs` comment.
- **Impact**: The precise foot-gun the `#1511`/`#1840` doctrine warns about — a contributor who adopts `ShaderFlags` in a new parser, trusting the "typed view" docstring, reintroduces a `variant()`-based gate that answers `false` on the `Unknown` hybrid-header corner (the same class of bug `#1838`/`#1839` just reverted).
- **Related**: `#1840`, `#1841`, `#1511`; `#1838`/`#1839` (the hybrid-corner reversion this could recreate).
- **Suggested Fix**: Either wire `ShaderFlags::classify` into the `import/material` decal/two-sided path so it becomes a genuine consumer, or delete `ShaderFlags` + `has_shader_property_fo3_fields` + the `variant()` accessor. Either way, remove the incorrect "live consumer" comment. Scoped as a follow-through on `#1840`, not a regression of it.

### NIF-D2-06: `Max Filepath` BSStreamHeader gate uses a raw magic literal `103` instead of a named `bsver::*` constant
- **Severity**: LOW
- **Dimension**: Version Gating
- **Location**: `crates/nif/src/header.rs` (`if user_version_2 >= 103 { … max_filepath … }`)
- **Status**: NEW
- **Game Affected**: FO4 / FO76 / Starfield (bsver ≥ 103; header-parse path).
- **Description**: Every other `bsver` comparison in the crate routes through a named constant in `version::bsver`, and the two sibling gates immediately above this line already do (`user_version_2 > bsver::FALLOUT4`, `user_version_2 <= bsver::FALLOUT4`). The `Max Filepath` field's nif.xml threshold `#BS_GTE_103#` is hardcoded here as a bare `103`. The value is correct; only the naming violates the crate's own "named constants, not bare decimal literals" convention.
- **Impact**: None functional — maintenance/consistency only. A bare literal survives cross-file refactors less well than a symbol.
- **Related**: `#1842` (a separate doc-token issue on the same header path).
- **Suggested Fix**: Add `pub const MAX_FILEPATH: u32 = 103;` to `version::bsver` and reference it here, matching the two adjacent gates.

### NIF-D5-01: `parse_fo4` backlight-power presence threshold is `3.0e38`, not nif.xml `FLT_MAX`
- **Severity**: LOW
- **Dimension**: Collision/Shader Parsing
- **Location**: `crates/nif/src/blocks/shader.rs` (`BSLightingShaderProperty::parse_fo4`)
- **Status**: NEW
- **Game Affected**: Fallout 4 (bsver 130–139, `BSLightingShaderProperty` full-body path).
- **Description**: `parse_fo4` reads the optional `Backlight Power` field with `if rim >= 3.0e38 && rim.is_finite()`. nif.xml gates the field on `Rimlight Power >= FLT_MAX (3.4028235e38) && Rimlight Power < FLT_INF`. The code's threshold `3.0e38` is looser than `FLT_MAX` by ~0.4e38, so a rimlight value in the half-open interval `[3.0e38, 3.4028235e38)` would make the parser read a 4-byte `backlight_power` that nif.xml says is absent — a 4-byte over-read on that block.
- **Impact**: Nil in practice — `Rimlight Power` ships either as its `#FLT_MAX#` sentinel (comfortably `>= 3.0e38`) or as a normal-range authored value (`<< 3.0e38`); no vanilla or plausible modded content authors a value in the affected gap, and FO4's `block_sizes` table would absorb the drift even if it did. `.is_finite()` correctly mirrors the `< FLT_INF` clause (and additionally rejects NaN).
- **Related**: `#1175` (the doc comment on this field already cites nif.xml 6609 / openmw / nifly).
- **Suggested Fix**: Change the constant to `rim >= f32::MAX` (exactly nif.xml `FLT_MAX`). One-line change; the `.is_finite()` guard already covers the upper clause.

### NIF-D6-01: `BhkMultiSphereShape::parse` fills a `Vec<[f32;4]>` with a per-element push loop instead of the bulk `read_pod_vec` path
- **Severity**: LOW
- **Dimension**: Allocation Hygiene
- **Location**: `crates/nif/src/blocks/collision/shape_primitive.rs` (`BhkMultiSphereShape::parse`)
- **Status**: NEW
- **Game Affected**: All Havok-bearing variants (`bhkMultiSphereShape` is a classic pre-FO4 Havok primitive).
- **Description**: The sphere array is a contiguous run of `[f32;4]` `(cx, cy, cz, r)` with no per-element transform — byte-identical to what `read_ni_color4_array` / `read_pod_vec::<[f32;4]>` produces. The current code does `allocate_vec::<[f32;4]>(num_spheres)` then a `for` loop of four `read_f32_le()` + `push`. This is the "direct allocate-then-loop-and-fill" shape the `read_pod_vec<T>` discipline (`#833`/`#873`) was meant to collapse for POD arrays.
- **Impact**: Negligible runtime cost — **not a `#833` regression** (the site is already single-allocation; it does not reintroduce the old intermediate-`Vec<u8>` double allocation), and `num_spheres` is tiny (≤~8 in shipped content). Filed LOW purely for idiom consistency with the rest of the crate's POD-array sites.
- **Related**: `#833`, `#873`.
- **Suggested Fix**: `let spheres = stream.read_ni_color4_array(num_spheres as usize)?;` (one `read_exact` instead of N per-element reads). Byte-for-byte equivalent on the LE host the crate already requires.
