# NIF Parser Audit — 2026-05-10

Orchestrator: `/audit-nif`. Six dimensions (Block Parsing, Version
Handling, Stream Position, Import Pipeline, Coverage Gaps, Stream
Allocation Hygiene). Methodology and per-finding format from
`.claude/commands/_audit-common.md`.

## Executive Summary

**14 NEW findings** (2 HIGH, 6 MEDIUM, 6 LOW) + 4 regression-guard
VERIFIED entries. The headline finding is a **convention question**
that crosses three dimensions: the post-#765/#769 sweep interpreted
nif.xml's `until=X` attribute as **exclusive** (`< X`), but niftools
itself (token table) and nifly (the C++ port) consistently treat it as
**inclusive** (`<= X`). The bug is silent for shipping Bethesda content
because every affected gate sits at versions older than any shipping
title. It bites on legacy Gamebryo content (Civ4, IndustryGiant,
Morrowind-era mods) and is the most plausible root cause of the
long-tracked "1-byte shortfall" on `NiSourceTexture` / NiTexturingProperty.

The 2026-05-04 performance batch's regression guards (`#[must_use]` on
`allocate_vec`, `read_pod_vec` + LE compile gate, `get_mut/insert`
counter split, #408 blanket sweep) are all intact. Open follow-up:
**no allocation-counter regression test exists** today (dhat-infra gap).

Coverage: 5 unmapped block types span FNV through FO76, mostly silent
skips on Oblivion+ where `block_size` rescues the parse loop. The HIGH-
severity gap is `NiBSplineCompFloatInterpolator` /
`NiBSplineCompPoint3Interpolator` — paired companions to the dispatched
transform variant, reachable on FNV/FO3 (not just Skyrim+).

Coverage of import-pipeline correctness yielded one MEDIUM: the parser
ships dedicated `TileShader / SkyShader / WaterShader / TallGrass*`
property structs (post-#455/#474/#550), but the import walker never
consumes them — FO3/FNV sky/water/grass meshes silently lose their
texture path.

## Findings by Severity

### HIGH (2)

#### NIF-D3-NEW-01: `until=X` semantic mismatch with niftools / nifly

- **Dimension**: Stream Position
- **Game Affected**: pre-Bethesda content (Civ4, IndustryGiant 2,
  Morrowind-era mods, v10.0.1.2 BSStreamHeader files); shipping
  Bethesda titles unaffected because every gate is older than 20.0.0.5
- **Status**: NEW
- **Locations**: `crates/nif/src/blocks/texture.rs:53-58`,
  `crates/nif/src/blocks/properties.rs:451`, and ~10 other call sites
  carrying a `// see #765 sweep` comment

niftools' own token table (`/mnt/data/src/reference/nifxml/nif.xml:6-9`)
defines `#NI_BS_LTE_FO3#` with the operator `#LTE#` and the description
"All NI + BS *until* Fallout 3" — i.e. `until` is colloquial for
**inclusive** `<=`. nifly mirrors this:

```cpp
// /mnt/data/src/reference/nifly/src/Shaders.cpp:25
if (fileVersion <= NiFileVersion::V10_0_1_2)
// /mnt/data/src/reference/nifly/src/Objects.cpp:217
if (fileVersion <= NiFileVersion::V10_0_1_3)
```

ByroRedux's #765 sweep chose `<` (exclusive), so on `v=10.0.1.3` exactly,
`NiSourceTexture::Use Internal` is skipped (1 byte under-read). On
`v=10.4.0.1`, `TexDesc::PS2 L/K` is skipped (4 bytes under-read).

**Bethesda content unaffected**: Oblivion is 20.0.0.5, FO3+ is 20.2.0.7;
both are well past every `until=10.X.X.X` gate, so the predicate
collapses to `false` either way. The bug shows up only on legacy
content the engine doesn't currently ship.

**Suggested Fix**: Audit every site with `// see #765 sweep` or
`// exclusive` comment. Flip `< NifVersion(0xN)` → `< NifVersion(N+1)`
(or add a `<=` overload). Document the inclusive semantic at the top
of `version.rs`. Pair with a regression test using a sample v10.0.1.3
NiSourceTexture — Civ4 mod content provides plenty.

#### NIF-D5-NEW-01: `NiBSplineCompFloatInterpolator` + `NiBSplineCompPoint3Interpolator` not dispatched

- **Dimension**: Coverage
- **Game Affected**: FNV, FO3, Skyrim LE/SE, FO4 (per "B-splines aren't
  Skyrim+ only" feedback memory; reachable on FO3/FNV)
- **Status**: NEW
- **Location**: `crates/nif/src/blocks/mod.rs` (only the transform
  variant at line 683 is wired)

Companion blocks to the dispatched `NiBSplineCompTransformInterpolator`.
They cannot alias the transform parser — they inherit
`NiBSplineFloatInterpolator` / `NiBSplinePoint3Interpolator` and add
2× `f32` (offset + half-range) on top of a different parent layout.

**Impact**: Silent `NiUnknown` skip via `block_size` (no drift on
20.0.1.0+). Animation importer loses these curves; affected float
(alpha/scale) and Point3 (color/translation) channels collapse to
constant or rest pose.

**Suggested Fix**: Two new parsers next to the transform variant
(~40 LOC each). Wire dispatch entries. Animation channel emitter in
`anim.rs` already supports the transform pattern; copy that path.

### MEDIUM (6)

#### NIF-D2-NEW-01: `NifVariant::Fallout3.bsver()` returns 21, not 34

- **File**: `crates/nif/src/version.rs:125`
- **Status**: NEW. Not observable today (nothing outside unit tests
  queries this), but contradicts the audit-spec hard-pin and traps
  future contributors.

`Fallout3` variant only ever applies to FO3 mod/dev builds (retail FO3
NIFs at bsver=34 detect as `FalloutNV` since the two are binary-
identical at that BSVER). Whatever single value `Fallout3.bsver()`
returns mis-represents most of the FO3 dev-bsver fan-out (14, 16, 21,
24-28, 30-33, 34).

**Suggested Fix**: Either return 34 to match the hard-pin (and document
that `Fallout3` variant = "dev/mod FO3"), or rename the helper to
`canonical_dev_bsver` and mark non-authoritative. The latter matches
the existing recommendation that callers use `stream.bsver()` directly.

#### NIF-D2-NEW-04: Feature-flag predicates duplicate parse-time bsver checks but are documented as authoritative

- **File**: `crates/nif/src/version.rs:140-205`
- **Status**: NEW

`compact_material()` / `has_emissive_mult()` / `has_shader_emissive_color()`
are public predicates that *no production parse site uses* (every site
calls `stream.bsver() >= N` directly). They coexist with predicates
that ARE used externally (`has_shader_alpha_refs`, `has_material_crc`)
under the same naming scheme — no way for a future contributor to know
which are blessed and which are advisory. The predicates' results
disagree with the parse path by one bsver step at boundary versions.

**Suggested Fix**: Either delete the unused predicates, or split
`AuthoritativeFeatures` (variant-only) from `BsverFeatures` (called on
`stream`, queries header bsver). Current "pick whichever flavor you
like" surface invites the exact regression class as #323.

#### NIF-D3-NEW-03: Drift detector + `block_size` adjustment masks bugs; no telemetry quantifies how much

- **Dimension**: Stream Position
- **Status**: NEW (architectural / observability)
- **Location**: `crates/nif/src/lib.rs` parse_nif loop

When a block parser under-/over-reads relative to declared `block_size`,
the outer loop pads/skips. Parse rates LOOK clean while quietly papering
over byte-level bugs. There's no log distinction between "consumed
exactly `block_size`" and "padded 1 byte." The known `NiTexturingProperty`
1-byte shortfall has been "tracked separately" precisely because there's
no automated way to root-cause it — NIF-D3-NEW-01 is one candidate root
cause but only telemetry will confirm.

**Suggested Fix**: Per-block-type drift histogram. In `parse_nif`, log
`block_type=X, declared=N, consumed=M, drift=N-M` at debug level. Aggregate
into a `--drift-histogram` CLI flag on the `nif_stats` example. Reuse
the #832 `bump_counter` pattern to avoid alloc churn.

#### NIF-D4-NEW-01: Import walker drops `file_name` from TileShader / SkyShader / TallGrass / BSSky / BSWater

- **Dimension**: Import Pipeline
- **Game Affected**: FO3, FNV (HUD tiles, sky, grass, water), Oblivion
  (some sky), Skyrim (`BSSkyShader` / `BSWaterShader`)
- **Status**: NEW
- **Location**: `crates/nif/src/import/material/walker.rs:438-811`

Parser-side fixes shipped in #455 (TileShader), #474 (Water), #550 (Sky)
added dedicated structs with `file_name` fields, but the import walker
was never updated to consume them. `grep "TileShader\|SkyShader\|WaterShader\|TallGrass"`
in the walker returns zero hits.

**Impact**: FO3/FNV HUD tiles, sky domes, water surfaces, tall grass
silently fall through with `info.texture_path = None` — render as
magenta-checker placeholder. Identical failure mode to the
"chrome posterized walls" feedback memo, on sky/UI surfaces. Sky and
water visibly affected.

**Suggested Fix**: Add `scene.get_as::<TileShaderProperty>(idx)` and
sibling arms inside the per-prop loop in `extract_material_info_from_refs`.
For each, route `file_name` into `info.texture_path`. Pattern matches
the existing `BSShaderNoLightingProperty` arm at `walker.rs:695`. Test
parity by extending `mesh_material_path_capture_tests.rs` with one
fixture per shader type.

#### NIF-D5-NEW-02: `BSTreadTransfInterpolator` not dispatched

- **Dimension**: Coverage
- **Game Affected**: FO3, FNV, Skyrim LE/SE, FO4 (per nif.xml `#FO3_AND_LATER#`)
- **Status**: NEW (re-confirmed open from 2026-04-18 audit's NIF-COV-11)

Used on tread/wheel rolling animation. Liberty Prime (FO3), Ranger
Vertibird (FNV), Skyrim mammoth/horse, FO4 Power Armor. Limited count
but visually critical when present.

**Impact**: Silent skip via block_size (FO3+). Affected vehicle/creature
tread animation imports as static — wheels/treads don't roll.

**Suggested Fix**: Add `BsTreadTransfInterpolator` parser in
`interpolator.rs` (~20 LOC: `num` + array<BSTreadTransform> + `Ref<NiFloatData>`).
Animation importer needs a new "tread-uv" channel — defer if pipeline
isn't ready, parser-only is still worth landing.

#### NIF-D5-NEW-03: `BSDistantObjectInstancedNode` (FO76) + `BSDistantObjectLargeRefExtraData` (SSE) not dispatched

- **Dimension**: Coverage
- **Game Affected**: FO76 (instanced node), Skyrim SE (large-ref extra)
- **Status**: NEW

`BSDistantObjectInstancedNode` is the canonical container for FO76
foliage/rock cluster instancing in worldspace LOD. `BSDistantObjectLargeRefExtraData`
rides on every SSE large-ref worldspace object that participates in
precombined-LOD scheduling.

**Impact**: Silent skip via block_size. FO76: distant LOD instances
drop, only the multi-bound shell renders → "ghost foliage." SSE:
large-ref flag lost, large refs miss precombined-LOD scheduling
(renderer treats them as plain refs, re-uploads geometry per cell).

**Suggested Fix**: Two small parsers in `node.rs` / `extra_data.rs`. SSE
LargeRef is one bool — pair with a `LargeRefMarker` ECS component. FO76
InstancedNode parses `BSDistantObjectInstance` array (per nif.xml
compound def) and emits per-instance transforms.

### LOW (6)

#### NIF-D1-NEW-01: `NiStencilProperty` Flags gate inconsistent with siblings

- **File**: `crates/nif/src/blocks/properties.rs:1521`
- **Status**: NEW (subordinate to NIF-D3-NEW-01; bundled into that fix)

NiStencilProperty uses `version < V10_0_1_3` (inclusive boundary at
v10.0.1.2). Siblings (`NiMaterialProperty:41`, `NiFogProperty:1307`)
use `version < V10_0_1_2` (exclusive). Per niftools/nifly, NiStencilProperty
has the correct semantic — siblings need to be flipped, not vice versa.

**Suggested Fix**: Bundle into NIF-D3-NEW-01 sweep.

#### NIF-D1-NEW-02: `NiMaterialColorController` / `NiLightColorController` `target_color` read unconditionally

- **Location**: `crates/nif/src/blocks/controller/legacy.rs`
- **Game Affected**: pre-Gamebryo NetImmerse content at v < 10.1.0.0 (no shipping Bethesda title)
- **Status**: NEW

`target_color` (u16) is `since="10.1.0.0"` per nif.xml; parsers read it
unconditionally. For v < 10.1.0.0 the field is absent — 2-byte mis-read.
Bethesda impact zero; bundle with NIF-D3-NEW-01 sweep.

**Suggested Fix**: Gate `target_color` with `version >= NifVersion(0x0A010000)`.

#### NIF-D2-NEW-02: `bsver_values` test misses Starfield + Unknown assertions

- **File**: `crates/nif/src/version.rs:481-488`
- **Status**: NEW

Test asserts FO3-FO76 but never checks `Starfield.bsver() == 172`
(audit spec) nor `Unknown.bsver() == 0`. Typo regression would slip CI.

**Suggested Fix**: Two extra `assert_eq!` lines.

#### NIF-D2-NEW-03: v20.0.0.4 always routes to Oblivion, ignoring early FO3

- **File**: `crates/nif/src/version.rs:91-94`
- **Status**: NEW

nif.xml `<version id="V20_0_0_4__11" user="11" bsver="11">Oblivion, Fallout 3</version>`
documents v20.0.0.4 as covering FO3 dev builds. Current code
unconditionally returns `Oblivion`. No retail FO3 NIF ships at v20.0.0.4
so impact is pre-release content only.

**Suggested Fix**: When `version == V20_0_0_4 && user_version == 11`,
prefer Fallout3.

#### NIF-D3-NEW-04: `NiSourceTexture::Use Internal` cond gate has no regression test

- **File**: `crates/nif/src/blocks/texture.rs:57`
- **Status**: NEW

Cond on `Use Internal` is `Use External == 0`. Current code is correct
but the cond order is unprotected — a future flip would silently drift
1 byte on every external texture (most of them). Defensive note.

**Suggested Fix**: Regression test that builds a v10.0.1.2 `NiSourceTexture`
with `use_external == true` and verifies parse consumes exactly the
expected bytes.

#### NIF-D5-NEW-04 / NIF-D5-NEW-05: FO76 long-tail + legacy NiVectorExtraData

- **NIF-D5-NEW-04 (LOW)**: `BSCollisionQueryProxyExtraData` + `NiPSysRotDampeningCtlr`
  re-confirms still-open issue **#728**. Per AUDIT_NIF_2026-04-26 corpus:
  2× collision-proxy + 5× rot-damping failures across 22 sampled FO76
  archives. Silent skip via block_size — falls in residual ~2.66%
  recoverable tail.
- **NIF-D5-NEW-05 (LOW)**: `NiVectorExtraData` (deprecated, pre-20.5)
  not dispatched. Closes loop on **#329** string-read sweep.

**Suggested Fix**: 5-line parsers each + 1-line dispatch wires; bundle
with #728 / #329 closeout.

## Regression Guards Verified

Dimension 6 found **0 NEW bugs**. All four 2026-05-04 perf-batch pins
intact:

- NIF-D6-NEW-01: `#[must_use]` on `NifStream::allocate_vec` (`stream.rs:207`)
  — every call site binds the result.
- NIF-D6-NEW-02: `read_pod_vec` single-alloc bulk path + LE compile gate;
  six bulk-array helpers route through it; bytemuck NOT introduced;
  four regression tests (`bulk_reads_*`) pin the byte-order contract.
- NIF-D6-NEW-03: `bump_counter` `get_mut / insert` split (`lib.rs:222-232`)
  — no `entry().or_insert(name.to_string())` regression.
- NIF-D6-NEW-04: Remaining 10 `Vec::with_capacity` hits in `blocks/`
  are derived from already-validated counts or compile-time constants,
  not file input.

## Open Follow-up: dhat-infra Gap

**No allocation-counter regression test exists today.** The 2026-05-04
batch's "expected savings" (#831/#832/#833) are guarded by source
review + commit-history inspection only. A `dhat::Profiler`-based smoke
harness inside a single-NIF parse benchmark, snapshotting the
global-alloc count into a JSON baseline that gates CI, would close the
gap.

## Methodology Notes

- Dimension 1 (block parsing) yielded fewer findings than expected
  because Sessions 11/12 (April audit cycle) already swept most
  CRITICAL/HIGH-class block-parse bugs. Remaining surface is dominated
  by edge-version content the engine doesn't target.
- Dimensions 1, 3, and 5 share a common thread: pre-Bethesda Gamebryo
  / NetImmerse content is the test bed where convention bugs become
  visible. The engine doesn't ship that content today, but the
  dispatch and parse paths exist, and audit hygiene benefits from
  fixing them.
- Dimension 4's single finding is the highest-impact USER-visible bug
  in this audit — sky, water, grass, and HUD overlays in FO3/FNV
  silently lose their textures. Verify with `tex.missing` console
  command on Megaton or any FNV exterior cell.
