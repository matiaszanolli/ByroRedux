## Findings by Severity

### HIGH (3)

#### F3-01 — Terrain uses `upload()` instead of `upload_scene_mesh()` — RT reflection sampling reads wrong geometry
- **Severity**: HIGH | **Dim**: 3 | **Status**: NEW
- **Location**: `byroredux/src/cell_loader.rs:373-393` (terrain) vs `:836-854` + `byroredux/src/scene.rs:727-747` (clutter)
- Cell-placed clutter uses `MeshRegistry::upload_scene_mesh`, which appends vertex/index data to the global SSBO and records `global_vertex_offset` / `global_index_offset`. Terrain calls plain `MeshRegistry::upload`, leaving both offsets at 0. Those offsets are later copied into every draw's `GpuInstance.vertex_offset` / `index_offset`, and the RT ray-query path uses them to fetch vertex UVs from the global geometry SSBO. **For every RT reflection/GI/shadow ray hitting terrain, the shader indexes into the SSBO at offset 0 — reading whatever the first `upload_scene_mesh` put there.**
- **Impact**: On exteriors (WastelandNV 3×3 / 7×7) the ground is the dominant surface. Every reflection ray hitting terrain reads garbage UVs → reflections in puddles/wet rocks/scope glass sample the wrong texture. Indirect-lighting GI bounces off terrain inherit wrong albedo. Shadow rays are unaffected (occlusion-only). Silent — no validation layer catches it since the index is in bounds.
- **Fix**: Route terrain through `upload_scene_mesh` (or refactor both into one `upload` that always accumulates). Confirm `geometry_dirty` flips.

#### F3-04 — No despawn/cell-unload path → reloading a cell leaks all prior entities, meshes, textures, BLASes
- **Severity**: HIGH | **Dim**: 3 | **Status**: NEW (M40 scope, not yet tracked)
- **Location**: `crates/core/src/ecs/world.rs:66-70` (monotonic spawn, no despawn) + `byroredux/src/cell_loader.rs` (no teardown before second load)
- `World::spawn` only increments `next_entity`; there is no `World::despawn` and no `CellRoot` owner marker. Every mesh uploaded to `MeshRegistry` stays for the process lifetime; each owns a BLAS. Loading Cell A → Cell B → Cell A uploads the mesh set 2×, texture set 2×, stamps 3× entity count.
- **Impact**: First cell switch doubles live entities + VRAM + TLAS build cost. Second triples. Blocks doorwalking, fast-travel, `coc` commands. Current always-load-once usage hides the bug — M40 will explode on it.
- **Fix**: Land M40. Short-term: stamp `CellRoot(EntityId)` on every spawn; expose `World::despawn(entity)` that calls `remove` on every storage; add `cell_loader::unload(root)` + `MeshRegistry::drop_mesh(handle)` + `AccelerationManager::drop_blas(handle)` + `TextureRegistry::drop(handle)`.

#### D5-F3 — parry3d `Nested composite shapes are not allowed` panic storm in exterior runs
- **Severity**: HIGH | **Dim**: 5 | **Status**: NEW
- **Location**: `crates/physics/src/sync.rs` (wraps with `catch_unwind`) + the code constructing nested Compound shapes from NIF collision data
- **9,555 `parry3d panic: Nested composite shapes are not allowed.`** messages in 30s of exterior runtime. Each caught by `catch_unwind`. Panicking is ~10-100× slower than a normal return; log is flooded. Interior run: zero panics. Exterior-specific — LAND terrain + placed statics create a "compound of compounds" path parry3d rejects.
- **Impact**: Exterior still 400+ FPS but every frame pays a non-trivial panic tax. On slower hardware this pushes frame time over budget. Log noise is a diagnostic hazard.
- **Fix**: In `byroredux_physics::sync`, pre-flatten nested Compound shapes before handing to parry3d. Walk the Compound tree and produce a flat `Vec<(Isometry, SharedShape)>` where each child is a primitive (TriMesh, Ball, Cuboid, etc.). Stop using `catch_unwind` as input-validation substitute.

### MEDIUM (4)

#### F3-02 / D5-F1 — ESM re-parsed 3× per exterior cell load
- **Severity**: MEDIUM | **Dim**: 3 + 5 | **Status**: NEW
- **Location**: `byroredux/src/cell_loader.rs:133-139`, `crates/plugin/src/esm/records/mod.rs:94-95`, `byroredux/src/scene.rs:102`
- `load_exterior_cells` calls `parse_esm_cells(&esm_data)` on line 133. Three lines later it calls `parse_esm(&esm_data)` for WTHR/CLMT; `parse_esm`'s first action is `parse_esm_cells(data)` a second time over the same 62 MB buffer. Dim 5 live-runtime observation showed **3× parses** (extra debug pass) — 3 `ESM parsed:` log lines in <1s vs 1 for interior.
- **Impact**: ~0.4s × 3 = ~1.2s avoidable load stall on release build. Scales linearly with ESM size (Skyrim.esm 244 MB, Fallout4.esm 500 MB — bigger hit).
- **Fix**: Call `parse_esm` once, use `record_index.cells` as the cell index. Drop the separate line-133 call. Or teach `parse_esm_cells` and the record-walker to share a single `EsmReader` pass.

#### F3-08 — `Material::classify_pbr` allocates `String` + `to_ascii_lowercase` per draw per frame
- **Severity**: MEDIUM | **Dim**: 3 | **Status**: NEW
- **Location**: `crates/core/src/ecs/components/material.rs:138-139` + `byroredux/src/render.rs:289`
- `build_render_data` calls `m.classify_pbr(m.texture_path.as_deref())` for every draw every frame. `classify_pbr` begins with `let path = texture_path.unwrap_or("").to_ascii_lowercase();` — one heap allocation + full string copy per draw per frame. For 809 entities at 48 FPS → ~39k `String` allocations/sec.
- **Impact**: Allocator churn scales linearly with entity count. On 3×3 exterior grid with thousands of cell refs the allocator becomes measurable frame time. Classification result is path-stable — recomputing per frame is pure waste.
- **Fix**: Compute `PbrMaterial` once at cell load, cache onto `Material` component. Remove per-frame call from `build_render_data`. Or use byte-wise case-insensitive comparison without allocation.

#### F2-02 — CONT DATA gate accepts 4 bytes but FNV CONT DATA is 5 bytes — respawn flag silently ignored
- **Severity**: LOW→MEDIUM (gameplay) | **Dim**: 2 | **Status**: NEW
- **Location**: `crates/plugin/src/esm/records/container.rs:86-89`
- FNV CONT DATA: `weight (f32) + flags (u8)` = 5 bytes. Code gates at `>= 4` and reads only weight. The `flags` byte (bit 0 = respawns, bit 1 = unknown) never extracted. `ContainerRecord` has no `flags` field.
- **Impact**: Respawn behavior on containers (vendor restock) unavailable to gameplay. Renderer unaffected.
- **Fix**: Add `pub flags: u8` to `ContainerRecord`; tighten gate to `>= 5`; read `sub.data[4]`.

#### F2-03 — NPC ACBS disposition reads u8 from i16 field — truncates sign
- **Severity**: MEDIUM (future gameplay) | **Dim**: 2 | **Status**: NEW
- **Location**: `crates/plugin/src/esm/records/actor.rs:167, 50`
- FNV ACBS `disposition_base` is an i16 at offset 20 (per UESP/FalloutSnip). Code reads `sub.data[20]` as u8 into a u8 field. For 0-127 this works; negative/>127 values become low-byte-only.
- **Impact**: NPC base disposition corrupted for values outside 0-127. FNV default 50 limits damage; disposition logic not yet wired.
- **Fix**: Change field to `i16`; use `read_i16_at(&sub.data, 20)`.

### LOW (7)

- **F2-01 / F3-10** — `b"TXST"` duplicate match arm at `cell.rs:202` + `cell.rs:211` → `unreachable_patterns` warning on every plugin-crate build. Behaviorally correct. Drop `| b"TXST"` from line 211.

- **F3-11** — `CachedNifImport` cache is scoped to one `load_references` call; cell reload re-parses every NIF. Low today; HIGH once unload (F3-04) lands.
- **F3-12** — Terrain BLAS built one-at-a-time inside `spawn_terrain_mesh` (one submit per cell tile × 49 tiles on 7×7 grid). Clutter uses batched path. Thread `blas_specs` through terrain loader.
- **D1-F1** — NiTriShape tangent/bitangent data is parsed-then-skipped at `tri_shape.rs:685-691` when `data_flags & 0xF000 != 0` (NBT method). - **D5-F2** — **4,411 / 5,837 NIF warnings** on the FNV archive (76%) are NiPSys particle sub-block under-reads: `NiParticleSystem` (1,252), `NiPSysRotationModifier` (1,149), `NiPSysGrowFadeModifier` (890), `NiPSysBoxEmitter` (807), etc. NEW — separate from #331 (which covers only bhk*Constraint, 1,159 of the 5,837). - **D5-F4** — FPS / entity / draw telemetry silent in release without `--debug` (`systems.rs:876-879` early-returns when `debug_logging` is false). Regression detection from end-user reports nearly impossible.
- **D5-F5** — Single FNV LAND record form 00150FC0 fails to parse on every ESM open. Either a format variant or genuinely malformed vanilla data. One exterior tile affected.
- **D5-F6** — 83/3,003 (2.76%) base-form misses in exterior placed-refs lookup (Saloon: 5/460 = 1.1%). Placed refs whose base form type isn't in the statics table (LVLI/FLOR/ACTI/DOOR/etc.). Quiet misses; some scenery not rendering.
- **F2-04** — Informational: AVIF, QUST, DIAL, INFO, PERK, SPEL, MGEF, ENCH, PROJ, IMOD, CAMS, CPTH, SCPT, IMGS, LSCR not parsed. Never claimed — not a regression.
- **F2-05** — Comment arithmetic "16 + 4 = 20" at `actor.rs:245` is misleading; actual offset 28 is correct. Cosmetic.
- **FNV-D4-01** — Roadmap claims "1 GB BLAS budget" but `acceleration.rs:118` is **4 GB**. Git history: 32ae4bf raised 256 MB → 1 GB, a later commit bumped to 4 GB. On 12 GB dev GPU eviction virtually never fires; on 6 GB consumer GPUs this will OOM before eviction. Update docs, or derive budget from detected VRAM (`max_vram / 3`).
