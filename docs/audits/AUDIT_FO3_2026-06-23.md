# Fallout 3 Compatibility Audit ‚Äî 2026-06-23

**Scope**: Per-game audit of ByroRedux readiness for Fallout 3 content.
FO3 rides the FNV path end-to-end (NIF v20.2.0.7 / BSVER 34, BSA v104,
shared ESM parser, shared cell loader, shared RT lighting). This audit
hunts the FO3-vs-FNV **divergences** ‚Äî not a re-run of `/audit-fnv`.

**Method**: All 7 dimensions run inline against the live repo at
`/mnt/data/src/gamebyro-redux`, with real-data validation against
`/mnt/data/SteamLibrary/steamapps/common/Fallout 3 goty/Data/`. Findings
verified against current code (anchored on symbols, not line numbers) and
deduplicated against the cached OPEN-issue list (`/tmp/audit/issues.json`,
28 open) plus prior `docs/audits/` reports. Each candidate finding was
challenged adversarially.

## Executive Summary

**Compatibility level: HIGH (parse + import), with the known scripting-runtime gap.**

FO3 is in strong shape and the shared FNV surface holds up on real FO3
data. Concrete validation this run:

- **NIF parse: 100.00% / 10 989 files** (`nif_stats` over `Fallout - Meshes.bsa`, 0 failures, 0 recovered). Confirms the ROADMAP compat-matrix figure.
- **ESM parse: OK** ‚Äî `Fallout3.esm` (288 MB) parses in 1.39 s to `game_kind = Fallout3NV`; captures 421 interior cells (229 666 REFRs), 32 worldspaces (41 976 exterior cells, 338 339 REFRs), 11 450 STAT-family base objects, 1 647 NPCs, 243 TXST, 326 factions.
- **FO3 body/skeleton meshes present** at the expected `meshes\characters\_male\` paths (upperbody / lefthand / righthand / skeleton).

**One NEW finding** (a real FO3-vs-FNV divergence the prior FO3 audit

records that trip a false "lighting may be mis-computed" warning even

### Delta vs FNV
- **Shared and verified-on-FO3-data**: inline shader stack (`BSShaderPPLightingProperty` / `BSShaderNoLightingProperty`), typed particle emitters, Classic-bhk collision, NPC body/hand spawn, REFR texture overlays, SCPT (u16 SCHR flags), cell loader.
- **Diverges**: FO3 ships 36-byte XCLL (FNV ships 40 only) ‚Äî NEW finding D3-FO3-01. FO3's inline-shader-only material universe (no BGSM, no BSLightingShaderProperty) is provably the only path taken. Capital Wasteland is one of 32 FO3 worldspaces (FNV has fewer).

## Findings

### D3-FO3-01: FO3 36-byte XCLL trips a spurious "lighting may be mis-computed" warning
- **Severity**: LOW
- **Dimension**: 3 (ESM Record Coverage) ‚Äî FO3-divergent
- **Location**: `crates/plugin/src/esm/cell/walkers.rs` (`XCLL_SIZES_FALLOUT_ERA` / `xcll_size_sanity_warn`); decode at the `b"XCLL"` arm
- **Status**: NEW
- **Description**: The canonical XCLL size set for `GameKind::Fallout3NV` is `XCLL_SIZES_FALLOUT_ERA = &[28, 40]`. Real `Fallout3.esm` ships **36-byte** XCLL records on 17 interior cells (Oblivion-style tail: shared 28-byte prefix + `Directional Fade`@28 + `Fog Clip Dist`@32, but **no** `Fog Power`@36 ‚Äî that 40-byte field is the FNV-era addition). Because 36 is absent from the canonical set, `xcll_size_sanity_warn` fires 17√ó during a vanilla `Fallout3.esm` parse with the message "‚Ä¶cell lighting may be mis-computed. Often indicates a malformed authoring or cross-game plugin injection." The data is **vanilla and correct**, not malformed.
- **Evidence**: `sf_parse_check` over `Fallout3.esm` emits 17√ó `ESM XCLL with 36 bytes is non-canonical for Fallout3NV (expected one of [28, 40])`. The same harness over `FalloutNV.esm` emits **zero** non-canonical XCLL ‚Äî confirming this is a clean FO3-vs-FNV divergence, not a shared bug. Crucially, the decode itself is **correct**: the `b"XCLL"` arm uses per-field length gating (#1312) ‚Äî `dir_fade = (len >= 32)`, `fog_clip = (len >= 36)`, `fog_power = (len >= 40)` ‚Äî so a 36-byte FO3 record decodes `dir_fade` + `fog_clip` and skips `fog_power`, exactly as intended. The in-code comment at that arm even says "A 36-byte Oblivion XCLL is the FULL TES4 Lighting (dir_fade + fog_clip, no fog_power)."
- **Impact**: Log spam (17 WARN lines per FO3 parse) plus an actively **misleading** message claiming cell lighting may be wrong and suggesting a cross-game injection, when the lighting is decoded correctly from vanilla data. No render impact. Misleads future FO3 cell-lighting debugging ‚Äî the prior FO3 audit (`AUDIT_FO3_2026-06-14.md`) asserted XCLL "identity with FNV (`[28,40]`, warn-only sanity)", which is the premise this finding refutes.
- **Related**: #1312 (per-field XCLL gating that makes the decode already correct); prior `AUDIT_FO3_2026-06-14.md` line 141.
- **Suggested Fix**: Add `36` to `XCLL_SIZES_FALLOUT_ERA` (‚Üí `&[28, 36, 40]`) and update the `fo3_fnv_xcll_sizes_pinned` test + doc comment to note that FO3 ships the 36-byte (no-fog_power) tail while FNV ships 40. The decode needs no change; only the canonical-size set and the warning gate are wrong. (Consider whether FO3 and FNV warrant separate `GameKind`s long-term, since they already diverge here.)

### D5-FO3-01: bhk Hinge / Prismatic / Breakable / StiffSpring constraints dropped silently in ragdoll chains
- **Severity**: MEDIUM
- **Dimension**: 5 (Collision Import)
- **Location**: `crates/nif/src/import/collision.rs` (`BhkConstraintData::Other => continue` in the ragdoll constraint loop)
- **Status**: Existing: #1539
- **Description**: The constraint extractor decodes only `Ragdoll` and `LimitedHinge` CInfo (including the FO3 `bhkMalleableConstraint` unwrap to its inner type); every other constraint type is `BhkConstraintData::Other` and is `continue`d with no telemetry. FO3 base meshes carry these other types ‚Äî confirmed by `nif_stats`: `bhkHingeConstraint` (6), `bhkPrismaticConstraint` (2), `bhkBreakableConstraint` (1), plus `bhkMalleableConstraint` (60, which IS unwrapped). A dropped joint inside a multi-body chain fragments that ragdoll.
- **Evidence**: `nif_stats --all` over `Fallout - Meshes.bsa`. Code confirms `bhkMalleableConstraint` is correctly unwrapped (`constraints.rs` line ~288), so the live gap is the genuinely-unsupported kinds. The drop is inside the ragdoll path (guarded by `bodies.len() >= 2`), so it affects ragdoll chains, not static architecture.
- **Impact**: A handful of FO3 ragdoll/articulated objects using these constraint types lose a joint with no diagnostic.
- **Related**: #1539, #1540 (degenerate inertia).
- **Suggested Fix**: Per #1539 ‚Äî at minimum emit a debug log when an unsupported constraint type is encountered in a ragdoll chain.

### D6-FO3-01: Uncompressed 16/24-bpp DDS (FO3 fonts + HUD) fall back to placeholder
- **Severity**: LOW
- **Dimension**: 6 (BSA v104 + Real-Data)
- **Location**: `crates/renderer/src/vulkan/dds.rs` (uncompressed `DDPF_RGB` branch, `bpp == 32` ensure)
- **Status**: Existing: #1542
- **Description**: The DDS parser only uploads 32-bpp uncompressed RGBA directly; FO3's 16-bpp font-glyph atlases (`textures\fonts\*_lod_a.dds`) and 24-bpp HUD compass (`hud_comp_direction_vertical.dds`) are rejected and render as the checker placeholder. UI-only, never world geometry, graceful (no crash). Accurately documented in-code as 9 of FO3's 12 261 textures (0.07%).
- **Evidence**: The `ensure!(bpp == 32, ‚Ä¶)` with the #1542 message; verified the branch and comment are unchanged.
- **Impact**: A few FO3 UI fonts + the HUD compass render as placeholder. Cosmetic, UI-layer only.
- **Related**: #1542.
- **Suggested Fix**: Per #1542 ‚Äî CPU-expand 16/24-bpp to R8G8B8A8 in the upload path (deferred as low-value).

### D4-FO3-01: Exterior worldspace selection prefers the default list before grid-containment
- **Severity**: LOW
- **Dimension**: 4 (Cell Loading ‚Äî exterior)
- **Location**: `byroredux/src/cell_loader/exterior.rs` (worldspace selection: `preferred = ["wastelandnv", "wasteland", "tamriel", "skyrim"]` evaluated before the grid-containment fallback)
- **Status**: Existing: #1655
- **Description**: When no `--wrld` override is given, the selector picks the first preferred-default worldspace (FO3 = "wasteland" / Capital Wasteland) BEFORE checking which worldspace actually contains the requested `--grid` coord. FO3 has 32 worldspaces (DLC: Point Lookout, The Pitt, Anchorage, Mothership Zeta‚Ä¶), so loading a DLC exterior grid without `--wrld` mis-selects Capital Wasteland. Base-game Capital Wasteland loads are unaffected.
- **Evidence**: Code path read directly; ESM parse confirms 32 FO3 worldspaces. The preferred list correctly includes "wasteland" (#444 added it), so the base case is right; the ordering bug is #1655.
- **Impact**: DLC exterior grids load the wrong worldspace unless `--wrld` is passed. Workaround exists (`--wrld`).
- **Related**: #1655, #444.
- **Suggested Fix**: Per #1655 ‚Äî run grid-containment before (or to disambiguate) the preferred-default list.

## FNV-Shared Surface (verified on FO3 data)

Record / block / shader paths FO3 inherits from FNV coverage, confirmed
exercised on real FO3 content with no FO3-only gap inside them:

- **Inline shader material path (Dim 1)** ‚Äî `BSShaderPPLightingProperty` (29 078 instances in FO3 meshes) and `BSShaderNoLightingProperty` (3 400) are the only shader stack. Verified:
  - Normal map read from the dedicated `BSShaderTextureSet` slot 1 (PP-lighting branch), not the Oblivion bump slot; the pre-Skyrim `NiTexturingProperty` path reads `normal_texture` first then falls back to `bump_texture`, so FO3 doesn't regress into Oblivion's bump-slot behavior.
  - NoLighting routes through the fullbright path (`material_kind = 102`) and honors the FO3/FNV-only `Alpha_Decal` (flags2 bit 21) via `is_decal_from_legacy_shader_flags` ‚Äî self-illumination does not dim with distance.
  - Decal flag positions use the FO3/FNV legacy single-u32 vocabulary (`shader_flags::fo3nv_f1` / `fo3nv_f2`), with compile-time assertions pinning bit-26/27 cross-era equivalence and the `ShaderFlags::LegacyFo3Fnv` typed view keeping bit-21 out of the Skyrim/FO4 vocabularies.
  - NIFAL single boundary holds: FO3 materials flow through `material_translate::translate_material`; `metalness`/`roughness` exit as concrete `f32` (NaN sentinel ‚Üí `Material::resolve_pbr` keyword classifier). FO3 authors no BGSM, so `MAT_FLAG_PBR_BSDF` is unreachable. FO3 PP/NoLighting self-illumination maps to `EmissiveSource::Material`.
- **Typed particle emitters (Dim 2)** ‚Äî FO3 DOES use the typed path: `NiPSysEmitter` (422), `NiPSysEmitterCtlr` (361), `NiPSysGrowFadeModifier` (282), `NiParticleSystem` (422). `extract_emitter_params` / `extract_emitter_rate` decode authored kinematics + birth rate + GrowFade base_scale with finite/positivity guards (NIFAL-S3/S5). The legacy `NiPSysBlock` (4 041) is the opaque container, not an emitter type. Known scope caveats (#1333 local transform, #1402 multi-emitter first-match) are shared with FNV, not FO3-specific.
- **Classic-bhk collision (Dim 5)** ‚Äî `extract_from_classic` is bit-for-bit shared and handles FO3's dominant chains (`bhkCollisionObject` 9 001, `bhkRigidBody` 8 957, MOPP/PackedNiTriStrips, ConvexVertices, Box/Capsule/Sphere). `havok_motion_type` maps the full canonical enum (#1652) ‚Äî BOX_INERTIA(4) clutter simulates, not mis-typed kinematic. `bhkConvexListShape` (55 in FO3) ‚Üí `CollisionShape::Compound` (#1277). The dispatch/resolve parity guard test passes. `bhkMultiSphereShape` is absent from FO3 base meshes (its #1277 translator is correct but not exercised on FO3 base).
- **NPC spawn (Dim 7)** ‚Äî FO3 body/hand paths (`humanoid_body_paths`) and skeleton path resolve to real BSA entries (verified via `listbsa`). FO3 kf-era spawn loads upperbody + lefthand + righthand (#793). `AnimationClipRegistry` dedup (#790) and B-spline pose-fallback (#772) are in place; B-splines absent from FO3 base meshes but reachable on DLC/anims (not ruled out by era).
- **ESM records (Dim 3)** ‚Äî SCPT SCHR flags read as u16 at cursor 18 (#1654); REFR XATO/XTNM/XTXR texture overlays implemented (`cell_loader/refr.rs`); no `unreachable_patterns` warning in the plugin crate. XCLL decode is per-field gated and correct for FO3's 36-byte tail (see D3-FO3-01 ‚Äî only the warning is wrong).

## FO3-Distinctive Gaps

- **Scripting runtime gap (largest FO3-specific functional gap, NOT a bug to file)**: FO3 SCPT records parse but no runtime executes them. M47.0 event-hook + M47.1 condition-eval foundations ship; the M47.2 compiled-Papyrus recognizer slice is in progress. FO3 quest/world interactivity is blocked on this. Owned by `/audit-scripting`; out of scope here.
- **Inline-shader-only material universe**: confirmed ‚Äî zero `BSLightingShaderProperty`, zero BGSM/BGEM in FO3 meshes. The Disney-BSDF / BGSM paths are provably unreachable for FO3.
- **36-byte XCLL** (D3-FO3-01) ‚Äî the one concrete FO3-vs-FNV authoring divergence found this run.
