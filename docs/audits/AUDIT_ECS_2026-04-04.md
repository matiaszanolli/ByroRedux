## Dimension 2: Query Safety

### E-1: `World::get()` — unsound lifetime extension via dropped RwLockReadGuard
- **Severity**: CRITICAL
- **Dimension**: Query Safety / Unsafe Code Review
- **Location**: `crates/core/src/ecs/world.rs:65-77`
- **Status**: NEW
- **Description**: `World::get<T>(&self)` acquires a `RwLockReadGuard`, extracts a raw pointer, drops the guard at function exit, and returns the pointer as `&T` with lifetime `'self`. After the guard drops, `query_mut()` (which also takes `&self`) can acquire a write lock, mutate/remove the component, and invalidate the reference.
- **Evidence**:
  ```rust
  pub fn get<T: Component>(&self, entity: EntityId) -> Option<&T> {
      let lock = self.storages.get(&TypeId::of::<T>())?;
      let guard = lock.read().expect("storage lock poisoned");
      let storage = guard.downcast_ref::<T::Storage>().expect("...");
      let ptr = storage.get(entity)? as *const T;
      // guard DROPPED here — lock released
      Some(unsafe { &*ptr })
  }
  ```
  The SAFETY comment claims the guard prevents write-lock acquisition, but the guard is a stack local dropped at function return. A caller can do:
  ```rust
  let ref_a = world.get::<Health>(e).unwrap();  // guard dropped
  let mut q = world.query_mut::<Health>().unwrap();  // write lock succeeds
  q.remove(e);  // frees component
  println!("{}", ref_a.0);  // use-after-free
  ```
- **Impact**: Use-after-free if caller holds returned `&T` across a `query_mut()` call. Currently latent — no existing call site triggers it.
- **Suggested Fix**: Return a guard-owning wrapper (like `ResourceRead`) that holds the `RwLockReadGuard` and derefs to `&T`. Or deprecate `get()` in favor of `query().get(entity)`.

## Dimension 3: World Integrity

### E-5: Entity ID overflow wraps silently in release mode
- **Severity**: MEDIUM
- **Dimension**: World Integrity
- **Location**: `crates/core/src/ecs/world.rs:33-36`
- **Status**: NEW
- **Description**: `spawn()` uses `self.next_entity += 1` with no overflow check. At u32::MAX, wraps to 0 in release mode (debug panics with cryptic message). Entity ID aliasing causes component data corruption.
- **Impact**: After 4B spawns. Unlikely in gameplay, plausible in test harnesses or procedural generation.
- **Suggested Fix**: Use `checked_add` with descriptive panic, or document limit.

### E-6: `insert()` accepts non-spawned entity IDs without validation
- **Severity**: LOW
- **Dimension**: World Integrity
- **Location**: `crates/core/src/ecs/world.rs:52-54`
- **Status**: NEW
- **Description**: Any `u32` can be passed as entity ID, even if never returned by `spawn()`. No provenance tracking.
- **Suggested Fix**: Add `debug_assert!(entity < self.next_entity)` or document limitation.

### E-7: `remove()`/`get_mut()` create storage as side effect
- **Severity**: LOW
- **Dimension**: World Integrity
- **Location**: `crates/core/src/ecs/world.rs:57-59, 80-82`
- **Status**: NEW
- **Description**: Both call `storage_write()` which lazily creates empty storage. A no-op `remove::<Foo>()` permanently changes `query::<Foo>()` from `None` to `Some(empty)`.
- **Suggested Fix**: Check `storages.get()` first, return `None` early if missing.

## Dimension 4: Resource Safety

### E-8: `insert_resource()` silently overwrites without returning old value
- **Severity**: MEDIUM
- **Dimension**: Resource Safety
- **Location**: `crates/core/src/ecs/world.rs:238-241`
- **Status**: NEW
- **Description**: Uses `HashMap::insert()` which drops the old value silently. Safe due to `&mut self` exclusivity, but callers cannot detect accidental double-initialization.
