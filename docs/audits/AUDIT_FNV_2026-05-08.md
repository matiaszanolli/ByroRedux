## Executive Summary

FNV is the engine's reference title. This audit measured observed numbers against the roadmap's claimed baselines — three fresh findings, all rooted in the **session-31 M41-EQUIP scaffold** (#896 Phases A.0 → B.2) materially changing the visible-actor mix on Prospector.

The single load-bearing finding is **FNV-D5-NEW-01** — a per-frame WARN-spam on Prospector: `skin_compute` descriptor pool's `max_slots = 32` cap is exhausted by the new body+armor entity volume, and the retry path at `draw.rs:506-513` has no per-entity sticky-failure marker, so two entities re-fire `create_slot` failures every frame (58 WARN lines / 300 frames). Severity MEDIUM because RT shadows on those NPCs are silently lost, but raster + AABB cull stay correct so the scene still renders.

The other two findings are doc drift:

- **FNV-D5-NEW-02** (LOW): ROADMAP cites `62 219` FNV ESM records in two places; observed total is `73 054` post-session-29 long-tail closeout (#808/#809/#810). Stale by ~10 800.
- **FNV-D5-NEW-03** (LOW): R6a-stale-7's gating clause "Refresh deferred until M41 lands the visible-actor workload" is falsified — M41-EQUIP B.2 has landed and Prospector now spawns 2562 entities (vs 1200 baseline). The bench-staleness Known Issue itself isn't a finding (already filed); the rationale text is stale and can be tightened.

**No CRITICAL, no HIGH.** Parse rate holds at 100.00% (14 881 / 14 881). Bench shows a -17% wall-clock FPS regression (172.6 → 143.7) but with +114% entity volume — fence_ms scaled +18% on 2.1× the geometry, which is sub-linear and consistent with RT cost amortising across the BLAS hierarchy.

## Findings

### FNV-D5-NEW-01: skin_compute descriptor pool exhausts on M41-EQUIP Prospector — per-frame WARN spam, RT shadows silently lost on overflowing entities

**Severity**: MEDIUM
**Dimension**: 5 (Real-Data Validation)
**Location**: [skin_compute.rs:248-260](crates/renderer/src/vulkan/skin_compute.rs#L248-L260) (`max_slots = 32` cap), [context/mod.rs:1102](crates/renderer/src/vulkan/context/mod.rs#L1102) (`SKIN_MAX_SLOTS = 32` constant), [context/draw.rs:506-512](crates/renderer/src/vulkan/context/draw.rs#L506-L512) (retry-every-frame WARN site)
**Status**: NEW · CONFIRMED at HEAD `318fcaf`

Prospector at HEAD spawns enough skinned meshes — NPC body + per-slot armor pieces from #896 Phase B.2 — that the 32-slot descriptor pool fills up. Two specific entities (1088 and 1095 in the captured run) never get a slot: `vkAllocateDescriptorSets` returns OUT_OF_POOL_MEMORY. The dispatch-side retry at `draw.rs:506-513` has no per-entity sticky-failure marker, so each frame re-fires `create_slot` on these two entities, each failure logs a fresh WARN — observed at **58 WARN lines / 300 frames**.

The pool-sizing comment at `skin_compute.rs:248-249` reads `max_slots == 32 (matches MAX_TOTAL_BONES / MAX_BONES_PER_MESH) covers every realistic interior cell` — this comment was authored before #896 Phase B.2 widened the visible-actor mix. Same code path as #871 (which fixed the buffer leak when `allocate_descriptor_sets` fails after buffer alloc succeeds), but #871 left the cap-and-retry semantics untouched.

**Impact**: RT shadows for body+armor on the two overflow entities silently disabled (raster path correct via inline skinning). Log noise on the cell-load tail; not a render-correctness bug for the 30 / 32 entities that fit, but a measurable regression on the M41-EQUIP load.

**Suggested fix** (pick one; sketches only — do not ship without verifying):
1. Bump `SKIN_MAX_SLOTS` to a value sized for the M41-EQUIP visible-actor mix (likely 64 or scale at cell-load from the actor count). Each slot costs 3 storage-buffer descriptors × `MAX_FRAMES_IN_FLIGHT` = 6 descriptors total — bumping to 64 stays well under the device limit.
2. Add a `failed_skin_slots: HashSet<EntityId>` cache on the renderer state so `draw.rs` skips the retry on a known-bad entity and the WARN fires once per cell-load, not per frame. Cleared on cell unload.
3. Both — (1) for capacity, (2) for the WARN-spam invariant on the next time something pushes past the new ceiling.

**Dedup-check**: Searched `descriptor` / `pool` / `slot` / `create_slot` against `/tmp/audit/issues.json` (200 issues, all states). Only #871 hits the same call site — that issue fixed the `output_buffer` leak when `allocate_descriptor_sets` fails after buffer-alloc succeeds; it did not address the cap or the retry semantics. This is a distinct defect.

### FNV-D5-NEW-02: ROADMAP FNV ESM record count (62 219) stale — actual is 73 054 post-#808/#809/#810

**Severity**: LOW (documentation drift)
**Dimension**: 5 (Real-Data Validation)

**Impact**: Doc drift only. No code or behaviour change. The Repro commands table's authority claim ("source of truth") becomes load-bearing when an audit reports against ROADMAP without a corresponding code-side reverify.

**Suggested fix**:
1. Update ROADMAP.md:70 → `73 054 records`.
2. Update ROADMAP.md:561 → `73 054`.
3. Re-run `parse_real_esm` for FO3 in a follow-up to surface the matching FO3 drift (out of dim-5 scope).

**Dedup-check**: No issue with `62 219` / `73 054` / "record count" in title across the 200-issue baseline. Distinct from #843 / #844 / #847 (audio audit residue) and the various NIF-PERF rows.
