# NIF Parser Audit — 2026-05-29

**Scope:** Deep correctness/coverage/perf audit of the NIF binary-format parser
across all 7 supported game variants (Oblivion → Starfield).
**Architecture:** 7 dimension agents (Block Parsing, Version Handling, Stream
Position, Import Pipeline, Coverage, Stream Allocation, NIFAL Translation).
**Corpus:** All 7 game data dirs present; Dim 3 and Dim 5 ran real-archive
sweeps via the in-tree `nif_stats --drift-histogram` and `d5_coverage` examples.
**Dedup baseline:** 19 open / 1325 total GitHub issues (`/tmp/audit/issues*.json`).

---

## Executive Summary

The NIF parser is in **excellent** shape. Every one of the ~20 architectural
regression guards across Dims 1/2/4/6/7 **holds**. Parse-block dispatch is
effectively complete (308 dispatch keys; ~99.999%+ true instance coverage on
every game). Stream-allocation hygiene (Dim 6) and the NIFAL translation
boundary (Dim 7) are clean with zero new findings.

**8 findings: 1 HIGH, 3 MEDIUM, 4 LOW.** No CRITICAL.

The single HIGH is a real data-loss bug: two undispatched Oblivion Havok
collision shapes trigger mid-file truncation (no block_size to realign),
silently dropping render geometry for 3 vanilla meshes.

**Cross-dimension theme — the `avobject_flags_u32()` misuse (Findings 02 & 03):**
Two independent dimensions (Block Parsing/Stream Position, and Version Handling)
converged on the same anti-pattern: the *variant-level* helper
`stream.variant().avobject_flags_u32()` is used as a stand-in for nif.xml's
*per-file* `BSVER > 26` gate (`bsver::FLAGS_U32_THRESHOLD`, which already exists
but is unused at both sites). One site corrupts FO3/FNV NoLighting falloff data
(corpus-confirmed, 22 instances, −16 drift); the other mis-reads NiAVObject flag
width on transitional-export Oblivion content. **Both should be fixed in one pass.**

## Findings

### NIF-2026-05-29-01: Oblivion `bhkConvexSweepShape` / `bhkMeshShape` undispatched → hard mid-file truncation
- **Severity:** HIGH
- **Dimension:** Coverage
- **Game Affected:** Oblivion (TES4, v20.0.0.5 — sizeless, no block_sizes table)
- **Location:** `crates/nif/src/blocks/mod.rs:1153` (`_ =>` fallback returns `Err`
  when `block_size == None`); recovery in `crates/nif/src/lib.rs:557-620`.
- **Status:** NEW
- **Description:** `bhkConvexSweepShape` (nif.xml `inherit="bhkConvexShapeBase"`,
  `V10_0_1_0`) and `bhkMeshShape` (`inherit="bhkShape"`) have no dispatch arm.
  On Oblivion there is no per-block size to realign, so the failed parse cannot be
  skipped — the loader truncates the remainder of the file.
- **Evidence:** Probe (header `num_blocks` vs parsed count) on `Oblivion - Meshes.bsa`:
  - `meshes\clutter\farm\handscythe01.nif` — header=47, parsed=3 → **lost 44**
  - `meshes\clutter\farm\oar01.nif` — header=17, parsed=3 → **lost 14**
  - `meshes\architecture\basementsections\ungrdltraphingedoor.nif` — header=31, parsed=3 → **lost 28**
  The bhk shape sits early (block ~3, under the collision object), so the *render
  geometry* that follows is discarded. `parse_nif` returns `Ok` (truncation =
  "successful recovery"), masking the loss from the clean-rate gate.
- **Impact:** A scythe, an oar, and a hinged trapdoor render as empty/stub geometry
  in Oblivion. Low file count but total data loss for those files; this is the
  residual tail of the Oblivion sizeless-cascade class (#474, #979, #980 — CLOSED).
- **Related:** #474, #979, #980 (same mechanism, CLOSED), #224 (`oblivion_skip_sizes`),
  NIF-2026-05-29-04 (the missing regression pin that would have caught this).
- **Suggested Fix:** Add dispatch arms for both shapes. Even a structured
  skip-to-known-size stub registered in `oblivion_skip_sizes` (#224 mechanism)
  stops the truncation and lets render geometry load. Layouts: `bhkConvexSweepShape`
  = bhkShape ref + material + radius + unknown vec; `bhkMeshShape` ≈ legacy
  bhkNiTriStripsShape. Verify against nifly `BSHavok` decoders / nif.xml.

### NIF-2026-05-29-02: `BSShaderNoLightingProperty` over-reads 16 bytes on FO3/FNV bsver≤26 — falloff fields gated on variant helper instead of per-file `BSVER > 26`
- **Severity:** MEDIUM
- **Dimension:** Block Parsing / Stream Position *(merged: Dim 1 NIF-D1-01 + Dim 3 NIF-D3-NEW-08 — same lines)*
- **Game Affected:** Fallout 3, Fallout New Vegas (NoLighting content at bsver ≤ 26;
  22 instances each in `Fallout - Meshes.bsa`)
- **Location:** `crates/nif/src/blocks/shader.rs:165-175` (`BSShaderNoLightingProperty::parse`)
- **Status:** NEW (distinct from #451 CLOSED — that was the *importer dropping* the
  values, not the parse-time version gate)
- **Description:** The four falloff fields (`falloff_start_angle/stop_angle/start_opacity
  /stop_opacity` = 4×f32 = 16 bytes) are read when `stream.variant().avobject_flags_u32()`
  is true. That predicate is game-enum keyed (`version.rs:480-491`: true for
  FO3/FNV/Skyrim+/FO4/FO76/Starfield) and is semantically the *NiAVObject flag-width*
  test, not the falloff-presence test. nif.xml (6236-6239) gates these on
  `vercond="#BSVER# #GT# 26"`. FO3/FNV ship NoLighting content down to bsver 11/14/24/26;
  for those files the parser reads 16 bytes that aren't there, consuming the first
  16 bytes of the *following* block.
- **Evidence:** Corpus drift sweep: `22× BSShaderNoLightingProperty drift=-16` on
  **both** FO3 and FNV mesh archives (`-16` = consumed 16 over the declared block_size;
  `16 = 4×sizeof(f32)`). The sibling `BSShaderPPLightingProperty::parse` (same file,
  shader.rs:82-101) does this correctly via per-file `bsver > FO3_PARALLAX(24)` gates.
  The matching constant `bsver::FLAGS_U32_THRESHOLD = 26` exists (`version.rs:224`) but
  is unused here.
- **Impact:** (1) The four falloff floats are populated with garbage bytes read from
  the next block → wrong cone angles / opacities on affected FO3/FNV decals & glow-map
  surfaces (per #451's importer wiring, the falloff reaches the GPU). (2) block_size
  realigns the stream so no downstream desync — a data-correctness bug, not a crash.
  Dim 1 initially rated this LOW assuming no shipping content was affected; Dim 3's
  corpus sweep **empirically disproved that** (22 real instances per game), so it is
  MEDIUM.
- **Related:** #451 (CLOSED, importer-side capture), #429 (CLOSED, analogous
  NiTexturingProperty over-read), #939 (CLOSED, drift histogram that surfaced this),
  NIF-2026-05-29-03 (same anti-pattern, different site).
- **Suggested Fix:** Replace the `shader.rs:166` predicate
  `if stream.variant().avobject_flags_u32() {` with
  `if stream.bsver() > crate::version::bsver::FLAGS_U32_THRESHOLD {`. Add an FO3/FNV
  bsver≤26 fixture asserting `consumed == declared` and falloff defaults (0,0,1,0).
  `Sky`/`TileShaderProperty` share the `parse_fo3` base but don't read falloff —
  unaffected, no change needed.

### NIF-2026-05-29-03: `NiAVObject` flag width uses variant helper, not per-file `BSVER > 26` — mis-reads Oblivion content detected as the Fallout3 variant
- **Severity:** MEDIUM
- **Dimension:** Version Handling *(Dim 2 NIF-D2-01)*
- **Game Affected:** Oblivion (and non-Bethesda Gamebryo) content authored/exported
  at v20.2.0.7 / user_version=11 / BSVER ≤ 26 — any file that detects as the
  `Fallout3` variant with in-file bsver ≤ 26.
- **Location:** `crates/nif/src/blocks/base.rs:75-82` (`NiAVObjectData::parse`)
- **Status:** NEW (#332/#328 are about flag *masks*, not flag *width*)
- **Description:** nif.xml defines `NiAVObject.Flags` as `uint` for `#BSVER# #GT# 26`
  (line 3442) and `ushort` for `#LTE# 26` (line 3483) — a strict per-file gate. The
  code branches on `stream.variant().avobject_flags_u32()`, which returns `true` for
  the *entire* `Fallout3` variant regardless of the file's actual BSVER. An Oblivion
  file exported at v20.2.0.7 / uv=11 / bsver=11 detects as `Fallout3` (`version.rs:362-365`),
  so the helper says u32 where nif.xml wants u16.
- **Evidence:** The code's own comment (base.rs:75-78) prescribes *"Use actual BSVER
  from header, not the variant's hardcoded value"* — then line 78 uses the variant
  helper, contradicting itself. `avobject_flags_u32()` (`version.rs:480-491`) is
  variant-level. Standard Oblivion (v20.0.0.4/5) is caught by the unconditional
  Oblivion branch *before* the uv match, so the bug is confined to the v20.2.0.7
  transitional-export edge (NifSkope/newer-tool exports). `bsver::FLAGS_U32_THRESHOLD = 26`
  (version.rs:224) exists for exactly this gate but is unused here.
- **Impact:** Reads 4 bytes where 2 are on disk, slipping the stream +2 from `flags`
  onward; block_size realignment masks the slip (parse rate unaffected, contents wrong
  — same failure mode as #342). Subtly wrong NiAVObject flags (cull/visibility) on
  affected files.
- **Related:** #1277-T5, #342 (analogous masked-slip), #437 (GameVariant),
  NIF-2026-05-29-02 (same anti-pattern, the other surviving site).
- **Suggested Fix:** `let flags = if stream.bsver() > bsver::FLAGS_U32_THRESHOLD { u32 } else { u16 };`.
  Add a regression test `(V20_2_0_7, uv=11, bsver=11)` → u16 and `(…, bsver=34)` → u32.
  This is the last variant-level predicate shadowing a strict per-file BSVER gate —
  fix together with F-02.

### NIF-2026-05-29-04: No parse-block coverage regression pin distinct from material-translation completeness
- **Severity:** MEDIUM
- **Dimension:** Coverage *(Dim 5 NIF-D5-04)*
- **Game Affected:** All (test infrastructure)
- **Location:** `crates/nif/tests/translation_completeness.rs::cross_game_translation_completeness`;
  `crates/nif/tests/per_block_baselines.rs`
- **Status:** NEW
- **Description:** The translation-completeness harness measures whether parsed blocks
  *translate* to a canonical Material — it does NOT catch parse-block truncation
  (the truncated collision blocks in F-01 never produced a Material anyway). No test
  asserts `parsed_block_count == header.num_blocks` (Oblivion) or a NiUnknown-rate
  ceiling (sized games). That is why the 3 Oblivion truncations are invisible to
  `cargo test` and only surfaced via the ad-hoc probe.
- **Evidence:** `translation_completeness.rs:198-235` iterates 5 games measuring
  material fields and `structurally_inconsistent` with no block-count parity assertion.
  `d5_coverage`'s `parse_nif ok` metric counts truncation-recovered files as Ok.
- **Impact:** Coverage regressions of the sizeless-cascade class land silently.
- **Related:** #568 (nif_stats clean metric counted NiUnknown-recovered as clean —
  CLOSED, same blind spot), #601, F-01.
- **Suggested Fix:** Add a per-game pin (extend `per_block_baselines.rs` or a new
  `block_coverage_baselines.rs`) asserting `parsed_blocks == header.num_blocks`
  (Oblivion) + a NiUnknown-rate ceiling (sized games) over a sampled vanilla set.

### NIF-2026-05-29-05: Modern `NiParticleSystem` local transform discarded at parse → emitter spawn ignores host-relative offset
- **Severity:** LOW
- **Dimension:** Import Pipeline *(Dim 4 NIF-D4-01)*
- **Game Affected:** All (NiParticleSystem is universal; most visible on Skyrim+/FO4
  multi-emitter FX where smoke/spark sits above the fire)
- **Location:** `crates/nif/src/blocks/particle.rs:903` and `:1219`
  (`let _av = NiAVObjectData::parse(stream)?;`); consumed-side
  `crates/nif/src/import/walk/mod.rs:1148` and `:512`
- **Status:** NEW
- **Description:** The `NiParticleSystem` parser reads the block's `NiAVObjectData`
  base (incl. local TRS relative to its parent NiNode) into `_av` and immediately
  discards it; the retained struct is only `{ original_type, modifier_refs }`. Both
  walkers then position the emitter purely from the host-node world transform. Any
  non-zero local translation/rotation authored on the particle block is lost. The
  legacy path (`legacy_particle.rs:522`) *retains* `av`, confirming the field is
  meaningful and the modern drop is the outlier.
- **Evidence:** `particle.rs:903`/`:1219` bind to `_av`; `ImportedParticleEmitter`
  (types.rs:880) has no translation/rotation field; `walk_node_particle_emitters_flat`
  composes only the parent chain.
- **Impact:** Emitters with a non-zero local offset spawn at the host node origin
  instead of the offset position. Vanilla torches/fires place the system at the node
  origin (zero offset) → invisible on the common case; surfaces on offset multi-emitter
  NIFs (campfire smoke above fire, FO4 steam stacks).
- **Related:** #401 (emitter import), #984, #707.
- **Suggested Fix:** Retain the parsed local transform on `NiParticleSystem`, compose
  it into `parent_transform` in the flat walker, and add a `local_translation`/`rotation`
  field to `ImportedParticleEmitter` for the hierarchical path. The data is already on
  the wire and merely thrown away.

### NIF-2026-05-29-06: Skyrim SE `bhkPlaneShape` undispatched — single block, block_size-recovered (no cascade)
- **Severity:** LOW
- **Dimension:** Coverage *(Dim 5 NIF-D5-03)*
- **Game Affected:** Skyrim SE (has block_sizes)
- **Location:** `crates/nif/src/blocks/mod.rs:1153` fallback
- **Status:** NEW
- **Description:** `bhkPlaneShape` (nif.xml `inherit="bhkHeightFieldShape"`, `#BETHESDA#`)
  has no dispatch arm. SSE carries a block_sizes table, so the fallback skips the block

- **Evidence:** Probe on `Skyrim - Meshes1.bsa`:
  `meshes\plants\switchnodechildren\slaughterfisheggcluster01_1.nif` — header=9,
  parsed=9, niunknown=1 → complete. 1 instance / 1 file in all vanilla SSE meshes.
- **Impact:** The egg-cluster loses its planar collision shape; the mesh renders fully.
  Cosmetic/physics-only, single vanilla file.
- **Related:** #766 (SE Havok long-tail incl. bhkPlaneShape — CLOSED; verify scope).
- **Suggested Fix:** Add a `bhkPlaneShape` arm (bhkHeightFieldShape layout: material +
  plane normal + constant), or accept the LOW NiUnknown skip. Confirm #766 scope first.

### NIF-2026-05-29-07: `d5_coverage` `DISPATCH_KEYS` snapshot is stale — over-reports uncovered types ~5×
- **Severity:** LOW
- **Dimension:** Coverage *(Dim 5 NIF-D5-01)*
- **Game Affected:** All (audit-tool accuracy)
- **Location:** `crates/nif/examples/d5_coverage.rs:18-319`
- **Status:** NEW
- **Description:** The example hardcodes a ~300-entry `DISPATCH_KEYS` snapshot of the
  dispatch table. Live `parse_block` carries ~308 keys and has gained arms since the
  snapshot, so the tool flags already-dispatched types (`NiAlphaController`,
  `BSKeyframeController`, `BSBoneLODExtraData`, `NiIntegersExtraData`, `NiFloatsExtraData`)
  as "UNCOVERED".
- **Evidence:** Of 7 reported-uncovered types, 4 are dispatched in live source
  (`mod.rs` lines 555, 568, 675, 775). Only 3 are genuine gaps.
- **Impact:** Inflated uncovered counts could mask the real (smaller) gap or trigger
  wasted follow-up; coverage % figures are pessimistic but directionally correct.
- **Related:** #601 (nif_stats top-20 blindspot — same family).
- **Suggested Fix:** Derive `DISPATCH_KEYS` at runtime (probe `parse_block` → check
  `!= NiUnknown`), generate from `mod.rs` via build script, OR add a unit test diffing
  the array against the live match arms so it can't drift silently.
