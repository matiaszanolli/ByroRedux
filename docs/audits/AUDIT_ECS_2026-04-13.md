## Dimension 2: Query Safety — 1 LOW

All query/resource APIs use RAII guards for lock tracking (fix #137), with one exception:

### E26-05: World::get() uses raw track_read instead of TrackedRead RAII guard
- **Severity**: LOW
- **Dimension**: Query Safety
- **Location**: `crates/core/src/ecs/world.rs:125-126`
- **Status**: NEW
- **Description**: `World::get()` is the only query/resource method that still calls
  `lock_tracker::track_read()` directly instead of using the `TrackedRead` RAII scope guard
  introduced in fix #137. If `lock.read()` panics (poisoned lock), the tracker row leaks,
  causing false deadlock detection on subsequent same-thread acquisitions. Every other method
  correctly uses the RAII pattern.
- **Impact**: Only triggers on poisoned lock + `catch_unwind` recovery — hardening only.
- **Related**: #137, E26-01 (prior audit)
- **Suggested Fix**: Replace `lock_tracker::track_read(type_id, ...)` with `TrackedRead::new(type_id, ...)`
  and `mem::forget` the guard after successful lock acquisition (same pattern as all other methods).

Prior fixes verified:
- **#137**: RAII `TrackedRead`/`TrackedWrite` scope guards correctly implemented in `query()`,
  `query_mut()`, `resource()`, `resource_mut()`, `try_resource()`, `try_resource_mut()`, and
  all `*_2_mut*` variants. ✓
- **#138**: `has()` and `count()` now use `self.query::<T>()` returning `QueryRead` with
  automatic Drop-based untrack. ✓
- TypeId-sorted lock acquisition in `query_2_mut`, `query_2_mut_mut`, `resource_2_mut`. ✓
- Same-type double-lock panics with clear message. ✓

### D5-01: Duplicate system names produce warning but are still added
- **Severity**: LOW
- **Dimension**: System & Scheduler
- **Location**: `crates/core/src/ecs/scheduler.rs:86-91`
- **Status**: NEW
- **Description**: `Scheduler::add()` logs `warn!` on duplicate name but adds the system anyway.
  Could confuse debug tooling (byro-dbg) which may expect unique system names.
- **Impact**: Debug tooling confusion. No safety impact.
- **Suggested Fix**: Either reject duplicates (return `Err`) or append a suffix.

### D5-02: Lock tracker cannot detect cross-thread ABBA deadlocks
- **Severity**: LOW
- **Dimension**: System & Scheduler
- **Location**: `crates/core/src/ecs/lock_tracker.rs:10-17`
- **Status**: NEW
- **Description**: Thread-local tracker only detects same-thread re-entrant deadlocks. If two
  systems on different rayon threads independently acquire multiple single-type queries outside
  `query_2_mut`, the tracker cannot detect ABBA ordering. Mitigated by `query_2_mut` /
  `query_2_mut_mut` APIs which enforce TypeId-sorted acquisition. No current built-in system
  triggers this pattern.
- **Impact**: Theoretical. All multi-query patterns in the codebase use the safe `_2_mut` APIs.
- **Suggested Fix**: Document the limitation more prominently. Consider a global lock-order
  validator in debug builds (lower priority — current mitigations are sufficient).
