# NIF-D5-NEW-02: pre_parse_cell rayon into_par_iter has no with_min_len threshold

**Labels**: bug, nif-parser, medium, performance

## Severity
**MEDIUM**

## Status
**NEW**

## Location
- \`byroredux/src/streaming.rs:555\` — \`extracted.into_par_iter()\` in \`pre_parse_cell\`'s Phase 2 parallel parse + import

## Description
Phase 2 of \`pre_parse_cell\` does \`extracted.into_par_iter().map(...).collect()\` over the per-NIF list. No \`.with_min_len(threshold)\` is configured, so rayon dispatches even N=1 work onto its worker pool.

Per the audit-finding-hygiene memory: post-#862 the NIF import cache means most cells fetch ALL their NIFs from the cache and only fresh-parse 0-6. The Riverwood log confirms: \`6 new unique meshes parsed, NIF cache hits/misses 156/6 this cell\`. At N=0-6 work items, rayon's worker-wake overhead (typically ~50-200 µs per dispatch to wake threads + sync the join) approaches or exceeds the actual parse cost of those 0-6 small NIFs.

This was fine when \`pre_parse_cell\` regularly saw N=100+ on session-start (#830 NIF-PERF-06 win was real), but the steady-state cell-stream case after #862 cache-skip has shifted the typical N way down. The min-len threshold should reflect that.

## Evidence
\`\`\`rust
// streaming.rs:552-562
// Phase 2: parallel parse + import. No shared mutex on the hot
// path; each worker owns its `Vec<u8>` for the whole closure.
let results: Vec<(String, Option<PartialNifImport>)> = extracted
    .into_par_iter()
    .map(|(path, bytes)| {
        // Per-NIF panic guard — ...
        let parsed = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
            // ... parse + import work ...
        }));
\`\`\`

No \`with_min_len\`, \`with_max_len\`, or other rayon thresholding.

## Trigger Conditions
- Steady-state cell streaming where the NIF cache absorbs most of the workload — typical post-warmup
- Player walking across an exterior grid (radius-3 → 49 cells initial + per-frame stream-in/out)

## Impact
At N=0-3 the parallel dispatch is pure loss. At N=4-6 it's break-even at best. Above N≈10-20 (session-start fresh cells) the parallelism still wins.

Estimated saving: 50-200 µs per cell-load worker frame. Visible as smoother cell-crossing transitions; not visible in steady-state FPS.

## Suggested Fix
Add a min-len guard:

\`\`\`rust
let results: Vec<(String, Option<PartialNifImport>)> = if extracted.len() < 8 {
    // Serial fast path — rayon dispatch overhead exceeds the work for small N.
    // Cache hits in the steady-state stream mean we're typically here.
    extracted.into_iter().map(|(path, bytes)| { /* same closure body */ }).collect()
} else {
    extracted.into_par_iter().map(|(path, bytes)| { /* same closure body */ }).collect()
};
\`\`\`

Or use rayon's \`.with_min_len(N)\` chunk-control if available for IntoParallelIterator. The threshold (8) is a reasonable starting point — empirically tune against the dhat fixture or a streaming-stress micro-bench.
