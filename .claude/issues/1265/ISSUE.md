# NIF-D5-NEW-05: extract_mesh (NiTriShape path) clones geom.tangents despite geom being dead after

**Labels**: bug, import-pipeline, low, performance

## Severity
**LOW**

## Status
**NEW**

## Location
- \`crates/nif/src/import/mesh/ni_tri_shape.rs:180\` — \`let tangents_yup = geom.tangents.clone();\`

## Description
Inside \`extract_mesh\` (NiTriShape import path), \`geom.tangents.clone()\` is called to populate \`ImportedMesh.tangents\`. After this line, \`geom\` is unused — it's a one-shot \`GeomData<'a>\` borrow that ends at the function boundary.

A trivial \`std::mem::take(&mut geom.tangents)\` or moving the field out via destructuring would eliminate the clone. The clone is a per-vertex \`Vec<[f32; 3]>\` memcpy — typically ~16-40 KB per FNV/FO3/Oblivion NiTriShape mesh (which is the only era where authored tangents are commonly present; Skyrim+ inline them on BSTriShape via the bs_tri_shape path).

## Evidence
\`\`\`rust
// import/mesh/ni_tri_shape.rs:175-181
    // #783 / M-NORMALS — pre-decoded tangents from the NIF's
    // \`NiBinaryExtraData("Tangent space (binormal & tangent vectors)")\`.
    // Empty when the source mesh has no authored tangents; the
    // renderer falls back to screen-space derivative TBN in that case.
    let tangents_yup = geom.tangents.clone();   // <-- one-char fix: mem::take

    Some(ImportedMesh {
        positions,
        // ...
        tangents: tangents_yup,
        // ...
    })
\`\`\`

\`geom\` is never read after line 180 — the function returns immediately after constructing \`ImportedMesh\`.

## Trigger Conditions
- Every NiTriShape mesh import with authored tangents (\`NiBinaryExtraData("Tangent space...")\` extra data block present)
- Common on FNV / FO3 / Oblivion content (the classic Bethesda tangent-space carrier)
- Skyrim+ inline tangents on BSTriShape and don't hit this path

## Impact
~16-40 KB redundant memcpy per FNV/FO3/Oblivion NiTriShape import. Cumulative across a cell-load worker phase. Visible as smoother streaming on those games; not visible in steady-state frames.

## Suggested Fix
One-character change — replace \`.clone()\` with \`std::mem::take(&mut geom.tangents)\`:

\`\`\`rust
let tangents_yup = std::mem::take(&mut geom.tangents);
\`\`\`

Verify \`geom\` is \`&mut GeomData\` (not \`&GeomData\`) at this site; if it's borrowed shared, the fix needs the function signature to accept \`&mut\` instead. Looking at the function signature should confirm.

Alternative if \`geom\` is shared: destructure on entry into individually-moved fields, or take ownership of \`geom\` as \`GeomData<'a>\` and read fields directly without clone.
