# NIF-D5-NEW-01: Unknown-block-type dispatch fallback allocates fresh Arc<str> per call (partial regression of #834)

**Labels**: bug, nif-parser, medium, performance

## Severity
**MEDIUM**

## Status
**NEW** — partial regression of #834. The lib.rs sites were fixed but `blocks/mod.rs:1131` was missed in the original sweep.

## Location
- `crates/nif/src/blocks/mod.rs:1131` — inside the `_ =>` (unknown block type) fallback of `parse_block`

## Description
When `parse_block` encounters a block type name not in the dispatch table, it constructs a `NiUnknown { type_name: Arc::from(type_name), data: Vec::new() }` placeholder. The `Arc::from(&str)` call allocates a fresh `Arc<str>` heap allocation per call.

On a Skyrim Whiterun / Riverwood cell load, this fallback fires hundreds of times per session-start phase as the parser walks blocks it doesn't dispatch yet (FO76 / Starfield extension types appearing in modded Skyrim content, internal Bethesda blocks not yet implemented, etc.). Each fires an `Arc::from(type_name)` allocation.

The fix from #834 (NIF-PERF Arc<str> interning) covered the `lib.rs` per-name allocation paths, but the dispatch-fallback site at `blocks/mod.rs:1131` was missed — it's still constructing a fresh `Arc<str>` rather than going through an interned cache.

## Evidence
\`\`\`rust
// blocks/mod.rs:1119-1133
_ => {
    if let Some(size) = block_size {
        let start = stream.position();
        stream.skip(size as u64)?;
        log::debug!(
            "Skipping unknown block type '{}' ({} bytes at offset {})",
            type_name, size, start
        );
        Ok(Box::new(NiUnknown {
            type_name: Arc::from(type_name),  // fresh Arc heap alloc per call
            data: Vec::new(),
        }))
    } else { /* error path */ }
}
\`\`\`

## Trigger Conditions
- Any NIF parse that contains a block type not in the `match` dispatch — common on modded Skyrim, Starfield, FO76 content
- Hot on session-start when many NIFs first-parse (49-cell Riverwood = ~500 unique NIF parses)

## Impact
Hundreds of `Arc::from(&str)` allocations per Riverwood load, each a small heap allocation. Cumulatively measurable in the cell-stream worker but not visible in steady-state frames. Same shape as the alloc patterns #834 closed elsewhere.

## Suggested Fix
Reuse the same interning cache or `Arc<str>` mechanism that #834 wired through for the lib.rs sites. If the cache is a per-parser `HashMap<&str, Arc<str>>`, plumb it through `parse_block`'s signature. If it's a static `OnceLock<RwLock<HashSet<Arc<str>>>>`, just look it up before falling back to `Arc::from`. ~10 LOC.

## Allocation regression coverage
Gap acknowledged — the dhat infrastructure from #1247 covers only a single-NiNode synthetic fixture. A real unknown-type fallback workload (modded Skyrim NIF) is not in the dhat baseline. Until that lands, the saving estimate is qualitative.
