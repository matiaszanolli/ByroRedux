# PERF-D3-NEW-03: DebugStats::draw_call_count is mislabelled — reports input to batcher, not GPU draw calls

**Labels**: bug, renderer, high, performance

## Severity
**HIGH** (misdiagnosis blocker, not a perf bug per se)

## Status
**NEW**

## Locations
- `byroredux/src/main.rs:1589` — writer
- `crates/core/src/ecs/resources.rs:224-225` — field declaration + doc-comment
- `byroredux/src/commands.rs:72` — surfaced via `stats` console command
- `byroredux/src/main.rs:2041` — surfaced via `--bench-frames` summary
- `crates/renderer/src/vulkan/context/draw.rs:2909` — real batch count computed but never reported
- `crates/renderer/src/vulkan/context/draw.rs:2278-2296` — real indirect-call count never reported

## Description
The field `DebugStats::draw_call_count` is documented as "Draw calls last frame" and surfaced to the user as `"Draws: {n}"` in the `stats` console command and `draws={n}` in the bench-frames summary. But the only writer at `main.rs:1589` sets it from `self.draw_commands.len()`, which is the **pre-batch DrawCommand vector** built by `build_render_data`.

The actual `cmd_draw_indexed` / `cmd_draw_indexed_indirect` call count after batching + multi-draw-indirect grouping is never recorded. The recent Skyrim Riverwood "12,277 draws/frame ~ 44 µs/draw" arithmetic derives from this mislabelled number and is therefore unsound — at 12,277 input commands the batcher could be collapsing them into anywhere from ~5 to ~12,277 actual GPU calls and the metric can't tell us which.

This has been actively misleading performance diagnoses across multiple sessions and audits.

## Evidence
```rust
// main.rs:1587-1590
// Record draw call count for diagnostics.
world_resource_set::<DebugStats>(&self.world, |s| {
    s.draw_call_count = self.draw_commands.len() as u32;
});
```
```rust
// resources.rs:224-225
/// Draw calls last frame.
pub draw_call_count: u32,
```

Meanwhile `draw.rs:2909` computes `working_batches = batches.len()` but only uses it for the scratch-shrink heuristic — never reports it up to DebugStats. The inner indirect grouping loop at `draw.rs:2278-2296` further compresses `batches.len()` into fewer `cmd_draw_indexed_indirect` calls (`group_size` batches per call), and that final number isn't tracked anywhere either.

## Trigger Conditions
Always reachable — every frame's stats / bench output is misreporting.

## Impact
Zero CPU/GPU. But unblocks correct diagnosis of every future "we're too slow" investigation — the next audit dimension will know whether the bottleneck is GPU-side draw cost (need to fix batching) or CPU-side command-building cost (need to fix `build_render_data`) without resorting to a custom RenderDoc capture.

## Suggested Fix
Two options:

**Option (a) — recommended**: rename the existing field to `draw_command_count` and add two new fields, `batch_count` (populated from `batches.len()` after the merge loop) and `indirect_call_count` (incremented inside the `while i < batches.len()` loop at `draw.rs:2097`, counting actual `cmd_draw_indexed` + `cmd_draw_indexed_indirect` invocations). Preserves the existing numbers in the rolling audit history while exposing the new metrics.

**Option (b)**: leave the field name as-is and have `draw_frame` return the actual GPU call count, with `main.rs:1589` writing that number into the field. Loses the input-to-batcher metric, which is also useful.

Same fix applies to the `debug-protocol` crate (`debug-protocol/src/lib.rs:100`) so byro-dbg renders all three fields.
