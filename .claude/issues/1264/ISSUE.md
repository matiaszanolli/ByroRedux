# NIF-D5-NEW-04: extract_bs_tri_shape clones uvs/vertex_colors/triangles from one-shot reference

**Labels**: bug, import-pipeline, low, performance

**LOW**

- \`crates/nif/src/import/mesh/bs_tri_shape.rs:161\` — \`shape.triangles.clone()\` (conditional)

## Description
\`extract_bs_tri_shape\` takes \`shape: &BsTriShape\` and clones three of its Vec fields into the returned \`ImportedMesh\` payload. The function is the one-shot import path — \`shape\` is never read after \`extract_bs_tri_shape\` returns, so the clones are pure waste.

Per the BSTriShape import sweep at the Skyrim/SSE Riverwood test scenario, every skinned NPC mesh (Whiterun guards, draugr, wildlife, Riverwood villagers) hits this path during cell load. The cloned fields are:

- \`uvs\`: \`Vec<[f32; 2]>\`, typically 1000-5000 verts × 8 B = 8-40 KB per mesh
- \`vertex_colors\`: \`Vec<[u8; 4]>\`, typically 1000-5000 verts × 4 B = 4-20 KB per mesh
- \`triangles\`: \`Vec<[u16; 3]>\`, typically 2000-10000 tris × 6 B = 12-60 KB per mesh

Riverwood radius-3 has ~20 NPCs + numerous static meshes. At ~80-200 KB per BSTriShape mesh cloned redundantly × 20-50 meshes = ~2-10 MB of redundant memcpy across the cell-load worker phase.

## Trigger Conditions

## Impact
~80-200 KB of redundant memcpy per imported BSTriShape mesh. At Riverwood scale: ~2-10 MB of memcpy across the cell-load worker phase. Visible as smoother cell-stream latency; not visible in steady-state frames.

## Suggested Fix
Change the function signature to take \`shape: BsTriShape\` (by value) where possible — then the fields can be moved out (no allocation, no memcpy) via direct field access or \`std::mem::take(&mut shape.uvs)\` patterns.

If \`extract_bs_tri_shape\` must keep the \`&\` signature (e.g. called from a context where the BsTriShape is needed afterwards — verify the call sites in \`import/mod.rs\`), then a two-overload approach: add \`extract_bs_tri_shape_owned(shape: BsTriShape, ...)\` for one-shot callers and keep the \`&\`-ref variant for read-only callers.

~20-40 LOC depending on call-site count.

**Caveat**: verify the call sites first. If \`shape\` is borrowed from a longer-lived NifScene that retains the BsTriShape for other passes (e.g. tangent synthesis cross-reference, BLAS build), the move-out is not safe. Per the audit's read of \`import/mod.rs\`, the BsTriShape is one-shot from the import path's perspective, but verify before changing the signature.
