# R1-N6: GpuMaterial lacks per-field offset tests — only the 272 B size invariant is pinned

Labels: bug renderer low 
State: OPEN

## Severity
**LOW** — Defense-in-depth gap. No live bug today; surfaces only when a refactor reorders fields within a vec4 slot.

## Status
**NEW**

## Locations
- `crates/renderer/src/vulkan/material.rs:395` — only `gpu_material_size_is_272_bytes` exists
- `crates/renderer/src/vulkan/scene_buffer.rs:1418` — by contrast, `GpuInstance` has `gpu_instance_field_offsets_match_shader_contract` that asserts every named offset (model=0, texture_index=64, …, _pad_albedo=108)

## Description
A reorganization within the 17 vec4 slots of `GpuMaterial` that preserves total size (272 B) but reorders fields would not trip any test. The shader-side `struct GpuMaterial` (in `triangle.frag:83`) has matching offset comments **at vec4-group level only** ("vec4 #4: textureIndex, normalMapIndex, darkMapIndex, glowMapIndex"), not per-field. So a swap between two same-type fields within a vec4 (e.g., `textureIndex ↔ normalMapIndex`) would produce wrong shader reads with no test failure.

`GpuInstance` is hardened against this by `gpu_instance_field_offsets_match_shader_contract` at `scene_buffer.rs:1418` — which asserts every named field's `offset_of!`. `GpuMaterial` has no equivalent.

## Evidence
```rust
// scene_buffer.rs:1418 — GpuInstance per-field offset test
fn gpu_instance_field_offsets_match_shader_contract() {
    assert_eq!(offset_of!(GpuInstance, model), 0);
    assert_eq!(offset_of!(GpuInstance, texture_index), 64);
    assert_eq!(offset_of!(GpuInstance, bone_offset), 68);
    // ... 13 named fields
}
```

No equivalent exists in `material.rs` (only the size pin at `:395`).

## Trigger Conditions
A refactor that:
1. Reorders fields within a single vec4 slot of GpuMaterial (e.g., to swap `textureIndex` and `normalMapIndex`)
2. Preserves total size (272 B)
3. Doesn't update the matching shader-side struct in lockstep

The size pin passes; the shader-side reads `mat.normalMapIndex` from what's now the `textureIndex` slot. Silent corruption.

## Impact
Latent. A future refactor (e.g., to group fields more semantically, or to align with std140 vs std430) could hit this. Today's R1 closeout state is correct — just unprotected.

## Suggested Fix
Add a sibling test in `material.rs::tests`:

```rust
use std::mem::offset_of;

/// Pin every named field's offset against the shader-side GLSL struct
/// at `triangle.frag:83-127`. The size invariant (272 B) catches
/// additions/removals; this catches reorderings within the existing
/// 17 vec4 slots that the size pin alone would miss.
///
/// Mirrors the `gpu_instance_field_offsets_match_shader_contract`
/// pattern at `scene_buffer.rs:1418` — every named field on the
/// Rust side gets an explicit `offset_of!` assertion against its
/// documented offset. The shader's per-vec4-group comments at
/// `triangle.frag:83-127` are the source of truth for these offsets.
#[test]
fn gpu_material_field_offsets_match_shader_contract() {
    // vec4 #1 — PBR scalars
    assert_eq!(offset_of!(GpuMaterial, roughness), 0);
    assert_eq!(offset_of!(GpuMaterial, metalness), 4);
    assert_eq!(offset_of!(GpuMaterial, emissive_mult), 8);
    assert_eq!(offset_of!(GpuMaterial, material_flags), 12);

    // vec4 #2 — Emissive RGB + specular_strength
    assert_eq!(offset_of!(GpuMaterial, emissive_r), 16);
    assert_eq!(offset_of!(GpuMaterial, emissive_g), 20);
    // ... pin all 56 fields
}
```

~20 lines, mirrors the existing `GpuInstance` test exactly.
