# NIF-D5-NEW-03: BSGeometry has three remaining per-vertex push loops post #873 consolidation (Starfield only)

**Labels**: bug, nif-parser, low, performance

## Severity
**LOW** — Starfield-only; Skyrim/FNV/FO3/FO4 don't trigger BSGeometry.

## Status
**NEW**

## Location
- \`crates/nif/src/blocks/bs_geometry.rs:411-432\` — three sibling per-vertex push loops in \`BSGeometry::parse\`

## Description
#873 (NIF-PERF-09) consolidated bulk POD reads in BSGeometry — colors/normals_raw/tangents_raw now go through \`read_pod_vec<T>\`. But three sibling loops at lines 411-432 (vertices, uvs0, uvs1) were left as the original per-vertex push-loop pattern, doing a manual \`for _ in 0..n { stream.read_u16_le() × 3; vertices.push([..]); }\`.

Each loop does \`n × 3 stream.read_u16_le()\` calls + an \`unpack_norm_i16\` or \`half_to_f32\` on each component, then pushes a \`[f32; 3]\` (or \`[f32; 2]\` for UVs) into the pre-allocated Vec. This is functionally equivalent to the bulk-read pattern but does N call-overheads instead of one.

The reason these weren't consolidated by #873: they involve a per-element transform (i16 → f32 normalization, half-float → f32 unpack), so they can't trivially use \`read_pod_vec\` as-is. A new \`read_pod_vec_into_mapped\` helper or a per-block bulk staging buffer would be the consolidation pattern.

## Evidence
\`\`\`rust
// bs_geometry.rs:411-432
let n_vertices = stream.read_u32_le()?;
let mut vertices = stream.allocate_vec::<[f32; 3]>(n_vertices)?;
for _ in 0..n_vertices {
    let x = unpack_norm_i16(stream.read_u16_le()? as i16, scale, Self::HAVOK_SCALE);
    let y = unpack_norm_i16(stream.read_u16_le()? as i16, scale, Self::HAVOK_SCALE);
    let z = unpack_norm_i16(stream.read_u16_le()? as i16, scale, Self::HAVOK_SCALE);
    vertices.push([x, y, z]);
}

let n_uv1 = stream.read_u32_le()?;
let mut uvs0 = stream.allocate_vec::<[f32; 2]>(n_uv1)?;
for _ in 0..n_uv1 {
    let u = half_to_f32(stream.read_u16_le()?);
    let v = half_to_f32(stream.read_u16_le()?);
    uvs0.push([u, v]);
}

let n_uv2 = stream.read_u32_le()?;
let mut uvs1 = stream.allocate_vec::<[f32; 2]>(n_uv2)?;
for _ in 0..n_uv2 {
    let u = half_to_f32(stream.read_u16_le()?);
    let v = half_to_f32(stream.read_u16_le()?);
    uvs1.push([u, v]);
}

## Impact
LOW today because Starfield isn't the active dev target. When Starfield support gets exercised, ~10-20 µs saved per BSGeometry block × N geometries per Starfield cell.

## Suggested Fix
Two options:

**Option (a)** — read raw \`Vec<u16>\` bulk via \`read_pod_vec\`, then post-process in a tight \`map\` into the final \`Vec<[f32; 3]>\` / \`Vec<[f32; 2]>\`. Single stream read per array; per-element transform cost stays the same but avoids the per-read call overhead.

**Option (b)** — add a \`read_pod_vec_into_mapped<T, U>(n, mapper: impl Fn(T) -> U)\` helper to \`stream.rs\`. Generalizes the pattern across BSGeometry's three loops + any future BSGeometry-variant blocks.

Either ~20-30 LOC.

## Allocation regression coverage
The 3 \`allocate_vec\` calls are already correct (\`#[must_use]\`-checked). The fix doesn't change allocation count — only stream-read call count. Outside the dhat-infra gap scope.
